# Architectural Deployment Plan for Secure Static Website on S3 with CloudFront

## Key AWS Services Required

1. **S3** - For hosting static website content
2. **CloudFront** - For global CDN distribution with HTTPS enforcement
3. **Lambda** - For automated content deployment and cleanup
4. **IAM** - For service permissions and OAI configuration
5. **CloudFormation** - For infrastructure as code deployment

## Architecture Overview

The solution implements a secure static website architecture where:
- Content is stored in an S3 bucket with restricted public access
- CloudFront serves content globally via HTTPS with an Origin Access Identity (OAI)
- Lambda functions handle automated deployment and cleanup
- No direct internet access to the S3 bucket is possible

## Detailed Component Analysis

### 1. S3 Bucket Configuration

**Properties:**
- Bucket name: Generated using `Fn::Join` with stack name and random suffix to ensure uniqueness
- Public access: Blocked via `PublicAccessBlockConfiguration`
- Versioning: Enabled for content management
- Website configuration: Set with `IndexDocument` as "index.html"
- Tags: Standard AWS tags for cost allocation

**Security Implementation:**
- No `AccessControl` property used (modern approach)
- Bucket policy grants access only to CloudFront OAI
- Ownership controls not required since we're using bucket policies

### 2. CloudFront Distribution

**Properties:**
- Viewer protocol policy: "redirect-to-https" (enforces HTTPS)
- Cache behavior: `MinTTL`, `DefaultTTL`, and `MaxTTL` all set to 0 (disables caching)
- Origin: S3 bucket with OAI
- Price class: "PriceClass_All" for global distribution
- SSL certificate: AWS-managed certificate

**Dependencies:**
- S3 bucket must exist before CloudFront creation
- OAI must be created before bucket policy attachment

### 3. Origin Access Identity (OAI)

**Properties:**
- Comment: Descriptive text for identification
- Automatically created by CloudFront when referenced

### 4. Lambda Functions for Automation

**Deployment Lambda:**
- Runtime: `python3.12` (latest supported)
- Handler: Custom function to upload index.html
- Permissions: S3:PutObject on the target bucket
- Execution role: Custom IAM role with least privilege

**Cleanup Lambda:**
- Runtime: `python3.12`
- Handler: Custom function to delete all objects
- Permissions: S3:DeleteObject on the target bucket
- Execution role: Custom IAM role with least privilege

### 5. Custom Resources for Lifecycle Management

**Deployment Trigger:**
- `Create` event: Uploads default index.html
- `Update` event: No operation (idempotent)
- `Delete` event: No operation (handled by cleanup Lambda)

**Cleanup Trigger:**
- `Create` event: No operation
- `Update` event: No operation
- `Delete` event: Empties bucket before deletion

## Parameter Configuration with Safe Defaults

```yaml
Parameters:
  Environment:
    Type: String
    Default: "production"
    AllowedValues: ["development", "staging", "production"]
  
  Region:
    Type: String
    Default: !Ref "AWS::Region"
    AllowedPattern: "^[a-z]{2}-[a-z]+-\d$"
```

## Dynamic Resource Fetching Strategy

**AMIs:** Not applicable for this static website architecture.

**AZs:** Using `Fn::GetAZs` and `Fn::Select` for any future VPC requirements:
```yaml
!Select [0, !GetAZs !Ref "AWS::Region"]
```

## Security Best Practices Implementation

1. **IAM Roles:**
   - Custom execution roles for Lambda functions with explicit S3 permissions
   - No wildcard permissions
   - Principle of least privilege

2. **Secrets Management:**
   - No secrets required for this static website
   - If needed in future, use `{{resolve:ssm-secure:...}}`

3. **Network Security:**
   - S3 bucket policies restrict access to only CloudFront OAI
   - No public bucket policies

## Deletion Policies

**S3 Bucket:**
- `DeletionPolicy: Delete` (default)
- `UpdateReplacePolicy: Delete`
- Cleanup Lambda ensures bucket is empty before deletion

**Lambda Functions:**
- Default deletion policy (no state to preserve)

## Error Handling and Validation

1. **Template Validation:**
   - All parameters have safe defaults
   - No hardcoded values
   - Proper intrinsic function usage

2. **Resource Dependencies:**
   - Explicit `DependsOn` clauses where necessary
   - Proper ordering of resource creation

3. **Linting Compliance:**
   - No `Fn::Sub` without variables
   - No hallucinated properties
   - Proper runtime selection

## Deployment Sequence

1. Create S3 bucket
2. Create Lambda execution roles
3. Create Lambda functions
4. Create OAI
5. Create bucket policy
6. Create CloudFront distribution
7. Trigger deployment Lambda
8. Validate deployment

## Cost Optimization

- CloudFront price class set to "PriceClass_All" (can be adjusted)
- S3 storage costs minimized with proper lifecycle policies
- Lambda provisioned concurrency not used (on-demand sufficient)

This architecture ensures a secure, automated, and production-ready static website deployment that adheres to all AWS best practices and CloudFormation linting rules.
