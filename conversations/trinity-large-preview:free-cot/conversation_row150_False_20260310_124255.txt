# Architectural Deployment Plan for Redis Caching Solution with SSM-Enabled EC2 Access

## 1. Key AWS Services Required

Based on the business requirements, the following AWS services are essential:

- **Amazon VPC** - To create the isolated network environment
- **Amazon EC2** - For the application server instance
- **Amazon ElastiCache** - For the Redis cluster
- **AWS Identity and Access Management (IAM)** - For creating the SSM-enabled role
- **AWS Systems Manager (SSM)** - For secure EC2 instance access
- **AWS SSM Parameter Store** - For dynamic AMI fetching
- **Amazon Route 53** - For DNS resolution (if needed)
- **AWS CloudWatch** - For monitoring and logging

## 2. Properties and Dependencies

### VPC Dependencies
- VPC must be created before subnets
- Internet Gateway must be attached to VPC before route tables can reference it
- Route tables must be associated with subnets

### Security Dependencies
- Security groups must reference VPC ID
- IAM role must be created before EC2 instance can reference it

### ElastiCache Dependencies
- Subnet Group must reference existing subnets
- Security group must reference VPC ID

### EC2 Dependencies
- IAM role must exist before EC2 instance creation
- Security group must exist before EC2 instance creation
- AMI must be resolved before EC2 instance creation

## 3. Safe AWS Defaults for Missing Properties

### Network Configuration
- **VPC CIDR**: `10.0.0.0/16` (standard private range)
- **Public Subnet CIDR**: `10.0.1.0/24` and `10.0.2.0/24`
- **Private Subnet CIDR**: `10.0.3.0/24` and `10.0.4.0/24`
- **Availability Zones**: Dynamically selected using `Fn::GetAZs` and `Fn::Select`

### EC2 Configuration
- **Instance Type**: `t3.micro` (free tier eligible, suitable for testing)
- **KeyPair**: `default-key` (safe default, user can override)
- **AMI**: Dynamically fetched using SSM Parameter Store
- **SSH Location**: `0.0.0.0/0` (can be restricted later)

### ElastiCache Configuration
- **Node Type**: `cache.t3.micro` (cost-effective for testing)
- **Engine Version**: `7.2` (latest stable Redis version)
- **Port**: `6379` (Redis default port)

## 4. Dynamic Resource Fetching Strategy

### AMI Resolution
Use SSM Parameter Store to fetch the latest Amazon Linux 2 AMI:
```yaml
!Sub 'resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2'
```

### Availability Zones
Dynamically select AZs based on region:
```yaml
!Select [0, !GetAZs '']
!Select [1, !GetAZs '']
```

## 5. Security Architecture

### IAM Role for SSM
Create an IAM role with the AWS managed policy `AmazonSSMManagedInstanceCore` to enable SSM access without SSH keys.

### Security Groups
- **EC2 Security Group**: Allow SSH from SSM and HTTP/HTTPS from anywhere
- **ElastiCache Security Group**: Allow Redis traffic only from EC2 security group

### Network ACLs
Use default NACL with appropriate rules for public and private subnets.

## 6. Resource Dependencies Flow

1. VPC creation
2. Internet Gateway creation and attachment
3. Subnet creation
4. Route table creation and association
5. Security group creation
6. IAM role creation
7. ElastiCache subnet group creation
8. ElastiCache cluster creation
9. EC2 instance creation

## 7. Monitoring and Logging

- CloudWatch alarms for ElastiCache metrics (CPU, memory, connections)
- EC2 instance monitoring via CloudWatch
- SSM session logging enabled

## 8. Cost Optimization

- Use `t3.micro` instances for both EC2 and ElastiCache nodes
- Enable ElastiCache cluster mode disabled (single shard)
- Use GP2 SSD for any EBS volumes

## 9. Template Structure Plan

The CloudFormation template will be organized as follows:

1. **Parameters** - With safe defaults for all configurable values
2. **Mappings** - For any static mappings needed
3. **Resources** - All AWS resources in logical dependency order
4. **Outputs** - For important resource identifiers (Redis endpoint, EC2 public IP)

This architecture ensures a secure, scalable, and maintainable Redis caching solution with SSM-enabled EC2 access, following AWS best practices and avoiding common deployment failures.
