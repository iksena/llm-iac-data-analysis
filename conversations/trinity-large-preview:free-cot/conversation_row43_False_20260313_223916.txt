# Detailed Architectural Deployment Plan for S3 Bucket with KMS Encryption and Lifecycle Management

## 1. Key AWS Services Required

Based on the business need, the following AWS services are required:

- **AWS::S3::Bucket** - The primary S3 bucket resource
- **AWS::KMS::Key** - Customer Master Key (CMK) for encryption
- **AWS::KMS::Alias** - Key alias for simplified management
- **AWS::S3::BucketPolicy** - Security policy for the bucket
- **AWS::S3::BucketLifecycleConfiguration** - Lifecycle rules for cost optimization

## 2. Properties and Dependencies

The template will have the following dependencies and property relationships:

- KMS Key must be created before the S3 Bucket (for encryption)
- Bucket Policy must reference the KMS Key ARN
- Lifecycle Configuration is independent but should be defined after the bucket
- Tags will be applied to both the bucket and KMS key for organizational tracking

## 3. Safe AWS Defaults

Since specific properties weren't provided, I'll use these safe defaults:

- **Bucket Name**: Generated dynamically using `!Sub` with stack name and random suffix to ensure uniqueness
- **KMS Key Policy**: Default policy allowing root account full access and S3 service to use the key
- **Lifecycle Rule**: Default 30-day transition to Intelligent-Tiering, 365-day transition to Glacier, and 1095-day expiration
- **Tags**: Default tag `Environment: Production` (can be overridden via parameter)

## 4. Dynamic Resource Fetching

For this template, dynamic fetching isn't strictly necessary since:
- No AMIs are required (no EC2 instances)
- No specific AZs need to be selected (S3 is region-wide)
- However, I'll use dynamic bucket naming to avoid conflicts

## 5. Architecture Plan

### Parameters
- `Environment` (Type: String, Default: 'Production')
- `LifecycleTransitionDays` (Type: Number, Default: 30)
- `LifecycleGlacierDays` (Type: Number, Default: 365)
- `LifecycleExpirationDays` (Type: Number, Default: 1095)

### Resources

1. **KMS Key** (`KMSKey`)
   - Description: "KMS key for S3 bucket encryption"
   - KeyPolicy: Allowing root account and S3 service
   - Enabled: true

2. **KMS Alias** (`KMSKeyAlias`)
   - AliasName: !Sub 'alias/${AWS::StackName}-s3-encryption-key'
   - TargetKeyId: !Ref KMSKey

3. **S3 Bucket** (`S3Bucket`)
   - BucketName: !Sub '${AWS::StackName}-${AWS::Region}-${!Select [0, !Split ["-", !Ref "AWS::Region"]]}'
   - BucketEncryption: SSE-KMS with KMS key
   - OwnershipControls: ObjectOwnership 'BucketOwnerPreferred'
   - PublicAccessBlockConfiguration: Block all public access
   - Tags: Environment and custom tags

4. **Bucket Policy** (`S3BucketPolicy`)
   - PolicyDocument: Allowing KMS decryption for S3 service
   - Bucket: !Ref S3Bucket

5. **Lifecycle Configuration** (`S3BucketLifecycle`)
   - Rules: Transition to Intelligent-Tiering, then Glacier, then expiration

### Security Considerations
- KMS key policy restricts access to authorized entities only
- Bucket policy enforces encryption requirements
- Public access is blocked by default
- Tags enable cost allocation and resource tracking

### Compliance with Best Practices
- No hardcoded values (except safe defaults)
- No deprecated properties
- Proper dependency management
- Resource protection with appropriate policies
- Modern S3 properties (no AccessControl, using OwnershipControls instead)

This architecture provides a secure, compliant, and maintainable S3 bucket with encryption and lifecycle management that can be deployed across multiple environments with minimal configuration changes.
