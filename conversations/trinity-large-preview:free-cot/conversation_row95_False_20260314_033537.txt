### SSM Automation Document Dependencies:
- IAM role with SSM permissions must exist
- EC2 instance with the correct tag must exist
- SSM Agent must be installed and running on the EC2 instance

### Network Dependencies:
- VPC must exist before creating Internet Gateway
- Internet Gateway must be attached to VPC before creating route tables
- Route tables must be associated with subnets

## 3. Safe AWS Defaults

- **CIDR Blocks**: 10.0.0.0/16 for VPC, 10.0.0.0/24 for subnet

- **Region**: Current region where template is deployed (using Fn::Sub and AWS::Region)

## 4. Dynamic Resource Fetching Strategy

### AMI Fetching:
I will use SSM Parameter Store to dynamically fetch the latest Amazon Linux 2 AMI:
- Parameter path: `/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`
- This ensures the template always uses the latest AMI without hardcoding

### AZ Selection:
I will use:
- `Fn::GetAZs`: To get all AZs in the current region
- `Fn::Select`: To select the first AZ (index 0)

## 5. Architecture Overview

The deployment will follow this sequence:

1. Create VPC with Internet Gateway and route table
2. Create Security Group allowing HTTP (port 80) and SSH (port 22)
3. Create IAM role with SSM permissions
4. Deploy EC2 instance with the latest Amazon Linux 2 AMI
5. Apply tag "nginx: Yes" to the instance
6. Create SSM Automation Document with three steps:
   - Step 1: Update OS packages with reboot handling
   - Step 2: Install Nginx web server
   - Step 3: Validate Nginx is running using curl
7. Execute the automation document targeting instances with the "nginx: Yes" tag

## 6. SSM Automation Document Design

The custom SSM Automation document will be designed as follows:

- **Document Type**: Automation
- **Target**: Instances with tag Key=nginx, Value=Yes
- **Steps**:
  1. **UpdateOS**: Execute AWS-RunPatchBaseline with reboot if needed
  2. **InstallNginx**: Install Nginx using yum
  3. **ValidateNginx**: Use curl to verify Nginx is serving content

## 7. Security Considerations

- IAM role will have only necessary SSM permissions
- Security Group will restrict access to necessary ports
- No hardcoding of sensitive information
- Principle of least privilege applied to all resources

## 8. Error Handling and Validation

- Each SSM step will have error handling
- The validation step will confirm Nginx is accessible
- The template will include appropriate DependsOn clauses to ensure proper sequencing

This architecture ensures a robust, tag-driven deployment that can be easily scaled and managed through AWS Systems Manager automation.
