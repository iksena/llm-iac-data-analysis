# IoT Data Pipeline Architecture Plan

## Key AWS Services Required

1. **AWS IoT Core** - For MQTT message ingestion from IoT devices
2. **Amazon SNS** - Pub/sub messaging intermediary
3. **Amazon SQS** - Message queuing for reliable storage
4. **AWS Systems Manager (SSM) Parameter Store** - For dynamic resource fetching

## Architecture Overview

The pipeline will follow this flow:
1. IoT devices publish messages to the `device/data` MQTT topic
2. IoT Rule captures messages from this topic
3. IoT Rule publishes to SNS topic
4. SNS topic fans out to SQS queue
5. SQS queue stores messages for downstream processing

## Resource Dependencies & Properties

### SQS Queue
- **Properties**: QueueName (with prefix), MessageRetentionPeriod (default: 345600 seconds/4 days)
- **Dependencies**: None
- **Default**: Queue name will use stack name prefix

### SNS Topic
- **Properties**: TopicName (with prefix)
- **Dependencies**: None
- **Default**: Topic name will use stack name prefix

### IoT Rule
- **Properties**: 
  - RuleName (with prefix)
  - SQL query: `SELECT * FROM 'device/data'`
  - Actions: SNS topic ARN
- **Dependencies**: SNS topic must exist first
- **Default**: Rule name uses stack name prefix

### IoT Core Configuration
- **Properties**: No explicit VPC configuration needed for IoT Core
- **Dependencies**: None
- **Default**: Uses AWS managed IoT service

## Dynamic Resource Fetching Strategy

Since this template doesn't require AMIs or specific AZs, we won't need SSM Parameter Store for those resources. However, we'll use dynamic references for any future extensibility.

## Security & Permissions

### IAM Roles
- **IoT Rule Role**: Needs permission to publish to SNS
- **SNS Topic Policy**: Allow IoT Core service to publish

### Security Groups
- Not required as IoT Core is a managed service

## Best Practices Implementation

### Parameter Handling
- No parameters required for this basic template
- If parameters were needed, they would include safe defaults

### Resource Naming
- All resources will use stack name prefixes to ensure uniqueness
- Example: `${AWS::StackName}-iot-data-queue`

### Deletion Policies
- SQS Queue: Retain (to prevent accidental data loss)
- SNS Topic: Delete (can be recreated easily)
- IoT Rule: Delete (stateless)

### Linting Compliance
- No hardcoded values
- All resource names follow CloudFormation naming conventions
- Proper intrinsic function usage

## Missing Information & Safe Defaults

Since specific properties weren't provided, I'll use these safe AWS defaults:

- **SQS Message Retention**: 345600 seconds (4 days)
- **Resource Names**: Stack name prefixes
- **IoT Rule SQL**: `SELECT * FROM 'device/data'` (captures all message attributes)
- **Deletion Policies**: Retain for SQS, Delete for SNS and IoT Rule

## Error Prevention Strategy

1. **Resource Dependencies**: SNS topic created before IoT Rule
2. **Naming Conflicts**: Stack name prefixes prevent collisions
3. **Permission Issues**: Explicit IAM role for IoT Rule
4. **Service Quotas**: Standard SNS/SQS limits should suffice for typical IoT workloads

This architecture provides a robust, scalable IoT data pipeline that follows AWS best practices and will pass cfn-lint validation.
