# EventBridge Stripe Product Creation Integration - Architecture Plan

## Key AWS Services Required

1. **AWS::Events::EventBus** - Custom event bus for PartnerApp events
2. **AWS::Events::ApiDestination** - API destination for Stripe integration
3. **AWS::IAM::Role** - Execution role for EventBridge to call Stripe API
4. **AWS::SSM::Parameter** - Secure storage for Stripe API key (optional enhancement)
5. **AWS::Events::Rule** - Event pattern matching for PartnerApp events

## Properties and Dependencies

1. **EventBus** - No dependencies, can be created standalone
2. **IAM Role** - Must be created before ApiDestination
3. **ApiDestination** - Depends on IAM Role and EventBus
4. **Event Pattern** - Depends on EventBus

## Parameter Design with Safe Defaults

```yaml
Parameters:
  StripeApiKey:
    Type: String
    Description: Stripe API key for authentication
    Default: dummy-key # Safe default for linting
    NoEcho: true
    
  EventBusName:
    Type: String
    Description: Name of the custom event bus
    Default: PartnerAppEventBus
    
  ApiDestinationName:
    Type: String
    Description: Name of the API destination
    Default: StripeProductApi
```

## Architecture Plan

### 1. Event Bus Configuration
- Create a custom event bus named "PartnerAppEventBus"
- Use default settings (no specific policy required for API destinations)

### 2. IAM Role Configuration
- Create an IAM role with permissions to:
  - Call `events:InvokeApiDestination`
  - Log events to CloudWatch (optional for debugging)
- Use AWS managed policies where possible
- Trust relationship allowing EventBridge service to assume role

### 3. API Destination Configuration
- Target endpoint: `https://api.stripe.com/v1/products`
- HTTP method: POST
- Authentication: API key in Authorization header
- Connection timeout: 5 seconds (default)
- Invocation rate limit: 1 request per second (default)

### 4. Event Pattern Configuration
- Source: `PartnerApp`
- DetailType: `PartnerAppDetailType`
- Match all events from this source/detail type combination

## Dynamic Resource Fetching Strategy

No AMIs or AZs are needed for this integration, so dynamic fetching isn't required. However, if we were to extend this with Lambda functions, we would:

- Use SSM Parameter Store for latest Amazon Linux AMIs
- Use `Fn::GetAZs` and `Fn::Select` for AZ selection
- Use dynamic references for secrets instead of plain parameters

## Security Considerations

1. **API Key Protection**
   - Use `NoEcho: true` for the Stripe API key parameter
   - Consider using AWS Secrets Manager for production (dynamic reference)
   - Never log or expose the API key in any outputs

2. **IAM Permissions**
   - Principle of least privilege: only allow necessary EventBridge permissions
   - No VPC or network resources needed (EventBridge is serverless)

3. **Event Validation**
   - Event pattern ensures only PartnerApp events trigger the integration
   - Consider adding schema validation if available

## Error Handling and Monitoring

1. **CloudWatch Logs**
   - Enable logging for the IAM role if debugging is needed
   - Monitor API destination invocation failures

2. **Dead Letter Queue**
   - Consider adding a DLQ for failed invocations (though EventBridge API destinations don't support DLQs directly)

## Deployment Sequence

1. Create IAM Role
2. Create Event Bus
3. Create API Destination (depends on IAM Role and Event Bus)
4. Create Event Rule (depends on Event Bus)

## Linting Compliance Strategy

1. All parameters have safe defaults
2. No hardcoded values (except safe defaults)
3. No deprecated runtimes or properties
4. Proper IAM role configuration with trust relationships
5. No use of Fn::Sub without variables
6. All required properties included, no hallucinated properties

## Extension Points

This architecture can be extended with:

1. **Lambda Function** - For complex event transformation before Stripe API call
2. **Event Schema** - For structured event validation
3. **Multiple API Destinations** - For different Stripe endpoints or other services
4. **Event Replay** - For testing and recovery scenarios

The plan adheres to all specified linting rules and AWS best practices while providing a robust, production-ready EventBridge integration with Stripe's product API.
