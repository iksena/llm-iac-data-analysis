# Architectural Deployment Plan for AWS Security Hub Controls Management

## 1. Key AWS Services Required

Based on the business need, the following AWS services are essential:

- **AWS Lambda**: To host the function that programmatically updates Security Hub controls
- **AWS Identity and Access Management (IAM)**: To create the execution role for Lambda with appropriate permissions
- **Amazon CloudWatch Logs**: To capture Lambda execution logs with configurable retention
- **AWS Security Hub**: The target service for control management
- **AWS Systems Manager (SSM) Parameter Store**: To dynamically fetch the latest Lambda runtime AMI (if needed) and potentially store configuration parameters
- **AWS CloudFormation**: The orchestration service for infrastructure as code

## 2. Properties and Dependencies

The deployment follows this logical sequence:

1. **IAM Role Creation**: The Lambda execution role must be created first, as it's a dependency for the Lambda function
2. **Lambda Function**: Deploys after the IAM role is available
3. **CloudWatch Log Group**: Created alongside or after Lambda to capture logs
4. **Security Hub API Calls**: Executed by Lambda at runtime (not during stack creation)

Key properties and their dependencies:

- **Lambda Execution Role**: Requires `AWS::IAM::Role` with `AssumeRolePolicyDocument` allowing `lambda.amazonaws.com`
- **Lambda Function**: Requires `Code` (inline or S3), `Handler`, `Runtime` (python3.12), and `Role` (referencing the IAM role)
- **CloudWatch Log Group**: Requires `LogGroupName` (typically `/aws/lambda/<function-name>`) and `RetentionInDays`
- **IAM Permissions**: Must include `securityhub:BatchUpdateStandardsControlAssociations` and potentially `securityhub:DescribeStandardsControls`

## 3. Safe AWS Defaults for Missing Properties

For properties not explicitly specified in the business need, I will use these safe defaults:

- **Lambda Runtime**: `python3.12` (latest supported runtime)
- **Lambda Timeout**: `30` seconds (default, sufficient for Security Hub API calls)
- **CloudWatch Log Retention**: `30` days (balances cost and compliance needs)
- **IAM Role Path**: `/` (default path)
- **IAM Permissions Boundary**: Optional, only if provided as a parameter
- **Security Hub Standards**: CIS AWS Foundations Benchmark and AWS Foundational Security Best Practices (hardcoded as these are the only standards with controls that can be disabled)

## 4. Dynamic Resource Fetching Strategy

To avoid hardcoding and ensure the template remains current:

- **Lambda Runtime**: Use `python3.12` directly (no AMI needed for Lambda)
- **Availability Zones**: Use `Fn::GetAZs` and `Fn::Select` to dynamically fetch AZs for the region
- **Security Hub Control ARNs**: These are static and well-documented, so they can be hardcoded in the Lambda function code
- **SSM Parameters**: Not required for this use case, but could be used for storing configuration if needed

## 5. Architecture Overview

The solution consists of:

1. **IAM Role**: Grants Lambda permission to call Security Hub APIs
2. **Lambda Function**: Contains Python code to update Security Hub controls based on input parameters
3. **CloudWatch Log Group**: Captures execution logs with configurable retention
4. **CloudFormation Template**: Orchestrates the deployment with parameters for customization

## 6. Parameter Design

The template will include these parameters with safe defaults:

- **DisableCIS140Controls**: `Boolean` (Default: `false`) - Whether to disable CIS 1.4.0 controls
- **DisableAWSFSPControls**: `Boolean` (Default: `false`) - Whether to disable AWS Foundational Security Best Practices controls
- **PermissionsBoundary**: `String` (Default: `""`) - Optional IAM permissions boundary ARN
- **LogRetentionDays**: `Number` (Default: `30`) - CloudWatch Logs retention period

## 7. Lambda Function Logic

The Lambda function will:

1. Accept the CloudFormation custom resource request
2. Parse the `DisableCIS140Controls` and `DisableAWSFSPControls` parameters
3. Use the Security Hub Boto3 client to call `batch_update_standards_control_associations`
4. Construct the control ARNs for CIS 1.4.0 and AWS Foundational Security Best Practices
5. Update each control's `AssociationStatus` to `DISABLED` if the corresponding parameter is `true`
6. Return success or failure to CloudFormation

## 8. Security Considerations

- **Least Privilege**: The IAM role grants only the permissions needed (`securityhub:BatchUpdateStandardsControlAssociations` and `securityhub:DescribeStandardsControls`)
- **No Hardcoded Secrets**: All configuration is passed via parameters or environment variables
- **Secure Defaults**: Lambda runs with the minimum required permissions and logs are retained for a reasonable period

## 9. Error Handling and Idempotency

The Lambda function will:

- Handle cases where controls are already in the desired state
- Return appropriate error messages to CloudFormation for debugging
- Ensure the custom resource is idempotent (repeated executions with the same parameters produce the same result)

This architecture ensures a robust, automated solution for managing Security Hub controls that can be deployed via CloudFormation with minimal manual intervention.
