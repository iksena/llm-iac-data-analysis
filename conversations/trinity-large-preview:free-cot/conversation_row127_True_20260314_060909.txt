# Architectural Deployment Plan for Amazon EFS Testing Environment

## Key AWS Services Required

1. **Amazon VPC** - To provide the network foundation
2. **Amazon EFS** - The primary shared storage solution
3. **AWS Systems Manager (SSM) Parameter Store** - For dynamic AMI fetching (though not needed for EFS)
4. **AWS CloudFormation** - For infrastructure as code deployment

## Architecture Overview

The deployment will create a basic VPC with a single subnet in us-east-1a, then provision an Amazon EFS file system within that subnet. This provides a shared storage solution suitable for testing environments.

## Detailed Component Analysis

### 1. VPC Configuration

**Properties:**
- CIDR Block: 10.0.0.0/16 (safe default)
- Enable DNS Support: True (default)
- Enable DNS Hostnames: True (default)

**Dependencies:**
- None (VPC is the foundation)

**Availability Zone Handling:**
- Will use `Fn::GetAZs` and `Fn::Select` to dynamically fetch us-east-1a
- This ensures the template works across different AWS accounts where AZ naming might differ

### 2. Subnet Configuration

**Properties:**
- VPC ID: References the created VPC
- CIDR Block: 10.0.0.0/24 (safe default within VPC range)
- Availability Zone: Dynamically selected us-east-1a

**Dependencies:**
- Depends on VPC creation

### 3. Amazon EFS File System

**Properties:**
- PerformanceMode: generalPurpose (safe default for testing)
- Encrypted: False (safe default for testing environments)
- ThroughputMode: bursting (safe default)

**Dependencies:**
- Depends on Subnet creation

### 4. EFS Mount Target

**Properties:**
- FileSystemId: References the created EFS
- SubnetId: References the created subnet
- SecurityGroups: Will create a basic security group allowing NFS traffic

**Dependencies:**
- Depends on EFS and Subnet creation

## Dynamic Resource Fetching Strategy

While EFS doesn't require AMI fetching, I'll demonstrate the SSM Parameter Store approach for best practices:

- **AMI fetching (if needed for future EC2 instances):** Use `resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`
- **This ensures the template always uses the latest AMI without hardcoding**

## Security Considerations

**Security Group for EFS:**
- Create a security group allowing TCP port 2049 (NFS) from trusted sources
- For testing, allow from the VPC CIDR (10.0.0.0/16)
- In production, this would be restricted to specific security groups or IP ranges

## Missing Properties and Safe Defaults

**Explicitly stated defaults:**
- VPC CIDR: 10.0.0.0/16
- Subnet CIDR: 10.0.0.0/24
- Performance Mode: generalPurpose
- Throughput Mode: bursting
- Encrypted: False (testing environment)

**No Parameters required** - all values have safe defaults, ensuring the template deploys without manual input.

## Template Structure Plan

The CloudFormation template will follow this logical order:
1. VPC creation
2. Subnet creation in us-east-1a
3. Security Group creation
4. EFS File System creation
5. Mount Target creation

Each resource will have appropriate `DependsOn` clauses to ensure proper creation order.

## Linting Compliance Strategy

The template will:
- Use dynamic AZ selection with `Fn::GetAZs` and `Fn::Select`
- Avoid hardcoded values
- Use only supported properties per CloudFormation schema
- Include proper dependency management
- Use safe defaults for all configurable properties

This architecture provides a minimal, deployable EFS testing environment that adheres to AWS best practices and will pass cfn-lint validation.
