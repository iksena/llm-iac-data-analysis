# Detailed Architectural Deployment Plan for VPC with Public Subnet

## 1. Key AWS Services Required

Based on the business need, the following AWS services are required:

- **VPC (AWS::EC2::VPC)** - The core networking construct
- **Internet Gateway (AWS::EC2::InternetGateway)** - Provides internet connectivity
- **VPC Gateway Attachment (AWS::EC2::VPCGatewayAttachment)** - Attaches the Internet Gateway to the VPC
- **Public Subnet (AWS::EC2::Subnet)** - A single public subnet for resources
- **Route Table (AWS::EC2::RouteTable)** - Manages routing for the public subnet
- **Route (AWS::EC2::Route)** - Defines the default route through the Internet Gateway
- **Subnet Route Table Association (AWS::EC2::SubnetRouteTableAssociation)** - Associates the subnet with the route table
- **EC2 Instance (AWS::EC2::Instance)** - For demonstration purposes (e.g., CloudWatch auto-alarms)
- **IAM Role (AWS::IAM::Role)** - For EC2 instance permissions
- **IAM Instance Profile (AWS::IAM::InstanceProfile)** - Attaches the IAM role to the EC2 instance

## 2. Properties and Dependencies

The deployment follows this dependency chain:

1. **VPC** must be created first (provides the networking context)
2. **Internet Gateway** can be created in parallel with VPC
3. **VPC Gateway Attachment** connects the Internet Gateway to the VPC
4. **Route Table** is created within the VPC context
5. **Route** is added to the Route Table, pointing to the Internet Gateway
6. **Public Subnet** is created within the VPC
7. **Subnet Route Table Association** links the subnet to the Route Table
8. **EC2 Instance** is launched in the public subnet with the instance profile

## 3. Safe AWS Defaults for Missing Properties

Since specific properties like CIDR blocks or instance types are not provided, I will use these safe defaults:

- **VPC CIDR Block**: `10.0.0.0/16` (standard private range, provides 65,534 usable IPs)
- **Public Subnet CIDR**: `10.0.1.0/24` (24-bit mask, provides 256 usable IPs)
- **EC2 Instance Type**: `t3.micro` (free tier eligible, suitable for demonstrations)
- **EC2 AMI**: Use SSM Parameter Store to fetch the latest Amazon Linux 2 AMI dynamically
- **Availability Zone**: Use `Fn::Select` with `Fn::GetAZs` to automatically select the first AZ in the region
- **EC2 Key Pair**: Default to 'default-key' (will need to be created in the region beforehand)

## 4. Dynamic Resource Fetching Strategy

To avoid hardcoding and ensure the template remains current:

- **AMI ID**: Use `AWS::SSM::Parameter::Value<AWS::EC2::Image::Id>` with the path `/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2` to fetch the latest Amazon Linux 2 AMI
- **Availability Zones**: Use `Fn::GetAZs` to get all AZs in the region, then `Fn::Select` to pick the first one (or allow parameter override)
- **Region-specific resources**: Use `AWS::Region` pseudo-parameter where needed

## 5. Security and Permissions

- **IAM Role**: Create a minimal role with CloudWatch permissions for EC2 monitoring
- **Instance Profile**: Attach the IAM role to the EC2 instance
- **Security Group**: Create a default security group allowing SSH from anywhere (for demonstration purposes)

## 6. Template Structure

The CloudFormation template will be organized as follows:

1. **Parameters Section**: Define configurable options with safe defaults
2. **Resources Section**: Define all AWS resources in dependency order
3. **Outputs Section**: Export key resource IDs for reference

## 7. Linting Compliance Strategy

To ensure the template passes cfn-lint:

- All Parameters will have `Default` values matching their `AllowedPattern` if specified
- No hardcoded AMI IDs or AZs
- No use of deprecated properties or runtimes
- Proper use of intrinsic functions for dynamic values
- No hallucinated properties in resource definitions

This architecture provides a solid foundation for internet-accessible resources while maintaining best practices for CloudFormation template design.
