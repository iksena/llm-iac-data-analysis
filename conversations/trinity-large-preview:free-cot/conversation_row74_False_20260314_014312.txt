# ECS Fargate Task Definition Deployment Plan

## Key AWS Services Required
1. **AWS CloudFormation** - Infrastructure as Code deployment
2. **Amazon ECS** - Container orchestration platform
3. **AWS IAM** - Identity and access management for task roles
4. **AWS CloudWatch Logs** - Centralized logging with retention
5. **AWS SSM** - Session Manager for container command execution
6. **AWS ECR** - Container registry (for execution role permissions)
7. **AWS VPC** - Network infrastructure (implicitly required for ECS)

## Architecture Analysis

### Task Role Requirements
The task role needs SSM Session Manager permissions to enable `ExecuteCommand` functionality. Based on AWS documentation, this requires:
- `ssmmessages:CreateControlChannel`
- `ssmmessages:CreateDataChannel`
- `ssmmessages:OpenControlChannel`
- `ssmmessages:OpenDataChannel`

### Task Execution Role Requirements
The execution role needs standard ECS permissions:
- ECR image pulling: `AmazonEC2ContainerRegistryReadOnly`
- CloudWatch Logs: `CloudWatchLogsFullAccess` (or more specific permissions)

### Container Configuration
- **Image**: Public NGINX image (for demo simplicity)
- **Port**: 80 (standard HTTP)
- **Logging**: CloudWatch Logs integration
- **Retention**: 7 days

## Property Dependencies & Order of Operations

1. **IAM Roles must be created before Task Definition**
   - Task role references IAM role ARN
   - Execution role references IAM role ARN

2. **Log Group must exist before Task Definition**
   - Container definition references log group name

3. **No VPC dependencies** (Fargate is serverless, no explicit VPC needed for this basic task definition)

## Safe Defaults Selection

Since specific properties weren't provided, I'll use these safe defaults:

- **Network Mode**: `awsvpc` (standard for Fargate)
- **Requires Compatibilities**: `FARGIDABLE`
- **CPU**: `256` (1 vCPU equivalent minimum)
- **Memory**: `512` (minimum for basic web serving)
- **Log Retention**: `7` days (as specified)

## Dynamic Resource Fetching Strategy

For this template, dynamic fetching isn't strictly necessary since:
- We're using a public NGINX image (no ECR fetch needed)
- No AMIs are required (Fargate is serverless)
- However, I'll demonstrate best practices by using dynamic references where applicable

## Security & Best Practices Implementation

1. **Principle of Least Privilege**: IAM roles will have only necessary permissions
2. **No hardcoded secrets**: Using AWS managed policies
3. **Resource naming**: Using stack name prefixes for uniqueness
4. **Logging**: Centralized CloudWatch Logs with retention policy

## Missing Information Handling

The business need doesn't specify:
- Task family name → Will use stack name as prefix
- Container name → Will use "nginx-container"
- Log group name → Will use stack name as prefix
- Task role name → Will use stack name as prefix
- Execution role name → Will use stack name as prefix

## Template Structure Plan

The CloudFormation template will follow this logical structure:

1. **Parameters** (with safe defaults)
2. **Resources** (in dependency order):
   - IAM Task Role
   - IAM Execution Role
   - CloudWatch Log Group
   - ECS Task Definition
3. **Outputs** (optional, for reference)

## Key Design Decisions

1. **Public NGINX Image**: Using `nginx:latest` for simplicity as requested
2. **Port Mapping**: Single port 80 mapping for HTTP
3. **Network Mode**: `awsvpc` for Fargate compatibility
4. **Log Configuration**: JSON format with auto-retry policy
5. **IAM Policies**: AWS managed policies where possible, custom inline policies for SSM

## Potential Pitfalls Avoided

1. **No hardcoded ARNs**: Using `!Sub` for dynamic ARN construction
2. **No hardcoded region**: Using `!Ref AWS::Region`
3. **No hardcoded account ID**: Using `!Ref AWS::AccountId`

5. **Valid JSON structure**: Ensuring all required properties are present
