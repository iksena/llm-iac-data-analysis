# Detailed Architectural Deployment Plan for Self-Configuring EC2 with Nginx and SSM

## 1. Key AWS Services Required

Based on the business requirements, the following AWS services are essential:

- **EC2**: To provision the Linux 2 instance
- **IAM**: To create the instance profile and role with SSM permissions
- **SSM (Systems Manager)**: For automated configuration and command execution
- **S3**: For centralized logging storage
- **SSM Parameter Store**: To dynamically fetch the latest Amazon Linux 2 AMI

## 2. Resource Properties and Dependencies

### Resource Dependencies:
1. **VPC and Networking**: While not explicitly required, the EC2 instance needs networking. I'll use the default VPC for simplicity.
2. **IAM Role Creation**: Must be completed before EC2 instance creation
3. **S3 Bucket Creation**: Must exist before SSM Association configuration
4. **SSM Association**: Depends on both the EC2 instance and the IAM role being ready

### Critical Properties:
- **EC2 Instance**: Amazon Linux 2 AMI, t2.micro (free tier eligible), appropriate security group
- **IAM Role**: Permissions for SSM and S3 access
- **S3 Bucket**: Unique name with appropriate policy for SSM logging
- **SSM Association**: Targets the EC2 instance with the shell script

## 3. Safe AWS Defaults for Missing Properties

Since specific properties aren't provided, I'll use these safe defaults:

- **Instance Type**: `t2.micro` (free tier eligible, sufficient for Nginx)
- **CIDR Blocks**: `0.0.0.0/0` for internet access (can be restricted later)
- **Key Pair**: `default-key` (placeholder - user must create or specify)
- **Region**: `us-east-1` (default region)
- **VPC**: Default VPC in the specified region
- **Security Group**: Allow HTTP (port 80) and SSH (port 22) from anywhere

## 4. Dynamic Resource Fetching Strategy

### AMI Retrieval:
I'll use SSM Parameter Store to fetch the latest Amazon Linux 2 AMI dynamically:
```
/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2
```
This ensures the template always uses the most recent AMI without hardcoding.

### Availability Zones:
I'll use `Fn::GetAZs` and `Fn::Select` to dynamically retrieve availability zones instead of hardcoding.

## 5. Architecture Overview

The deployment will follow this sequence:

1. Create IAM Role with SSM and S3 permissions
2. Create S3 bucket for centralized logging
3. Create EC2 instance with the IAM role
4. Configure SSM Association to install Nginx
5. Set up SSM logging to S3 bucket

## 6. Security Considerations

- **IAM Permissions**: Least privilege principle - only necessary permissions for SSM and S3
- **S3 Bucket Policy**: Restrict access to authorized entities only
- **Security Group**: Minimal required ports open (HTTP and SSH)
- **SSM Documents**: Use AWS-managed documents where possible

## 7. SSM Configuration Details

The SSM Association will:
- Target the specific EC2 instance by instance ID
- Use the AWS-RunShellScript document
- Execute commands to update packages and install Nginx
- Ensure Nginx starts automatically

## 8. S3 Logging Configuration

SSM will be configured to:
- Store command outputs in the specified S3 bucket
- Use structured paths under a `logs/` prefix
- Maintain audit trails of all SSM operations

## 9. Error Handling and Validation

The template will include:
- Proper dependency management to prevent race conditions
- Validation of IAM permissions
- Error handling for SSM operations
- Cleanup procedures for failed deployments

This architecture ensures a fully automated, self-configuring EC2 instance that meets all specified requirements while following AWS best practices and security principles.
