# Detailed Architectural Deployment Plan for Amazon Aurora MySQL Cluster

## 1. Key AWS Services Required

Based on the business need, the following AWS services are required:

- **Amazon VPC** - To provide network isolation and security
- **Amazon RDS Aurora MySQL** - The primary database service with cluster and instances
- **AWS SSM Parameter Store** - To dynamically fetch the latest Aurora MySQL engine version
- **AWS Secrets Manager** (optional, for secure password management) - To store database credentials securely
- **AWS Identity and Access Management (IAM)** - For service permissions and roles

## 2. Properties and Dependencies

### Network Dependencies
- VPC must be created before Aurora cluster
- Subnets must exist before Aurora instances can be provisioned
- Security groups must be created before attaching to Aurora instances

### Aurora Dependencies
- Parameter group must be created before attaching to Aurora instances
- Database cluster must be created before instances can join it
- IAM roles must exist before Aurora instances can assume them

### Critical Properties
- Database parameter group settings (time_zone, sql_mode) must be defined
- Instance class must be db.t3.large as specified
- Public accessibility must be enabled for both instances
- Engine version must be dynamically fetched using SSM

## 3. Safe AWS Defaults for Missing Properties

### VPC Configuration
- CIDR Block: `10.0.0.0/16` (standard private network)
- Public Subnet CIDR: `10.0.0.0/24` and `10.0.1.0/24`
- Availability Zones: Dynamically selected using `Fn::GetAZs` and `Fn::Select`

### Security Group Configuration
- Default VPC security group will be used for simplicity
- Port 3306 (MySQL) will be open to the VPC CIDR range

### Database Configuration
- Database name: `aurora_db` (default)
- Master username: `admin` (default)
- Master password: Will use dynamic reference to Secrets Manager or SSM Secure String
- Engine version: Dynamically fetched using SSM Parameter Store

## 4. Dynamic Resource Fetching Strategy

### Aurora Engine Version
```yaml
EngineVersion: !Ref AWS::NoValue
```

The engine version will be dynamically fetched using SSM Parameter Store:
```yaml
!Sub '/aws/rds/aurora-mysql/${EngineMajorVersion}/latest`
```

### AMI Fetching (if needed for any related resources)
For any EC2 instances or related resources:
```yaml
!Sub '/aws/service/ami-amazon-linux-latest/amzn2-ami-hvn-x86_64-gp2'
```

## 5. Architecture Design

### Network Architecture
- Single VPC with two public subnets across two AZs
- Public subnets will host the Aurora instances with public accessibility
- Security group allowing MySQL traffic from within VPC

### Database Architecture
- Aurora MySQL cluster with cluster endpoint
- Two db.t3.large instances in separate AZs for redundancy
- Custom parameter group with time_zone and sql_mode settings
- Public accessibility enabled for both instances

### Security Considerations
- Database credentials will be stored in Secrets Manager
- IAM roles will be used for Aurora service access
- Security group will restrict access to VPC CIDR range

### Parameter Group Configuration
```yaml
DBParameterGroupName: aurora-mysql-params
Parameters:
  time_zone: "UTC"
  sql_mode: "STRICT_TRANS_TABLES,NO_ZERO_DATE,NO_ZERO_IN_DATE,ERROR_FOR_DIVISION_BY_ZERO"
```

## 6. Template Structure Planning

The CloudFormation template will follow this logical structure:

1. **Parameters** - With safe defaults for all configurable properties
2. **Mappings** - For any static configuration (if needed)
3. **Resources** - In dependency order:
   - VPC and subnets
   - Security group
   - Database parameter group
   - Aurora cluster
   - Aurora instances
4. **Outputs** - For cluster endpoint and instance endpoints

## 7. Compliance with Best Practices

### Parameter Handling
- All Parameters will have `Default` values matching their `AllowedPattern`
- Example: KeyPair name default will be 'default-key' if pattern requires alphanumeric

### Dynamic References
- Database password will use `{{resolve:secretsmanager:...}}` or `{{resolve:ssm-secure:...}}`
- Engine version will use SSM Parameter Store
- No hardcoded AMI IDs or AZ names

### Resource Protection
- DeletionPolicy and UpdateReplacePolicy will be set for Aurora resources
- This ensures data protection during stack updates

### Security Implementation
- Secrets Manager for credentials
- IAM roles for service permissions
- Security groups for network isolation

This architectural plan ensures a production-ready, secure, and maintainable Aurora MySQL deployment that adheres to AWS best practices and will pass cfn-lint validation.
