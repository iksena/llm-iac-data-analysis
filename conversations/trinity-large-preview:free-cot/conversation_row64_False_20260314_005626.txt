# AWS CloudFormation Deployment Plan for Web Application Environment

## Key AWS Services Required

1. **VPC** - Core networking construct
2. **Internet Gateway** - Provides internet connectivity
3. **Route Tables** - Directs traffic within VPC
4. **Subnets** - Public and private network segments
5. **EC2 Instance** - Web server hosting the application
6. **Security Groups** - Network access control
7. **IAM Role** - Instance permissions for SSM and CloudWatch
8. **SSM Parameter Store** - Dynamic AMI fetching

## Architecture Overview

The deployment will create a VPC with:
- One public subnet for the web server
- One private subnet (for future expansion)
- Internet Gateway attached to VPC
- Route table directing internet traffic to the gateway
- EC2 instance in public subnet with Apache, PHP, and inventory app

## Detailed Resource Properties and Dependencies

### VPC Configuration
- **CIDR Block**: Default to `10.0.0.0/16` (safe default for small environments)
- **Dependencies**: None (base resource)

### Internet Gateway
- **Properties**: Must be attached to VPC
- **Dependencies**: Created after VPC, attached immediately after creation

### Route Tables
- **Main Route Table**: Created with VPC, routes internal traffic
- **Public Route Table**: Created after IGW, routes 0.0.0.0/0 to IGW
- **Dependencies**: IGW must exist before public route table creation

### Subnets
- **Public Subnet**: CIDR `10.0.1.0/24`, Availability Zone dynamically selected
- **Private Subnet**: CIDR `10.0.2.0/24`, same AZ as public
- **Dependencies**: VPC must exist first, then subnets, then route table associations

### EC2 Instance
- **Instance Type**: Default to `t3.micro` (free tier eligible)
- **AMI**: Dynamically fetched using SSM Parameter Store
- **Dependencies**: VPC, subnets, security groups, IAM role must exist first

## Dynamic Resource Fetching Strategy

### AMI Selection
Use SSM Parameter Store to fetch the latest Amazon Linux 2 AMI:
```yaml
Fn::ImportValue: !Sub 'resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2'
```

This ensures we always get the latest patched AMI without hardcoding IDs.

### Availability Zones
Use `Fn::GetAZs` and `Fn::Select` to dynamically select AZs:
```yaml
Fn::Select: [0, Fn::GetAZs: !Ref 'AWS::Region']
```

This makes the template portable across regions.

## Security Group Configuration

### Web Server Security Group
- **Ingress**: HTTP (port 80) from 0.0.0.0/0, SSH (port 22) from 0.0.0.0/0 (for testing)
- **Egress**: All traffic (0.0.0.0/0) on all ports
- **Dependencies**: Created after VPC

### User Data Script Strategy

The user data will:
1. Update system packages
2. Install Apache HTTP Server
3. Install PHP
4. Download and configure a sample inventory application
5. Start Apache service

Script will be base64-encoded and include error handling.

## IAM Role Configuration

Create an IAM role with:
- **SSM Managed Policy**: `AmazonSSMManagedInstanceCore` (for Systems Manager access)
- **CloudWatch Managed Policy**: `CloudWatchAgentServerPolicy` (for monitoring)
- **Custom Policy**: Allow EC2 actions for instance management

## Parameter Strategy

Define parameters with safe defaults:
- **KeyName**: Default to 'default-key' (user must create this key pair first)
- **InstanceType**: Default to 't3.micro'
- **WebServerPort**: Default to 80
- **SSHLocation**: Default to '0.0.0.0/0' (for testing, should be restricted in production)

## Template Structure and Dependencies

1. **Parameters** - Define all configurable options
2. **Resources** - Create in dependency order:
   - VPC
   - Internet Gateway + VPC attachment
   - Route Tables
   - Subnets
   - Security Groups
   - IAM Role + Instance Profile
   - EC2 Instance (with user data)
3. **Outputs** - Return public IP and DNS name

## Error Prevention Measures

- All parameters have defaults matching their allowed patterns
- No hardcoded AMIs or AZs
- Security groups properly reference VPC
- User data script includes error handling and logging
- IAM policies use exact ARNs
- Resources have proper dependency chains

## Production Considerations (Not in Initial Scope)

- Use Elastic IP for static public IP
- Add NAT Gateway for private subnet internet access
- Implement Auto Scaling Group for high availability
- Add Application Load Balancer for better traffic distribution
- Implement proper SSH access restrictions (security group rules)
- Add CloudWatch alarms for monitoring
- Implement IAM policies with least privilege

This architecture provides a solid foundation that can be extended for production workloads while maintaining CloudFormation best practices and ensuring successful deployments.
