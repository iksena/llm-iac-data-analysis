# Architectural Deployment Plan for Scheduled SSM Automation Workflow

## Key AWS Services Required

Based on the business requirements, I'll need to provision the following AWS services:

1. **Amazon EC2** - For Linux and Windows instances with SSM agent installed
2. **AWS Systems Manager (SSM)** - For cross-platform script execution and State Manager
3. **EventBridge Scheduler** - For triggering the automation workflow
4. **IAM** - For security roles and permissions
5. **Amazon S3** - For storing execution logs
6. **AWS SSM Parameter Store** - For dynamic AMI fetching
7. **AWS CloudFormation** - For infrastructure as code deployment

## Properties and Dependencies Analysis

### VPC and Network Dependencies
- VPC must be created before EC2 instances
- Internet Gateway must be attached to VPC before route table creation
- Route table must be associated with public subnet before EC2 launch
- Security groups must exist before EC2 instance creation

### SSM Dependencies
- IAM roles must be created before EC2 instances
- SSM document must be created before State Manager association
- State Manager association requires SSM document to exist

### EventBridge Dependencies
- IAM role for EventBridge must exist before scheduler creation
- Scheduler requires SSM State Manager association to exist

## Safe AWS Defaults (where properties were missing)

### Network Configuration
- VPC CIDR: `10.0.0.0/16` (standard private range)
- Public Subnet CIDR: `10.0.1.0/24`
- Availability Zone: First AZ from `Fn::GetAZs` (dynamic selection)
- EC2 Instance Type: `t3.micro` (free tier eligible, sufficient for testing)

### Security Configuration
- SSH/RDP access: Restricted to 0.0.0.0/0 for simplicity (consider tightening in production)
- Key Pair: Parameter with default 'default-key' (user must create this beforehand)

### SSM Configuration
- Linux AMI: Latest Amazon Linux 2 from SSM Parameter Store
- Windows AMI: Latest Windows Server 2019 from SSM Parameter Store
- SSM Document: Cross-platform document with OS-specific commands

## Dynamic Resource Fetching Strategy

### AMI Selection via SSM Parameter Store
I'll use the following SSM parameters to fetch the latest AMIs dynamically:
- Linux: `/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`
- Windows: `/aws/service/ami-windows-latest/Windows_Server-2019-English-Full-Base`

This approach ensures we always use the latest patched AMIs without hardcoding IDs.

### Availability Zone Selection
Using `Fn::GetAZs` and `Fn::Select` to dynamically select the first available AZ:
```yaml
Fn::Select:
  - 0
  - Fn::GetAZs: !Ref 'AWS::Region'
```

## Security Architecture

### IAM Roles
1. **EC2InstanceRole** - For EC2 instances with SSM and S3 permissions
2. **EventBridgeSchedulerRole** - For EventBridge to invoke SSM actions
3. **S3AccessRole** - For SSM to write logs to S3

### Permissions Breakdown
- EC2 instances: SSM agent, S3 read/write for logs
- EventBridge: SSM StartAssociationsExecution permission
- S3 bucket: Public access blocked, versioning enabled for security

## Resource Dependencies Flow

```
1. IAM Roles → 2. VPC/Network → 3. Security Groups
   ↓                ↓                ↓
4. EC2 Instances → 5. SSM Document → 6. State Manager Association
   ↓                ↓                ↓
7. S3 Bucket → 8. EventBridge Scheduler
```

## Tag-Based Targeting Strategy

The SSM State Manager Association will target instances using the tag `HelloWorld: true`. This requires:
- EC2 instances to be created with this tag
- SSM document to support cross-platform execution
- EventBridge schedule to trigger the association

## Output Strategy

The template will output:
- Instance IDs (for verification)
- S3 bucket name (for log access)
- EventBridge schedule ARN (for monitoring)
- Public IP addresses (for SSH/RDP access)

## Error Handling Considerations

- All resources will have appropriate `DeletionPolicy` and `UpdateReplacePolicy` where applicable
- IAM policies will use least privilege principle
- S3 bucket will have versioning enabled for data protection
- EventBridge schedule will have appropriate retry policies

This architecture ensures a secure, scalable, and maintainable solution that adheres to AWS best practices while meeting all specified requirements.
