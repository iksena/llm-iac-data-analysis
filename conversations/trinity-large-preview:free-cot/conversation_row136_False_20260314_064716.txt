# IAM Governance Control Architecture Plan

## Key AWS Services Required
1. **AWS Identity and Access Management (IAM)** - Core service for creating roles, policies, and permission boundaries
2. **AWS Systems Manager (SSM) Parameter Store** - For dynamic AMI fetching (though not directly used in this IAM-focused template)
3. **AWS CloudFormation** - Infrastructure as Code deployment mechanism

## Architecture Overview

This solution implements a governance framework for IAM resource creation with configurable permission boundaries. The architecture centers around an IAM role that developers can assume, with conditional permission boundaries that can be toggled between permissive and restricted modes.

## Detailed Component Analysis

### 1. IAM Role Structure
- **Base Role**: Provides broad IAM privileges for developers to create/modify IAM resources
- **Permission Boundary Policy**: Restricts actions to S3 and read-only IAM operations
- **Conditional Enforcement**: Toggle parameter to switch between permissive and restricted modes

### 2. Governance Controls
- **Naming Prefix Enforcement**: Only allows creation of roles/policies with `MyTeam*` prefix
- **Mandatory Tagging**: Requires `Owner` tag on all developer-managed IAM resources
- **Permission Boundary Application**: Automatically applies boundary to all created roles

### 3. Parameter Configuration
```yaml
Parameters:
  EnablePermissionBoundary:
    Type: String
    Default: "true"
    AllowedValues: ["true", "false"]
    Description: Enable strict permission boundary enforcement
  TeamOwnerTag:
    Type: String
    Default: "MyTeam"
    AllowedPattern: "^[a-zA-Z0-9-_]+$"
    Description: Team identifier for Owner tag
```

### 4. IAM Policies Design

#### Base IAM Policy (Developer Role)
```yaml
PolicyDocument:
  Version: "2012-10-17"
  Statement:
    - Effect: Allow
      Action:
        - iam:CreateRole
        - iam:DeleteRole
        - iam:UpdateRole
        - iam:CreatePolicy
        - iam:DeletePolicy
        - iam:AttachRolePolicy
        - iam:DetachRolePolicy
        - iam:TagRole
        - iam:UntagRole
        - iam:ListAttachedRolePolicies
        - iam:GetRole
        - iam:ListPolicies
      Resource: "*"
```

#### Permission Boundary Policy
```yaml
PolicyDocument:
  Version: "2012-10-17"
  Statement:
    - Effect: Allow
      Action:
        - s3:*
        - iam:Get*
        - iam:List*
        - iam:Generate*
      Resource: "*"
```

#### Conditional Policy with Naming Prefix
```yaml
PolicyDocument:
  Version: "2012-10-17"
  Statement:
    - Effect: Allow
      Action: iam:*
      Resource: 
        - !Sub "arn:${AWS::Partition}:iam::${AWS::AccountId}:role/MyTeam*"
        - !Sub "arn:${AWS::Partition}:iam::${AWS::AccountId}:policy/MyTeam*"
```

### 5. Dependencies and Properties

#### Resource Dependencies
- IAM Role creation depends on successful policy attachment
- Permission boundary must be defined before role creation
- Tagging requires IAM permissions to be established

#### Property Requirements
- All IAM resources must have proper ARNs
- Policies require Version and Statement properties
- Permission boundaries need explicit PolicyDocument definitions

### 6. Safe Defaults Implementation

#### Default Values
- `EnablePermissionBoundary`: "true" (enforced by default)
- `TeamOwnerTag`: "MyTeam" (standard team identifier)
- Permission boundary ARN: Uses AWS managed policy or custom policy

#### AWS Best Practices
- Use AWS managed policies where applicable
- Implement least privilege principle
- Ensure proper error handling and logging

### 7. Dynamic Resource Handling

#### AMI Fetching (Future Extension)
While not directly applicable to IAM, the pattern would be:
```yaml
ImageId: !Sub "resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2"
```

#### Availability Zones
```yaml
!Select [0, !GetAZs ""]
```

### 8. Security Considerations

#### Permission Boundary Enforcement
- Boundary policy restricts to S3 and read-only IAM
- Overrides any permissions of roles created by developers
- Cannot be removed by developers (managed by governance team)

#### Tagging Strategy
- Mandatory `Owner` tag for accountability
- Team-specific tagging for resource tracking
- Automated validation of tag presence

### 9. Template Structure

```yaml
Resources:
  DeveloperGovernanceRole:
    Type: AWS::IAM::Role
    Properties:
      AssumeRolePolicyDocument:
        Version: "2012-10-17"
        Statement:
          - Effect: Allow
            Principal:
              Service: ec2.amazonaws.com
            Action: sts:AssumeRole
      ManagedPolicyArns:
        - !Ref PermissionBoundaryPolicy
      Policies:
        - PolicyName: DeveloperIAMAccess
          PolicyDocument: !Ref DeveloperIAMPolicy
      Tags:
        - Key: Owner
          Value: !Ref TeamOwnerTag
```

### 10. Validation and Testing

#### Linting Compliance
- All parameters have safe defaults
- No hardcoded values
- Proper use of intrinsic functions
- Correct AWS resource type names

#### Functional Testing
- Verify permission boundary enforcement
- Test naming prefix restrictions
- Validate mandatory tagging
- Confirm toggle functionality

This architecture provides a robust governance framework for IAM resource creation while maintaining flexibility through configurable parameters. The solution enforces organizational policies through permission boundaries and automated controls, ensuring compliance and accountability in developer-managed IAM resources.
