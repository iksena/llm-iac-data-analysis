# Architectural Deployment Plan for EC2 Instance with Security Group and S3 Bucket

## 1. Key AWS Services Required

Based on the business need, I'll need the following AWS services:

- **EC2 (Elastic Compute Cloud)**: To create the virtual server instance
- **VPC (Virtual Private Cloud)**: To provide network isolation (required for EC2)
- **Internet Gateway**: To enable internet connectivity for the EC2 instance
- **Route Table**: To direct traffic to the internet gateway
- **EC2 Security Group**: To control inbound/outbound traffic to the instance
- **S3 (Simple Storage Service)**: To create the storage bucket
- **IAM Role**: For EC2 instance profile (best practice for S3 access)

## 2. Properties and Dependencies

### Critical Dependencies:
1. **VPC must be created before** EC2 instance, Security Group, and Internet Gateway
2. **Internet Gateway must be attached to VPC** before creating route table entries
3. **Security Group must be created** before EC2 instance (dependency)
4. **S3 bucket can be created independently** but should be created before any resources that need to reference it

### Property Dependencies:
- EC2 instance requires:
  - VPC ID (for subnet)
  - Security Group ID
  - KeyPair name (for SSH access)
  - AMI ID (for instance image)
- Security Group requires:
  - VPC ID
- Route Table requires:
  - VPC ID
  - Internet Gateway ID

## 3. Safe AWS Defaults (for missing properties)

Since specific properties aren't provided, I'll use these safe defaults:

- **CIDR Block**: `10.0.0.0/16` (standard private VPC range)
- **Subnet CIDR**: `10.0.1.0/24` (single subnet in first AZ)
- **EC2 Instance Type**: `t3.micro` (free tier eligible, suitable for basic web serving)
- **SSH KeyPair**: `default-key` (will require user to create this key pair)
- **AMI**: Latest Amazon Linux 2 using SSM Parameter Store
- **Availability Zone**: Dynamically selected using `Fn::GetAZs` and `Fn::Select`
- **S3 Bucket Name**: Dynamically generated using `AWS::AccountId` and `AWS::Region` to ensure uniqueness

## 4. Dynamic Resource Fetching Strategy

### AMI Retrieval using SSM Parameter Store:
I'll use the SSM Parameter Store to fetch the latest Amazon Linux 2 AMI dynamically:
```yaml
ImageId: !Sub 'arn:aws:ssm:${AWS::Region}:amazon/linux-ami/amzn2-ami-hvm-x86_64-gp2'
```

This ensures the template always uses the latest patched AMI without hardcoding.

### Availability Zone Selection:
```yaml
!Select [0, !GetAZs '']
```
This selects the first available AZ in the region, avoiding hardcoded AZ names.

## 5. Security Group Configuration

The security group will:
- Allow HTTP (port 80) from anywhere (0.0.0.0/0)
- Allow SSH (port 22) from the user's IP (using a Parameter with safe default)
- Allow all outbound traffic

## 6. S3 Bucket Configuration

The S3 bucket will:
- Use a unique name based on account ID and region
- Have a bucket policy allowing public read access (if needed for web hosting)
- Include OwnershipControls if using legacy AccessControl property (though I'll use modern BucketPolicy approach)

## 7. EC2 Instance Configuration

The EC2 instance will:
- Use the latest Amazon Linux 2 AMI
- Be placed in the created subnet
- Have the security group attached
- Include an IAM role for potential S3 access
- Have a public IP address assigned

## 8. Network Architecture

The VPC will have:
- One public subnet (10.0.1.0/24)
- Internet Gateway attached
- Route table directing 0.0.0.0/0 to Internet Gateway
- Route table association with the public subnet

## 9. Template Structure Plan

The CloudFormation template will be organized as:

1. **Parameters**: For customizable values (SSH location, KeyPair name)
2. **Resources**: All AWS resources in logical order
3. **Outputs**: For displaying instance public IP and S3 bucket URL

This architecture ensures a secure, scalable, and maintainable deployment that follows AWS best practices and will pass cfn-lint validation.
