# IoT Data Pipeline Architecture Plan

## Key AWS Services Required

1. **AWS IoT Core** - For MQTT message ingestion from IoT devices
2. **Amazon SNS** - Pub/sub messaging service to act as intermediary
3. **Amazon SQS** - Message queuing for reliable message storage
4. **AWS Systems Manager (SSM) Parameter Store** - For dynamic resource fetching

## Architecture Overview

The pipeline will follow this flow:
Device → IoT Core (MQTT topic: `device/data`) → IoT Rule → SNS Topic → SQS Queue

## Detailed Resource Plan

### 1. SQS Queue
- **Resource Type**: `AWS::SQS::Queue`
- **Properties**:
  - `QueueName`: Use a descriptive name like `IoTDataQueue`
  - `VisibilityTimeout`: Default to 30 seconds (safe AWS default)
  - `MessageRetentionPeriod`: Default to 345,600 seconds (4 days)
  - `MaximumMessageSize`: Default to 262,144 bytes (256 KB)

### 2. SNS Topic
- **Resource Type**: `AWS::SNS::Topic`
- **Properties**:
  - `TopicName`: Use a descriptive name like `IoTDataTopic`
  - `DisplayName`: Optional, but recommended for identification

### 3. IoT Rule
- **Resource Type**: `AWS::IoT::TopicRule`
- **Properties**:
  - `RuleName`: Use a descriptive name like `IoTDataRule`
  - `TopicRulePayload`: Contains the rule configuration
  - `SqlVersion`: Use `2016-03-31` (latest stable version)
  - `Sql`: Query to capture messages from `device/data` topic

### 4. IAM Role for IoT Rule
- **Resource Type**: `AWS::IAM::Role`
- **Properties**:
  - `RoleName`: Use a descriptive name like `IoTDataRuleRole`
  - `AssumeRolePolicyDocument`: Allow IoT Core to assume this role
  - `ManagedPolicyArns`: Attach `AWSIoTLogging` policy
  - `Policies`: Custom policy allowing SNS publish and SQS send

## Dependencies and Properties

1. **SNS Topic must be created before IoT Rule** - The IoT Rule needs the SNS Topic ARN
2. **SQS Queue must be created before IoT Rule** - The IoT Rule needs the SQS Queue ARN
3. **IAM Role must be created before IoT Rule** - The IoT Rule needs the IAM Role ARN

## Safe AWS Defaults

Since specific properties weren't provided:

- **CIDR Blocks**: Not applicable for this template
- **Instance Types**: Not applicable (no EC2 instances)
- **Key Pairs**: Not applicable (no EC2 instances)
- **AMI IDs**: Not applicable (no EC2 instances)
- **Availability Zones**: Not applicable (no regional resources requiring AZs)

## Dynamic Resource Fetching

For this specific template, we don't need to fetch AMIs or AZs since we're not provisioning EC2 instances or VPCs. However, if we were to extend this template in the future:

- **AMI IDs**: Use SSM Parameter Store with `resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`
- **Availability Zones**: Use `Fn::GetAZs` and `Fn::Select` functions

## Security Considerations

1. **IAM Policies**: The IoT Rule IAM Role will need explicit permissions:
   - `sns:Publish` to the SNS Topic ARN
   - `sqs:SendMessage` to the SQS Queue ARN
   - `iot:Connect`, `iot:Publish`, `iot:Subscribe`, `iot:Receive` for IoT Core access

2. **Resource Policies**: Consider adding resource-based policies if cross-account access is needed

3. **Encryption**: SQS and SNS support server-side encryption by default

## Error Handling and Best Practices

1. **Deletion Policies**: Not critical for this template since all resources are replaceable
2. **UpdateReplace Policies**: Not critical for this template
3. **Naming Conventions**: Use consistent, descriptive names for all resources
4. **Resource Tagging**: Consider adding tags for cost allocation and management

## Template Structure

The CloudFormation template will follow this logical order:

1. Parameters (if any, with safe defaults)
2. Resources in dependency order:
   - SQS Queue
   - SNS Topic
   - IAM Role
   - IoT Rule
3. Outputs (optional, for resource ARNs)

This architecture provides a robust, scalable IoT data pipeline that leverages AWS's managed services for reliability and security. The use of SNS as an intermediary provides flexibility for future expansion (e.g., adding more subscribers to the topic).
