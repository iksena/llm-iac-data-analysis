# AWS CloudFormation Architecture Plan: Secure Parameter Storage and Retrieval for Serverless Applications

## 1. Key AWS Services Required

Based on the business requirements, I'll need the following AWS services:

- **AWS Systems Manager (SSM) Parameter Store** - For storing non-sensitive configuration parameters
- **AWS Secrets Manager** - For storing sensitive credentials with automatic rotation
- **AWS Lambda** - For the serverless function that retrieves parameters and secrets
- **AWS IAM** - For creating least-privilege roles and policies
- **AWS CloudWatch Logs** - For Lambda function logging
- **AWS CloudFormation** - For infrastructure as code deployment

## 2. Properties and Dependencies

### Critical Dependencies:
1. IAM Role must be created before Lambda function (Lambda needs to reference the role)
2. Lambda function must be created before any test events or permissions are added
3. SSM Parameters and Secrets Manager resources can be created in parallel
4. Lambda function needs appropriate IAM permissions to access SSM and Secrets Manager

### Properties to Define:
- Lambda function runtime: `python3.12` (latest supported)
- Lambda timeout: 30 seconds (safe default)
- Lambda memory: 128 MB (minimum, sufficient for parameter retrieval)
- SSM parameter namespace: Stack-specific prefix (e.g., `/myapp/${AWS::StackName}/`)
- Secrets Manager secret name: Stack-specific prefix (e.g., `${AWS::StackName}-db-credentials`)

## 3. Safe AWS Defaults

Since specific properties weren't provided, I'll use these safe defaults:

- **Lambda Runtime**: `python3.12` (avoiding deprecated runtimes)
- **Lambda Timeout**: `30` seconds
- **Lambda Memory**: `128` MB
- **SSM Parameter Type**: `String` (for non-sensitive config)
- **Secrets Manager Secret Type**: `Credentials` (with username and password)
- **Password Length**: `16` characters
- **Excluded Characters**: `"/@"` (as specified in requirements)
- **IAM Managed Policies**:
  - `AWSLambdaBasicExecutionRole` for CloudWatch logging
  - Custom inline policies for SSM and Secrets Manager access

## 4. Dynamic Resource Fetching Strategy

### AMI Handling:
Not applicable for this serverless architecture (no EC2 instances).

### Availability Zones:
Will use `Fn::GetAZs` and `Fn::Select` to avoid hardcoding AZs:
```yaml
Fn::Select:
  - 0
  - Fn::GetAZs: !Ref 'AWS::Region'
```

## 5. Security Architecture

### Least Privilege IAM Policies:
- Lambda execution role will have:
  - `AWSLambdaBasicExecutionRole` (managed policy)
  - Custom inline policy for SSM access (GetParametersByPath)
  - Custom inline policy for Secrets Manager access (GetSecretValue)

### Parameter Organization:
- SSM Parameters will use hierarchical naming: `/stackname/config/key`
- This enables path-based retrieval using `GetParametersByPath`

### Secret Security:
- Secrets Manager will enforce password complexity
- Excluded characters will prevent issues with JSON parsing
- Secret will be encrypted at rest using AWS KMS

## 6. Lambda Function Design

### Boto3 Workflows:
1. **SSM Parameter Retrieval**:
   - Use `get_parameters_by_path` with recursive=True
   - Decrypt secure strings automatically
   - Handle pagination for large parameter sets

2. **Secrets Manager Retrieval**:
   - Use `get_secret_value` with appropriate exception handling
   - Parse JSON response for credentials
   - Implement retry logic for eventual consistency

### Error Handling:
- Implement comprehensive try-catch blocks
- Return meaningful error messages
- Log all operations to CloudWatch

## 7. CloudFormation Resource Structure

The template will follow this logical order:

1. **Parameters** (with safe defaults)
2. **IAM Role and Policies**
3. **SSM Parameters**
4. **Secrets Manager Secret**
5. **Lambda Function**
6. **Permissions and Test Events**

## 8. Compliance with Best Practices

### Linting Compliance:
- All Parameters have `Default` values matching `AllowedPattern`
- No hardcoded AMIs or AZs
- No deprecated Lambda runtimes
- Proper `DeletionPolicy` and `UpdateReplacePolicy` pairs
- No unnecessary `Fn::Sub` usage
- All required properties included per CloudFormation schema

### Security Best Practices:
- No hardcoded secrets in template
- Least-privilege IAM policies
- Encrypted parameters and secrets
- Secure Lambda execution environment
- Comprehensive logging

## 9. Testing Strategy

The architecture will include:
- Test Lambda invocation with sample parameters
- Validation of parameter retrieval logic
- Verification of secret decryption
- IAM permission testing
- CloudWatch log verification

This architecture provides a secure, scalable foundation for centralized configuration management in serverless applications while adhering to AWS best practices and ensuring successful CloudFormation deployment.
