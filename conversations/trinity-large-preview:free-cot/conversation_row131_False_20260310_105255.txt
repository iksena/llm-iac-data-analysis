# Centralized Alerting System Architecture Plan

## Key AWS Services Required

1. **Amazon SNS** - Primary notification service with topics and subscriptions
2. **Amazon CloudWatch** - Monitoring and alarm creation for failed notifications
3. **AWS Systems Manager (SSM) Parameter Store** - Dynamic resource fetching (though not directly used in this template)
4. **AWS Identity and Access Management (IAM)** - Service policies for SNS topic access

## Architecture Overview

The solution creates a centralized alerting system with the following components:

1. **Primary SNS Topic** - Receives notifications from multiple AWS services
2. **Subscription Management** - Configurable endpoints (email, HTTP/HTTPS) with retry policies
3. **Fallback Mechanism** - Secondary SNS topic for failed notification delivery
4. **Monitoring** - CloudWatch Alarm to detect and route failed notifications

## Detailed Component Design

### 1. SNS Topic Configuration

**Primary Topic:**
- Topic name: `CentralizedAlertingTopic`
- Policy: Allows authorized AWS services to publish messages
- Subscriptions: User-configurable (email, HTTP/HTTPS endpoints)
- Retry policy: Configurable for HTTP/HTTPS targets

**Fallback Topic:**
- Topic name: `AlertingFallbackTopic`
- Used when primary delivery fails
- Default subscription: Email address for critical alerts

### 2. Subscription Management

**Email Subscriptions:**
- Parameter: `EmailSubscriptions` (comma-separated list)
- Default: `admin@example.com` (safe default)
- Validation: Email format pattern matching

**HTTP/HTTPS Subscriptions:**
- Parameter: `WebhookEndpoints` (comma-separated list)
- Default: Empty (no default webhook)
- Retry policy: Configurable (default: 3 retries, 1-minute intervals)

**Conditions:**
- Email subscriptions only created if email addresses are provided
- Webhook subscriptions only created if endpoints are provided

### 3. CloudWatch Alarm for Failure Monitoring

**Alarm Configuration:**
- Metric: `NumberOfNotificationsFailed` from SNS
- Threshold: ≥ 1 failure
- Period: 60 seconds
- Evaluation periods: 1
- Actions: Publish to fallback SNS topic

### 4. IAM Policy for Service Access

**Topic Policy:**
- Version: 2012-10-17
- Statement: Allow `sns:Publish` from authorized AWS services
- Services: CloudWatch Alarms, RDS, S3, Budgets, etc.
- Condition: Source ARN matching allowed services

## Parameter Design with Safe Defaults

### Required Parameters

1. **EmailSubscriptions**
   - Type: CommaDelimitedList
   - Default: `admin@example.com`
   - Description: Email addresses for alert delivery

2. **WebhookEndpoints**
   - Type: CommaDelimitedList
   - Default: Empty list
   - Description: HTTP/HTTPS endpoints for webhook delivery

3. **RetryPolicy**
   - Type: Number
   - Default: 3
   - Description: Number of retry attempts for HTTP/HTTPS endpoints

4. **RetryDelay**
   - Type: Number
   - Default: 60
   - Description: Retry delay in seconds

### Optional Parameters

1. **FallbackEmail**
   - Type: String
   - Default: `fallback@example.com`
   - Description: Email for fallback notifications

## Resource Dependencies and Ordering

1. **SNS Topics** → Created first (no dependencies)
2. **IAM Policy** → Depends on SNS topic creation
3. **Subscriptions** → Depends on SNS topic and parameters
4. **CloudWatch Alarm** → Depends on SNS topic and fallback topic

## Dynamic Resource Fetching Strategy

While this template doesn't require AMIs or AZs, the approach would be:

- **AMIs**: Use `resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`
- **AZs**: Use `Fn::GetAZs` with `Fn::Select` for specific index

## Security Considerations

1. **IAM Policy Scope**: Restrict publish permissions to specific AWS services only
2. **Subscription Validation**: Email confirmation required for all email subscriptions
3. **HTTPS Endpoints**: Require valid SSL certificates for webhook endpoints
4. **Retry Policy**: Prevent infinite retry loops with reasonable limits

## Failure Handling Strategy

1. **Primary Failure**: CloudWatch Alarm triggers on any failed notification
2. **Fallback Routing**: Failed notifications routed to secondary SNS topic
3. **Escalation**: Fallback topic delivers to critical email address
4. **Monitoring**: Both primary and fallback delivery failures monitored

## Template Structure Plan

```yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: Centralized Alerting System with SNS and CloudWatch Monitoring

Parameters:
  - EmailSubscriptions
  - WebhookEndpoints
  - RetryPolicy
  - RetryDelay
  - FallbackEmail

Resources:
  - PrimarySNSTopic
  - TopicPolicy
  - EmailSubscriptions (conditional)
  - WebhookSubscriptions (conditional)
  - FallbackSNSTopic
  - CloudWatchAlarm

Conditions:
  - HasEmailSubscriptions
  - HasWebhookEndpoints
```

## Linting Compliance Strategy

1. **All Parameters**: Will have safe defaults matching allowed patterns
2. **No Hardcoding**: No hardcoded AMIs, AZs, or resource names
3. **S3 Modernization**: Not applicable (no S3 buckets in this template)
4. **Resource Protection**: Will include both DeletionPolicy and UpdateReplacePolicy where applicable
5. **Security**: Will use dynamic references for any secrets (though none required here)
6. **Lambda Runtimes**: Not applicable (no Lambda functions in this template)

This architecture provides a robust, production-ready alerting system that meets all specified requirements while adhering to AWS best practices and CloudFormation linting rules.
