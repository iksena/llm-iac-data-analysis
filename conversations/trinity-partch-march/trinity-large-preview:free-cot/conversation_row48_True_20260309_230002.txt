# AWS Glue ETL Job Infrastructure Deployment Plan

## 1. Key AWS Services Required

Based on the business need, the following AWS services are required:

- **AWS Glue** - For creating and executing ETL jobs
- **Amazon S3** - For storing Glue scripts and Spark logs
- **AWS IAM** - For creating the service role with necessary permissions
- **AWS Systems Manager (SSM) Parameter Store** - For dynamic resource fetching (though not explicitly needed here, it's a best practice for future extensibility)

## 2. Properties and Dependencies

The template will have the following dependencies and properties:

- **S3 Bucket** must be created before the IAM role can reference it in its policy
- **IAM Role** must exist before the Glue Job can reference it
- **Glue Job** depends on both the IAM role and S3 bucket being available

## 3. Safe AWS Defaults

Since specific properties weren't provided, I'll use these safe defaults:

- **S3 Bucket Name**: A unique name using the stack name and a timestamp to ensure uniqueness
- **IAM Role Name**: Generated using the stack name
- **Glue Job Name**: Generated using the stack name
- **Region**: `AWS::Region` pseudo parameter (inferred from the stack's deployment region)
- **CIDR Blocks**: Not applicable for this template
- **Instance Types**: Not applicable (Glue is serverless)

## 4. Dynamic Resource Fetching

For this specific template, dynamic fetching isn't strictly necessary since:
- We're not using EC2 instances
- The Glue job uses a managed runtime

However, I'll implement best practices by:
- Using `AWS::Region` for region-specific resources
- Using stack name prefixes for resource naming to ensure uniqueness

## 5. Architecture Plan

The CloudFormation template will be structured as follows:

### Resources to be Created:

1. **S3 Bucket** - For storing Glue scripts and Spark logs
   - Properties: Bucket name, versioning enabled for script backup
   - Policy: Public access blocked, versioning enabled

2. **IAM Role** - For AWS Glue service permissions
   - Properties: Role name, trusted entity (glue.amazonaws.com)
   - Managed Policies: `AWSGlueServiceRole`, `AmazonS3FullAccess` (or more restrictive S3 policies)
   - Custom Policy: Inline policy allowing Glue to access the specific S3 bucket

3. **AWS Glue Job** - The ETL job itself
   - Properties: Job name, IAM role ARN, script location (S3 path), Glue version 4.0, Python language
   - Command: Python script location in S3
   - Execution properties: Timeout, max capacity

### Security Considerations:

- The IAM role will have the minimum necessary permissions
- S3 bucket will have versioning enabled for script recovery
- Public access will be blocked on the S3 bucket
- The Glue job will only have access to its designated S3 bucket

### Template Structure:

The template will follow this logical flow:
1. Define Parameters (with safe defaults)
2. Create S3 Bucket
3. Create IAM Role with policies
4. Create Glue Job referencing the role and S3 scripts

### Key Implementation Details:

- **S3 Bucket Policy**: Will use `AWS::S3::BucketPolicy` instead of the deprecated `AccessControl` property
- **IAM Role**: Will use the correct AWS Glue service principal
- **Glue Job**: Will specify Glue version 4.0 and Python as the language
- **Script Location**: Will reference the S3 path `s3://bucket-name/gue_scripts/sample_convert.py`

This architecture ensures a secure, scalable, and maintainable Glue ETL infrastructure that follows AWS best practices and will pass cfn-lint validation.
