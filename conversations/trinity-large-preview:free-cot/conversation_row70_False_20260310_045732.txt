# Multi-AZ VPC Architecture Deployment Plan

## 1. Key AWS Services Required

Based on the business need, the following AWS services are required:

- **VPC** (AWS::EC2::VPC) - The core networking construct
- **Internet Gateway** (AWS::EC2::InternetGateway) - Provides internet connectivity
- **VPC Gateway Attachment** (AWS::EC2::VPCGatewayAttachment) - Attaches the IGW to the VPC
- **Subnets** (AWS::EC2::Subnet) - Four subnets total: 2 public and 2 private across multiple AZs
- **Route Tables** (AWS::EC2::RouteTable) - One for public subnets and one for private subnets
- **Routes** (AWS::EC2::Route) - Routes for internet traffic
- **Subnet Route Table Associations** (AWS::EC2::SubnetRouteTableAssociation) - Associates subnets with route tables
- **Elastic IP** (AWS::EC2::EIP) - Optional, for NAT Gateway (if needed)
- **NAT Gateway** (AWS::EC2::NatGateway) - Optional, for private subnet internet access

## 2. Properties and Dependencies

The deployment order and dependencies are critical:

1. **VPC** must be created first with appropriate CIDR blocks
2. **Internet Gateway** must be created and attached to the VPC before any routing can occur
3. **Subnets** must be created with proper CIDR blocks and Availability Zone assignments
4. **Route Tables** must be created after subnets exist
5. **Routes** must be created after Route Tables exist
6. **Subnet Route Table Associations** must be created last to connect everything

Key properties to consider:
- VPC requires a CIDR block (e.g., 10.0.0.0/16)
- Subnets require unique CIDR blocks within the VPC range
- Route Tables need routes pointing to the Internet Gateway
- All resources must be tagged appropriately

## 3. Safe AWS Defaults

Since specific properties weren't provided, I'll use these safe defaults:

- **VPC CIDR**: `10.0.0.0/16` (standard private range)
- **Public Subnet CIDRs**: `10.0.1.0/24` and `10.0.2.0/24`
- **Private Subnet CIDRs**: `10.0.3.0/24` and `10.0.4.0/24`
- **Availability Zones**: Dynamically fetched using `Fn::GetAZs` and `Fn::Select`
- **Tags**: Standard Name tags for all resources

## 4. Dynamic Resource Fetching

For this VPC template, dynamic fetching isn't strictly necessary since we're not provisioning instances. However, I'll plan for best practices:

- **Availability Zones**: Use `Fn::GetAZs` to get all AZs in the region, then `Fn::Select` to pick the first two
- **AMIs**: Not needed for this VPC-only template, but if instances were added later, I'd use SSM Parameter Store (e.g., `/aws/service/ami-amazon-linux-latest/amzn2-ami-hvn-x86_64-gp2`)

## Architecture Plan

The template will create:

1. **VPC**: 10.0.0.0/16 with DNS support enabled
2. **Internet Gateway**: Attached to the VPC
3. **Public Subnets**: 
   - Subnet 1: 10.0.1.0/24 in AZ1
   - Subnet 2: 10.0.2.0/24 in AZ2
4. **Private Subnets**:
   - Subnet 3: 10.0.3.0/24 in AZ1
   - Subnet 4: 10.0.4.0/24 in AZ2
5. **Public Route Table**: Routes 0.0.0.0/0 to Internet Gateway
6. **Private Route Table**: No default route (can be extended later)
7. **Associations**: All subnets properly associated with their respective route tables

The public subnets will have direct internet access through the route table, while private subnets will be isolated by default. This architecture supports high availability by spanning two AZs and provides the foundation for deploying EC2 instances, RDS databases, or other AWS services with proper network isolation.
