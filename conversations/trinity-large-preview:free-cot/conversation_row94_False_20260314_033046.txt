# Detailed Architectural Deployment Plan

## 1. Key AWS Services Required

Based on the business requirements, the following AWS services are essential:

- **EC2**: To host the Linux 2 instance
- **IAM**: For creating the instance profile and role with SSM permissions
- **SSM (Systems Manager)**: For automation documents, command execution, and associations
- **S3**: For centralized logging bucket
- **CloudWatch**: For monitoring and logging integration
- **VPC**: To provide network isolation (default VPC will be used)
- **Security Groups**: To control network access to the EC2 instance

## 2. Properties and Dependencies

The deployment follows this logical sequence:

1. **IAM Role & Instance Profile**: Created first to grant SSM permissions
2. **S3 Bucket**: Created for centralized logging before EC2 instance
3. **EC2 Instance**: Deployed with the IAM role and SSM agent
4. **SSM Automation Document**: Created to orchestrate the multi-step process
5. **SSM Association**: Links the automation document to the EC2 instance

Critical dependencies:
- IAM role must exist before EC2 instance creation
- S3 bucket must exist before EC2 instance creation (for logging configuration)
- SSM Automation document must exist before association

## 3. Safe AWS Defaults

Since specific properties weren't provided, I'll use these safe defaults:

- **Instance Type**: `t3.micro` (free tier eligible, suitable for testing)
- **VPC**: Default VPC in the region
- **Subnets**: First two subnets from default VPC
- **CIDR Blocks**: Default VPC CIDR (172.31.0.0/16)
- **KeyPair**: Parameter with default 'default-key' (user must have this key imported)
- **AMI**: Latest Amazon Linux 2 using SSM Parameter Store
- **Availability Zones**: Dynamically fetched using `Fn::GetAZs`

## 4. Dynamic Resource Fetching Strategy

**AMI Resolution**: Use SSM Parameter Store to fetch the latest Amazon Linux 2 AMI:
```
resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2
```

**Availability Zones**: Use `Fn::GetAZs` to dynamically fetch available AZs in the region, then `Fn::Select` to pick the first two:
```
Fn::Select: [0, Fn::GetAZs: ""]
```

## 5. Architecture Overview

The solution implements a secure, automated deployment with these key components:

**IAM Configuration**: A least-privilege IAM role with policies for SSM (AmazonSSMManagedInstanceCore), S3 (for logging bucket access), and CloudWatch. This ensures the EC2 instance can only perform necessary operations.

**EC2 Instance**: Deployed with the latest Amazon Linux 2 AMI, configured with the IAM role, and placed in the default VPC with appropriate security groups allowing HTTP (port 80) and SSH (port 22) access.

**SSM Automation Document**: A custom document defining a three-step process:
1. **OS Updates**: Runs `sudo yum update -y` with reboot handling
2. **Nginx Installation**: Installs Nginx web server
3. **Validation**: Uses `curl` to verify the web server responds correctly

**Centralized Logging**: An S3 bucket configured for SSM logs with appropriate bucket policy, ensuring auditability of all automation operations.

**Security Groups**: Configured to allow HTTP traffic from anywhere (0.0.0.0/0) and SSH from the user's IP (using a parameter with validation).

## 6. Implementation Strategy

The CloudFormation template will be structured as follows:

1. **Parameters**: For KeyPair name, SSH CIDR block, and S3 bucket name
2. **Resources**:
   - IAM Role and Instance Profile
   - S3 Bucket and Policy
   - EC2 Instance with UserData for initial setup
   - SSM Automation Document
   - SSM Association
   - Security Groups

3. **Outputs**: EC2 instance ID, public IP, and S3 bucket name for easy access

The template adheres to all AWS best practices, including using dynamic references for AMIs, avoiding hardcoded values, implementing proper IAM policies, and ensuring all resources have appropriate deletion and update policies for safe cleanup.

This architecture provides a production-ready, automated deployment that ensures the EC2 instance is always updated, Nginx is properly installed, and the entire configuration can be validated through SSM automation, with all operations logged for audit purposes.
