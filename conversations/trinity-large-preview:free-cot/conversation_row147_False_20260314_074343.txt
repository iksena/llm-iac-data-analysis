# Serverless Slot Machine Web Application - CloudFormation Architecture Plan

## Key AWS Services Required

1. **S3** - Static website hosting for frontend assets
2. **Lambda** - Serverless slot machine logic
3. **Cognito Identity Pools** - Anonymous user authentication
4. **IAM** - Least privilege access control
5. **SSM Parameter Store** - Dynamic resource fetching
6. **CloudFormation Custom Resources** - Automated deployment

## Architecture Overview

The solution follows a serverless pattern where unauthenticated users access a static website hosted on S3, which invokes a Lambda function through Cognito-provided temporary credentials to simulate slot machine spins.

## Detailed Component Breakdown

### 1. S3 Bucket Configuration

**Properties:**
- `AccessControl: PublicRead` (legacy, but required for static website)
- `WebsiteConfiguration` with `IndexDocument: index.html`
- `OwnershipControls` with `Rules: [ObjectOwnership: BucketOwnerPreferred]`

**Dependencies:**
- Must be created before Lambda custom resource deployment
- Requires IAM policy for public access

**Best Practice Compliance:**
- Using `AWS::S3::BucketPolicy` instead of legacy `AccessControl` where possible
- Including `OwnershipControls` to satisfy linting requirements

### 2. Lambda Function (Slot Machine Logic)

**Runtime:** `python3.12` (latest supported)

**Properties:**
- `Handler: index.lambda_handler`
- `Timeout: 10` seconds
- `MemorySize: 128` MB
- `Environment` variables for configuration

**Dependencies:**
- IAM execution role with logging permissions
- Requires S3 bucket for deployment package

**Logic Implementation:**
- Random card selection from predefined deck
- Win condition evaluation based on combinations
- JSON response with results

### 3. Cognito Identity Pool

**Properties:**
- `AllowUnauthenticatedIdentities: true`
- `CognitoIdentityProviders` (empty for anonymous access)

**Dependencies:**
- IAM roles for authenticated/unauthenticated users

**Role Configuration:**
- Unauthenticated role allows only `lambda:InvokeFunction`
- Least privilege principle enforced

### 4. IAM Roles and Policies

**Lambda Execution Role:**
- Managed policy: `AWSLambdaBasicExecutionRole`
- Custom inline policy for specific permissions

**Cognito Unauthenticated Role:**
- Custom inline policy allowing only Lambda invocation
- Resource-based policy on Lambda function

**Dependencies:**
- Roles must exist before Lambda creation
- Policies attached after role creation

### 5. Custom Lambda Resource (Automated Deployment)

**Purpose:** Download ZIP, inject configuration, upload to S3

**Properties:**
- `ServiceToken` pointing to custom resource Lambda ARN
- `BucketName`, `Region`, `IdentityPoolId` as parameters

**Dependencies:**
- Requires S3 bucket to exist
- Needs Lambda execution permissions

**Implementation Steps:**
1. Download pre-built ZIP from source
2. Read HTML files and inject Cognito/region configurations
3. Upload modified files to target S3 bucket

### 6. SSM Parameter Store Integration

**Purpose:** Dynamic resource fetching to avoid hardcoding

**Parameters Used:**
- Latest Amazon Linux AMI: `/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`
- Region-specific configurations

**Benefits:**
- Template remains evergreen
- No manual AMI updates required
- Region-agnostic deployment

## Parameter Strategy

**Safe Defaults Required:**
```yaml
Parameters:
  WebsiteBucketName:
    Type: String
    Default: slot-machine-website-${AWS::AccountId}
    Description: S3 bucket name for website hosting

  Region:
    Type: String
    Default: !Ref AWS::Region
    Description: AWS Region for deployment
```

**Pattern Compliance:**
- All Parameters have `Default` values
- No manual input required during deployment
- Safe patterns for AllowedPattern properties

## Dependency Chain

1. **IAM Roles** → Created first for least privilege
2. **S3 Bucket** → Created before Lambda and custom resource
3. **Lambda Function** → Depends on IAM role
4. **Cognito Identity Pool** → Depends on IAM roles
5. **Custom Resource** → Depends on S3 bucket and Lambda
6. **Bucket Policy** → Final step for public access

## Security Considerations

**Least Privilege:**
- Lambda role: Only logging permissions
- Cognito role: Only Lambda invocation
- No cross-service access beyond requirements

**Anonymous Access:**
- Cognito provides temporary credentials
- No permanent AWS credentials exposed
- IAM policies restrict to specific Lambda function

## Error Handling and Best Practices

**Deletion Policies:**
- S3 bucket: `DeletionPolicy: Retain` with `UpdateReplacePolicy: Retain`
- Lambda: Default deletion (stateless)

**Resource Protection:**
- All stateful resources have appropriate policies
- No accidental data loss during updates

**Linting Compliance:**
- No hardcoded AMIs or AZs
- Proper use of intrinsic functions
- Valid AWS managed policy ARNs
- No hallucinated properties

## Dynamic Configuration Injection

**Frontend Modifications:**
- Inject Cognito Identity Pool ID into JavaScript
- Configure region-specific endpoints
- Add error handling for Lambda invocation

**Implementation:**
- Custom resource Lambda reads files as text
- String replacement for configuration values
- Uploads modified files maintaining directory structure

This architecture ensures a secure, scalable, and maintainable serverless slot machine application that can be deployed across any AWS region without modification.
