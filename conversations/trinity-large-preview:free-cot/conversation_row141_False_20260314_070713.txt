# Image Recognition Pipeline Architecture Plan

## Key AWS Services Required

1. **Amazon S3** - Storage bucket for image uploads
2. **AWS Step Functions** - Orchestrates the workflow
3. **Amazon Rekognition** - Image analysis for face detection
4. **Amazon SNS** - SMS notifications
5. **AWS Lambda** - Custom resource for S3 bucket cleanup
6. **AWS IAM** - Service roles and permissions
7. **Amazon EventBridge** - Triggers Step Functions on S3 events
8. **AWS Systems Manager (SSM)** - Dynamic AMI fetching (if needed)

## Architecture Overview

The solution follows an event-driven architecture where:
1. User uploads image to S3 bucket
2. EventBridge detects the upload and triggers Step Functions
3. Step Functions validates file type, calls Rekognition, and sends SMS if faces detected
4. Lambda cleanup resource ensures smooth stack deletion

## Detailed Component Breakdown

### 1. S3 Bucket Configuration
- **Properties**: Versioning enabled, server access logging (optional)
- **EventBridge Integration**: Create event rule with S3 pattern for `ObjectCreated:*` events
- **Security**: Bucket policy restricting access to authorized roles only

### 2. IAM Roles and Policies

**EventBridge Role (InvokeStepFunctionsRole)**:
- Trust policy: EventBridge service
- Permissions: `states:StartExecution` on the state machine ARN
- Least privilege: Only allows triggering the specific state machine

**Step Functions Role (StateMachineRole)**:
- Trust policy: Step Functions service
- Permissions:
  - S3: `s3:GetObject` on the bucket
  - Rekognition: `rekognition:DetectLabels`
  - SNS: `sns:Publish`
  - CloudWatch Logs: `logs:CreateLogGroup`, `logs:CreateLogStream`, `logs:PutLogEvents`

### 3. Step Functions State Machine

**States**:
1. **ValidateImage** (Choice): Check file extension (.jpg, .jpeg, .png)
2. **GetS3Object** (Task): Retrieve image from S3
3. **DetectFaces** (Task): Call Rekognition DetectLabels API
4. **CheckLabels** (Choice): Verify "Human" or "Person" labels exist
5. **SendSMS** (Task): Publish to SNS topic if faces detected
6. **Done** (Succeed): Workflow completion

**Error Handling**: Catch all failures and log to CloudWatch

### 4. SNS Topic Configuration
- **Properties**: Delivery policy for SMS throttling
- **Subscription**: Direct SMS to +14071113333
- **Encryption**: Optional KMS key for message protection

### 5. Lambda Cleanup Resource

**Function Properties**:
- Runtime: `nodejs20.x` (latest supported)
- Handler: `index.handler`
- Memory: 128 MB
- Timeout: 30 seconds

**Function Logic**:
```javascript
// Empty bucket by deleting all objects
// Use S3.listObjectsV2() and S3.deleteObjects()
// Return success when complete
```

**IAM Permissions**:
- S3: `s3:ListBucket`, `s3:DeleteObject` on the bucket

### 6. EventBridge Rule

**Properties**:
- **EventPattern**: S3 `ObjectCreated:*` events for the target bucket
- **State**: Enabled
- **Targets**: Step Functions state machine with input transformer

## Dependencies and Execution Order

1. IAM roles must be created before Step Functions
2. SNS topic must exist before state machine creation
3. S3 bucket must exist before EventBridge rule creation
4. Lambda function must be deployed before custom resource invocation

## Safe Defaults (Based on Best Practices)

**Missing Properties**:
- S3 Bucket: No default name provided - will use `Fn::Join` with stack name
- Step Functions: Default timeout of 1 hour
- Lambda: Default memory of 128 MB, timeout of 30 seconds
- SNS: No default topic name - will use stack name

**Dynamic Resource Fetching**:
- No AMIs required for this architecture
- AZs: Use `Fn::GetAZs` and `Fn::Select` for any future VPC needs

## Security Considerations

1. **Least Privilege**: Each role has only necessary permissions
2. **Encryption**: S3 bucket encryption at rest (SSE-S3)
3. **Logging**: CloudWatch Logs for all services
4. **Input Validation**: State machine validates file types before processing
5. **Error Handling**: Comprehensive error catching and logging

## Stack Deletion Strategy

1. Lambda cleanup resource runs first, emptying the S3 bucket
2. EventBridge rule and Step Functions are deleted
3. SNS topic is deleted
4. S3 bucket is deleted (now empty)

This architecture ensures:
- Automated processing without manual intervention
- Secure access with minimal permissions
- Reliable error handling and logging
- Smooth stack lifecycle management
- Compliance with AWS best practices and linting rules
