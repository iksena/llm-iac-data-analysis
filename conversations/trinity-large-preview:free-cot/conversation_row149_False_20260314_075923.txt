# Detailed Architectural Deployment Plan for EFS with EC2 Integration

## 1. Key AWS Services Required

Based on the business requirements, the following AWS services are essential:

- **Amazon VPC** - To create the network foundation with public subnets across two AZs
- **Amazon EFS** - For the shared, encrypted file system with mount targets
- **Amazon EC2** - To provision two instances in separate AZs with EFS access
- **AWS IAM** - To create roles and policies for EFS/EC2 access and SSM permissions
- **AWS Systems Manager (SSM)** - For secure instance management and AMI resolution
- **Amazon CloudWatch** - To monitor EFS performance metrics with alarms
- **AWS Key Management Service (KMS)** - For EFS encryption at rest
- **AWS Security Groups** - To control network traffic to EFS and EC2 instances

## 2. Properties and Dependencies

### Critical Dependencies:
1. **VPC Creation** → **Subnets** → **Internet Gateway** → **Route Tables** → **EFS Mount Targets** → **EC2 Instances**
2. **IAM Role Creation** → **Instance Profile Attachment** → **EC2 Launch**
3. **EFS Creation** → **Mount Target Creation** (requires VPC, subnets, security groups)
4. **EC2 Creation** → **User Data Execution** (requires EFS DNS name)

### Property Dependencies:
- EFS Mount Targets require VPC, Subnet, and Security Group IDs
- EC2 Instances require IAM Instance Profile, Key Pair, and Security Group IDs
- CloudWatch Alarms require EFS File System ID
- Security Groups require VPC ID

## 3. Safe AWS Defaults (for missing properties)

### Network Configuration:
- **VPC CIDR**: `10.0.0.0/16` (standard private range)
- **Public Subnet CIDRs**: 
  - AZ1: `10.0.1.0/24`
  - AZ2: `10.0.2.0/24`
- **Availability Zones**: Dynamically selected using `Fn::GetAZs` and `Fn::Select`

### EC2 Configuration:
- **Instance Type**: `t3.micro` (free tier eligible, suitable for testing)
- **AMI**: Latest Amazon Linux 2 using SSM Parameter Store
- **Key Pair**: Parameter with default value 'default-key' (user must create or override)
- **IAM Role**: Custom role with EFS and SSM permissions

### EFS Configuration:
- **Performance Mode**: `generalPurpose` (default, cost-effective)
- **Throughput Mode**: `bursting` (default)
- **Encrypted**: `true` with default KMS key
- **Lifecycle Policy**: `AFTER_30_DAYS` (cost optimization)

### CloudWatch Alarms:
- **Metric Namespaces**: `AWS/EFS`
- **Standard Alarms**: 
  - `PercentIOLimit` > 90% for 1 datapoint
  - `BurstCreditBalance` < 1GB for 1 datapoint
  - `PermittedThroughput` > 100MB/s for 1 datapoint

## 4. Dynamic Resource Fetching Strategy

### AMI Resolution:
```yaml
!Sub 'arn:${AWS::Partition}:ssm:${AWS::Region}:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2'
```

### AZ Selection:
```yaml
!Select [0, !GetAZs '']
!Select [1, !GetAZs '']
```

### EFS DNS Name:
Use `!Ref` on the EFS resource to get the DNS name for mounting in User Data.

## 5. Security Architecture

### IAM Role Design:
- **Policy 1**: EFS access (mount, read/write)
- **Policy 2**: SSM full access (for management)
- **Instance Profile**: Attach both policies

### Security Groups:
- **EFS Security Group**: Allow NFS (port 2049) from EC2 instances only
- **EC2 Security Group**: Allow SSH (port 22) from anywhere (for testing), HTTP/HTTPS if needed

### Encryption:
- EFS encrypted using default KMS key
- No hardcoded secrets; use SSM Parameter Store for AMI IDs

## 6. Bootstrapping Strategy

### EC2 User Data:
1. Install EFS client utilities
2. Create `/home` directory if not exists
3. Mount EFS using the resolved DNS name
4. Add mount to `/etc/fstab` for persistence
5. Verify mount success

### Mount Command:
```bash
mount -t efs ${EFS_ID}:/ /home
```

## 7. CloudWatch Monitoring Strategy

### Alarm Configuration:
- **PercentIOLimit**: Alert when EFS is approaching I/O capacity
- **BurstCreditBalance**: Warn when burst credits are low
- **PermittedThroughput**: Alert on unusually high throughput

### Alarm Actions:
- SNS topic for notifications (create with email subscription)
- Consider auto-scaling policies if applicable

## 8. Template Structure Plan

### Logical Sections:
1. **Parameters** - With safe defaults
2. **Mappings** - For any region-specific configurations
3. **Resources** - All AWS resources in dependency order
4. **Outputs** - EFS ID, EC2 public IPs, CloudWatch alarm ARNs

### Resource Dependencies:
```
VPC → Subnets → Internet Gateway → Route Tables
VPC → Security Groups
VPC + Subnets + Security Groups → EFS Mount Targets
IAM Role → Instance Profile
EFS + Instance Profile + Security Groups → EC2 Instances
EFS ID → CloudWatch Alarms
```

## 9. Compliance with Best Practices

### Linting Compliance:
- All Parameters have `Default` values
- No hardcoded AMIs or AZs
- No `AccessControl` on S3 (not applicable here)
- All stateful resources have `DeletionPolicy` and `UpdateReplacePolicy`
- No `Fn::Sub` without variables
- No hallucinated properties

### Security Compliance:
- IAM roles use least privilege principle
- EFS encrypted at rest
- No hardcoded secrets
- Security groups restrict network access

This architectural plan ensures a production-ready, automated deployment that meets all specified requirements while adhering to AWS best practices and CloudFormation linting rules.
