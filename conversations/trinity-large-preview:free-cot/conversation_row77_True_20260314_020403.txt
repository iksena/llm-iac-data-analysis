# DynamoDB Table Deployment Plan for Supporter Product Data

## Key AWS Services Required
1. **AWS::DynamoDB::Table** - Core service for creating environment-specific DynamoDB tables
2. **AWS::ApplicationAutoScaling::ScalableTarget** - For auto-scaling configuration in PROD
3. **AWS::ApplicationAutoScaling::ScalingPolicy** - Target-tracking policy for write capacity scaling
4. **AWS::SSM::Parameter** - For dynamic resource naming and configuration
5. **AWS::CloudFormation::WaitCondition** - Optional, for ensuring table creation completes before proceeding

## Architecture Overview

The template will create environment-specific DynamoDB tables with distinct configurations for PROD and CODE environments, using a single Stage parameter to control conditional resource provisioning.

## Detailed Implementation Plan

### 1. Parameters Definition
```yaml
Parameters:
  Stage:
    Type: String
    Default: CODE
    AllowedValues:
      - PROD
      - CODE
    Description: Environment stage (PROD or CODE)
```

### 2. Resource Naming Strategy
- Use dynamic naming with `${Stage}` suffix: `SupporterProductData-${Stage}`
- This ensures unique table names across environments
- Example: `SupporterProductData-PROD` and `SupporterProductData-CODE`

### 3. Conditional Resource Provisioning
Use `Condition` blocks to control resource creation based on the Stage parameter:

```yaml
Conditions:
  IsProduction: !Equals [!Ref Stage, "PROD"]
  IsCode: !Equals [!Ref Stage, "CODE"]
```

### 4. PROD Environment Configuration

**Table Properties:**
- BillingMode: PROVISIONED
- ProvisionedThroughput:
  - ReadCapacityUnits: 20
  - WriteCapacityUnits: 5
- Auto-scaling for write capacity:
  - ScalableTarget with MinCapacity: 5, MaxCapacity: 500
  - ScalingPolicy with TargetTrackingConfiguration (target utilization 70%)
- SSE enabled (default KMS key)
- TTL attribute: expiryDate
- Tags: Include backup-related tags

**Auto-scaling Configuration:**
```yaml
ScalableTarget:
  MinCapacity: 5
  MaxCapacity: 500
  ResourceId: !Sub "table/${TableName}"
  ScalableDimension: "dynamodb:table:WriteCapacityUnits"
  ServiceNamespace: "dynamodb"
```

### 5. CODE Environment Configuration

**Table Properties:**
- BillingMode: PAY_PER_REQUEST
- All other properties identical to PROD (SSE, TTL, tags)
- No auto-scaling configuration needed

### 6. Common Properties (Both Environments)

**Encryption:**
- SSESpecification with Enabled: true
- Uses default AWS-managed KMS key

**TTL Configuration:**
```yaml
TimeToLiveSpecification:
  AttributeName: expiryDate
  Enabled: true
```

**Backup Tags:**
```yaml
Tags:
  - Key: Backup
    Value: "true"
```

### 7. Security & Best Practices

**IAM Permissions:**
- Ensure template includes necessary IAM permissions for auto-scaling
- Use least-privilege principle for any additional roles

**Resource Protection:**
- Set DeletionPolicy: Retain on DynamoDB tables
- Set UpdateReplacePolicy: Retain to prevent accidental data loss

### 8. SSM Integration for Dynamic Resources

While not strictly necessary for DynamoDB, plan to use SSM Parameter Store for:
- Fetching latest AMI IDs if EC2 resources are added later
- Storing configuration parameters
- Dynamic naming conventions

### 9. Template Structure

```yaml
Resources:
  DynamoDBTable:
    Type: AWS::DynamoDB::Table
    Condition: IsProduction
    Properties: # PROD configuration

  DynamoDBTableCode:
    Type: AWS::DynamoDB::Table
    Condition: IsCode
    Properties: # CODE configuration

  ScalableTarget:
    Type: AWS::ApplicationAutoScaling::ScalableTarget
    Condition: IsProduction
    Properties: # Auto-scaling configuration

  ScalingPolicy:
    Type: AWS::ApplicationAutoScaling::ScalingPolicy
    Condition: IsProduction
    Properties: # Scaling policy
```

### 10. Validation & Testing Strategy

**Template Validation:**
- Use cfn-lint to validate syntax and best practices
- Test with both PROD and CODE parameter values
- Verify conditional resource creation

**Functional Testing:**
- Deploy to CODE environment first
- Test table creation, TTL functionality, and encryption
- Deploy to PROD with auto-scaling enabled
- Monitor scaling behavior and performance

### 11. Cost Optimization

**PROD:**
- Provisioned capacity with auto-scaling prevents over-provisioning
- Target tracking ensures optimal utilization

**CODE:**
- PAY_PER_REQUEST eliminates capacity planning overhead
- Ideal for unpredictable or low-traffic environments

### 12. Monitoring & Maintenance

**CloudWatch Metrics:**
- Monitor read/write capacity utilization
- Track throttling events
- Monitor backup success rates

**Backup Strategy:**
- Leverage DynamoDB point-in-time recovery (PITR)
- Schedule automated backups based on tags
- Test restore procedures regularly

This architecture ensures environment-specific optimization while maintaining consistent functionality across PROD and CODE environments, with proper cost controls and scalability for production workloads.
