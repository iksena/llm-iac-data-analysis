Based on the business need, the following AWS services are required:

- **Amazon DynamoDB** - To create the source table with streams enabled
- **Amazon EventBridge** - To receive and route filtered events
- **EventBridge Pipes** - To connect DynamoDB streams to EventBridge with filtering
- **AWS IAM** - To create necessary service roles and policies
- **AWS Systems Manager (SSM) Parameter Store** - For dynamic resource references (though not directly used in this template)

## 2. Properties and Dependencies

### Critical Dependencies:
1. **IAM Roles must be created before Pipes** - Pipes reference IAM roles for permissions
2. **DynamoDB Table must exist before Pipe creation** - Pipe needs the table ARN and stream ARN
3. **EventBridge Bus must exist before Pipe creation** - Pipe needs the bus ARN

### Required Properties:
- DynamoDB Table: StreamEnabled=true, StreamViewType=NEW_AND_OLD_IMAGES
- EventBridge Pipe: Source=DYNAMODB, Destination=EVENTBRIDGE, with FilterCriteria
- IAM Roles: Appropriate permissions for DynamoDB and EventBridge access

## 3. Safe AWS Defaults

Since specific properties are missing, I'll use these safe defaults:

- **DynamoDB Table**: Default name = "IntegrationTable", default read/write capacity = 5 (on-demand not used to avoid cost surprises)
- **EventBridge Bus**: Default name = "default" (the default AWS EventBridge bus)
- **IAM Role Names**: Use descriptive names with stack ID to avoid conflicts
- **Region**: Use `AWS::Region` pseudo-parameter for region-specific resources

## 4. Dynamic Resource Fetching Strategy

For this specific template, dynamic fetching is not required for AMIs or AZs since we're not provisioning EC2 instances or VPCs. However, I'll use:

- **Region-aware references**: Using `AWS::Region` for any region-specific configurations
- **Stack-aware naming**: Using `AWS::StackName` for resource naming to ensure uniqueness

## 5. Architecture Design

The solution will consist of:

1. **DynamoDB Table** with streams enabled
2. **IAM Role** for the Pipe to access DynamoDB and EventBridge
3. **EventBridge Pipe** that:
   - Sources from DynamoDB stream
   - Filters events where 'avatarUrl' ends with '.png'
   - Routes to EventBridge bus with specified metadata
4. **IAM Policy** attached to the role

## 6. Security Considerations

- **Least Privilege**: IAM role will have only necessary permissions
- **No hardcoded secrets**: No secrets required for this integration
- **Resource naming**: Stack-aware naming to avoid conflicts

## 7. Error Handling and Validation

- **Filter Criteria**: Will use JSON pattern matching for the '.png' filter
- **Event Metadata**: Will set DetailType="UserAvatarUpdated" and Source="com.company.avatar"
- **Error handling**: CloudFormation will handle rollback on failure

## 8. Template Structure Plan

The CloudFormation template will be structured as:

1. **Parameters** (with safe defaults)
2. **Resources**:
   - DynamoDB Table
   - IAM Role and Policy
   - EventBridge Pipe
3. **Outputs** (optional: table ARN, pipe ARN for reference)

## 9. Specific Implementation Details

### Filter Criteria Design:
```json
{
  "detail-type": ["DynamoDB Stream Record"],
  "detail": {
    "eventSource": ["aws:dynamodb"],
    "eventName": ["INSERT", "MODIFY"],
    "detail": {
      "NewImage": {
        "avatarUrl": [{"endsWith": ".png"}]
      }
    }
  }
}
```

### Event Metadata:
- **DetailType**: "UserAvatarUpdated"
- **Source**: "com.company.avatar"

### IAM Permissions:
- **DynamoDB**: `dynamodb:DescribeStream`, `dynamodb:GetRecords`, `dynamodb:GetShardIterator`
- **EventBridge**: `events:PutEvents`

This architecture ensures a robust, event-driven integration between DynamoDB and EventBridge with proper filtering and metadata enrichment, following AWS best practices and CloudFormation linting rules.
