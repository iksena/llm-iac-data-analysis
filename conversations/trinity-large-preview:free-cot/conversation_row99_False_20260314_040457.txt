# Detailed Architectural Deployment Plan for Secure Multi-AZ VPC

## Key AWS Services Required

1. **VPC** - Core networking construct
2. **Internet Gateway (IGW)** - Public internet access
3. **NAT Gateway** - Private subnet internet access
4. **Elastic IP (EIP)** - Static IP for NAT Gateway
5. **Subnets** - Public and private subnets across AZs
6. **Route Tables** - Separate routing for public/private traffic
7. **Route Table Associations** - Linking subnets to route tables
8. **AWS Systems Manager (SSM) Parameter Store** - Dynamic AMI fetching

## Properties and Dependencies

### Critical Dependencies
1. VPC must be created before IGW, subnets, and route tables
2. IGW must be attached to VPC before creating EIP/NAT Gateway
3. EIP must be created before NAT Gateway
4. Route tables must be created before associations
5. Subnets must exist before route table associations

### Property Dependencies
- NAT Gateway requires an EIP and a public subnet
- Route table associations require both route table and subnet to exist
- Security groups (if added later) require VPC to exist

## Safe AWS Defaults

### VPC Configuration
- **CIDR Block**: `10.0.0.0/16` (standard private range)
- **Region**: `us-east-1` (as specified)
- **AZ Selection**: Dynamic using `Fn::GetAZs` and `Fn::Select`

### Subnet Configuration
- **Public Subnets**: 
  - AZ1: `10.0.1.0/24`
  - AZ2: `10.0.2.0/24`
- **Private Subnets**:
  - AZ1: `10.0.3.0/24`
  - AZ2: `10.0.4.0/24`

### Route Table Configuration
- **Public Route Table**: Direct internet access via IGW
- **Private Route Table**: Internet access via NAT Gateway

## Dynamic Resource Fetching Strategy

### AMI Selection
Use SSM Parameter Store to fetch the latest Amazon Linux 2 AMI:
```yaml
Fn::ImportValue: !Sub "resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2"
```

### AZ Selection
Use dynamic AZ fetching to ensure cross-region compatibility:
```yaml
!Select [0, !GetAZs !Ref "AWS::Region"]
!Select [1, !GetAZs !Ref "AWS::Region"]
```

## Security Best Practices Implementation

### Network Security
1. **Least Privilege Routing**: Public subnets have direct internet access, private subnets route through NAT
2. **No Direct Internet Access**: Private resources cannot be accessed directly from internet
3. **Controlled Outbound**: Private resources can initiate outbound connections via NAT

### Tagging Strategy
Implement comprehensive tagging for governance:
- **Environment**: Production/Development
- **Owner**: Team/Department
- **Project**: Specific project identifier
- **CostCenter**: Financial tracking code

## Architecture Flow

1. **VPC Creation** → Establishes network boundary
2. **IGW Creation & Attachment** → Enables internet connectivity
3. **Subnet Creation** → Distributes resources across AZs
4. **EIP Creation** → Provides static IP for NAT
5. **NAT Gateway Creation** → Enables private internet access
6. **Route Table Creation** → Defines traffic flow rules
7. **Route Table Associations** → Links subnets to routing policies

## Compliance with Best Practices

### Parameter Handling
- All parameters will have safe defaults
- Allowed patterns will match default values
- No required manual input during deployment

### Resource Protection
- No stateful resources requiring DeletionPolicy/UpdateReplacePolicy
- All resources properly scoped and secured

### Modern AWS Practices
- No hardcoded values
- Dynamic resource fetching
- Latest AWS service configurations
- Proper IAM and security group configurations

This architecture ensures high availability, security, and compliance with AWS Well-Architected Framework while maintaining deployment automation and operational excellence.
