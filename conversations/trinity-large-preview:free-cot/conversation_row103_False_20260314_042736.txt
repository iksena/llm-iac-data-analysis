6. S3 Bucket must exist before Bucket Policy attachment

### Property Dependencies:
- Subnet requires VPC ID
- Route Table requires VPC ID and Internet Gateway attachment
- EC2 Instance requires IAM Instance Profile, Security Group, and Subnet

- **S3 Bucket Name**: Dynamically generated using `!Sub` with stack name to ensure uniqueness
- **Region**: `us-east-1` (default region, can be parameterized)

- Use SSM Parameter Store to fetch the latest Amazon Linux 2 AMI: `/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`

### Availability Zones:
- Use `Fn::GetAZs` to retrieve available AZs for the region

- This makes the template portable across regions

## 5. Security Architecture

### Network Security:
- Public subnet with Internet Gateway for web accessibility
- Security Group allowing only HTTP (port 80) from anywhere
- No SSH access exposed (installation via user data)

### Instance Security:
- IAM Role with least-privilege access to CloudWatch and SSM
- No hardcoded credentials in user data
- Server-side encryption enabled on S3 bucket

### Data Protection:
- S3 bucket with default encryption (AES-256)
- Public access blocked via ACL and bucket policy
- IAM policies restricting log access to authorized entities

## 6. Installation and Configuration Strategy

### EC2 User Data:
- Install Apache and PHP using `yum` package manager
- Create a dynamic PHP info page
- Configure AWS CLI for CloudWatch Logs agent installation
- Start Apache service and enable on boot

### CloudWatch Integration:
- Install CloudWatch Logs agent via user data
- Configure agent to send Apache logs to S3 bucket
- Use IAM role for authentication (no hardcoded credentials)

## 7. Template Structure Plan

The CloudFormation template will follow this logical structure:

1. **Parameters** (with safe defaults)
2. **Resources** (in dependency order)
3. **Outputs** (for easy access to important IDs/ARNs)

### Resource Creation Order:
1. VPC and Internet Gateway
2. Public Subnet
3. Route Table and Route
4. Security Group
5. IAM Role and Instance Profile
6. S3 Bucket and Policy
7. EC2 Instance

## 8. Compliance and Monitoring

### CloudWatch Monitoring:
- IAM role includes `CloudWatchAgentServerPolicy`
- SSM access for remote management
- Logs automatically forwarded to S3

### S3 Compliance:
- Server-side encryption enabled by default
- Public access blocked via multiple mechanisms
- Bucket policy restricting log ingestion to authorized sources

## 9. Failure Prevention Measures

### Linting Compliance:
- All Parameters have Default values matching AllowedPattern
- No hardcoded AMIs or AZs
- No legacy S3 properties used
- Proper DeletionPolicy/UpdateReplacePolicy pairs where needed
- No unnecessary Fn::Sub usage

### Runtime Safety:
- Latest Amazon Linux AMI ensures security patches
- Free tier instance type prevents unexpected costs
- Default key pair allows deployment without manual input

This architecture provides a secure, monitored, and compliant web application environment that can be deployed automatically without manual intervention, meeting all specified requirements while adhering to AWS best practices.
