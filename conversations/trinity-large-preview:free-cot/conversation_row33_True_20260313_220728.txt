# IAM Roles for EC2 with Read-Only IAM Access - Architecture Plan

## Key AWS Services Required
1. **AWS::IAM::Role** - To create the IAM roles for EC2 instances
2. **AWS::IAM::ManagedPolicy** - To attach the read-only IAM permissions
3. **AWS::IAM::Policy** - To define the trust relationships with EC2 service

## Architecture Overview

I'll create two IAM roles that EC2 instances can assume, each with read-only access to IAM metadata. The roles will trust the EC2 service and have permissions to perform `iam:Get*` and `iam:List*` actions across all IAM resources.

## Detailed Implementation Plan

### 1. IAM Role Configuration

**Role 1: EC2ReadOnlyIAMRole1**
- **Trust Relationship**: Configured to trust the EC2 service
- **Permissions**: Read-only access to IAM metadata using `iam:Get*` and `iam:List*` actions
- **Path**: Default path ("/")
- **Description**: EC2 role with read-only IAM access

**Role 2: EC2ReadOnlyIAMRole2**
- **Trust Relationship**: Configured to trust the EC2 service
- **Permissions**: Read-only access to IAM metadata using `iam:Get*` and `iam:List*` actions
- **Path**: Default path ("/")
- **Description**: EC2 role with read-only IAM access

### 2. Permissions Strategy

Instead of creating custom policies, I'll use the AWS managed policy `IAMReadOnlyAccess` which already includes the necessary permissions for `iam:Get*` and `iam:List*` actions across all IAM resources. This approach is more maintainable and follows AWS best practices.

### 3. Trust Relationship Configuration

Each role will have a trust policy that allows the EC2 service to assume the role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

### 4. Resource Properties

**Role Properties:**
- **RoleName**: Explicitly named for clarity and management
- **Description**: Clear description of the role's purpose
- **Path**: Default path ("/")
- **ManagedPolicyArns**: Reference to `arn:aws:iam::aws:policy/IAMReadOnlyAccess`

### 5. Dependencies and Order

Since this template only creates IAM resources, there are no external dependencies. IAM roles can be created independently without requiring other AWS services.

### 6. Safe Defaults and Best Practices

- **Role Names**: Using descriptive names like `EC2ReadOnlyIAMRole1` and `EC2ReadOnlyIAMRole2`
- **Permissions**: Leveraging the AWS managed `IAMReadOnlyAccess` policy instead of creating custom policies
- **Trust Relationships**: Properly configured to only allow EC2 service to assume these roles
- **No Hardcoding**: No hardcoded values needed for this IAM-only template

### 7. Template Structure

The CloudFormation template will include:
1. **Resources section** with two IAM::Role resources
2. **Each role** will have:
   - A trust relationship with EC2 service
   - The managed policy `IAMReadOnlyAccess` attached
   - Appropriate metadata and descriptions

### 8. Security Considerations

- **Principle of Least Privilege**: Using the managed `IAMReadOnlyAccess` policy ensures only read-only permissions
- **No Secret Management**: No secrets or passwords required for this template
- **Resource Isolation**: Each role is independently defined and can be managed separately

### 9. Validation and Testing

After deployment, you can:
- Verify the roles exist in the IAM console
- Check that EC2 instances can assume these roles
- Test that the roles have the expected read-only IAM permissions
- Confirm the trust relationships are properly configured

This architecture provides a clean, maintainable solution for granting EC2 instances read-only access to IAM metadata, following AWS best practices and CloudFormation conventions.
