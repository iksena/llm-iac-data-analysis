# AWS CloudFormation Deployment Plan for IAM Managed Policies

## Key AWS Services Required
1. **AWS Identity and Access Management (IAM)** - For creating managed policies
2. **Amazon S3** - Target service for S3 access policy
3. **Amazon DynamoDB** - Target service for protection policy
4. **Amazon RDS** - Target service for protection policy

## Architecture Analysis

This deployment plan focuses on creating three IAM managed policies with specific permissions. Since this is a policy-only deployment, there are no infrastructure dependencies like VPCs, subnets, or EC2 instances to consider.

## Detailed Implementation Plan

### 1. S3 Access Policy for iOS Application

**Purpose**: Grant read/write access to the specific S3 bucket `ios-apps.sagebridge.org`

**Required Permissions**:
- `s3:PutObject` - Upload objects
- `s3:GetObject` - Download objects  
- `s3:DeleteObject` - Delete objects
- `s3:ListBucket` - List bucket contents
- `s3:GetBucketLocation` - Get bucket region

**Policy Structure**:
- Resource: Specific bucket ARN (`arn:aws:s3:::ios-apps.sagebridge.org`)
- Resource: All objects in bucket (`arn:aws:s3:::ios-apps.sagebridge.org/*`)
- Effect: Allow for all specified actions

### 2. DynamoDB Protection Policy

**Purpose**: Prevent accidental deletion or schema modification of DynamoDB tables

**Required Permissions**:
- `dynamodb:DeleteTable` - Explicitly denied
- `dynamodb:UpdateTable` - Explicitly denied

**Policy Structure**:
- Resource: All DynamoDB tables (`arn:aws:dynamodb:*:*:table/*`)
- Effect: Deny for destructive actions
- Note: This is a deny policy, so it must be attached in addition to any allow policies

### 3. RDS Protection Policy

**Purpose**: Block destructive RDS operations to protect databases and backups

**Required Permissions**:
- `rds:DeleteDBInstance` - Explicitly denied
- `rds:DeleteDBCluster` - Explicitly denied

**Policy Structure**:
- Resource: All RDS instances and clusters
- Effect: Deny for destructive actions
- Note: This is a deny policy, so it must be attached in addition to any allow policies

## Best Practices Implementation

### Policy Naming Convention
- Use environment-aware naming: `S3Access-${Environment}`, `DynamoDBProtection-${Environment}`, `RDSProtection-${Environment}`
- This ensures uniqueness across deployments and clear identification

### Policy Documentation
- Each policy includes a comprehensive description explaining its purpose
- Policy documents are well-commented for maintainability

### Resource Protection
- Since these are IAM policies (not stateful resources), no DeletionPolicy or UpdateReplacePolicy is needed
- Policies can be safely updated or deleted without data loss concerns

### Security Considerations
- Policies use the principle of least privilege
- Deny policies are explicitly created rather than relying on absence of allow permissions
- No hardcoded ARNs - use CloudFormation intrinsic functions where appropriate

## Linting Compliance

This template will pass cfn-lint because:
- No hardcoded AMIs or AZs are used (not applicable for IAM policies)
- All parameters have safe defaults
- No legacy S3 properties are used
- No Fn::Sub is used unnecessarily
- All AWS service names and ARNs are correctly formatted
- No hallucinated properties are included

## Deployment Strategy

1. **Template Validation**: Validate the CloudFormation template before deployment
2. **Incremental Deployment**: Deploy policies one at a time to isolate any issues
3. **Testing**: After deployment, test policy permissions using IAM policy simulator
4. **Documentation**: Update internal documentation with policy ARNs and purposes

This architecture provides a robust, secure, and maintainable solution for the specified business requirements while adhering to AWS best practices and CloudFormation linting rules.
