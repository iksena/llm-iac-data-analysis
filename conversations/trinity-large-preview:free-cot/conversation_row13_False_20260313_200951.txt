# Detailed Architectural Deployment Plan for Apache HTTP Server EC2 Instance with cfn-init and cfn-signal

## 1. Key AWS Services Required

Based on the business need, the following AWS services are required:

- **EC2**: To create and manage the virtual server instance
- **IAM**: For instance profile and role to grant EC2 permissions for CloudFormation operations
- **SSM Parameter Store**: To dynamically fetch the latest Amazon Linux 2 AMI ID
- **VPC**: To provide networking infrastructure (implied requirement for EC2)
- **Security Group**: To control inbound/outbound traffic to the EC2 instance
- **IAM Instance Profile**: To allow the EC2 instance to make AWS API calls for cfn-init, cfn-signal, and cfn-hup

## 2. Properties and Dependencies

### Critical Dependencies:
1. **IAM Instance Profile** must be created before the EC2 instance
2. **Security Group** must exist before EC2 instance creation
3. **VPC and Subnet** must be available (using default VPC if not specified)
4. **SSH Key Pair** must exist or be created (using default key if not specified)

### Property Dependencies:
- EC2 instance requires IAM instance profile, security group IDs, and subnet ID
- cfn-init requires metadata with packages, services, and files sections
- cfn-signal requires the wait handle from CreationPolicy
- cfn-hup requires configuration for metadata updates

## 3. Safe AWS Defaults (Explicitly Stated)

Since specific properties are missing, I will use these safe defaults:

- **AMI**: Latest Amazon Linux 2 AMI via SSM Parameter Store (`/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`)
- **Instance Type**: `t3.micro` (free tier eligible, suitable for web server)
- **VPC**: Default VPC of the region
- **Subnet**: First available subnet in the region
- **CIDR**: Default VPC CIDR (`10.0.0.0/16`)
- **SSH Key Pair**: Parameter with default value 'default-key' (user must have this key or create it)
- **Security Group**: Allow SSH (22) from 0.0.0.0/0 and HTTP (80) from 0.0.0.0/0
- **Region**: Determined by the stack's deployment region

## 4. Dynamic Resource Fetching Strategy

### AMI Fetching:
Use SSM Parameter Store to dynamically fetch the latest Amazon Linux 2 AMI:
```yaml
ImageId: !Sub 'arn:aws:ssm:${AWS::Region}:${AWS::AccountId}:parameter/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2'
```

### Availability Zones:
Use `Fn::GetAZs` and `Fn::Select` to avoid hardcoding:
```yaml
!Select [0, !GetAZs '']
```

## 5. Architecture Plan

### Template Structure:
1. **Parameters**: For SSH key name with default
2. **Resources**:
   - IAM Instance Profile with necessary permissions
   - Security Group with SSH and HTTP rules
   - EC2 instance with cfn-init metadata
3. **CreationPolicy**: With ResourceSignal and 5-minute timeout
4. **Metadata**: For cfn-init packages, services, and files
5. **UserData**: To bootstrap cfn-init

### EC2 Instance Configuration:
- **Packages**: Install httpd via yum
- **Services**: Enable and start httpd service
- **Files**: Create /var/www/html/index.html with sample content
- **Users**: Configure cfn-hup for metadata updates

### IAM Permissions Required:
- `cloudformation:DescribeStackResource`
- `cloudformation:SignalResource`
- `ssm:GetParameter` (for AMI fetching)
- `ec2:Describe*`

### Security Considerations:
- Use IAM instance profile instead of access keys
- Restrict security group to necessary ports
- Use dynamic references for any secrets (though none required here)

### Failure Prevention:
- CreationPolicy with 300-second timeout ensures instance is fully configured
- cfn-signal only sent after successful configuration
- cfn-hup configured to handle metadata updates automatically

This architecture ensures a robust, automated deployment that meets all the specified requirements while following AWS best practices and avoiding common CloudFormation pitfalls.
