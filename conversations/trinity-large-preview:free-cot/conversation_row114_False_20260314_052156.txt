# AWS B2B Data Interchange (B2BI) Pipeline Architecture Plan

## Key AWS Services Required

1. **AWS B2B Data Interchange (B2B Integration)** - Core service for EDI processing
2. **Amazon S3** - Storage for input EDI files and output JSON data
3. **AWS Lambda** - Custom transformation logic (if needed beyond B2BI capabilities)
4. **AWS Identity and Access Management (IAM)** - Service roles and permissions
5. **AWS Systems Manager (SSM) Parameter Store** - Dynamic AMI fetching (future-proofing)
6. **Amazon CloudWatch** - Logging and monitoring

## Architecture Overview

The solution will create a complete B2B integration pipeline with:
- B2BI profile for business partner configuration
- Transformer resource for EDI X12_214 to JSON mapping
- Input S3 bucket for receiving EDI files
- Output S3 bucket for storing transformed JSON
- IAM roles for service permissions
- Event-driven processing (S3 upload triggers transformation)

## Detailed Resource Plan

### 1. B2BI Profile Resource
- **Type**: AWS::B2BIntegration::Profile
- **Properties**:
  - ProfileName: Dynamic with Fn::Sub
  - BusinessContact: Name, Email, Phone (with safe defaults)
  - Description: "EDI X12_214 Processing Profile"

### 2. Transformer Resource
- **Type**: AWS::B2BIntegration::Transformer
- **Properties**:
  - TransformerName: Dynamic naming
  - InputFormat: "EDI"
  - OutputFormat: "JSON"
  - MappingLogic: X12_214 specific transformation rules
  - ProfileId: Reference to created profile

### 3. S3 Input Bucket
- **Type**: AWS::S3::Bucket
- **Properties**:
  - BucketName: Dynamic with prefix
  - Versioning: Enabled for audit trail
  - ServerAccessLogging: Optional for compliance
  - PublicAccessBlockConfiguration: Block all public access

### 4. S3 Output Bucket
- **Type**: AWS::S3::Bucket
- **Properties**:
  - BucketName: Dynamic with suffix
  - Versioning: Enabled
  - Encryption: SSE-S3 for data protection
  - LifecycleConfiguration: Transition to IA after 30 days

### 5. IAM Roles and Policies
- **B2BI Service Role**: For B2BI to access S3
- **Lambda Execution Role**: If custom processing is needed
- **S3 Bucket Policies**: Fine-grained access control

### 6. Event Configuration
- **S3 Event Notifications**: Trigger transformation on file upload
- **SNS Topic**: For processing notifications (optional)

## Dependencies and Execution Order

1. Create IAM roles and policies first
2. Create S3 buckets (input and output)
3. Create B2BI profile
4. Create transformer resource
5. Configure event notifications
6. Validate integration

## Safe Defaults for Missing Properties

```yaml
    Description: "Input S3 bucket name"

    Type: String
    Default: !Sub "${AWS::StackName}-output"
    Description: "Output S3 bucket name"
```

### Dynamic Resource Fetching:
- **AMIs**: Not required for this service-based architecture
- **AZs**: Use Fn::GetAZs and Fn::Select for any future EC2 needs
- **ARNs**: Use dynamic references for any future secrets

## Security Best Practices

1. **Least Privilege**: IAM roles with minimal required permissions
2. **Encryption**: SSE-S3 for S3 buckets, TLS for any API calls
3. **Logging**: CloudWatch logs for all operations
4. **Monitoring**: CloudWatch alarms for bucket access and processing failures
5. **Compliance**: Enable versioning and access logging

## Error Handling and Validation

1. **Template Validation**: Use cfn-lint before deployment
2. **Resource Dependencies**: Explicit DependsOn declarations
3. **Error Notifications**: SNS topics for processing failures
4. **Rollback Triggers**: CloudFormation rollback on critical failures
