# Automated Image Analysis Pipeline Architecture Plan

## Key AWS Services Required

1. **S3** - Storage bucket for image uploads
2. **Lambda** - Serverless functions for image processing and notifications
3. **Step Functions** - Workflow orchestration for the analysis pipeline
4. **Rekognition** - Image analysis for human face detection
5. **SNS** - SMS notifications for alerts
6. **IAM** - Service permissions and role management
7. **SSM Parameter Store** - Dynamic resource fetching (AMIs, etc.)

## Architecture Overview

The solution follows an event-driven architecture where S3 triggers a Lambda function on image upload, which initiates a Step Functions workflow. The workflow orchestrates Rekognition analysis and conditional SMS notifications based on detection results.

## Detailed Component Breakdown

### 1. S3 Bucket Configuration
- **Properties**: Versioning enabled, server access logging
- **Event Notification**: Configured to trigger Lambda on `s3:ObjectCreated:*` events
- **Dependencies**: Must be created before Lambda event source mapping

### 2. Lambda Functions
**Image Processing Lambda**:
- Runtime: Python 3.12 (latest supported)
- IAM Role: S3 read access, Step Functions execution permission
- Function: Extracts image metadata, initiates Step Functions execution

**Rekognition Analysis Lambda**:
- Runtime: Python 3.12
- IAM Role: Rekognition access, S3 read access for image retrieval
- Function: Calls Rekognition DetectFaces API, returns analysis results

**Notification Lambda**:
- Runtime: Python 3.12
- IAM Role: SNS publish permission
- Function: Sends SMS via SNS when human faces detected

### 3. Step Functions State Machine
- **Type**: Standard workflow
- **Definition**: JSON-based state machine with error handling and retry logic
- **States**: 
  - Task state for Rekognition analysis
  - Choice state for conditional branching based on detection results
  - Task state for SNS notification (only executed if faces detected)

### 4. IAM Role Configuration (Critical for Circular Dependencies)

**Lambda Execution Role**:
```yaml
Policies:
  - PolicyName: RekognitionAccess
    PolicyDocument:
      Version: "2012-10-17"
      Statement:
        - Effect: Allow
          Action: rekognition:DetectFaces
          Resource: "*"
  - PolicyName: S3Access
    PolicyDocument:
      Version: "2012-10-17"
      Statement:
        - Effect: Allow
          Action: s3:GetObject
          Resource: !Sub "arn:aws:s3:::${ImageBucket}/*"
  - PolicyName: StepFunctionsAccess
    PolicyDocument:
      Version: "2012-10-17"
      Statement:
        - Effect: Allow
          Action: states:StartExecution
          Resource: !Sub "arn:aws:states:${AWS::Region}:${AWS::AccountId}:stateMachine:${StateMachine}"
```

**S3 Event Notification Permission**:
- Use Lambda's `Permission` resource to grant S3 service principal access
- Resource: Lambda function ARN
- Action: `lambda:InvokeFunction`

### 5. SNS Topic Configuration
- **Properties**: Delivery status logging enabled
- **Subscription**: SMS protocol with phone number parameter
- **IAM Policy**: Lambda notification function granted `sns:Publish` permission

## Properties and Dependencies

### S3 Dependencies
1. Bucket must exist before creating Lambda permission
2. Event notification configuration requires Lambda function ARN
3. Bucket policy (if needed) must reference existing resources

### Lambda Dependencies
1. IAM role must exist before function creation
2. Function code package must be available (via S3 or inline)
3. Event source mapping requires both S3 bucket and Lambda function

### Step Functions Dependencies
1. IAM role for state machine execution must exist
2. Lambda functions referenced in states must be created first
3. State machine definition must reference valid ARNs

## Safe AWS Defaults (Explicitly Stated)

**Parameters with Defaults**:
```yaml
Parameters:
  S3BucketName:
    Type: String
    Default: image-analysis-pipeline-<account-id>
    Description: S3 bucket name for image uploads
  PhoneNumber:
    Type: String
    Default: +1234567890
    Description: Phone number for SMS alerts
  LambdaTimeout:
    Type: Number
    Default: 30
    Description: Lambda function timeout in seconds
```

**Dynamically Fetched Resources**:
- **AMIs**: Not applicable (serverless architecture)
- **AZs**: Use `Fn::GetAZs` and `Fn::Select` for any future VPC expansion
- **Region-specific resources**: Use `AWS::Region` pseudo parameter

## Security & Best Practices Implementation

1. **Least Privilege**: Each Lambda function has only the permissions it needs
2. **No Hardcoding**: All ARNs constructed using intrinsic functions
3. **Secure Parameters**: Phone number passed as parameter (not sensitive), could use SSM SecureString for production
4. **Error Handling**: Step Functions include retry policies and catch blocks
5. **Logging**: All functions write to CloudWatch Logs with appropriate retention policies

## Circular Dependency Resolution Strategy

The main circular dependency is between S3 event notifications and Lambda permissions. This is resolved by:

1. Creating the S3 bucket first
2. Creating Lambda functions with execution roles
3. Creating Lambda permissions for S3 service principal
4. Finally, configuring S3 event notifications

This sequence ensures all ARNs exist before they're referenced.

## State Machine Definition Strategy

The Step Functions state machine will use Amazon States Language (ASL) with:

- **Input Path**: Extracts image key from S3 event
- **Result Path**: Stores Rekognition results
- **Choice State**: Evaluates `FaceDetails` array length to determine if faces detected
- **Parallel State**: Could be used for concurrent analysis if multiple images processed

This architecture ensures a robust, secure, and maintainable image analysis pipeline that meets all specified requirements while adhering to AWS best practices and CloudFormation linting rules.
