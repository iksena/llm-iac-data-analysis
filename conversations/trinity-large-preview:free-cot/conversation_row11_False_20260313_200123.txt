# DynamoDB Table for Mobile Notifications Reporting - Architecture Plan

## Key AWS Services Required
1. **AWS::DynamoDB::Table** - Core service for creating the DynamoDB table
2. **AWS::DynamoDB::Table/AttributeDefinition** - Define table attributes
3. **AWS::DynamoDB::Table/KeySchema** - Define primary key structure
4. **AWS::DynamoDB::Table/GlobalSecondaryIndex** - Create GSI for time-based queries
5. **AWS::DynamoDB::Table/ProvisionedThroughput** - Configure billing mode
6. **AWS::DynamoDB::Table/TimeToLiveSpecification** - Configure TTL for automatic cleanup

## Properties and Dependencies Analysis

### Table Configuration
- **TableName**: `MobileNotificationsReporting` (explicit naming for clarity)
- **BillingMode**: `PAY_PER_REQUEST` (cost-efficient for unpredictable workloads)
- **TimeToLiveSpecification**: Enabled with `AttributeName: "ttl"` and `Enabled: true`

### Primary Key Design
- **Partition Key**: `id` (String) - Unique identifier for each notification
- **Sort Key**: `timestamp` (Number) - Event timestamp for ordering

### Global Secondary Index (GSI)
- **IndexName**: `TypeSentTimeIndex`
- **Partition Key**: `type` (String) - Notification type/category
- **Sort Key**: `sentTime` (Number) - When notification was sent
- **Projection**: `ALL` - Include all attributes for comprehensive queries

### Attribute Definitions
1. `id` (String) - Primary partition key
2. `timestamp` (Number) - Primary sort key  
3. `type` (String) - GSI partition key
4. `sentTime` (Number) - GSI sort key
5. `ttl` (Number) - TTL attribute for automatic cleanup

## Safe Defaults and Dynamic Resource Fetching

### Missing Properties - Safe Defaults
- **Region**: Use `AWS::Region` pseudo-parameter for dynamic region detection
- **AccountID**: Use `AWS::AccountId` pseudo-parameter for account-specific references
- **Default TTL**: 30 days (2,592,000 seconds) - reasonable default for stale data

### Dynamic AMI Fetching (Not Applicable)
Since this template only creates DynamoDB resources, AMI fetching via SSM Parameter Store is not required. However, if Lambda functions were added later for data processing, we would use:
```yaml
ImageId: !Sub 'arn:${AWS::Partition}:ssm:${AWS::Region}/aws/service/lambda/runtime/nodejs20.x'
```

### Availability Zone Handling (Not Applicable)
DynamoDB is region-based, not AZ-based, so AZ handling is not required for this template.

## Security and Best Practices Implementation

### Resource Protection
- **DeletionPolicy**: `Retain` - Prevent accidental deletion of production data
- **UpdateReplacePolicy**: `Retain` - Maintain data during updates

### Cost Optimization
- **BillingMode**: `PAY_PER_REQUEST` - No capacity planning needed, pay only for what you use
- **TTL**: Automatic cleanup reduces storage costs and improves query performance

### Data Modeling Best Practices
- **GSI Design**: Composite key (`type` + `sentTime`) enables efficient time-based queries by notification type
- **TTL Attribute**: Using `ttl` field allows DynamoDB to automatically delete expired items

## Architecture Rationale

### Why This Design?
1. **Pay-per-request billing** eliminates capacity planning overhead and optimizes costs for unpredictable workloads
2. **Composite GSI key** (`type` + `sentTime`) enables efficient queries like "all notifications of type X sent in the last 24 hours"
3. **TTL auto-expiry** ensures the table doesn't grow indefinitely, maintaining performance and controlling costs
4. **Explicit naming** makes the template self-documenting and easier to maintain

### Query Patterns Supported
- Primary key queries: `id` + `timestamp` for exact item retrieval
- GSI queries: `type` + `sentTime` for time-based filtering by notification category
- Full table scans: For administrative or analytical purposes (less efficient)

### Scalability Considerations
- DynamoDB automatically scales with pay-per-request billing
- GSI provides additional query flexibility without impacting primary table performance
- TTL prevents table bloat, maintaining query performance over time

This architecture provides a robust, cost-effective foundation for mobile notifications reporting with automatic cleanup and efficient query patterns for common use cases.
