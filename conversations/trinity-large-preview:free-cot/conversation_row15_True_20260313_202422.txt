# Detailed Architectural Deployment Plan for API Gateway with Lambda Non-Proxy Integration

## 1. Key AWS Services Required

Based on the business need, the following AWS services are required:

- **AWS::Serverless-2016-10-31** (Transform for SAM)
- **AWS::Lambda::Function** - Lambda function to handle API requests
- **AWS::IAM::Role** - IAM role for Lambda execution permissions
- **AWS::ApiGateway::RestApi** - REST API Gateway
- **AWS::ApiGateway::Resource** - API Gateway resource (path)
- **AWS::ApiGateway::Method** - API Gateway method with non-proxy integration
- **AWS::ApiGateway::Deployment** - API deployment
- **AWS::ApiGateway::Stage** - API stage for deployment

## 2. Properties and Dependencies

The deployment follows this dependency chain:

1. IAM Role must exist before Lambda creation
2. Lambda function must exist before API Gateway method creation
3. API Gateway resources must exist before methods
4. Deployment must exist before Stage creation

## 3. Safe AWS Defaults (where values are missing)

- **Lambda Runtime**: `python3.12` (latest supported Python runtime)
- **Lambda Handler**: `index.handler` (standard handler format)
- **API Gateway Stage Name**: `prod` (production default)
- **Lambda Memory Size**: `128` MB (minimum allowed)
- **Lambda Timeout**: `30` seconds (maximum allowed)
- **IAM Role Path**: `/` (root path)
- **IAM Role Managed Policy**: `service-role/AWSLambdaBasicExecutionRole` (standard Lambda execution role)

## 4. Dynamic Resource Fetching Strategy

- **Lambda Code**: Will use inline code in the template (no external S3 bucket required)
- **AMI**: Not applicable for Lambda (serverless)
- **AZs**: Will use `Fn::GetAZs` and `Fn::Select` for any AZ-specific operations

## 5. Architecture Design

The template will create:

1. An IAM role with basic Lambda execution permissions
2. A Lambda function with inline Python code
3. A REST API with a single resource and method
4. A deployment and stage for the API

## 6. Security Considerations

- Lambda will have minimal permissions (only CloudWatch Logs)
- No VPC configuration (simpler, cost-effective for public APIs)
- No authentication/authorization (basic setup - can be enhanced later)

## 7. Error Handling and Best Practices

- All resources will have appropriate logical IDs
- Template will use YAML format with proper indentation
- No hardcoded values except safe defaults
- All required properties will be included based on AWS documentation

## 8. Template Structure

The CloudFormation template will be structured as follows:

1. **AWSTemplateFormatVersion**: 2010-09-09
2. **Transform**: AWS::Serverless-2016-10-31
3. **Resources**: All AWS resources defined here
4. **Outputs**: API endpoint URL for easy access

## 9. Lambda Function Design

The Lambda function will:
- Accept API Gateway events
- Return a simple JSON response
- Log execution details to CloudWatch

## 10. API Gateway Configuration

The API will:
- Use non-proxy integration (explicit integration configuration)
- Have a single GET method on the root resource
- Return a simple response from Lambda

This architecture provides a minimal, functional API Gateway with Lambda integration that follows AWS best practices and can be deployed successfully.
