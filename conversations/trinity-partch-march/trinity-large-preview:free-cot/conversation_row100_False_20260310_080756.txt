# IAM Baseline Architecture Deployment Plan

## Key AWS Services Required
1. **AWS Identity and Access Management (IAM)** - Core service for creating groups, roles, and policies
2. **AWS Systems Manager (SSM) Parameter Store** - For dynamic AMI fetching (though not directly used in IAM template)
3. **AWS CloudFormation** - For template deployment and infrastructure as code

## Architecture Overview

This template creates a secure IAM baseline with role-based access controls and cross-account trust relationships. The architecture follows the principle of least privilege with MFA enforcement for elevated actions.

## IAM Groups and Roles Design

### Core IAM Groups
1. **PrivilegedAdmin** - Full administrative access with MFA requirement
2. **RestrictedAdmin** - Administrative access with specific service restrictions
3. **IdentityAdmin** - Identity and access management permissions

### Cross-Account Trust Configuration
- **Identity Management Account** - Trusted for identity operations
- **Tooling Management Account** - Trusted for tooling operations
- **Centralized Management** - Single point for account governance

## Security Controls Implementation

### MFA Enforcement
- All privileged roles require MFA for console access
- MFA required for destructive operations
- Time-based one-time password (TOTP) support

### Permission Boundaries
- Explicit deny policies for sensitive operations (billing, CloudHSM)
- Granular permissions using AWS managed policies
- Custom policies for specific use cases

### Role Assumption
- Conditional access based on source account
- External ID support for enhanced security
- Session duration limits

## CloudFormation Execution Roles

### StackSet Administration
- Cross-account stack deployment capabilities
- Template validation permissions
- Resource creation and modification rights

### StackSet Execution
- Limited permissions for target accounts
- Resource-specific access controls
- Audit trail capabilities

## Properties and Dependencies

### IAM Role Properties
- `AssumeRolePolicyDocument` - Trust relationships
- `ManagedPolicyArns` - AWS managed policies
- `Policies` - Custom inline policies
- `PermissionsBoundary` - Permission boundaries
- `Tags` - Resource tagging

### Policy Properties
- `Version` - Policy language version (2012-10-17)
- `Statement` - Individual permissions
- `Effect` - Allow/Deny
- `Action` - Specific API actions
- `Resource` - Target resources
- `Condition` - Conditional requirements

## Safe Defaults and Dynamic Resource Fetching

### Missing Properties Handling
- **Default CIDR blocks**: 10.0.0.0/16 (for any network references)
- **Default KeyPair**: 'default-key' (for any SSH requirements)
- **Default Region**: 'us-east-1' (for any region-specific operations)

### Dynamic AMI Fetching
While IAM templates don't require AMIs, the pattern would be:
```yaml
Fn::ImportValue: !Sub "resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2"
```

### Dynamic AZ Selection
```yaml
Fn::Select:
  - 0
  - Fn::GetAZs: !Ref 'AWS::Region'
```

## Template Structure

### Parameters Section
- Account IDs for trusted accounts
- MFA requirement flags
- Permission boundary ARNs
- Custom policy overrides

### Resources Section
1. IAM Groups
2. IAM Roles
3. Managed Policies
4. Inline Policies
5. Permission Boundaries

### Outputs Section
- Role ARNs for cross-account access
- Policy ARNs for reference
- Group ARNs for team management

## Best Practices Implementation

### Security
- Principle of least privilege
- MFA enforcement
- Explicit deny policies
- Permission boundaries

### Maintainability
- AWS managed policies where possible
- Clear naming conventions
- Comprehensive tagging
- Documentation in comments

### Compliance
- Audit trail capabilities
- Resource naming standards
- Access logging configuration
- Regular review triggers

## Deployment Considerations

### Pre-requisites
- Target accounts must exist
- Trusted account IDs must be known
- MFA devices must be available

### Post-deployment
- Test role assumptions
- Verify cross-account access
- Validate permission boundaries
- Audit trail verification

This architecture provides a secure, scalable IAM baseline that can be deployed across multiple AWS accounts while maintaining centralized governance and control.
