# Architectural Deployment Plan for Self-Configuring EC2 with Nginx and SSM

## 1. Key AWS Services Required

Based on the business requirements, the following AWS services are essential:

- **Amazon EC2**: For hosting the Linux 2 instance
- **AWS Systems Manager (SSM)**: For automated configuration and logging
- **Amazon S3**: For centralized log storage
- **AWS Identity and Access Management (IAM)**: For instance role and permissions
- **Amazon VPC**: For network isolation (default VPC acceptable)
- **AWS SSM Parameter Store**: For dynamic AMI retrieval

## 2. Properties and Dependencies

### Critical Dependencies:
1. **VPC and Internet Gateway**: The EC2 instance requires internet access for SSM connectivity and package installation. The Internet Gateway must be attached to the VPC before the EC2 instance can be provisioned.
2. **IAM Role Creation**: The IAM role must be created before the EC2 instance to ensure proper permissions for SSM access and S3 logging.
3. **S3 Bucket Creation**: The S3 bucket must exist before the SSM Association can be configured to write logs to it.
4. **SSM Document**: The AWS-RunShellScript document must be available (it's a managed document, so no creation needed).

### Property Dependencies:
- EC2 instance depends on IAM role, VPC, and subnet
- SSM Association depends on EC2 instance and S3 bucket
- S3 bucket policy depends on the bucket itself

## 3. Safe AWS Defaults for Missing Properties

Since specific properties weren't provided, I'll use these safe defaults:

- **CIDR Blocks**: 10.0.0.0/16 for VPC, 10.0.0.0/24 for subnet
- **Instance Type**: t3.micro (free tier eligible)
- **Key Pair**: 'default-key' (will be a parameter with this default)
- **Region**: us-east-1 (will use Fn::Sub for region-specific resources)
- **Availability Zone**: Will use Fn::GetAZs and Fn::Select to dynamically select

## 4. Dynamic Resource Fetching Strategy

### AMI Retrieval:
- Use SSM Parameter Store to fetch the latest Amazon Linux 2 AMI: `/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`
- This ensures the template always uses the latest AMI without hardcoding

### Availability Zones:
- Use `Fn::GetAZs: ""` to get all AZs in the region
- Use `Fn::Select: [0, Fn::GetAZs: ""]` to select the first AZ

## Detailed Architecture Plan

### 1. IAM Role and Policy
- Create an IAM role with the following managed policies:
  - `AmazonSSMManagedInstanceCore` (for SSM access)
  - Custom policy for S3 logging permissions
- The custom policy will allow:
  - s3:PutObject on the specific log path (logs/*) in the designated bucket
  - s3:GetBucketLocation for the bucket

### 2. S3 Bucket for Logs
- Create a dedicated S3 bucket for storing SSM command outputs
- Configure bucket policy to allow the IAM role to write logs
- Use structured path "logs/" for organized storage
- Implement versioning for log retention

### 3. EC2 Instance Configuration
- Use Amazon Linux 2 AMI fetched dynamically from SSM Parameter Store
- Place in default VPC and a public subnet for internet access
- Associate the IAM role for SSM permissions
- Ensure security group allows HTTP (port 80) for Nginx

### 4. SSM Association for Nginx Installation
- Create an SSM Association that targets the EC2 instance
- Use the AWS-RunShellScript document
- Script commands:
  ```bash
  #!/bin/bash
  sudo yum update -y
  sudo amazon-linux-extras install nginx1 -y
  sudo systemctl start nginx
  sudo systemctl enable nginx
  ```
- Configure association to run during instance launch

### 5. SSM Logging Configuration
- Configure SSM to send command outputs to the S3 bucket
- Use structured path: `logs/${Date}/${Time}/${AWS::Region}/${AccountId}/${ExecutionId}/`
- This provides auditability and organized log storage

### 6. Resource Dependencies and Creation Order
1. VPC and subnet (if not using defaults)
2. S3 bucket and bucket policy
3. IAM role and instance profile
4. EC2 instance
5. SSM Association

### 7. Security Considerations
- Principle of least privilege for IAM policies
- Use VPC endpoints for S3 access if in private subnet
- Enable encryption for S3 bucket (SSE-S3 default)
- Use secure parameters for any sensitive data

### 8. Template Structure
The CloudFormation template will be organized as:
- Parameters (with safe defaults)
- Mappings (if needed)
- Resources (in creation order)
- Outputs (instance ID, public IP, S3 bucket name)

This architecture ensures a self-configuring EC2 instance that automatically installs and starts Nginx upon launch while maintaining centralized logging through SSM to S3, all with proper security and automation practices.
