# Detailed Architectural Deployment Plan for EFS with Cross-AZ EC2 Instances and Monitoring

## Key AWS Services Required

1. **VPC** - Virtual Private Cloud for network isolation
2. **EC2** - Elastic Compute Cloud instances for application hosting
3. **EFS** - Elastic File System for shared storage across AZs
4. **IAM** - Identity and Access Management for security roles
5. **CloudWatch** - Monitoring and alerting for performance metrics
6. **SSM** - Systems Manager for AMI resolution and instance management
7. **Auto Scaling** - Optional, for scaling EC2 instances
8. **S3** - Optional, for storing configuration files or backups
9. **CloudFormation** - Infrastructure as Code deployment

## Architecture Overview

The solution will create a VPC with public subnets across multiple AZs, deploy two EC2 instances in separate AZs with IAM roles for EFS access and SSM permissions, create an EFS file system with mount targets in each AZ, and configure CloudWatch alarms for performance monitoring.

## Detailed Component Breakdown

### 1. VPC Configuration

**Properties:**
- CIDR block: 10.0.0.0/16 (default)
- Public subnets: 10.0.0.0/24, 10.0.1.0/24 (one per AZ)
- Internet Gateway attached to VPC
- Network ACLs allowing inbound/outbound traffic on ports 22, 80, 443, and 2049 (NFS)
- Route tables with default route to Internet Gateway

**Dependencies:**
- Internet Gateway must be attached to VPC before creating route associations
- Subnets must exist before creating EFS mount targets

### 2. IAM Role Configuration

**Properties:**
- Role name: EFS-SSM-Role
- Managed policies: AmazonSSMManagedInstanceCore, AmazonElasticFileSystemClientReadWrite
- Instance profile for EC2 attachment

**Dependencies:**
- Role must exist before creating EC2 instances

### 3. EFS File System

**Properties:**
- Performance mode: GeneralPurpose
- Throughput mode: Bursting
- Encrypted: true (default)
- Lifecycle policies: Transition to IA after 30 days

**Dependencies:**
- VPC must exist before creating file system
- Mount targets require subnets to be available

### 4. EC2 Instances

**Properties:**
- AMI: Latest Amazon Linux 2 using SSM parameter store
- Instance type: t3.micro (default)
- Key pair: default-key (safe default)
- IAM instance profile: EFS-SSM-Role
- User data: EFS mounting script with retry logic
- CreationPolicy with cfn-signal for mount verification

**Dependencies:**
- IAM role must exist before instance creation
- EFS mount targets must be available before mounting

### 5. CloudWatch Alarms

**Properties:**
- Alarm for EFS I/O limits at 95% threshold
- Alarm for EFS throughput at 80% threshold
- SNS topic for notifications (dummy email: admin@example.com)
- Alarm actions: SNS topic ARN

**Dependencies:**
- EFS file system must exist before creating alarms
- SNS topic must exist before alarm creation

## Dynamic Resource Resolution

### AMI Resolution
Use SSM Parameter Store to fetch latest Amazon Linux 2 AMI:
```yaml
!Sub 'arn:aws:ssm:${AWS::Region}:${AWS::AccountId}:parameter/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2'
```

### Availability Zones
Use Fn::GetAZs and Fn::Select to dynamically fetch AZs:
```yaml
!Select [0, !GetAZs '']
!Select [1, !GetAZs '']
```

## EFS Mounting Script Logic

The user data script will:
1. Install EFS client utilities
2. Create mount point directory
3. Wait for EFS DNS name resolution
4. Attempt mount with retries (up to 10 attempts, 30-second intervals)
5. Send cfn-signal upon successful mount
6. Handle mount failures gracefully

## Security Considerations

- Network ACLs will allow NFS traffic (port 2049) only from EC2 instance subnets
- Security groups will restrict SSH access to specific IP ranges
- EFS will be encrypted at rest
- IAM roles will follow principle of least privilege

## Monitoring Strategy

- CloudWatch alarms will monitor:
  - PercentIOLimit metric at 95% threshold
  - BurstCreditBalance for throughput monitoring
  - CPU utilization on EC2 instances
- Alarms will trigger SNS notifications
- Metrics will be retained for 15 months

## Failure Handling

- CreationPolicy ensures instances successfully mount EFS before stack completion
- Auto-healing through instance recovery if mount fails
- Stack rollback on critical failures
- Graceful degradation for non-critical components

## Performance Optimization

- EFS performance mode set to GeneralPurpose for balanced performance
- Bursting throughput mode for cost optimization
- Proper subnet placement for minimal latency
- CloudWatch metrics for capacity planning

This architecture ensures high availability, security, and automated deployment while adhering to AWS best practices and CloudFormation linting rules.
