# Detailed Architectural Deployment Plan for AppSync GraphQL API with Lambda Backend

## 1. Key AWS Services Required

Based on the business need, the following AWS services are required:

- **AWS AppSync** - For creating the GraphQL API (`PetShopAPI`)
- **AWS Lambda** - For the serverless backend function returning mock pet data
- **AWS IAM** - For creating execution roles with least privilege
- **AWS CloudFormation** - For infrastructure as code deployment

## 2. Properties and Dependencies

The deployment follows a specific order of operations:

1. **IAM Roles** must be created first (Lambda execution role and AppSync data source role)
2. **Lambda Function** depends on the execution role
3. **AppSync API** can be created independently
4. **AppSync Data Source** depends on both the API and Lambda function
5. **AppSync Schema** and **Resolver** depend on the API and data source

## 3. Safe AWS Defaults

Since specific properties weren't provided, I'll use these safe defaults:

- **Lambda Runtime**: `nodejs20.x` (latest LTS Node.js runtime)
- **Lambda Timeout**: `10` seconds (sufficient for mock data)
- **Lambda Memory**: `128` MB (minimum, sufficient for static data)
- **AppSync Authentication**: API Key (with automatic expiration)
- **Mock Pet Data**: Static array with 3 sample pets (ID, name, type)

## 4. Dynamic Resource Fetching

For this specific use case, dynamic fetching isn't required for:
- Lambda runtime (we'll use a fixed runtime)
- AppSync API (no AMIs or AZs involved)

However, I'll follow best practices by:
- Using AWS managed policies for IAM roles
- Avoiding hardcoded values where possible

## 5. Architecture Design

### IAM Roles (Least Privilege)

**Lambda Execution Role**:
- Trust relationship allowing Lambda service to assume role
- Permissions: `AWSLambdaBasicExecutionRole` (for CloudWatch Logs)
- No S3 or VPC access needed (simple function)

**AppSync Data Source Role**:
- Trust relationship allowing AppSync service to assume role
- Permissions: `lambda:InvokeFunction` on the specific Lambda ARN
- Explicit resource-based policy for AppSync invocation

### Lambda Function

**Implementation**:
- Node.js 20.x runtime
- Handler: `index.handler`
- Code: Inline (for simplicity with mock data)
- Environment: No external dependencies needed

**Mock Data Structure**:
```javascript
[
  { id: "1", name: "Max", type: "Dog" },
  { id: "2", name: "Bella", type: "Cat" },
  { id: "3", name: "Charlie", type: "Bird" }
]
```

### AppSync GraphQL API

**API Configuration**:
- Name: `PetShopAPI`
- Authentication: API Key (with 7-day expiration)
- Schema: Simple query for `getPets` returning list of pets

**GraphQL Schema**:
```graphql
type Pet {
  id: ID!
  name: String!
  type: String!
}

type Query {
  getPets: [Pet]
}
```

### AppSync Data Source & Resolver

**Data Source**:
- Type: AWS Lambda
- Lambda function ARN
- IAM role ARN (AppSync data source role)

**Resolver Configuration**:
- Type: Query
- Field: `getPets`
- Request Mapping Template: Pass-through (no transformation needed)
- Response Mapping Template: Map Lambda array to GraphQL `[Pet]` type

## 6. Security Considerations

- **API Key**: Auto-generated with 7-day expiration (can be rotated)
- **IAM Roles**: Least privilege principle applied
- **Lambda**: No external dependencies or network access required
- **AppSync**: Anonymous access allowed via API Key

## 7. Deployment Strategy

The CloudFormation template will deploy in this sequence:

1. IAM Roles (parallel creation possible)
2. Lambda Function
3. AppSync API
4. AppSync Data Source
5. AppSync Schema
6. AppSync Resolver

This ensures all dependencies are satisfied before resources that depend on them are created.

## 8. Error Handling & Validation

- **IAM Role Creation**: Validate trust relationships
- **Lambda Deployment**: Validate handler and runtime
- **AppSync API**: Validate schema syntax
- **Resolver**: Validate mapping templates

The template will include appropriate `DependsOn` clauses to ensure proper sequencing and will use AWS best practices for all resource configurations.
