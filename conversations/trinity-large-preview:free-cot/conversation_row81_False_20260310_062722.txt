# AWS Resource Group Creation Architecture Plan

## Key AWS Services Required

1. **AWS Resource Groups** - Core service for creating and managing resource groups
2. **AWS Systems Manager (SSM)** - For dynamic AMI fetching and parameter store access
3. **AWS Identity and Access Management (IAM)** - For service roles and permissions
4. **AWS CloudFormation** - For template deployment and orchestration

## Architecture Analysis

Since the business need is specifically for creating an AWS Resource Group using tag-based filters, this is a relatively straightforward deployment that doesn't require complex infrastructure dependencies.

### Resource Group Requirements

The Resource Group will use tag-based filters to include instances from multiple AWS services:
- EC2 instances
- RDS instances  
- SageMaker resources
- SSM managed instances
- Other taggable AWS resources

### Properties and Dependencies

**No complex dependencies exist** for this deployment:
- Resource Groups are a management service that doesn't require VPCs, subnets, or internet connectivity
- No EC2 instances need to be launched
- No database or storage dependencies
- No IAM roles required for basic Resource Group functionality

### Safe Defaults Selection

Since this is a management-focused template without infrastructure components:

1. **Resource Group Name**: Default to "ProductionResources" 
2. **Resource Query Type**: Default to "TAG_FILTERS_1_0"
3. **Tag Filters**: Default to common production tags like:
   - Environment: "Production"
   - Project: "CoreServices"
   - Owner: "PlatformTeam"

### Dynamic Resource Handling

For this specific use case, dynamic resource fetching isn't required since:
- We're not launching any instances
- We're not creating any infrastructure
- The Resource Group will discover resources based on existing tags

However, if we were to extend this template to include resource creation, we would use:
- SSM Parameter Store for AMI IDs: `/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`
- `Fn::GetAZs` and `Fn::Select` for Availability Zones

## Template Structure Plan

### Parameters Section
```yaml
Parameters:
  ResourceGroupName:
    Type: String
    Default: ProductionResources
    Description: Name for the resource group
  
  TagKey:
    Type: String
    Default: Environment
    Description: Tag key to filter resources
  
  TagValue:
    Type: String
    Default: Production
    Description: Tag value to filter resources
```

### Resources Section
```yaml
Resources:
  ProductionResourceGroup:
    Type: AWS::ResourceGroups::Group
    Properties:
      Name: !Ref ResourceGroupName
      ResourceQuery:
        Type: TAG_FILTERS_1_0
        Query:
          TagFilters:
            - Key: !Ref TagKey
              Values:
                - !Ref TagValue
```

### Metadata and Description
Include clear documentation about:
- Purpose of the template
- How to modify tag filters
- Default values and their rationale

## Security Considerations

Since this template only creates a Resource Group:
- No IAM roles needed for basic functionality
- No security groups or network configurations required
- Resource Group permissions will be inherited from the deploying user/role

## Validation Strategy

The template will be validated using:
- AWS CloudFormation linter (cfn-lint) for syntax compliance
- AWS CloudFormation validation for schema compliance
- Manual review of tag filter logic

## Extension Possibilities

This basic template can be extended to:
- Include multiple tag filters for complex resource selection
- Add resource group tags for management purposes
- Create IAM roles for resource group-based permissions
- Integrate with AWS Config for compliance monitoring

The architecture is minimal and focused, adhering to AWS best practices by avoiding hardcoded values and using safe defaults that allow immediate deployment without manual parameter input.
