# ECS Fargate Task Definition Deployment Plan

## Key AWS Services Required
1. **AWS ECS (Elastic Container Service)** - Core service for running containerized applications
2. **AWS IAM (Identity and Access Management)** - For creating task execution and task roles with proper permissions
3. **AWS CloudWatch Logs** - For centralized logging with retention policies
4. **AWS SSM (Systems Manager)** - For session management capabilities
5. **AWS ECR (Elastic Container Registry)** - For container image storage (though using public NGINX in this case)

## Architecture Overview

The deployment will create:
- An ECS Fargate task definition with two IAM roles
- A task execution role for ECR access and logging
- A task role for SSM session management
- A container definition using public NGINX image
- CloudWatch Log Group with 7-day retention

## Detailed Implementation Plan

### 1. IAM Roles and Policies

**Task Execution Role:**
- Trust relationship allowing ECS to assume the role
- Permissions:
  - `AmazonECSTaskExecutionRolePolicy` (AWS managed policy)
  - `logs:CreateLogStream`, `logs:PutLogEvents` (for CloudWatch Logs)

**Task Role:**
- Trust relationship allowing ECS to assume the role
- Permissions:
  - `AmazonSSMManagedInstanceCore` (for SSM session management)
  - `logs:CreateLogStream`, `logs:PutLogEvents` (for CloudWatch Logs)

### 2. CloudWatch Log Group

- Name: `ecs-demo-log-group`
- RetentionInDays: 7 (as specified)
- No KMS key specified (default AWS-managed encryption)

### 3. Task Definition

**Properties:**
- Family: `ecs-demo-task-definition`
- NetworkMode: `awsvpc` (required for Fargate)
- RequiresCompatibilities: `FARGATE`
- Cpu: `256` (minimum for Fargate)
- Memory: `512` (minimum for Fargate)
- ExecutionRoleArn: Reference to task execution role
- TaskRoleArn: Reference to task role
- ContainerDefinitions:
  - Name: `nginx-container`
  - Image: `nginx:latest` (public image as specified)
  - PortMappings:
    - ContainerPort: 80
  - LogConfiguration:
    - LogDriver: `awslogs`
    - Options:
      - `awslogs-group`: Reference to CloudWatch Log Group
      - `awslogs-region`: `!Ref AWS::Region`
      - `awslogs-stream-prefix`: `ecs`

### 4. Best Practices Implementation

**Dynamic References:**
- No hardcoded AMIs needed (no EC2 instances)
- No hardcoded AZs needed (Fargate is serverless)
- Region references using `!Ref AWS::Region`

**Security:**
- Principle of least privilege for IAM roles
- No hardcoded secrets (none required for this demo)
- Public NGINX image is acceptable for demonstration

**Resource Protection:**
- No stateful resources requiring DeletionPolicy/UpdateReplacePolicy
- All resources are ephemeral or managed by AWS

**Linting Compliance:**
- All intrinsic functions properly formatted
- No hallucinated properties
- Proper use of AWS resource type names

## Missing Information & Safe Defaults

The following properties were not specified and will use AWS defaults:

1. **VPC/Network Configuration:** Not required for Fargate with awsvpc network mode
2. **Subnets:** Not required for this basic task definition
3. **Security Groups:** Not specified - will be omitted (can be added later if needed)
4. **Task CPU/Memory:** Using minimum Fargate values (256 CPU, 512 Memory)
5. **Log Group Name:** Using `ecs-demo-log-group` as default
6. **Task Definition Family:** Using `ecs-demo-task-definition` as default

## Dependencies

1. IAM roles must be created before the task definition
2. CloudWatch Log Group must exist before the task definition (handled by CloudFormation's dependency management)
3. No external dependencies like VPCs or gateways needed for this serverless Fargate deployment

- AWS CloudFormation linter (cfn-lint) for syntax and best practices
