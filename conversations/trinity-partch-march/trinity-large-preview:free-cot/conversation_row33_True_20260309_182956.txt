## Architecture Plan

### IAM Role Structure

The template will create two IAM roles with the following structure:

1. **Role 1**: `EC2ReadOnlyIAMRole1`
   - Trusted entity: ec2.amazonaws.com
   - Attached policy: EC2ReadOnlyIAMPolicy

2. **Role 2**: `EC2ReadOnlyIAMRole2`
   - Trusted entity: ec2.amazonaws.com
   - Attached policy: EC2ReadOnlyIAMPolicy

### Managed Policy Permissions

The managed policy will grant read-only access to IAM resources with the following permissions:

- `iam:Get*`: All Get actions (e.g., GetRole, GetPolicy, GetUser)
- `iam:List*`: All List actions (e.g., ListRoles, ListPolicies, ListUsers)

These permissions will be scoped to all IAM resources using the wildcard `*` resource.

### Trust Relationship

Both roles will have a trust relationship that allows the EC2 service to assume them:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

### Policy Document

The managed policy will have the following structure:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "iam:Get*",
        "iam:List*"
      ],
      "Resource": "*"
    }
  ]
}
```

### CloudFormation Resource Mapping

The template will use the following AWS CloudFormation resources:

1. **AWS::IAM::Role** (x2): For creating the two EC2-trusted roles
2. **AWS::IAM::ManagedPolicy**: For creating the read-only IAM policy
3. **AWS::IAM::Policy**: For attaching the managed policy to each role

### Best Practices Compliance

This template will adhere to all the critical AWS best practices and linting rules:

- No hardcoded values (except safe defaults for names)
- No deprecated runtimes (not applicable for IAM)
- No legacy S3 properties (not applicable for IAM)
- Proper use of AWS managed policy ARNs and resource type names
- No use of Fn::Sub where not needed
- No hallucinated properties

The final template will be a minimal, focused IAM template that provides exactly what the business need requires: two EC2-trusted roles with read-only IAM access.
