# Risk Management System - CloudFormation Architecture Plan

## 1. Key AWS Services Required

Based on the business need, the following AWS services are required:

- **AWS Lambda** - For serverless compute to process risk management logic
- **Amazon DynamoDB** - For storing workload and risk-related data with composite keys
- **AWS IAM** - For creating the execution role with necessary permissions
- **Amazon SNS** - For notifications and alerts
- **AWS Systems Manager (SSM) Parameter Store** - For dynamic AMI fetching (future-proofing)
- **AWS Secrets Manager** - For secure credential storage (future-proofing)

## 2. Properties and Dependencies

### Critical Dependencies:
1. IAM Role must be created before Lambda functions can reference it
2. DynamoDB table must exist before Lambda functions can be created (due to IAM permissions)
3. SNS Topic must be created before Lambda functions can publish to it
4. Lambda functions must be created before any triggers or permissions can be attached

### Property Requirements:
- DynamoDB table requires a primary key schema (partition key: workload_id, sort key: question_id)
- Lambda functions need appropriate IAM permissions for DynamoDB operations
- SNS Topic requires a policy for SSM OpsCenter access
- Lambda functions need appropriate runtime versions and timeout configurations

## 3. Safe AWS Defaults

Since specific properties are not provided, I'll use these safe defaults:

- **Region**: `us-east-1` (safe default for most deployments)
- **DynamoDB Table Read/Write Capacity**: `5` (on-demand scaling not available in all regions for older templates)
- **Lambda Timeout**: `30` seconds (default safe value)
- **Lambda Memory**: `128` MB (minimum safe value)
- **SNS Topic Display Name**: `RiskManagementNotifications`
- **VPC**: No VPC required (Lambda functions can run in default VPC)
- **Subnet**: Not required for this configuration

## 4. Dynamic Resource Fetching Strategy

### AMI Fetching (Future-Proofing):
Although Lambda functions don't require AMIs, if we need to provision EC2 instances in the future, we'll use:
```yaml
Fn::ImportValue: !Sub "arn:${AWS::Partition}:ssm:${AWS::Region}:amazon/linux/latest/amzn2-ami-hvm-x86_64-gp2"
```

### Availability Zones:
```yaml
Fn::GetAZs: !Ref "AWS::Region"
```

## 5. Detailed Architecture Plan

### IAM Role Structure:
- Create an IAM role with the following policies:
  - AWSLambdaBasicExecutionRole (managed policy)
  - Custom inline policy for DynamoDB access (PutItem, GetItem, UpdateItem, DeleteItem)
  - Custom inline policy for SNS publish access

### DynamoDB Table Design:
- Table Name: `RiskManagementData`
- Partition Key: `workload_id` (String)
- Sort Key: `question_id` (String)
- Billing Mode: `PAY_PER_REQUEST` (on-demand scaling)

### Lambda Functions:
1. **WorkloadProcessor Lambda**:
   - Runtime: `python3.12`
   - Handler: `workload_processor.lambda_handler`
   - Permissions: DynamoDB write, SNS publish
   - Environment Variables: DYNAMODB_TABLE, SNS_TOPIC_ARN

2. **RiskEvaluator Lambda**:
   - Runtime: `python3.12`
   - Handler: `risk_evaluator.lambda_handler`
   - Permissions: DynamoDB read/write, SNS publish
   - Environment Variables: DYNAMODB_TABLE, SNS_TOPIC_ARN

### SNS Topic Configuration:
- Topic Name: `RiskManagementAlerts`
- Display Name: `Risk Management System Notifications`
- Policy: Allow SSM OpsCenter to publish alerts
- Subscription: Email endpoint (using Parameter with default dummy value)

### Security Considerations:
- Use AWS Secrets Manager for any sensitive configuration
- Implement least privilege IAM policies
- Enable CloudWatch Logs for Lambda functions
- Use environment variables for configuration (not hardcoded values)

### Error Handling:
- Implement retry logic in Lambda functions for DynamoDB operations
- Set up dead-letter queues for failed Lambda invocations
- Configure CloudWatch alarms for function errors

This architecture ensures a robust, scalable, and secure risk management system that meets the business requirements while following AWS best practices and CloudFormation linting rules.
