# AWS CloudFormation Architecture Plan for Windows EC2 CPU Stress Testing

## Key AWS Services Required

1. **EC2** - Windows Server 2019 instance for CPU stress testing
2. **IAM** - Service roles and instance profiles for SSM and FIS integration
3. **SSM (Systems Manager)** - For remote command execution and custom document management
4. **VPC** - Virtual network infrastructure (with fallback to default VPC)
5. **CloudWatch Logs** - For logging SSM command execution and stress test results
6. **FIS (Fault Injection Simulator)** - For orchestrating the stress testing workflow
7. **SSM Parameter Store** - For dynamic AMI retrieval

## Architecture Overview

The deployment will create a self-contained stack with:
- A Windows Server 2019 EC2 instance with SSM agent pre-installed
- IAM roles for both SSM access and FIS integration
- A custom SSM document containing PowerShell CPU stress script
- Optional VPC infrastructure with fallback to default VPC
- CloudWatch log group for monitoring

## Detailed Resource Planning

### 1. Parameters with Safe Defaults

**AMI Parameter:**
- Use SSM Parameter Store to fetch latest Windows Server 2019 AMI
- Parameter: `AMIId` with default: `resolve:ssm:/aws/service/ami-windows-latest/Windows_Server-2019-English-Full-Base`

**Instance Type:**
- Parameter: `InstanceType` with default: `t3.nano`
- Allowed values: `t3.nano`, `t3.micro`, `t3.small`, `t3.medium`, `t3.large`

**Subnet Configuration:**
- Parameter: `UseDefaultVPC` (Boolean) with default: `false`
- If `false`, require `SubnetId` parameter with default: `null` (user must provide)
- If `true`, create resources in default VPC

**Key Pair:**
- Parameter: `KeyName` with default: `default-key` (user must create this key pair)

**Stress Test Duration:**
- Parameter: `StressDurationSeconds` with default: `300` (5 minutes)
- Min: 60, Max: 3600

### 2. IAM Role Design

**SSM Role for EC2 Instance:**
- Permissions: `AmazonSSMManagedInstanceCore` managed policy
- Allows SSM agent to communicate with AWS Systems Manager

**FIS Integration Role:**
- Trust relationship: `fault-injection-simulator.amazonaws.com`
- Permissions:
  - `ssm:SendCommand` - Execute SSM commands
  - `logs:CreateLogGroup`, `logs:CreateLogStream`, `logs:PutLogEvents` - CloudWatch logging
  - `ec2:DescribeInstances` - Instance discovery
  - `ssm:GetDocument`, `ssm:ListCommands`, `ssm:ListCommandInvocations` - SSM operations

### 3. SSM Document Creation

**Document Type:** Automation
**Content:** PowerShell script that:
- Detects logical processors using `Get-WmiObject -Class Win32_Processor`
- Spawns background threads equal to processor count
- Each thread runs CPU-intensive calculation for specified duration
- Logs start/end times and processor count to CloudWatch

**Document Schema:**
```yaml
Type: AWS::SSM::Document
Properties:
  Name: CPUStressTest
  DocumentType: Automation
  Content:
    schemaVersion: '0.3'
    description: CPU Stress Test for Windows
    parameters:
      Duration:
        type: String
        default: '300'
    mainSteps:
      - name: ExecuteStressTest
        action: aws:runPowerShellScript
        inputs:
          runCommand:
            - # PowerShell script here
```

### 4. VPC Fallback Logic

**Conditional Resource Creation:**
- If `UseDefaultVPC` is `true`:
  - Use `AWS::EC2::VPC::DEFAULT` (no explicit VPC creation)
  - Use default security group
- If `UseDefaultVPC` is `false`:
  - Create new VPC with CIDR: `10.0.0.0/16`
  - Create public subnet with CIDR: `10.0.0.0/24`
  - Create Internet Gateway and attach to VPC
  - Create route table with default route to IGW
  - Create security group allowing RDP (port 3389) from 0.0.0.0/0

### 5. EC2 Instance Configuration

**Properties:**
- ImageId: SSM-resolved Windows AMI
- InstanceType: Parameter value
- KeyName: Parameter value
- Network configuration based on VPC logic
- IAM instance profile: SSM role
- UserData: Configure SSM agent if needed (though Windows AMIs typically have it pre-installed)

**Logical Processor Detection:**
- Script will use PowerShell: `(Get-WmiObject -Class Win32_Processor).NumberOfLogicalProcessors`

### 6. CloudWatch Log Group

**Purpose:** Capture stress test execution logs
**Retention:** 30 days (configurable parameter)
**Resource Policy:** Allow SSM service to write logs

### 7. Dependencies and Resource Ordering

**Critical Dependencies:**
1. VPC and subnet must exist before EC2 creation
2. IAM roles must be created before instance profile
3. Security group must exist before EC2 creation
4. CloudWatch log group should be created before SSM document execution

**Resource References:**
- Use `!Ref` for cross-resource references
- Use `!GetAtt` for attributes like VPC IDs and security group IDs
- Use `!Sub` for constructing ARNs and resource names

### 8. Error Handling and Validation

**Parameter Validation:**
- Instance type validation against allowed values
- Duration validation (60-3600 seconds)
- Subnet validation if not using default VPC

**Resource Validation:**
- Ensure AMI resolution succeeds before instance creation
- Validate security group rules are correctly applied
- Confirm IAM role trust relationships are properly configured

### 9. Security Considerations

**Network Security:**
- Restrict RDP access to specific IP ranges (parameterizable)
- Use security group instead of NACLs for instance-level control
- Consider VPN or bastion host for production environments

**IAM Security:**
- Principle of least privilege for all roles
- No wildcard permissions in IAM policies
- Use managed policies where appropriate

**Data Protection:**
- No sensitive data hardcoded in templates
- Use secure parameters for any credentials
- Enable CloudWatch log encryption if required

### 10. Template Structure

**Logical Organization:**
1. Parameters section with all configurable options
2. Mappings section (if needed for region-specific configurations)
3. Resources section with all AWS resources
4. Outputs section (optional - instance ID, public IP, etc.)

**Naming Convention:**
- Prefix all resources with stack name for uniqueness
- Use descriptive names for easy identification
- Include environment identifier if multi-environment deployment

This architecture ensures a robust, secure, and maintainable CloudFormation template that meets all business requirements while adhering to AWS best practices and avoiding common deployment failures.
