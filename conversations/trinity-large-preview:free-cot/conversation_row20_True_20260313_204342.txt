# AWS Client VPN Monitoring Architecture Plan

## Key AWS Services Required

1. **AWS CloudWatch** - For dashboard creation and metrics aggregation
2. **AWS CloudWatch Logs** - For storing and querying Client VPN logs
3. **AWS Systems Manager (SSM) Parameter Store** - For dynamic resource fetching
4. **AWS IAM** - For service permissions and roles
5. **AWS CloudFormation** - For infrastructure as code deployment

## Architecture Overview

The solution will create a CloudWatch Dashboard with pre-configured Logs Insights queries to monitor Client VPN usage patterns. The architecture follows a modular approach with separate components for dashboard creation, query definitions, and data visualization.

## Component Breakdown

### 1. CloudWatch Dashboard Configuration

**Properties:**
- Dashboard name: `ClientVPNAnalyticsDashboard`
- Dashboard body: JSON structure containing widgets for:
  - Connection duration metrics
  - Data transfer volumes (ingress/egress)
  - User activity heatmaps
  - Authentication method breakdowns

**Dependencies:**
- CloudWatch Logs Insights queries must exist before dashboard creation
- IAM permissions for CloudWatch full access

### 2. CloudWatch Logs Insights Query Definitions

**Query Categories:**
- **Connection Analytics**: Duration, frequency, endpoint distribution
- **Data Transfer**: Ingress/egress volumes, bandwidth usage
- **User Activity**: Distinct users, login patterns, session counts
- **Authentication Analysis**: AD/SAML vs Mutual Auth vs Mixed Auth breakdown

**Query Structure:**
```yaml
Type: AWS::Logs::QueryDefinition
Properties:
  Name: ConnectionDurationQuery
  LogGroupNames:
    - /aws/vpn/logs
  QueryString: |
    fields @timestamp, @message
    | filter @message like /connection/
    | stats count(*) as TotalConnections, avg(duration) as AvgDuration
    | sort by TotalConnections desc
```

### 3. Dynamic Resource Fetching Strategy

**SSM Parameters for Latest Resources:**
- Use `resolve:ssm:/aws/service/cloudwatch/cloudwatch-agent/amazon-linux-2` for any agent-related resources
- Use `resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2` for any EC2 instances (if needed)

**Availability Zones:**
- Use `Fn::GetAZs: ""` to dynamically fetch available AZs
- Use `Fn::Select` to pick specific AZs from the list

## Properties and Dependencies

### CloudWatch Dashboard Dependencies
1. **IAM Role**: CloudWatchFullAccess for dashboard creation
2. **Log Groups**: Client VPN logs must exist in `/aws/vpn/logs`
3. **Query Definitions**: All Logs Insights queries must be created first

### Logs Insights Queries Dependencies
1. **Log Group**: `/aws/vpn/logs` must exist
2. **IAM Permissions**: Logs:QueryLogsInsights for query execution

## Safe AWS Defaults

### Missing Properties Handling

**CIDR Blocks:**
- Default VPC CIDR: `10.0.0.0/16`
- Default Subnet CIDR: `10.0.0.0/24`

**Instance Types:**
- Default: `t3.micro` (if any EC2 instances are needed)

**Key Pairs:**
- Default: `default-key` (with warning if not found)

**Region:**
- Default: `us-east-1` (with parameter option)

## Security and Permissions

### IAM Role Configuration
```yaml
Resources:
  CloudWatchDashboardRole:
    Type: AWS::IAM::Role
    Properties:
      AssumeRolePolicyDocument:
        Version: "2012-10-17"
        Statement:
          - Effect: Allow
            Principal:
              Service: cloudformation.amazonaws.com
            Action: sts:AssumeRole
      ManagedPolicyArns:
        - arn:aws:iam::aws:policy/CloudWatchFullAccess
        - arn:aws:iam::aws:policy/CloudWatchLogsReadOnlyAccess
```

### Query Execution Permissions
- Use `AWS::Logs::QueryDefinition` with appropriate IAM permissions
- Ensure logs are accessible in the specified log group

## Query Organization Strategy

### Folder Structure (Logical Organization)
1. **Connection Analytics**
   - ConnectionDurationQuery
   - ConnectionFrequencyQuery
   - EndpointDistributionQuery

2. **Data Transfer**
   - IngressVolumeQuery
   - EgressVolumeQuery
   - BandwidthUsageQuery

3. **User Activity**
   - DistinctUsersQuery
   - LoginPatternsQuery
   - SessionCountsQuery

4. **Authentication Analysis**
   - AD_SAML_BreakdownQuery
   - MutualAuthBreakdownQuery
   - MixedAuthBreakdownQuery

## Dashboard Visualization Strategy

### Widget Types
- **Tables**: For endpoint-specific usage statistics
- **Bar Charts**: For authentication method comparisons
- **Line Graphs**: For connection duration trends over time
- **Heat Maps**: For user activity patterns

### Dashboard Layout
- 12-column grid system
- Responsive design for different screen sizes
- Logical grouping of related metrics

## Error Handling and Validation

### CloudFormation Best Practices
1. **Deletion Policies**: Set to `Retain` for dashboard and query definitions
2. **Update Replace Policies**: Match deletion policies for consistency
3. **Resource Validation**: Ensure all referenced resources exist before creation

### Query Validation
- Test queries against sample log data
- Include error handling for missing log groups
- Provide default values for query parameters

## Deployment Strategy

### Sequential Resource Creation
1. Create IAM roles and permissions
2. Create CloudWatch Logs Insights queries
3. Create CloudWatch Dashboard
4. Validate dashboard functionality

