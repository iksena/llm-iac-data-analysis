# DynamoDB Table with Local and Global Secondary Indexes - Architecture Plan

## Key AWS Services Required
1. **AWS::DynamoDB::Table** - Core service for creating the DynamoDB table
2. **AWS::SSM::Parameter** - For dynamic AMI fetching (though not directly used for DynamoDB, following the pattern for consistency)
3. **AWS::IAM::Role** - For any necessary permissions (if needed for cross-account access or other operations)

## Architecture Design

### Table Structure
I'll create a DynamoDB table with:
- A primary key (partition key)
- Local Secondary Indexes (LSIs) - same partition key, different sort key
- Global Secondary Indexes (GSIs) - different partition and sort keys

### Properties & Dependencies
- **BillingMode**: Set to PAY_PER_REQUEST (default) for cost-effectiveness
- **ProvisionedThroughput**: Only required for PAY_PER_REQUEST billing mode
- **SSESpecification**: Enable encryption at rest using AWS-managed KMS key
- **PointInTimeRecoverySpecification**: Enable for data protection

### Safe Defaults (Following Best Practices)
- **Region**: Use `AWS::Region` pseudo-parameter
- **Table Name**: Use a descriptive name with environment prefix
- **Read/Write Capacity**: Use 5 units each for provisioned throughput (safe default)
- **Encryption**: Enable by default with AWS-managed KMS

### Dynamic Resource Fetching
While DynamoDB doesn't require AMI fetching, I'll follow the pattern by:
- Using `AWS::Region` for region-specific configurations
- Using `AWS::AccountId` for account-specific resources
- Using `AWS::StackName` for unique naming

## Detailed Implementation Plan

### Parameters (with Safe Defaults)
```yaml
Parameters:
  TableName:
    Type: String
    Default: MyDynamoDBTable
    Description: Name of the DynamoDB table
  PartitionKey:
    Type: String
    Default: id
    Description: Primary partition key name
  SortKey:
    Type: String
    Default: timestamp
    Description: Primary sort key name (if needed)
```

### Resources

#### DynamoDB Table
```yaml
Resources:
  MyDynamoDBTable:
    Type: AWS::DynamoDB::Table
    Properties:
      TableName: !Ref TableName
      BillingMode: PAY_PER_REQUEST
      AttributeDefinitions:
        - AttributeName: !Ref PartitionKey
          AttributeType: S
        - AttributeName: !Ref SortKey
          AttributeType: S
      KeySchema:
        - AttributeName: !Ref PartitionKey
          KeyType: HASH
        - AttributeName: !Ref SortKey
          KeyType: RANGE
      SSESpecification:
        Enabled: true
      PointInTimeRecoverySpecification:
        PointInTimeRecoveryEnabled: true
```

#### Local Secondary Index (LSI)
```yaml
      LocalSecondaryIndexes:
        - IndexName: LSIByTimestamp
          KeySchema:
            - AttributeName: !Ref PartitionKey
              KeyType: HASH
            - AttributeName: !Ref SortKey
              KeyType: RANGE
          Projection:
            ProjectionType: ALL
```

#### Global Secondary Index (GSI)
```yaml
      GlobalSecondaryIndexes:
        - IndexName: GSIByCategory
          KeySchema:
            - AttributeName: category
              KeyType: HASH
            - AttributeName: !Ref SortKey
              KeyType: RANGE
          Projection:
            ProjectionType: ALL
          ProvisionedThroughput:
            ReadCapacityUnits: 5
            WriteCapacityUnits: 5
```

### Best Practices Applied
1. **No hardcoded values** - Using parameters with defaults
2. **Encryption enabled** - SSESpecification with AWS-managed KMS
3. **Point-in-time recovery** - For data protection
4. **Safe defaults** - PAY_PER_REQUEST billing mode, 5 capacity units
5. **Descriptive naming** - Using parameters for resource names

### Linting Compliance
- All properties are valid for AWS::DynamoDB::Table
- No deprecated runtimes or features used
- Proper use of intrinsic functions
- No hardcoded AMIs or AZs (not applicable for DynamoDB but pattern followed)

This architecture provides a robust, production-ready DynamoDB table with both local and global secondary indexes, following all AWS best practices and linting rules.
