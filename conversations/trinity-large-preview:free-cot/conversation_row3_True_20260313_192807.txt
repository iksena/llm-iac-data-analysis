# Detailed Architectural Deployment Plan for S3 Website Hosting with DeletionPolicy

## 1. Key AWS Services Required

Based on the business need, the following AWS services are required:

- **AWS::S3::Bucket** - Core service for creating the S3 bucket
- **AWS::S3::BucketPolicy** - For configuring public access to enable website hosting
- **AWS::CloudFormation::WaitCondition** - Optional, for ensuring bucket creation completes before proceeding

## 2. Properties and Dependencies

The S3 bucket for website hosting requires the following properties:

- **BucketName** - Unique identifier for the bucket (will use a combination of stack name and timestamp)
- **AccessControl** - Set to "PublicRead" to enable website access (though we'll use BucketPolicy instead)
- **WebsiteConfiguration** - Contains IndexDocument and ErrorDocument for static website hosting
- **DeletionPolicy** - Set to "Retain" to prevent accidental deletion
- **UpdateReplacePolicy** - Must match DeletionPolicy for linting compliance

Dependencies are minimal since S3 is a standalone service, but the bucket must be fully created before applying the BucketPolicy.

## 3. Safe AWS Defaults

Since specific properties weren't provided, I'll use these safe defaults:

- **BucketName**: `${AWS::StackName}-${AWS::Region}-${AWS::AccountId}-${Timestamp}` - Ensures uniqueness
- **IndexDocument**: "index.html" - Standard default for web hosting
- **ErrorDocument**: "error.html" - Standard error page
- **Region**: `${AWS::Region}` - Uses the stack's region
- **Timestamp**: `${AWS::NoValue}` with a custom resource or timestamp function - Ensures unique bucket names

## 4. Dynamic Resource Fetching

For this S3-only template, dynamic fetching isn't strictly necessary since we're not using AMIs or other dynamic resources. However, I'll demonstrate best practices by:

- Using `${AWS::Region}` instead of hardcoding region names
- Using `${AWS::AccountId}` for resource identification
- Using `${AWS::StackName}` for naming consistency

## 5. Architecture Plan

The template will follow this structure:

1. **Parameters Section**: Define safe defaults for any configurable properties
2. **Resources Section**:
   - Create S3 bucket with website configuration
   - Apply DeletionPolicy and UpdateReplacePolicy
   - Create BucketPolicy for public access
3. **Outputs Section**: Return the website URL and bucket name

## 6. Security Considerations

- The BucketPolicy will grant public read access only to the website objects
- No IAM roles or policies are needed for basic website hosting
- The DeletionPolicy ensures data retention even if the stack is deleted

## 7. Compliance with Best Practices

The template will adhere to all specified linting rules:

- ✅ All Parameters will have safe Default values
- ✅ No hardcoded AMIs or AZs (not applicable for S3)
- ✅ Will use BucketPolicy instead of AccessControl property
- ✅ Will include both DeletionPolicy and UpdateReplacePolicy
- ✅ No unnecessary Fn::Sub usage
- ✅ No hallucinated properties
- ✅ Uses safe defaults and dynamic references where appropriate
