# EventBridge Pipes SQS to API Enrichment Architecture Plan

## Key AWS Services Required

1. **Amazon EventBridge Pipes** - Core workflow orchestration
2. **Amazon SQS** - Event source queue
3. **Amazon CloudWatch Logs** - Target for enriched data
4. **AWS IAM** - Service permissions and role management
5. **AWS Systems Manager (SSM) Parameter Store** - Dynamic resource resolution (if needed)

## Architecture Overview

The EventBridge Pipe will create an automated data enrichment pipeline:
- **Source**: SQS queue receives messages containing zip codes
- **Transformation**: EventBridge Pipes fetches location data from external API
- **Target**: Enriched data logged to CloudWatch Logs
- **Permissions**: IAM role grants necessary access to all services

## Detailed Resource Properties & Dependencies

### 1. SQS Queue (Event Source)
- **Type**: Standard queue (default)
- **Properties**:
  - QueueName: `EventBridgePipeSourceQueue`
  - MessageRetentionPeriod: 345600 (4 days default)
  - VisibilityTimeout: 30 seconds (default)
  - MaximumMessageSize: 262144 bytes (256KB default)

### 2. CloudWatch Log Group (Target)
- **Type**: Log group for enriched data storage
- **Properties**:
  - LogGroupName: `/aws/eventbridge/pipes/enriched-data`
  - RetentionInDays: 30 (default, can be adjusted)

### 3. IAM Role for EventBridge Pipes
- **Trust Policy**: Allows `events.amazonaws.com` to assume role
- **Permissions**:
  - SQS: `sqs:ReceiveMessage`, `sqs:DeleteMessage`
  - CloudWatch Logs: `logs:CreateLogStream`, `logs:PutLogEvents`
  - EventBridge Pipes: `pipes:*` (if needed for internal operations)

### 4. EventBridge Pipe Configuration
- **Source**: SQS queue ARN
- **Enrichment**: API destination invocation
- **Target**: CloudWatch Log Group
- **Transformation**: Dynamic data appending using API response

## Dynamic Resource Resolution Strategy

### API Destination Configuration
- **Endpoint**: `https://api.zippopotam.us/us/{zipcode}`
- **Method**: GET
- **Path Parameter**: `{zipcode}` extracted from SQS message body
- **Connection**: HTTP connection (no VPC required for public API)

### Message Transformation Logic
The pipe will:
1. Read SQS message containing JSON with `zipcode` field
2. Extract zipcode value
3. Call API with dynamic path parameter
4. Append API response to original message
5. Forward enriched data to CloudWatch Logs

## Safe Defaults & Missing Values

### Parameters (if needed)
- **Default Values**: Use safe defaults for any optional parameters
- **Allowed Patterns**: Ensure defaults match patterns if specified

### Resource Defaults
- **SQS**: Standard queue with default retention
- **Log Group**: 30-day retention (standard)
- **IAM Role**: Least-privilege permissions only

## Security & Best Practices Implementation

### IAM Role Design
- **Principle of Least Privilege**: Only grant required permissions
- **Trust Relationship**: Restrict to EventBridge service only
- **No Hardcoded Secrets**: Use secure parameters if credentials needed

### API Security
- **Public API**: No authentication required for zippopotam.us
- **Data Validation**: Ensure zipcode format validation before API call
- **Error Handling**: Log failures without exposing sensitive data

## Dependencies & Deployment Order

1. **IAM Role** - Must exist before pipe creation
2. **SQS Queue** - Event source for pipe
3. **CloudWatch Log Group** - Target for enriched data
4. **EventBridge Pipe** - Final resource connecting all components

## Error Handling & Monitoring

### Failure Scenarios
- **API Unreachable**: Log error, forward original message
- **Invalid Zipcode**: Log validation error, skip enrichment
- **SQS Issues**: Standard retry mechanisms apply
- **IAM Permissions**: Monitor for denied operations

### Monitoring Strategy
- **CloudWatch Metrics**: SQS queue depth, pipe invocation counts
- **Log Analysis**: Search enriched data patterns in log group
- **Error Tracking**: Monitor for enrichment failures

## Scalability Considerations

### EventBridge Pipes
- **Auto-scaling**: Built-in scaling based on SQS message volume
- **Concurrency**: Default concurrency limits apply
- **Throttling**: API rate limits respected automatically

### SQS Queue
- **Message Volume**: Standard queue handles high throughput
- **DLQ**: Consider dead-letter queue for failed messages
- **Visibility**: Default timeout sufficient for API calls

This architecture provides a robust, automated data enrichment pipeline using managed AWS services with proper security, error handling, and monitoring capabilities.
