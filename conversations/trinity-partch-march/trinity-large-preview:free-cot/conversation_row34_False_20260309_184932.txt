# Detailed Architectural Deployment Plan for MySQL RDS with Provisioned IOPS

## 1. Key AWS Services Required

Based on the business requirements, the following AWS services are essential:

- **AWS::EC2::VPC** - To create the isolated network environment
- **AWS::EC2::Subnet** (x2) - Two private subnets in different AZs for high availability
- **AWS::EC2::InternetGateway** - For VPC internet connectivity
- **AWS::EC2::VPCGatewayAttachment** - To attach the Internet Gateway to the VPC
- **AWS::EC2::RouteTable** - For routing traffic within the VPC
- **AWS::EC2::Route** - To define routes (including the default route to the Internet Gateway)
- **AWS::EC2::SubnetRouteTableAssociation** (x2) - To associate subnets with the route table
- **AWS::RDS::DBSubnetGroup** - To group the private subnets for RDS
- **AWS::EC2::SecurityGroup** - To control network access to the RDS instance
- **AWS::RDS::DBInstance** - The MySQL RDS database with Provisioned IOPS
- **AWS::RDS::DBParameterGroup** - Optional, for MySQL-specific configuration

## 2. Properties and Dependencies

### Critical Dependencies:
1. **VPC must be created before** any subnets, security groups, or route tables
2. **Internet Gateway must be attached to VPC** before creating routes that use it
3. **DBSubnetGroup depends on subnets being available**
4. **DBInstance depends on DBSubnetGroup and SecurityGroup**

### Property Dependencies:
- Subnet CIDR blocks must be unique within the VPC
- Security Group must reference the VPC ID
- DBInstance must reference the DBSubnetGroup and SecurityGroup

## 3. Safe AWS Defaults (for missing properties)

Since specific properties like CIDR blocks aren't provided, I'll use these safe defaults:

- **VPC CIDR**: `10.0.0.0/16` (standard private range)
- **Subnet CIDRs**: 
  - Subnet 1: `10.0.1.0/24` (AZ1)
  - Subnet 2: `10.0.2.0/24` (AZ2)
- **DB Instance Class**: `db.r5.large` (as specified)
- **Storage Type**: `io1` (Provisioned IOPS as specified)
- **Allocated Storage**: `100` GB (reasonable default for production)
- **IOPS**: `1000` (standard for db.r5.large with io1)
- **Engine**: `mysql`
- **Engine Version**: `8.0` (latest stable)
- **Master Username**: `admin` (default, though should be parameterized)
- **Master Password**: Will use dynamic reference to Secrets Manager (best practice)

## 4. Dynamic Resource Fetching Strategy

### AMI Fetching:
For this RDS deployment, we don't need to fetch AMIs since RDS manages the database instance's OS. However, if we were deploying EC2 instances for application layers, we would use:

```yaml
Fn::ImportValue: !Sub "resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2"
```

### Availability Zones:
Instead of hardcoding AZs, we'll use:

```yaml
!GetAZs: !Ref "AWS::Region"
```

Then select two AZs dynamically:

```yaml
!Select: [0, !GetAZs: !Ref "AWS::Region"]
!Select: [1, !GetAZs: !Ref "AWS::Region"]
```

## 5. Security Considerations

- **Password Management**: Use AWS Secrets Manager with dynamic reference instead of plain text parameters
- **Network Isolation**: All resources in private subnets with no public IP assignment
- **Security Group**: Restrict MySQL port (3306) to internal VPC CIDR only
- **IAM Permissions**: Minimal required for RDS operations

## 6. Template Structure Plan

The CloudFormation template will follow this logical structure:

1. **Parameters** (with safe defaults)
2. **Resources** in dependency order:
   - VPC and networking components
   - Security Group
   - DB Subnet Group
   - RDS Instance
3. **Outputs** (optional: endpoint, ARN, etc.)

## 7. Linting Compliance Strategy

To ensure cfn-lint compliance:

- All Parameters will have `Default` values matching their `AllowedPattern`
- No hardcoded AMI IDs or AZs
- No use of `Fn::Sub` without variables
- All stateful resources will have both `DeletionPolicy` and `UpdateReplacePolicy` set to `Snapshot`
- Security credentials will use dynamic references

## 8. High Availability & Disaster Recovery

The architecture inherently supports:
- **Multi-AZ Deployment**: By placing subnets in different AZs
- **Provisioned IOPS**: For consistent high-performance I/O
- **Network Isolation**: Through dedicated VPC and security groups

## 9. Cost Optimization Considerations

- **Instance Class**: db.r5.large is appropriately sized for moderate workloads
- **Storage**: 100GB with 1000 IOPS is cost-effective for many applications
- **Provisioned IOPS**: Only pay for what's allocated, not burst capacity

This plan ensures a production-ready, secure, and compliant MySQL RDS deployment that meets all specified requirements while following AWS best practices and CloudFormation linting rules.
