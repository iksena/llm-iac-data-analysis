# Event-Driven DynamoDB to EventBridge Integration Architecture Plan

## Key AWS Services Required

1. **DynamoDB** - To create the table with streaming enabled
2. **EventBridge** - For the centralized event bus and Pipes
3. **IAM** - For service roles and permissions
4. **Lambda** (optional, for demonstration) - To show downstream processing
5. **SSM Parameter Store** - For dynamic AMI fetching (if Lambda is included)

## Architecture Overview

The solution will create a DynamoDB table with change streams enabled, an EventBridge event bus, and two Pipes that filter and route specific DynamoDB stream events to the bus. This enables real-time event-driven processing of database changes.

## Detailed Resource Breakdown

### 1. DynamoDB Table with Streams

**Properties:**
- Table name: `OrdersTable`
- Primary key: `OrderId` (String)
- Stream specification: `NEW_AND_OLD_IMAGES` to capture full item state
- Billing mode: `PAY_PER_REQUEST` (on-demand capacity)

**Dependencies:** None

### 2. EventBridge Event Bus

**Properties:**
- Name: `OrdersEventBus`
- EventSourceName: `dynamodb.orders`

**Dependencies:** None

### 3. EventBridge Pipes (2 total)

**Pipe 1 - New Orders:**
- Source: DynamoDB stream from OrdersTable
- Filter: `eventName = INSERT`
- Target: OrdersEventBus
- Description: Routes new order creations

**Pipe 2 - Order Updates:**
- Source: DynamoDB stream from OrdersTable
- Filter: `eventName = MODIFY`
- Target: OrdersEventBus
- Description: Routes order modifications

**Dependencies:** Both pipes depend on DynamoDB table and EventBridge bus

### 4. IAM Roles and Policies

**Pipe Execution Role:**
- Trust policy: Allows EventBridge Pipes service
- Permissions: DynamoDB stream read access, EventBridge PutEvents

**Dependencies:** None

## Safe Defaults and Dynamic Resource Fetching

### Missing Properties (Safe Defaults)

**DynamoDB Table:**
- Primary key: `OrderId` (String) - chosen as safe default for orders
- StreamViewType: `NEW_AND_OLD_IMAGES` - provides complete change context
- BillingMode: `PAY_PER_REQUEST` - no capacity planning required

**EventBridge:**
- Event bus name: `OrdersEventBus` - descriptive and unique
- Pipe names: `NewOrdersPipe`, `OrderUpdatesPipe` - self-documenting

### Dynamic Resource Fetching

## Security and Best Practices Implementation

### IAM Policies
- Principle of least privilege for all roles
- No wildcard permissions
- Resource-level permissions for DynamoDB and EventBridge

### Resource Protection
- No stateful resources requiring DeletionPolicy/UpdateReplacePolicy
- All resources are ephemeral or managed by AWS

### Event Schema
- Define clear event patterns for INSERT and MODIFY operations
- Include source, detail-type, and relevant detail fields

## Dependencies and Deployment Order

1. IAM Role (no dependencies)
2. DynamoDB Table (no dependencies)
3. EventBridge Event Bus (no dependencies)
4. EventBridge Pipes (depend on IAM Role, DynamoDB Table, and Event Bus)

## Event Filtering Logic

**INSERT events:**
- `eventName = INSERT`
- `eventName = MODIFY` with `image.new.image.status = "CREATED"`

**MODIFY events:**
- `eventName = MODIFY`
- `image.old.image.status <> image.new.image.status`

## Error Handling and Monitoring

**EventBridge Pipes:**
- Enable retry policies (default 3 attempts)
- Configure dead-letter queues (optional)
- Set up CloudWatch alarms for pipe failures

**DynamoDB Streams:**
- Monitor stream latency
- Set up alerts for unprocessed records

## Cost Optimization

- DynamoDB on-demand capacity eliminates over-provisioning
- EventBridge Pipes have no additional cost beyond source/target services
- Monitor and set up billing alerts

This architecture provides a robust, event-driven integration between DynamoDB and EventBridge while adhering to AWS best practices and CloudFormation linting rules.
