# Event-Driven AWS Batch Processing Pipeline Architecture Plan

## Key AWS Services Required

1. **VPC & Networking**: `AWS::EC2::VPC`, `AWS::EC2::Subnet`, `AWS::EC2::SecurityGroup`, `AWS::EC2::InternetGateway`, `AWS::EC2::VPCGatewayAttachment`
2. **Batch Processing**: `AWS::Batch::ComputeEnvironment`, `AWS::Batch::JobQueue`, `AWS::Batch::JobDefinition`
3. **Event Processing**: `AWS::SQS::Queue`, `AWS::EventSchemas::Schema`, `AWS::Pipes::Pipe`
4. **IAM**: `AWS::IAM::Role`, `AWS::IAM::Policy`
5. **ECS**: `AWS::ECS::Cluster` (for Batch compute resources)
6. **SSM**: For dynamic AMI fetching

## Architecture Overview

The pipeline will follow this flow:
1. SQS receives messages (e.g., "OrderCreated")
2. EventBridge Pipe filters messages based on content
3. Pipe triggers AWS Batch job execution
4. Batch processes the job in isolated VPC resources
5. Results are returned to the originating service

## Detailed Component Breakdown

### 1. VPC Configuration
- **CIDR Blocks**: Use default `10.0.0.0/16` for VPC, `10.0.1.0/24` and `10.0.2.0/24` for private subnets
- **Availability Zones**: Use `Fn::GetAZs` and `Fn::Select` to dynamically select AZs
- **Security Groups**: Create separate SG for Batch compute resources with minimal egress rules
- **Internet Gateway**: Required for SSM access to Batch instances

### 2. IAM Roles & Policies
- **Batch Service Role**: Trust Batch service, permissions for Batch actions
- **Batch Instance Role**: For ECS container instances, SSM access, CloudWatch logging
- **Pipe Role**: Permissions to read SQS and trigger Batch jobs
- **ECS Execution Role**: For container image pulls and logging

### 3. AWS Batch Configuration
- **Compute Environment**: Managed, on-demand, private subnets
- **Job Queue**: Connected to compute environment
- **Job Definition**: Container properties, command overrides, environment variables

### 4. EventBridge Pipe Configuration
- **Source**: SQS queue ARN
- **Filter Criteria**: JSON path matching "OrderCreated" events
- **Target**: Batch job queue with job definition override

### 5. SQS Queue
- **Visibility Timeout**: 300 seconds (default)
- **Message Retention**: 345,600 seconds (4 days)
- **Encryption**: SSE-SQS enabled

## Properties & Dependencies

### Critical Dependencies
1. VPC must be created before subnets
2. Internet Gateway must be attached to VPC before VPCGatewayAttachment
3. IAM roles must exist before referencing in Batch/Compute resources
4. Security groups must be created before referencing in compute resources

### Property Specifications

#### VPC Properties
```yaml
CidrBlock: 10.0.0.0/16
EnableDnsSupport: true
EnableDnsHostnames: true
```

#### Subnet Properties
```yaml

AvailabilityZone: !Select [0, !GetAZs ""]
MapPublicIpOnLaunch: false
```

#### Batch Compute Environment
- **Type**: MANAGED
- **AllocationStrategy**: BEST_FIT
- **MinvCpus**: 0
- **DesiredvCpus**: 0
- **MaxvCpus**: 256
- **InstanceTypes**: optimal

#### Job Definition
- **Type**: container
- **ContainerProperties**: Image, memory, vcpus, command
- **RetryStrategy**: Attempts: 1

#### Pipe Properties
- **Source**: !GetAtt SqsQueue.Arn
- **FilterCriteria**: JSON path matching OrderCreated
- **Target**: Batch job queue with job definition override

## Dynamic Resource Fetching

### AMI Retrieval
Use SSM Parameter Store to fetch latest Amazon Linux 2 AMI:
```yaml

```

### AZ Selection
```yaml

!Select [1, !GetAZs ""]
```

## Security Best Practices

1. **Principle of Least Privilege**: Each IAM role has only necessary permissions
2. **VPC Isolation**: Batch compute resources in private subnets
3. **Encryption**: SQS SSE-SQS, CloudWatch logs encryption
4. **No Public Access**: No public IPs assigned to Batch instances

## Error Handling & Resilience

1. **Retry Strategy**: Single attempt for Batch jobs
2. **Dead Letter Queue**: Optional SQS DLQ for failed messages
3. **CloudWatch Alarms**: Monitor Batch queue depth, job failures
4. **Auto Scaling**: Managed compute environment handles scaling
