# Architectural Deployment Plan for AWS Organizations Serverless Pipeline

## Key AWS Services Required

1. **S3** - For storing organizational data with encryption and access controls
2. **Lambda** - For fetching AWS Organizations metadata via cross-account role assumption
3. **CloudWatch Events** - For scheduling weekly Lambda execution
4. **IAM** - For cross-account role assumption and permissions management
5. **Glue** - For automatic schema discovery and cataloging
6. **Athena** - For SQL-based querying of organizational data
7. **SSM Parameter Store** - For dynamic Lambda function code storage

## Architecture Overview

The solution follows a serverless event-driven architecture where CloudWatch Events triggers Lambda weekly, which fetches Organizations data via cross-account role assumption, stores it in S3, and Glue automatically catalogs it for Athena querying.

## Detailed Component Analysis

### 1. S3 Bucket Configuration

**Properties:**
- Bucket name: Parameterized with default value (e.g., `organizations-data-${AWS::AccountId}`)
- Server-side encryption: SSE-S3 enabled by default
- Access control: Bucket Policy restricting access to authorized roles
- Versioning: Enabled for data protection
- Block public access: Enabled with all settings

**Dependencies:** None - S3 is independent

**Linting Compliance:**
- No `AccessControl` property used
- `OwnershipControls` not required as we're using bucket policies
- `DeletionPolicy: Retain` with `UpdateReplacePolicy: Retain` for data protection

### 2. Lambda Function

**Properties:**
- Runtime: `python3.12` (latest supported)
- Handler: `index.lambda_handler`
- Memory: 512 MB (default)
- Timeout: 300 seconds (default)
- Code: Stored in SSM Parameter Store (not hardcoded)

**Dependencies:**
- Requires IAM role with Organizations read permissions
- Requires S3 write permissions
- Requires STS assume role permissions for cross-account access

**Linting Compliance:**
- No deprecated runtimes
- Uses dynamic code reference via SSM

### 3. CloudWatch Events Rule

**Properties:**
- Schedule: `rate(7 days)` for weekly execution
- Target: Lambda function
- Enabled: True by default

**Dependencies:** Lambda function must exist first

### 4. IAM Roles and Policies

**Lambda Execution Role:**
- Managed policy: `service-role/AWSLambdaBasicExecutionRole`
- Custom policy: Organizations read access
- Custom policy: S3 write access to target bucket
- Custom policy: STS assume role for cross-account access

**Cross-account Role (in management account):**
- Trust relationship: Lambda execution role
- Permissions: `organizations:Describe*` actions

**Glue Role:**
- Managed policy: `AWSGlueServiceRole`
- Custom policy: S3 read access to target bucket

**Linting Compliance:**
- Exact AWS managed policy ARNs used
- Proper trust relationships configured

### 5. Glue Crawler

**Properties:**
- Name: Parameterized with default (e.g., `OrganizationsCrawler`)
- Database: Parameterized Athena database name
- Role: Glue IAM role
- Schedule: `on-demand` (triggered manually or via Lambda)
- S3 target: Target bucket path
- Schema change policy: Update all

**Dependencies:** S3 bucket and Glue role must exist

### 6. SSM Parameter for Lambda Code

**Properties:**
- Name: `/organizations-pipeline/lambda-code`
- Type: `String`
- Tier: `Standard`
- Value: Base64-encoded Lambda function code

**Dependencies:** None - independent resource

## Parameter Design

```yaml
Parameters:
  S3BucketName:
    Type: String
    Default: !Sub 'organizations-data-${AWS::AccountId}'
    Description: S3 bucket name for storing organizational data

  TagsToInclude:
    Type: String
    Default: 'Environment,Owner,Project'
    Description: Comma-separated list of tags to include in reports

  ManagementAccountId:
    Type: String
    Default: ''
    Description: Management account ID for cross-account role assumption

  AthenaDatabaseName:
    Type: String
    Default: 'managementcur'
    Description: Athena database name for cataloged data
```

## Security Considerations

1. **Least Privilege:** IAM roles grant only necessary permissions
2. **Encryption:** S3 encryption at rest, Lambda environment variables encrypted
3. **Cross-account Security:** Temporary credentials via STS, minimal trust relationships
4. **Data Protection:** S3 versioning, deletion policies with retention

## Dynamic Resource Resolution

**Lambda Code Storage:**
- Lambda function code stored in SSM Parameter Store
- Base64-encoded Python script that:
  - Assumes cross-account role using STS
  - Fetches Organizations data (accounts, OUs, tags)
  - Writes JSON files to S3 with timestamp prefixes

**AMI Resolution (if needed for future extensions):**
- Use `Fn::ImportValue` or SSM parameters for any EC2-based components

## Execution Flow

1. **Weekly Trigger:** CloudWatch Events triggers Lambda every 7 days
2. **Cross-account Access:** Lambda assumes role in management account
3. **Data Collection:** Fetches Organizations metadata (accounts, OUs, tags)
4. **Data Storage:** Writes JSON files to S3 with date-based prefixes
5. **Schema Discovery:** Glue crawler runs to catalog new data
6. **Query Ready:** Athena database updated for SQL queries

## Error Handling and Monitoring

- **CloudWatch Logs:** Lambda logs captured automatically
- **Dead Letter Queue:** Optional SQS queue for failed executions
- **Metrics:** CloudWatch metrics for Lambda invocations and errors
- **Alarms:** Optional alarms for failed executions or high error rates

## Integration Points

- **Exported ARNs:** S3 bucket ARN, Lambda function ARN, Glue database name
- **Tags:** Resource tagging for cost allocation and organization
- **Cross-account Role ARN:** Exported for management account setup

This architecture provides a secure, automated, and scalable solution for centralized AWS Organizations data collection and analysis, meeting the business requirements for governance and reporting while adhering to AWS best practices and CloudFormation linting rules.
