# DynamoDB Stream to EventBridge Integration Architecture Plan

## Key AWS Services Required

1. **DynamoDB** - To create the table with stream enabled
2. **EventBridge** - To create the event bus and pipe
3. **IAM** - To create necessary service roles and policies
4. **Lambda** (optional) - For advanced processing if needed beyond simple filtering

## Architecture Overview

The solution will create a DynamoDB table with streams enabled, an EventBridge Pipe to consume the stream, and an EventBridge event bus to receive filtered events. The pipe will use a filter pattern to match events where the 'avatarUrl' field ends with '.png'.

## Detailed Implementation Plan

### 1. DynamoDB Table Configuration

- Create a DynamoDB table with `StreamSpecification` enabled
- Use `NEW_AND_OLD_IMAGES` view type to capture both before and after states
- Set appropriate read/write capacity (provisioned or on-demand)

### 2. EventBridge Pipe Configuration

- Create an EventBridge Pipe that consumes from the DynamoDB stream
- Configure the pipe with a filter pattern to match `avatarUrl` ending with '.png'
- Transform the event to include `DetailType` and `Source` metadata
- Route matching events to the EventBridge bus

### 3. IAM Roles and Policies

- Create a pipe role with permissions to:
  - Read from DynamoDB stream
  - Write to EventBridge bus
- Create a DynamoDB table role if needed for stream management

### 4. EventBridge Bus

- Create a custom event bus or use the default bus
- Configure appropriate permissions if using a custom bus

## Properties and Dependencies

### DynamoDB Table Properties
- `TableName`: Dynamically generated or parameterized
- `AttributeDefinitions`: Required for key schema
- `KeySchema`: Partition key (and sort key if needed)
- `BillingMode`: 'PAY_PER_REQUEST' (default) or 'PROVISIONED'
- `StreamSpecification`: Must be enabled with `NEW_AND_OLD_IMAGES`

### EventBridge Pipe Properties
- `Source`: 'AWS::DynamoDB::Table'
- `SourceConfig`: Points to DynamoDB stream ARN
- `Filter`: JSON pattern matching `avatarUrl` ending with '.png'
- `Target`: EventBridge bus
- `Enrichment`: Optional Lambda function for complex transformations

### IAM Role Properties
- `AssumeRolePolicyDocument`: Service trust for EventBridge
- `Policies`: Inline or managed policies for stream and bus access

## Safe Defaults (Due to Missing Specific Values)

### DynamoDB
- **BillingMode**: 'PAY_PER_REQUEST' (on-demand capacity)
- **Partition Key**: 'id' (string type)
- **Table Name**: 'IntegrationTable' (or parameterized)

### EventBridge
- **Event Bus**: Use default bus if not specified
- **Source**: 'com.company.dynamodb' (or parameterized)
- **DetailType**: 'UserAvatarUpdate' (or parameterized)

### IAM
- **Role Name**: Dynamically generated with stack name prefix
- **Policy Names**: Similarly prefixed

## Dynamic Resource Fetching

### AMI Usage
Not applicable for this use case (no EC2 instances).

### Availability Zones
Not applicable for this use case (no VPC resources).

## Security Considerations

- Use least privilege IAM policies
- Enable encryption at rest for DynamoDB
- Consider VPC endpoints for private connectivity if needed

## Error Handling and Monitoring

- Enable CloudWatch Logs for pipe execution
- Set up CloudWatch Alarms for failed deliveries
- Consider dead-letter queues for failed events

## Template Structure

The CloudFormation template will be organized as follows:

1. **Parameters** (with safe defaults)
2. **Resources** (in logical order)
3. **Outputs** (for key ARNs and IDs)

## Linting Compliance

The template will adhere to all specified linting rules:
- No hardcoded AMIs or AZs
- No legacy S3 properties
- Proper use of dynamic references
- Correct IAM policy ARNs
- Valid runtime versions (if Lambda is used)

This architecture provides a robust, production-ready integration between DynamoDB streams and EventBridge while following AWS best practices and ensuring template validity.
