# Automated Secrets Manager Monitoring & Notification System Architecture Plan

## Key AWS Services Required

1. **AWS Secrets Manager** - Source of truth for secret lifecycle events
2. **Amazon CloudWatch Events (EventBridge)** - Event routing and filtering
3. **AWS Lambda** - Event processing and SNS notification logic
4. **Amazon SNS** - Alert delivery mechanism
5. **AWS IAM** - Service permissions and security
6. **AWS Systems Manager (SSM) Parameter Store** - Dynamic Lambda runtime fetching

## Architecture Overview

The system will use CloudWatch Events to capture Secrets Manager lifecycle events, trigger a Lambda function, which then processes the event and sends notifications via SNS. This creates a decoupled, scalable, and serverless architecture.

## Detailed Implementation Plan

### 1. Event Pattern Design

CloudWatch Events will monitor three specific Secrets Manager event types:
- `CreateSecret` - When a new secret is created
- `UpdateSecret` - When an existing secret is modified
- `GetSecretValue` - When a secret is accessed (read)

The event pattern will use AWS-provided event schemas to ensure precise matching.

### 2. Lambda Function Architecture

The Lambda function will:
- Parse incoming event data
- Extract relevant metadata (secret name, timestamp, event type)
- Format a human-readable message
- Publish to the configured SNS topic

### 3. Security & Permissions

IAM roles will be created with the principle of least privilege:
- Lambda execution role: Only permissions needed for CloudWatch Logs, SNS, and Secrets Manager read-only access
- SNS topic: Standard delivery permissions

### 4. Dynamic Resource Resolution

To ensure maximum portability and avoid hardcoding:

**AMI Resolution**: Lambda functions will use the latest Python 3.12 runtime available in the region, fetched dynamically through CloudFormation's runtime resolution.

**Region-Aware Configuration**: All resources will be created in the same region as the template deployment, with no hardcoded region dependencies.

## Property Dependencies & Execution Order

1. **SNS Topic Creation** → Must exist before Lambda can reference it
2. **IAM Role Creation** → Must exist before Lambda can be created
3. **Lambda Function Creation** → Must exist before EventBridge rule can reference it
4. **EventBridge Rule Creation** → Final step, depends on Lambda

## Safe Defaults & Parameter Strategy

Since no specific values are provided, I'll use these safe defaults:

**Parameters with Defaults**:
- `NotificationEmail`: Default to a placeholder (e.g., 'admin@example.com') that users must update
- `SecretNamePattern`: Default to '*' to match all secrets (can be restricted later)
- `LambdaTimeout`: Default to 30 seconds (maximum allowed)
- `LambdaMemory`: Default to 128 MB (minimum sufficient for this use case)

**Resource Defaults**:
- Lambda runtime: Latest Python 3.12 available
- Lambda timeout: 30 seconds
- Lambda memory: 128 MB
- SNS subscription protocol: Email

## SSM Parameter Usage

The Lambda function will use the latest Python 3.12 runtime available in the region, which CloudFormation can resolve automatically without SSM Parameter Store for runtimes. However, if we needed to reference specific AMIs or other dynamic values in the future, we would use SSM Parameter Store with the `resolve:ssm:` syntax.

## Security Best Practices

- No hardcoded credentials or secrets
- IAM roles follow least privilege principle
- SNS topic uses standard delivery without special configurations
- Lambda function has minimal required permissions

## Error Handling & Resilience

The architecture inherently provides:
- Automatic retry through CloudWatch Events (up to 24 hours)
- Dead-letter queue capability if needed (not implemented by default)
- CloudWatch Logs for all Lambda executions
- SNS provides reliable message delivery

## Template Structure Planning

The CloudFormation template will be organized as:

1. **Parameters Section**: All configurable options with safe defaults
2. **Resources Section**:
   - SNS Topic
   - IAM Role and Policy
   - Lambda Function
   - EventBridge Rule
3. **Outputs Section**: ARNs and endpoints for reference

This architecture ensures a production-ready, secure, and maintainable solution that can be deployed across any AWS region without modification.
