# EventBridge-to-Kinesis Integration Architecture Plan

## Key AWS Services Required

1. **Amazon EventBridge** - Event bus for capturing application events
2. **Amazon Kinesis Data Stream** - Scalable event ingestion and buffering
3. **AWS Identity and Access Management (IAM)** - Service-to-service permissions
4. **AWS Systems Manager (SSM) Parameter Store** - Dynamic resource fetching (if needed for future extensions)

## Architecture Overview

The integration will create a unidirectional data flow:
`Application Events → EventBridge Event Bus → Kinesis Data Stream`

## Detailed Resource Breakdown

### 1. EventBridge Event Bus
- **Resource Type**: `AWS::Events::EventBus`
- **Name**: `my-application-events`
- **Purpose**: Capture events from the application source
- **Properties**: 
  - `Name`: `my-application-events`
  - `EventSourceName`: `my-application`

### 2. Kinesis Data Stream
- **Resource Type**: `AWS::Kinesis::Stream`
- **Name**: `application-events-stream`
- **Properties**:
  - `Name`: `application-events-stream`
  - `ShardCount`: 1 (default for low-volume testing)
  - `RetentionPeriodHours`: 24 (default)

### 3. IAM Role for EventBridge
- **Resource Type**: `AWS::IAM::Role`
- **Name**: `EventBridgeKinesisIntegrationRole`
- **Purpose**: Grant EventBridge permissions to publish to Kinesis
- **Properties**:
  - `AssumeRolePolicyDocument`: Allow EventBridge service to assume role
  - `ManagedPolicyArns`: Include `AWSKinesisFullAccess` or custom policy
  - `Policies`: Custom inline policy for Kinesis publishing

### 4. EventBridge Rule
- **Resource Type**: `AWS::Events::Rule`
- **Purpose**: Route events from the bus to Kinesis
- **Properties**:
  - `EventPattern`: Match events from `my-application`
  - `EventBusName`: Reference to the created event bus
  - `Targets`: Kinesis stream with the IAM role

## Properties and Dependencies

### Critical Dependencies
1. **IAM Role must be created before EventBridge Rule** - The rule references the role ARN
2. **Event Bus must exist before Rule creation** - Rule targets the specific bus
3. **Kinesis Stream must exist before Rule creation** - Rule sends events to the stream

### Property Defaults (Safe AWS Defaults)
- **ShardCount**: 1 (can be adjusted based on throughput requirements)
- **RetentionPeriodHours**: 24 (standard default)
- **EventBusName**: `default` (if not specified, uses the default bus)
- **EventSourceName**: `my-application` (application-specific identifier)

## Dynamic Resource Fetching

For this specific template, dynamic fetching isn't required since we're using managed services with stable ARNs. However, if we were to extend this with Lambda functions or other compute resources, we would use:

- **AMIs**: `resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`
- **AZs**: `Fn::GetAZs` with `Fn::Select` for specific index
- **Secrets**: `{{resolve:secretsmanager:SecretName:SecretString:}}`

## Security Considerations

1. **Principle of Least Privilege**: Instead of `AWSKinesisFullAccess`, create a custom policy allowing only `kinesis:PutRecord` and `kinesis:PutRecords`
2. **Resource-Level Permissions**: Restrict IAM permissions to the specific Kinesis stream ARN
3. **Event Pattern Security**: Ensure EventBridge rule only matches expected event schemas

## Error Handling and Best Practices

1. **Deletion Policies**: Not critical for this integration (stateless services)
2. **UpdateReplacePolicies**: Not required for these resources
3. **Resource Naming**: Use consistent naming conventions with prefixes
4. **Tags**: Include standard tags (Environment, Owner, Project) for resource management

## Validation Strategy

The template will be validated using:
- AWS CloudFormation Linter (cfn-lint) for syntax and best practices
- AWS CloudFormation ValidateTemplate API for schema compliance
- Manual review of IAM permissions for least privilege adherence

This architecture provides a minimal, production-ready EventBridge-to-Kinesis integration that can be deployed without manual intervention and scaled based on actual event volume requirements.
