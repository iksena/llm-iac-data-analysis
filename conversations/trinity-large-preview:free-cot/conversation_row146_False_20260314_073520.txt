# Automated Image Analysis Pipeline Architecture Plan

## Key AWS Services Required

1. **S3** - Storage bucket for image uploads and Lambda triggers
2. **Lambda** - Serverless functions for image processing and notification
3. **Step Functions** - Workflow orchestration for the analysis pipeline
4. **Rekognition** - Image analysis for human face detection
5. **SNS** - SMS notification service
6. **IAM** - Service-to-service permissions and role management
7. **SSM Parameter Store** - Dynamic resource fetching (AMIs, if needed)

## Architecture Overview

The solution follows an event-driven architecture where S3 uploads trigger a Lambda function, which initiates a Step Functions workflow. The workflow orchestrates Rekognition analysis and conditional SNS notifications based on detection results.

## Detailed Component Breakdown

### 1. S3 Bucket Configuration
- **Properties**: Versioning enabled, server access logging (optional)
- **Event Notification**: Configured to trigger Lambda on `s3:ObjectCreated:*` events
- **Dependencies**: Must exist before Lambda can be configured with notification permissions

### 2. Lambda Functions (3 total)

**A. S3 Trigger Lambda (Python 3.12)**
- **Purpose**: Entry point for S3 events, initiates Step Functions
- **Runtime**: `python3.12` (latest supported)
- **IAM Role**: S3 read access, Step Functions start execution permission
- **Dependencies**: Requires S3 bucket and Step Functions state machine to exist

**B. Image Analysis Lambda (Python 3.12)**
- **Purpose**: Invoked by Step Functions to analyze images with Rekognition
- **Runtime**: `python3.12`
- **IAM Role**: Rekognition access, S3 read access for image retrieval
- **Dependencies**: Requires Rekognition service and S3 bucket

**C. Notification Lambda (Python 3.12)**
- **Purpose**: Sends SMS via SNS when human faces detected
- **Runtime**: `python3.12`
- **IAM Role**: SNS publish permission
- **Dependencies**: Requires SNS topic and valid phone number parameter

### 3. Step Functions State Machine

**Properties**:
- **Definition**: JSON-based workflow definition
- **IAM Role**: Permissions for Step Functions execution, Lambda invocation
- **Dependencies**: Requires all Lambda functions to be created first

**Workflow Logic**:
1. Start execution from S3 Lambda
2. Invoke Image Analysis Lambda with S3 object details
3. Conditional branch: If human detected → Invoke Notification Lambda
4. End execution

### 4. IAM Role Strategy (Resolving Circular Dependencies)

**Primary IAM Role** (for S3 Trigger Lambda):
- S3: `s3:GetObject`, `s3:ListBucket` (source bucket)
- Step Functions: `states:StartExecution`
- Lambda: `lambda:InvokeFunction` (for self-triggers if needed)

**Secondary IAM Role** (for Step Functions):
- Lambda: `lambda:InvokeFunction` (to invoke both analysis and notification Lambdas)
- CloudWatch: `logs:CreateLogGroup`, `logs:CreateLogStream`, `logs:PutLogEvents`

**Rekognition Lambda Role**:
- Rekognition: `rekognition:DetectLabels`
- S3: `s3:GetObject` (to read images)

**Notification Lambda Role**:
- SNS: `sns:Publish`

**Circular Dependency Resolution**:
- Create IAM roles first with minimal permissions
- Use `DependsOn` to ensure proper creation order
- S3 bucket policy grants Lambda service permission to invoke Lambda
- Lambda permissions granted via resource-based policy rather than IAM role alone

### 5. SNS Topic Configuration

**Properties**:
- **DisplayName**: "ImageDetectionAlerts"
- **Subscription**: SMS endpoint (phone number parameter)
- **IAM Role**: SNS publish permission for Notification Lambda

### 6. SSM Parameter Store Usage

**Parameters**:
- `ImageAnalysisThreshold`: Default "HUMAN" label detection threshold
- `NotificationPhoneNumber`: Default "+1234567890" (dummy value)
- `MaxRetries`: Default "3" for retry logic

## Creation Order & Dependencies

1. **IAM Roles** (independent, created first)
2. **S3 Bucket** (depends on IAM roles for bucket policy)
3. **Step Functions State Machine** (depends on Lambda ARNs)
4. **Lambda Functions** (depends on S3 bucket, Step Functions, IAM roles)
5. **SNS Topic** (depends on IAM roles)
6. **S3 Event Notification Configuration** (final step, depends on all resources)

## Security & Best Practices Implementation

- **Least Privilege**: Each IAM role has only necessary permissions
- **No Hardcoding**: All ARNs, IDs, and sensitive data use dynamic references
- **Runtime Selection**: Latest Python 3.12 for all Lambdas
- **Error Handling**: Step Functions includes retry and catch configurations
- **Logging**: All components integrated with CloudWatch Logs
- **Encryption**: S3 bucket uses SSE-S3 by default

## Parameter Configuration (with Safe Defaults)

```yaml
Parameters:
  S3BucketName:
    Type: String
    Default: "image-analysis-pipeline-unique-id"
    Description: "S3 bucket name for image uploads"
  
  PhoneNumber:
    Type: String
    Default: "+1234567890"
    Description: "Phone number for SMS alerts (E.164 format)"
  
  LogLevel:
    Type: String
    Default: "INFO"
    AllowedValues: ["DEBUG", "INFO", "WARNING", "ERROR"]
    Description: "Lambda logging level"
```

## Key Architectural Decisions

1. **Three-Lambda Approach**: Separates concerns (trigger, analysis, notification) for better maintainability
2. **Step Functions Workflow**: Provides visibility, error handling, and conditional logic
3. **Resource-Based Policies**: Resolves S3→Lambda circular dependencies
4. **Dynamic References**: Uses SSM parameters and dynamic ARNs instead of hardcoding
5. **Latest Runtimes**: Ensures security patches and performance improvements

This architecture ensures a robust, secure, and maintainable image analysis pipeline that meets all specified requirements while following AWS best practices and CloudFormation linting rules.
