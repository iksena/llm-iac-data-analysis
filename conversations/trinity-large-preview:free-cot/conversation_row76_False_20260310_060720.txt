- **IAM Role Properties**: Each role requires a unique name, description, and trust policy
- **Policy Properties**: Each role needs both managed and inline policies with specific permissions
- **Tag-Based Conditions**: Used for production environment restrictions
- **Resource-Based Permissions**: For role-passing capabilities to workshop-specific service roles
- **Service-Linked Role Creation**: Required for Security Admins

## 4. Dynamic Resource Fetching Strategy

While IAM roles don't require AMIs or AZs, I'll establish a pattern for dynamic resource fetching:

- **AWS Managed Policy ARNs**: Use the latest ARNs from AWS documentation
- **Service Namespaces**: Use AWS service namespaces rather than hardcoded values
- **Resource Identifiers**: Use AWS-provided identifiers and ARNs

## 5. Architecture Plan for Each IAM Role

### Security Admins Role
- **Permissions**: Full FIS access, role-passing privileges, service-linked role creation
- **Policies**: 
  - AWS managed policies (AWSReadOnlyAccess)
  - Custom inline policy for FIS full access
  - Custom inline policy for role-passing to workshop service roles
  - Custom inline policy for service-linked role creation

### Admins Role
- **Permissions**: Similar to Security Admins but with production FIS access denied
- **Policies**:
  - Inherits Security Admins policies
  - Additional inline policy denying FIS access to resources tagged "prod"

### Users Role
- **Permissions**: Limited to experiment operations and tagging
- **Policies**:
  - AWS managed policies (AWSReadOnlyAccess)
  - Custom inline policy for experiment start/stop
  - Custom inline policy for FIS resource viewing
  - Custom inline policy for experiment tagging
  - Custom inline policy denying production FIS access

### Non-Users Role
- **Permissions**: Read-only AWS access with FIS actions explicitly denied
- **Policies**:
  - AWS managed policies (AWSReadOnlyAccess)
  - Custom inline policy explicitly denying all FIS actions

## 6. Template Structure and Best Practices

### Parameters Section
- Define any necessary parameters with safe default values
- Use AllowedPattern where appropriate with matching default values

### Resources Section
- Create four IAM::Role resources with unique logical IDs
- Each role will have:
  - Trust policy allowing appropriate principals
  - Managed policies (AWSReadOnlyAccess)
  - Custom inline policies for granular permissions

### Policy Design Patterns
- Use IAM policy variables (like aws:ResourceTag) for tag-based conditions
- Implement explicit denies for production access where required
- Use IAM policy conditions for resource-based permissions

### Security Considerations
- Follow principle of least privilege for each role
- Use explicit deny statements where necessary to override inherited permissions
- Ensure proper separation of duties between roles

## 7. Implementation Strategy

The template will be implemented as a single CloudFormation template with:

1. **Parameters**: For any configurable values (though minimal for IAM roles)
2. **Resources**: Four IAM::Role resources with comprehensive policies
3. **Policies**: Combination of AWS managed policies and custom inline policies
4. **Conditions**: For tag-based and environment-based permissions

This architecture ensures:
- Clear separation of duties across the four roles
- Granular permissions aligned with business requirements
- Proper security controls for production environments
- Scalable and maintainable IAM structure

The final template will be production-ready, adhering to AWS best practices and passing cfn-lint validation.
