# Detailed Architectural Deployment Plan for EFS with EC2 Integration

## 1. Key AWS Services Required

Based on the business requirements, the following AWS services are essential:

- **Amazon VPC** - Network foundation with public subnets across two AZs
- **Amazon EFS** - Encrypted file system for shared storage
- **Amazon EC2** - Two instances in separate AZs for cross-AZ integration
- **AWS IAM** - Roles and policies for EC2 instances to access EFS and SSM
- **AWS Systems Manager (SSM)** - Secure instance management and AMI resolution
- **Amazon CloudWatch** - Monitoring and alerting for EFS performance metrics
- **AWS Security Groups** - Network security for EFS mount targets and EC2 instances
- **AWS Key Management Service (KMS)** - Encryption key for EFS data at rest

## 2. Properties and Dependencies

### Critical Dependencies:
1. **VPC Creation** → **Subnets** → **Internet Gateway** → **Route Tables** → **EFS Mount Targets**
2. **IAM Role Creation** → **Instance Profile** → **EC2 Launch**
3. **EFS Creation** → **Mount Target Creation** (requires subnets and security groups)
4. **EC2 Launch** → **User Data Execution** (requires EFS mount targets to be available)

### Property Dependencies:
- EFS Mount Targets require existing subnets and security groups
- EC2 instances require IAM instance profiles before launch
- CloudWatch Alarms require existing EFS file system
- Security Groups require VPC ID before creation

## 3. Safe AWS Defaults (Where Properties Were Not Specified)

### Network Configuration:
- **VPC CIDR**: `10.0.0.0/16` (standard private range)
- **Public Subnet CIDRs**: 
  - AZ1: `10.0.1.0/24`
  - AZ2: `10.0.2.0/24`
- **Availability Zones**: Dynamically selected using `Fn::GetAZs` and `Fn::Select`

### EC2 Configuration:
- **Instance Type**: `t3.micro` (free tier eligible, suitable for testing)
- **AMI**: Latest Amazon Linux 2 using SSM Parameter Store
- **KeyPair**: Parameter with default value 'default-key' (user must create or override)

### EFS Configuration:
- **Performance Mode**: `generalPurpose` (default, suitable for most workloads)
- **Throughput Mode**: `bursting` (default, cost-effective)
- **Encrypted**: `true` (required by business need)
- **KMS Key**: AWS-managed CMK (default for EFS encryption)

### Security Groups:
- **EFS Security Group**: Allows NFS traffic (port 2049) from EC2 instances
- **EC2 Security Group**: Allows SSH (port 22) from anywhere for management

## 4. Dynamic Resource Fetching Strategy

### AMI Resolution:
Use SSM Parameter Store to fetch the latest Amazon Linux 2 AMI:
```yaml
ImageId: !Sub 'resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2'
```

### Availability Zone Selection:
```yaml
!Select [0, !GetAZs '']
!Select [1, !GetAZs '']
```

### EFS DNS Name:
Use the `!Ref` intrinsic function to get the EFS file system ID, which AWS automatically resolves to the correct DNS name.

## 5. Architecture Design Details

### VPC Design:
- Single VPC with two public subnets across two AZs
- Internet Gateway attached to VPC
- Route table with default route (0.0.0.0/0) pointing to Internet Gateway
- Public subnets automatically assigned public IP addresses

### EFS Design:
- Encrypted file system with AWS-managed KMS key
- Two mount targets (one per AZ) in public subnets
- Security group allowing NFS traffic from EC2 instances only

### EC2 Design:
- Two instances in separate AZs and public subnets
- IAM instance profile with EFS and SSM permissions
- User data script that:
  1. Installs EFS client utilities
  2. Creates `/home` directory if it doesn't exist
  3. Mounts EFS file system using the DNS name
  4. Adds mount to `/etc/fstab` for persistence

### IAM Design:
- Instance role with managed policies:
  - `AmazonSSMManagedInstanceCore` (SSM access)
  - Custom policy allowing EFS mount operations
- Instance profile to attach role to EC2 instances

### CloudWatch Alarms Design:
- **BurstCreditBalance**: Alerts when credits drop below threshold
- **PercentIOLimit**: Alerts when IO is approaching 100% utilization
- **PermittedThroughputExceeded**: Alerts on throttling events
- **TotalSize**: Alerts when storage approaches provisioned capacity

### Security Considerations:
- EFS encryption at rest using AWS-managed KMS key
- Network ACLs and security groups to restrict access
- IAM roles with least privilege principle
- SSM for secure remote management without bastion hosts

## 6. Bootstrapping and Automation

### User Data Script:
The EC2 instances will execute a bash script during first boot that:
1. Updates system packages
2. Installs `amazon-efs-utils`
3. Creates mount point directory
4. Mounts EFS file system
5. Configures automatic mounting on reboot

### Mount Configuration:
- Uses NFS v4.1 protocol
- Mount options: `nfsvers=4.1,rsize=1048576,wsize=1048576,hard,timeo=600,retrans=2`
- Mount target DNS name: `fs-xxxxxxxx.efs.us-east-1.amazonaws.com:/`

## 7. Error Handling and Validation

### CloudFormation Best Practices Applied:
- All resources have appropriate `DependsOn` attributes
- Parameters have safe default values
- No hardcoded AMI IDs or AZs
- Proper deletion policies for stateful resources
- Security groups properly referenced

### Validation Steps:
1. Template validation using `aws cloudformation validate-template`
2. cfn-lint compliance check
3. Dependency graph verification
4. Security group rule validation
5. IAM permission boundary checks

This architecture ensures high availability through multi-AZ deployment, data persistence through EFS encryption, and operational efficiency through automated monitoring and management.
