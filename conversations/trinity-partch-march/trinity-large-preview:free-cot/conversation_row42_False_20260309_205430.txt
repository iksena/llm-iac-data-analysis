# Detailed Architectural Deployment Plan for VPC with Public and Private Subnets

## 1. Key AWS Services Required

Based on the business need, the following AWS services are required:

- **AWS::EC2::VPC** - The core VPC resource
- **AWS::EC2::InternetGateway** - For public internet access
- **AWS::EC2::VPCGatewayAttachment** - To attach the Internet Gateway to the VPC
- **AWS::EC2::Subnet** - Three subnets total (2 public, 1 private)
- **AWS::EC2::RouteTable** - Separate route tables for public and private subnets
- **AWS::EC2::Route** - Routes for internet traffic
- **AWS::EC2::SubnetRouteTableAssociation** - To associate subnets with route tables
- **AWS::EC2::EIP** - Elastic IP for NAT Gateway
- **AWS::EC2::NatGateway** - For private subnet internet access
- **AWS::EC2::SecurityGroup** - For network security
- **AWS::EC2::VPCEndpoint** - Optional, for private access to AWS services

## 2. Properties and Dependencies

The deployment follows this critical dependency chain:

1. **VPC** must be created first (provides the network foundation)
2. **InternetGateway** and **VPCGatewayAttachment** (IGW must attach to VPC)
3. **Subnets** (require VPC ID, can be created in parallel)
4. **RouteTables** (require VPC ID)
5. **Routes** (require RouteTable and IGW/NAT Gateway)
6. **SubnetRouteTableAssociations** (require Subnet and RouteTable)
7. **EIP** and **NatGateway** (EIP must exist before NAT Gateway)
8. **SecurityGroup** (can be created anytime after VPC exists)

## 3. Safe AWS Defaults for Missing Properties

Since specific properties weren't provided, I'll use these safe defaults:

- **VPC CIDR**: `10.0.0.0/16` (standard private range)
- **Public Subnet CIDRs**: `10.0.0.0/24` and `10.0.1.0/24`
- **Private Subnet CIDR**: `10.0.2.0/24`
- **Availability Zones**: Dynamically fetched using `Fn::GetAZs` and `Fn::Select`
- **Instance Type**: `t3.micro` (free tier eligible)
- **SSH Key Pair**: Parameter with default value 'default-key'
- **AMI**: Fetched dynamically using SSM Parameter Store

## 4. Dynamic Resource Fetching Strategy

**AMIs**: I'll use SSM Parameter Store to fetch the latest Amazon Linux 2 AMI:
- `resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`

**Availability Zones**: Using `Fn::GetAZs` to get all AZs in the region, then `Fn::Select` to pick the first two:
- `Fn::Select: [0, Fn::GetAZs: !Ref 'AWS::Region']`
- `Fn::Select: [1, Fn::GetAZs: !Ref 'AWS::Region']`

## 5. RegionMap Implementation

To ensure resources persist in given Availability Zones when zones are added/removed, I'll implement a static RegionMap:

```yaml
RegionMap:
  us-east-1:
    AZ1: 'us-east-1a'
    AZ2: 'us-east-1b'
  us-west-2:
    AZ1: 'us-west-2a'
    AZ2: 'us-west-2b'
  eu-west-1:
    AZ1: 'eu-west-1a'
    AZ2: 'eu-west-1b'
```

This map will be referenced using `Fn::FindInMap` to ensure consistent AZ placement across deployments.

## 6. Security Considerations

- **Security Groups**: Will create separate security groups for public and private resources
- **Network ACLs**: Will use default NACL with appropriate rules
- **IAM Roles**: Will create least-privilege IAM roles for any instances
- **Encryption**: Will enable encryption for any EBS volumes

## 7. Template Structure Plan

The CloudFormation template will be organized as follows:

1. **Parameters** section with all configurable options and safe defaults
2. **Mappings** section with RegionMap for AZ consistency
3. **Resources** section with all AWS resources in dependency order
4. **Outputs** section with useful information like VPC ID and subnet IDs

## 8. Linting Compliance Strategy

To ensure cfn-lint compliance:

- All Parameters will have `Default` values matching their `AllowedPattern`
- No hardcoded AMIs or AZs
- No use of `Fn::Sub` without variables
- All stateful resources will have both `DeletionPolicy` and `UpdateReplacePolicy`
- Latest Lambda runtimes will be used if Lambda is included
- Exact IAM policy ARNs and resource type names will be used

This architecture provides a robust, production-ready VPC setup that meets the business requirements while following AWS best practices and ensuring successful automated deployment.
