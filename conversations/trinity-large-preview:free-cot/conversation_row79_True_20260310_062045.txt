# AWS IAM Managed Policies Deployment Plan

## Key AWS Services Required
1. **AWS Identity and Access Management (IAM)** - Core service for creating and managing IAM policies
2. **Amazon S3** - Object storage service for the iOS application bucket
3. **Amazon DynamoDB** - NoSQL database requiring protection policies
4. **Amazon RDS** - Relational database service requiring protection policies

## Architecture Analysis

This deployment plan focuses on creating three IAM managed policies with specific permissions and restrictions. No infrastructure resources (EC2, VPC, etc.) are required for this use case.

## Detailed Policy Design

### 1. S3 Access Policy for iOS Application

**Purpose**: Grant read/write access to the ios-apps.sagebridge.org bucket for an iOS application

**Required Permissions**:
- `s3:PutObject` - Upload objects to the bucket
- `s3:GetObject` - Download objects from the bucket  
- `s3:DeleteObject` - Delete objects from the bucket
- `s3:ListBucket` - List objects in the bucket
- `s3:GetBucketLocation` - Get bucket location information

**Resource Specification**:
- Bucket: `arn:aws:s3:::ios-apps.sagebridge.org`
- Objects: `arn:aws:s3:::ios-apps.sagebridge.org/*`

### 2. DynamoDB Protection Policy

**Purpose**: Prevent accidental deletion or schema modifications in DynamoDB

**Required Permissions**:
- `dynamodb:DeleteTable` - Explicitly denied
- `dynamodb:UpdateTable` - Explicitly denied
- `dynamodb:UpdateTableReplicaAutoScaling` - Explicitly denied
- `dynamodb:UpdateGlobalTable` - Explicitly denied

**Resource Specification**: `*` (applies to all DynamoDB tables)

### 3. RDS Protection Policy

**Purpose**: Block destructive operations on RDS instances and clusters

**Required Permissions**:
- `rds:DeleteDBInstance` - Explicitly denied
- `rds:DeleteDBCluster` - Explicitly denied
- `rds:DeleteDBSnapshot` - Explicitly denied
- `rds:DeleteDBClusterSnapshot` - Explicitly denied

**Resource Specification**: `*` (applies to all RDS resources)

## Implementation Strategy

### Policy Creation Approach
- Use `AWS::IAM::ManagedPolicy` resources for each policy
- Apply `PolicyDocument` with explicit `Effect: Deny` statements for protection policies
- Use `Effect: Allow` with specific actions for the S3 access policy
- Set appropriate `Description` and `Path` properties for each policy

### Resource Dependencies
- No external dependencies required (IAM policies are standalone resources)
- No VPC, subnets, or other infrastructure needed
- No SSM Parameter Store lookups required (no dynamic resource fetching needed)

### Best Practices Applied
- Use descriptive policy names and descriptions
- Apply least privilege principle (only necessary permissions granted)
- Use explicit deny statements for protection policies
- Include proper versioning and documentation in policy descriptions

## Template Structure Plan

The CloudFormation template will contain:

1. **Parameters Section**: No parameters required (policies are static)
2. **Resources Section**: Three `AWS::IAM::ManagedPolicy` resources
3. **Outputs Section**: Optional outputs for policy ARNs if needed for reference

## Security Considerations

- Policies use explicit deny statements to prevent destructive actions
- S3 policy follows principle of least privilege (only required actions allowed)
- No hardcoded credentials or secrets required
- Policies can be attached to specific IAM roles or users as needed

## Validation Strategy

- Use AWS Policy Simulator to validate policy permissions
- Test deny statements by attempting restricted operations
- Verify S3 policy allows intended operations on the target bucket
- Ensure policies don't inadvertently block legitimate operations

This architecture provides a secure, production-ready approach to managing IAM policies for S3 access and database protection without requiring complex infrastructure dependencies.
