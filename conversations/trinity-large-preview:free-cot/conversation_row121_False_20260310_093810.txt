# Architectural Deployment Plan for Scheduled SSM Automation Workflow

## 1. Key AWS Services Required

Based on the business need, the following AWS services are essential:

- **Amazon Virtual Private Cloud (VPC)** - Network isolation for EC2 instances
- **Amazon EC2** - Provision Linux and Windows instances with SSM agent
- **AWS Systems Manager (SSM)** - Execute cross-platform scripts and manage associations
- **AWS EventBridge Scheduler** - Trigger SSM associations on a 15-minute schedule
- **AWS Identity and Access Management (IAM)** - Create roles for EC2 instances and EventBridge
- **Amazon S3** - Store execution logs
- **AWS SSM Parameter Store** - Dynamically fetch latest AMIs
- **AWS SSM Documents** - Define OS-specific commands
- **AWS SSM State Manager** - Create associations between schedules and instances

## 2. Properties and Dependencies

### Critical Dependencies:
1. **VPC Creation** → **Internet Gateway Attachment** → **Route Table Association** → **EC2 Instance Provisioning**
2. **IAM Roles** must be created before EC2 instances and EventBridge Scheduler
3. **SSM Document** must exist before creating the State Manager Association
4. **S3 Bucket** must be created before SSM State Manager Association (for logging)

### Property Dependencies:
- EC2 instances require IAM roles with SSM permissions
- EventBridge Scheduler requires IAM role with SSM invocation permissions
- SSM State Manager Association requires SSM Document and target instances
- S3 bucket policy requires bucket existence

## 3. Safe AWS Defaults (where properties are missing)

### Network Configuration:
- **VPC CIDR**: `10.0.0.0/16` (standard private range)
- **Public Subnet CIDR**: `10.0.0.0/24` (first subnet for public access)
- **Availability Zone**: `Fn::Select[0, Fn::GetAZs[us-east-1]]` (first AZ in region)

### EC2 Configuration:
- **Instance Type**: `t3.micro` (free tier eligible, sufficient for testing)
- **SSH Key Pair**: `default-key` (safe default, user can override)
- **Linux AMI**: `resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`
- **Windows AMI**: `resolve:ssm:/aws/service/ami-windows-latest/Windows_Server-2019-English-Full-Base`

### Security Configuration:
- **SSH Port**: `22` (Linux)
- **RDP Port**: `3389` (Windows)
- **SSM Port**: `443` (both platforms)

## 4. Dynamic Resource Fetching Strategy

### AMI Retrieval using SSM Parameter Store:
```yaml
LinuxAMI:
  Type: AWS::SSM::Parameter::Value<AWS::EC2::Image::Id>
  Default: /aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2

WindowsAMI:
  Type: AWS::SSM::Parameter::Value<AWS::EC2::Image::Id>
  Default: /aws/service/ami-windows-latest/Windows_Server-2019-English-Full-Base
```

### Availability Zone Selection:
```yaml
AZ:
  Fn::Select:
    - 0
    - Fn::GetAZs: !Ref 'AWS::Region'
```

## 5. Security Architecture

### IAM Roles:
1. **EC2InstanceRole** - For EC2 instances with SSM and S3 permissions
2. **EventBridgeSchedulerRole** - For EventBridge to invoke SSM
3. **S3BucketPolicy** - For S3 bucket access control

### Security Groups:
- **SSMSecurityGroup** - Allows inbound 443 from anywhere (SSM agent communication)
- **SSHSecurityGroup** - Allows inbound 22 from specific CIDR (optional)
- **RDPSecurityGroup** - Allows inbound 3389 from specific CIDR (optional)

## 6. SSM Document Design

The SSM document will use platform detection to execute appropriate commands:

```yaml
---
schemaVersion: '1.2'
description: Cross-platform Hello World script
mainSteps:
- action: aws:runShellScript
  name: runCommand
  inputs:
    runCommand:
      - |
        #!/bin/bash
        if [ -f /etc/os-release ]; then
          echo "Hello World from Linux!"
          echo "Linux instance ID: $(curl -s http://169.254.169.254/latest/meta-data/instance-id)"
        else
          echo "Hello World from Windows!"
          powershell.exe "Write-Host 'Windows instance ID: ' $((Get-InstanceId))"
        fi
```

## 7. EventBridge Scheduler Configuration

- **Schedule Expression**: `rate(15 minutes)`
- **Target**: SSM State Manager Association
- **IAM Role**: EventBridgeSchedulerRole with `ssm:SendCommand` permissions

## 8. SSM State Manager Association

- **Document**: The cross-platform SSM document
- **Targets**: EC2 instances with tag `HelloWorld: true`
- **Schedule**: EventBridge Scheduler ARN
- **Output Location**: S3 bucket for logs

## 9. S3 Bucket Configuration

- **Bucket Name**: Dynamically generated or user-provided
- **Policy**: Bucket policy allowing SSM to write logs
- **Encryption**: SSE-S3 (default)

## 10. Outputs

The template will provide:
- **LinuxInstanceID** and **WindowsInstanceID**
- **LogBucketName** for log access
- **EventBridgeScheduleARN** for monitoring
- **SSMDocumentName** for reference

## 11. Error Handling and Validation

- **DependsOn** clauses to ensure proper resource ordering
- **DeletionPolicy: Retain** for S3 bucket to prevent accidental data loss
- **UpdateReplacePolicy: Retain** for stateful resources
- **Validation** of tag existence before association creation

This architecture ensures a production-ready, secure, and maintainable deployment that follows AWS best practices and will pass cfn-lint validation.
