# AWS CloudFormation Deployment Plan for Windows EC2 CPU Stress Testing Environment

## Key AWS Services Required

1. **EC2** - Windows Server 2019 instance for CPU stress testing
2. **IAM** - Service roles and instance profiles for SSM and FIS integration
3. **SSM** - Systems Manager for remote command execution and parameter store access
4. **CloudWatch Logs** - Log storage and monitoring for stress test execution
5. **VPC** - Virtual network infrastructure (with fallback to default VPC)
6. **Security Groups** - Network access control for the Windows instance
7. **SSM Documents** - Custom PowerShell script for CPU stress testing

## Architecture Overview

The deployment will create a self-contained stack that provisions:
- A Windows Server 2019 EC2 instance with SSM agent pre-installed
- IAM roles for both SSM access and FIS integration
- Custom SSM document containing PowerShell CPU stress script
- Network infrastructure with intelligent VPC fallback logic
- CloudWatch log group for stress test execution monitoring

## Detailed Resource Planning

### 1. Parameters with Safe Defaults

**AMI Parameter:**
- Use SSM Parameter Store to dynamically fetch the latest Windows Server 2019 AMI
- Default: `/aws/service/ami-windows-latest/Windows_Server-2019-English-Full-Base`
- This ensures we always get the latest patched AMI without hardcoding

**Instance Type Parameter:**
- Allowed values: `t3.nano`, `t3.micro`, `t3.small`, `t3.medium`, `t3.large`
- Default: `t3.micro` (safe, cost-effective default for testing)

**Subnet Parameter:**
- Type: AWS::EC2::Subnet::Id
- Default: `""` (empty string to trigger default VPC logic)
- This allows users to specify a subnet or fall back to default VPC

**VPC Parameter:**
- Type: AWS::EC2::VPC::Id
- Default: `""` (empty string to trigger default VPC logic)

**KeyName Parameter:**
- Type: AWS::EC2::KeyPair::KeyName
- Default: `default-key` (safe dummy value that users must replace)

### 2. VPC and Network Logic

**Intelligent Fallback Mechanism:**
- If VPC parameter is empty, use `AWS::NoValue` to trigger default VPC creation
- If subnet parameter is empty, automatically create a subnet in the default VPC
- This ensures the template works in both specified and default VPC scenarios

**Security Group:**
- Allow RDP access from 0.0.0.0/0 (for testing purposes - should be restricted in production)
- Allow all outbound traffic
- Associate with the instance using the InstanceSecurityGroup property

### 3. IAM Role Architecture

**SSM Role:**
- Permissions: `AmazonSSMManagedInstanceCore` (AWS managed policy)
- Allows SSM agent to communicate with AWS Systems Manager
- Enables remote command execution via SSM

**FIS Integration Role:**
- Custom IAM role with permissions:
  - `ssm:SendCommand` - Trigger SSM documents
  - `logs:CreateLogGroup`, `logs:CreateLogStream`, `logs:PutLogEvents` - CloudWatch logging
  - `ec2:DescribeInstances` - EC2 instance management
- This role enables FIS to orchestrate the CPU stress testing

### 4. SSM Document and PowerShell Script

**Custom SSM Document:**
- Document type: `Automation`
- Content: PowerShell script that:
  - Detects logical processor count using `Get-WmiObject -Class Win32_Processor`
  - Spawns background threads equal to logical processor count
  - Each thread runs a CPU-intensive loop for configurable duration
  - Uses `Start-Job` for parallel execution
  - Implements configurable timeout (default: 300 seconds)

**Script Logic:**
```powershell
$duration = 300 # Default 5 minutes
$processors = (Get-WmiObject -Class Win32_Processor).NumberOfLogicalProcessors
1..$processors | ForEach-Object {
    Start-Job -ScriptBlock {
        $start = Get-Date
        while ((Get-Date) - $start -lt [TimeSpan]::FromSeconds($using:duration)) {
            # CPU-intensive operation
            $result = 1
            1..1000000 | ForEach-Object { $result *= $_ }
        }
    }
}
```

### 5. Instance Profile and Role Attachment

**Instance Profile Creation:**
- Create IAM instance profile resource
- Add SSM role to instance profile
- Attach instance profile to EC2 instance
- This allows the Windows instance to assume the SSM role automatically

### 6. CloudWatch Log Group

**Log Configuration:**
- Create dedicated log group for stress test execution
- Configure instance to stream PowerShell output to CloudWatch
- Set retention policy (default: 30 days)
- This provides visibility into stress test execution and results

### 7. Dependencies and Resource Ordering

**Critical Dependencies:**
1. VPC must exist before creating subnets
2. Security group must exist before creating EC2 instance
3. IAM role must exist before creating instance profile
4. Instance profile must exist before launching EC2 instance
5. SSM document must exist before it can be executed

**Resource Creation Order:**
1. IAM roles and instance profile
2. VPC and subnet (if not using default)
3. Security group
4. CloudWatch log group
5. SSM document
6. EC2 instance with all configurations

### 8. Error Handling and Validation

**Parameter Validation:**
- Instance type validation against allowed values
- Subnet/VPC validation to ensure they exist in the same region
- AMI validation through SSM parameter store

**Resource Validation:**
- Ensure instance profile has correct role attached
- Verify SSM document syntax before deployment
- Validate security group rules are correctly configured

### 9. Security Considerations

**Least Privilege Principle:**
- SSM role only has necessary SSM permissions
- FIS role has minimal required permissions for orchestration
- No admin privileges granted beyond what's necessary for stress testing

**Network Security:**
- RDP access should be restricted in production (currently open for testing)
- All outbound traffic allowed for SSM communication
- Consider adding NACLs for additional network control

### 10. Cost Optimization

**Instance Selection:**
- Use t3 instances for burstable performance and cost efficiency
- Default to t3.micro for minimal cost during testing
- Allow scaling up to t3.large for more intensive testing

**Resource Cleanup:**
- Ensure all resources are tagged for easy identification
- Consider adding deletion policies for stateful resources
- Provide clear documentation for manual cleanup if needed

This architecture provides a robust, production-ready CloudFormation template that meets all specified requirements while adhering to AWS best practices and ensuring successful deployment through proper parameterization and dynamic resource fetching.
