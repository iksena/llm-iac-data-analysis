# Detailed Architectural Deployment Plan for Secure Static Website on S3 with CloudFront

## 1. Key AWS Services Required

Based on the business requirements, the following AWS services are essential:

- **AWS CloudFormation** - Infrastructure as Code deployment
- **Amazon S3** - Static website hosting and content storage
- **Amazon CloudFront** - Global CDN for content delivery
- **AWS Lambda** - Custom resource for automated content deployment and cleanup
- **AWS Identity and Access Management (IAM)** - Service permissions and security policies
- **AWS Systems Manager (SSM) Parameter Store** - Dynamic resource fetching (though not directly used here, it's a best practice for future extensibility)

## 2. Properties and Dependencies

### Critical Dependencies and Order of Creation:

1. **S3 Bucket** → Must be created before CloudFront Origin Access Identity (OAI)
2. **Origin Access Identity (OAI)** → Created after S3 bucket, used by CloudFront
3. **S3 Bucket Policy** → Grants OAI read access, depends on both S3 bucket and OAI
4. **CloudFront Distribution** → Depends on S3 bucket, OAI, and bucket policy
5. **Lambda Function** → Created before custom resource invocation
6. **Custom Lambda Resource** → Depends on Lambda function and all resources it modifies

### Property Dependencies:

- CloudFront Origin must reference the S3 bucket's ARN
- S3 Bucket Policy must reference the OAI's Canonical User ID
- Lambda execution role must be created before the Lambda function
- Custom resource must have proper service token pointing to Lambda ARN

## 3. Safe AWS Defaults for Missing Properties

Since specific properties like CIDR blocks or instance types aren't mentioned, I'll use these safe defaults:

- **S3 Bucket Name**: `${AWS::StackName}-static-website-${AWS::Region}` (auto-generated)
- **CloudFront Price Class**: `PriceClass_100` (US, Canada, Europe - cost-effective)
- **Lambda Runtime**: `python3.12` (latest supported)
- **Lambda Timeout**: `30` seconds (default maximum)
- **CloudFront Viewer Protocol Policy**: `redirect-to-https` (enforces HTTPS)
- **CloudFront Default TTL**: `0` (caching disabled as requested)
- **S3 Bucket Versioning**: `Disabled` (unless retention is needed)
- **S3 Public Access Block**: `true` (blocks public access except for OAI)

## 4. Dynamic Resource Fetching Strategy

While AMIs aren't directly used in this static website deployment, the principle applies to Lambda layers or other resources:

- **Lambda Layers**: Use SSM Parameter Store for latest runtime layers
- **ACM Certificates**: Use `AWS::ACM::Certificate` with DNS validation
- **Latest Runtime**: Reference via `AWS::Lambda::Runtime::Python3_12` (if available)

## 5. Security Architecture

### Origin Access Identity (OAI) Implementation:
- Create OAI for CloudFront to access S3 bucket
- Apply S3 bucket policy granting OAI read permissions
- Block all other public access to the bucket
- This ensures content is only accessible via CloudFront, not directly from S3

### Lambda Security:
- Least-privilege IAM role for Lambda
- No direct internet access needed (S3 is accessed via VPC endpoints or public APIs)
- Environment variables for configuration (bucket name, file content)

## 6. Lambda Custom Resource Implementation

### Create Phase:
- Lambda function creates `index.html` with default content
- Uses S3 PutObject API
- Returns success/failure to CloudFormation

### Delete Phase:
- Lambda function lists all objects in bucket
- Deletes all objects to prevent retention conflicts
- Returns success/failure to CloudFormation

### Error Handling:
- Implement retries for S3 operations
- Return meaningful error messages to CloudFormation
- Use try-catch blocks for all operations

## 7. CloudFormation Template Structure

### Resources Section:
1. S3 Bucket
2. Origin Access Identity
3. S3 Bucket Policy
4. Lambda Execution Role
5. Lambda Function
6. CloudFront Distribution
7. Custom Lambda Resource (Create)
8. Custom Lambda Resource (Delete)

### Parameters Section:
- Optional: Custom domain name (with safe default)
- Optional: Default HTML content (with safe default)
- Optional: Price class (with safe default)

### Metadata Section:
- cfn-lint configuration
- AWS::CloudFormation::Interface for UI hints

## 8. Best Practices Implementation

### S3 Modernization:
- Use `AWS::S3::BucketPolicy` instead of `AccessControl` property
- Implement `OwnershipControls` if public access is ever needed
- Use `PublicAccessBlockConfiguration` to block public access

### Lambda Best Practices:
- Use `/tmp` directory for temporary files
- Implement proper error handling and logging
- Use environment variables for configuration

### CloudFormation Best Practices:
- Use `DeletionPolicy: Retain` for S3 bucket to prevent accidental deletion
- Use `UpdateReplacePolicy: Retain` to match deletion policy
- Implement proper DependsOn attributes
- Use dynamic references for any secrets (though none are needed here)

## 9. Testing and Validation Strategy

### Pre-deployment:
- Validate CloudFormation template with cfn-lint
- Test Lambda function locally with sam-cli
- Verify IAM permissions are minimal

### Post-deployment:
- Test CloudFront distribution with HTTPS
- Verify S3 bucket is not publicly accessible
- Test custom resource creation and deletion
- Validate content is served correctly via CloudFront

### Monitoring:
- Enable CloudFront access logs
- Monitor Lambda function logs
- Set up CloudWatch alarms for failures

This architecture ensures a secure, automated, and production-ready static website deployment that follows all AWS best practices and CloudFormation linting rules.
