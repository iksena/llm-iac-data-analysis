# Detailed Architectural Deployment Plan for EC2 Instance with Security Group

## Key AWS Services Required

1. **VPC** - Virtual Private Cloud for network isolation
2. **InternetGateway** - For internet connectivity
3. **VPCGatewayAttachment** - To attach the Internet Gateway to the VPC
4. **Subnet** - Network segment for the EC2 instance
5. **RouteTable** - For routing traffic
6. **Route** - To route traffic through the Internet Gateway
7. **SubnetRouteTableAssociation** - To associate the route table with the subnet
8. **SecurityGroup** - Network security controls for the EC2 instance
9. **EC2 Instance** - The compute resource
10. **IAM Role** (optional) - For instance profile if needed

## Properties and Dependencies

The deployment follows this critical dependency chain:
1. VPC must be created first
2. Internet Gateway must be created and attached to VPC
3. Route Table must be created with a route to the Internet Gateway
4. Subnet must be created
5. Subnet must be associated with the Route Table
6. Security Group must be created (can reference VPC)
7. EC2 Instance must be created (can reference Security Group and Subnet)

## Safe AWS Defaults

Since specific properties weren't provided, I'll use these safe defaults:

- **VPC CIDR**: `10.0.0.0/16` (standard private range)
- **Subnet CIDR**: `10.0.1.0/24` (single subnet within VPC)
- **EC2 Instance Type**: `t3.micro` (free tier eligible, suitable for testing)
- **AMI**: Latest Amazon Linux 2 using SSM Parameter Store
- **SSH Port**: `22` (standard for SSH access)
- **Region**: Will use `AWS::Region` pseudo parameter for dynamic region resolution
- **KeyName**: `default-key` (safe default parameter value)

## Dynamic Resource Fetching Strategy

**AMI Resolution**: I'll use SSM Parameter Store to fetch the latest Amazon Linux 2 AMI:
```
resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2
```

This ensures the template always uses the latest patched AMI without hardcoding.

**Availability Zones**: I'll use `Fn::GetAZs` and `Fn::Select` to dynamically fetch available AZs instead of hardcoding.

## Security Group Configuration

The Security Group will:
- Allow inbound SSH (port 22) from anywhere (0.0.0.0/0) - can be restricted later
- Allow all outbound traffic (0.0.0.0/0 on all ports)
- Reference the VPC created in the template

## EC2 Instance Configuration

The EC2 instance will:
- Use the dynamically resolved AMI
- Use `t3.micro` instance type
- Be placed in the created subnet
- Use the created Security Group
- Have a public IP address assigned
- Use the default key pair name parameter

## Parameter Strategy

I'll create a single parameter for KeyName with:
- Parameter name: `KeyName`
- Default value: `default-key`
- Description: "Name of an existing EC2 KeyPair to enable SSH access"

This follows the rule of never defining parameters without defaults.

## Template Structure

The template will follow this logical order:
1. Parameters section (KeyName)
2. Resources section with:
   - VPC
   - InternetGateway + VPCGatewayAttachment
   - RouteTable + Route
   - Subnet
   - SubnetRouteTableAssociation
   - SecurityGroup
   - EC2 Instance

## Linting Compliance

The template will:
- Use only allowed CloudFormation properties
- Avoid hardcoded values (AMIs, AZs)
- Use proper intrinsic functions
- Include all required properties for each resource type
- Follow AWS naming conventions
- Use current AWS managed policy ARNs if IAM is needed

This architecture ensures a fully functional, deployable CloudFormation template that creates a complete network stack with a secure EC2 instance, following all AWS best practices and linting rules.
