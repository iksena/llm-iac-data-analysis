# AWS Batch Event-Driven Pipeline Architecture Plan

## Key AWS Services Required

1. **VPC & Networking**: `AWS::EC2::VPC`, `AWS::EC2::Subnet`, `AWS::EC2::SecurityGroup`, `AWS::EC2::InternetGateway`, `AWS::EC2::VPCGatewayAttachment`
2. **Batch Processing**: `AWS::Batch::ComputeEnvironment`, `AWS::Batch::JobQueue`, `AWS::Batch::JobDefinition`
3. **Event Processing**: `AWS::SQS::Queue`, `AWS::EventSchemas::Schema`, `AWS::Pipes::Pipe`
4. **IAM**: `AWS::IAM::Role` (multiple roles for Batch, Pipes, ECS)
5. **Compute**: `AWS::ECS::Cluster` (managed by Batch)
6. **Storage**: `AWS::S3::Bucket` (for job artifacts/logs)

## Architecture Overview

The pipeline follows an event-driven architecture where:
1. SQS receives messages (e.g., "OrderCreated")
2. EventBridge Pipes filters and routes messages to Batch
3. Batch processes containerized jobs in isolated VPC subnets
4. ECS executes containers using Fargate or EC2 capacity

## Detailed Resource Properties & Dependencies

### 1. VPC Configuration
- **CIDR Blocks**: Default to `10.0.0.0/16` for VPC, `10.0.1.0/24` (public) and `10.0.2.0/24` (private) for subnets
- **Dependencies**: Internet Gateway must be attached to VPC before creating VPC Gateway Attachment
- **Security Groups**: Create separate groups for Batch compute and SQS access

### 2. IAM Roles & Policies
- **Batch Service Role**: Trust relationship with `batch.amazonaws.com`
- **Batch Instance Role**: For EC2 instances (if using EC2 capacity)
- **Pipes Execution Role**: Trust relationship with `pipes.amazonaws.com`
- **ECS Execution Role**: For Fargate task execution
- **Policies**: Use AWS managed policies where possible (e.g., `AWSBatchServiceRole`, `AmazonEC2ContainerServiceforEC2Role`)

### 3. Batch Environment
- **Compute Environment**: Managed type with `desiredvCpus: 0` for cost optimization
- **Instance Types**: Default to `optimal` for EC2, or use Fargate for serverless
- **Subnets**: Use private subnets for compute resources
- **Security Groups**: Allow outbound internet access via NAT Gateway

### 4. SQS Queue
- **Visibility Timeout**: Default to 30 seconds
- **Message Retention**: Default to 345,600 seconds (4 days)
- **Redrive Policy**: Configure dead-letter queue for failed messages

### 5. EventBridge Pipes
- **Source**: SQS queue ARN
- **Target**: Batch job queue ARN
- **Filter Criteria**: Match on message attributes (e.g., `detail-type` equals "OrderCreated")
- **Enrichment**: Optional - enrich messages before sending to Batch

### 6. Batch Job Definition
- **Container Properties**: Define image, memory, vCPU requirements
- **Command**: Default to empty array (overrideable)
- **Environment Variables**: Pass SQS message attributes as environment variables
- **Retry Strategy**: Configure retry attempts and intervals

## Dynamic Resource Fetching

### AMI Selection
```yaml
Fn::ImportValue: !Sub "aws/service/ecs/optimized-ami/amazon-linux-2/recommended"
```
Or use SSM Parameter Store:
```yaml
Fn::Sub: "aws:ssm:/aws/service/ecs/optimized-ami/amazon-linux-2/recommended"
```

### Availability Zones
```yaml
Fn::GetAZs: !Ref "AWS::Region"
Fn::Select: [0, !GetAZs '']
```

## Security Best Practices

1. **VPC Isolation**: All Batch compute in private subnets
2. **IAM Least Privilege**: Grant only necessary permissions to each role
3. **Network ACLs**: Configure inbound/outbound rules for Batch subnets
4. **Secrets Management**: Use AWS Secrets Manager for sensitive data
5. **Logging**: Enable CloudWatch logging for all components

## Cost Optimization

1. **Compute Environment**: Set `desiredvCpus: 0` to avoid idle costs
2. **Spot Instances**: Enable spot capacity for cost savings
3. **Fargate**: Use serverless compute to eliminate infrastructure management
4. **Lifecycle Policies**: Configure S3 bucket lifecycle rules for old logs

## Error Handling & Monitoring

1. **Dead Letter Queues**: Configure for SQS and Batch job failures
2. **CloudWatch Alarms**: Monitor queue depth, job failures, and resource utilization
3. **EventBridge Rules**: Create rules for error events and notifications
4. **Batch Job Retry**: Configure exponential backoff for transient failures

## Template Structure

The CloudFormation template will be organized as follows:

1. **Parameters**: All configurable options with safe defaults
2. **Mappings**: AMI mappings and other static configurations
3. **Resources**: All AWS resources in logical order
4. **Outputs**: Key ARNs and endpoints for integration

This architecture ensures a scalable, secure, and cost-effective event-driven processing pipeline that can handle varying workloads while maintaining isolation and compliance requirements.
