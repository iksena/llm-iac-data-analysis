# Detailed Architectural Deployment Plan

## Key AWS Services Required

1. **VPC** - Core networking infrastructure
2. **Internet Gateway** - Public internet access
3. **NAT Gateway** - Private subnet outbound internet access
4. **Elastic IP** - Static IP for NAT Gateway
5. **Subnets** - 3 public and 3 private across 3 AZs
6. **Route Tables** - Public and private routing
7. **Security Groups** - Network access control
8. **Application Load Balancer** - Internal load balancing
9. **ECR Repository** - Container image storage
10. **SSM Parameter Store** - Cross-service resource referencing

## Architecture Overview

The deployment will create a highly available VPC with:
- 1 public subnet per AZ (3 total) for internet-facing resources
- 2 private subnets per AZ (6 total) for backend services
- Internet Gateway for public subnet internet access
- NAT Gateway for private subnet outbound internet access
- Internal ALB for load balancing across private subnets
- ECR repository for container image storage
- SSM parameters for resource ID referencing

## Properties and Dependencies

### VPC Configuration
- **CIDR Block**: Default to `10.0.0.0/16` (safe default for production)
- **EnableDnsSupport**: `true`
- **EnableDnsHostnames**: `true`

### Subnet Configuration
- **Public Subnets**: 3 subnets with CIDR blocks `10.0.1.0/24`, `10.0.2.0/24`, `10.0.3.0/24`
- **Private Subnets**: 6 subnets with CIDR blocks `10.0.10.0/24`, `10.0.11.0/24`, `10.0.12.0/24`, `10.0.20.0/24`, `10.0.21.0/24`, `10.0.22.0/24`
- **Availability Zones**: Dynamically fetched using `Fn::GetAZs` and `Fn::Select`

### Internet Gateway
- Attached to VPC before provisioning Elastic IP
- **DeletionPolicy**: `Retain` with corresponding `UpdateReplacePolicy`

### NAT Gateway
- Requires Elastic IP allocation before creation
- Placed in public subnet for internet access
- **DeletionPolicy**: `Retain` with corresponding `UpdateReplacePolicy`

### Route Tables
- **Public Route Table**: Routes `0.0.0.0/0` to Internet Gateway
- **Private Route Table**: Routes `0.0.0.0/0` to NAT Gateway

### Security Groups
- **HTTP/HTTPS SG**: Ingress ports 80, 443 from `0.0.0.0/0`
- **SSH SG**: Ingress port 22 from `0.0.0.0/0` (for initial setup)
- **PostgreSQL SG**: Ingress port 5432 from VPC CIDR
- **Internal SG**: Ingress all ports from VPC CIDR
- **ALB SG**: Ingress ports 80, 443 from VPC CIDR

### Application Load Balancer
- **Scheme**: `internal`
- **Type**: `application`
- **Subnets**: All 6 private subnets
- **Default Action**: Fixed response on port 80
- **DeletionPolicy**: `Retain` with corresponding `UpdateReplacePolicy`

### ECR Repository
- **ImageScanningConfiguration**: `{ scanOnPush: true }`
- **DeletionPolicy**: `Retain` with corresponding `UpdateReplacePolicy`

## SSM Parameter Store Integration

Store the following resource IDs for cross-service referencing:
- `/vpc/id` - VPC ID
- `/vpc/public-subnets` - Comma-separated public subnet IDs
- `/vpc/private-subnets` - Comma-separated private subnet IDs
- `/security-groups/http-https` - HTTP/HTTPS security group ID
- `/security-groups/ssh` - SSH security group ID
- `/security-groups/postgres` - PostgreSQL security group ID
- `/security-groups/internal` - Internal security group ID
- `/alb/listener-arn` - ALB listener ARN

## Dynamic Resource Fetching

### AMI Selection
Use SSM Parameter Store to fetch latest Amazon Linux 2 AMI:
```yaml
Fn::ImportValue: !Sub 'resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2'
```

### Availability Zones
Dynamically fetch AZs for the region:
```yaml
Fn::GetAZs: !Ref 'AWS::Region'
```

## Security Best Practices

1. **Security Groups**: Use least privilege principles, allow only necessary traffic
2. **ECR**: Enable image scanning on push for vulnerability detection
3. **SSM Parameters**: Store resource IDs for secure cross-service referencing
4. **NAT Gateway**: Provides outbound internet access while blocking inbound traffic

## Error Handling and Validation

1. **CIDR Validation**: Ensure no overlapping CIDR blocks
2. **Dependency Management**: Use `DependsOn` for proper resource creation order
3. **Parameter Validation**: Use `AllowedPattern` for CIDR blocks and other inputs
4. **Resource Limits**: Check AWS service quotas for VPCs, subnets, and security groups

## Production Readiness Considerations

1. **High Availability**: Resources distributed across 3 AZs
2. **Scalability**: ALB can scale automatically based on traffic
3. **Security**: Multiple security groups for different access patterns
4. **Maintainability**: SSM parameters enable easy resource referencing
5. **Cost Optimization**: NAT Gateway only in public subnets, ECR with scanning

This architecture provides a solid foundation for deploying containerized applications with proper networking, security, and high availability considerations.
