# ECS Cluster with Configurable Monitoring - Architecture Plan

## Key AWS Services Required

1. **VPC** - Virtual Private Cloud for network isolation
2. **Internet Gateway** - Internet connectivity for public subnets
3. **Subnets** - Public and private subnets across multiple AZs
4. **Route Tables** - Routing configuration for subnets
5. **Security Groups** - Network access control for ECS instances
6. **IAM Roles & Instance Profiles** - Permissions for ECS instances
7. **Auto Scaling Group** - Managed EC2 instance scaling
8. **Launch Configuration** - EC2 instance configuration
9. **ECS Cluster** - Container orchestration
10. **CloudWatch Alarms & Dashboards** - Monitoring and alerting
11. **SSM Parameter Store** - Dynamic AMI fetching

## Properties and Dependencies

### Critical Dependencies:
- VPC must be created before Subnets
- Subnets must exist before Route Tables can reference them
- Internet Gateway must be attached to VPC before Route Table can use it
- IAM Roles must exist before Instance Profiles can reference them
- Security Groups must exist before Launch Configuration can reference them

### Property Dependencies:
- Auto Scaling Group requires Launch Configuration
- ECS Cluster requires IAM Role for service access
- CloudWatch Alarms require Metric Namespaces and Dimensions

## Safe AWS Defaults (where values are missing)

### Network Configuration:
- **VPC CIDR**: `10.0.0.0/16` (standard private range)
- **Public Subnet CIDRs**: `10.0.1.0/24`, `10.0.2.0/24`, `10.0.3.0/24`
- **Private Subnet CIDRs**: `10.0.11.0/24`, `10.0.12.0/24`, `10.0.13.0/24`
- **Availability Zones**: Dynamically fetched using `Fn::GetAZs` and `Fn::Select`

### EC2 Configuration:
- **Instance Type**: `t3.medium` (balanced performance/cost)
- **AMI**: Fetched dynamically using SSM Parameter Store
- **KeyPair**: Default parameter with 'default-key' fallback
- **Volume Size**: 50GB (standard for ECS workloads)

### ECS Configuration:
- **Cluster Name**: Derived from stack name
- **Desired Capacity**: 2 instances (minimum viable cluster)
- **Max Capacity**: 4 instances (allows for scaling)

## Dynamic Resource Fetching Strategy

### AMI Retrieval:
```yaml
AMALatestECSOptimized:
  Type: AWS::SSM::Parameter::Value<AWS::EC2::Image::Id>
  Default: /aws/service/ecs/optimized-ami/amazon-linux-2/recommended
```

### Availability Zones:
```yaml
Fn::Select:
  - 0
  Fn::GetAZs: !Ref 'AWS::Region'
```

## Security & Monitoring Architecture

### Security Groups:
- **ECS Instance SG**: Allows inbound 80/443 from ALB SG, SSH from admin IPs
- **ALB SG**: Allows inbound 80/443 from anywhere, outbound to ECS instances
- **Admin Access**: Configurable CIDR for SSH access

### IAM Roles:
- **ECSInstanceRole**: EC2 instance permissions for ECS agent
- **ECSServiceRole**: ECS service permissions for cluster management
- **CloudWatchRole**: Permissions for ECS metrics publishing

### Monitoring Configuration:
- **CloudWatch Alarms**: CPU utilization, memory reservation, container count
- **Dashboard**: ECS cluster metrics visualization
- **Retention Period**: 30 days (configurable)

## Template Structure Plan

### Parameters Section:
- VPC configuration (CIDR, subnets)
- EC2 configuration (instance type, key pair)
- ECS configuration (desired capacity, max capacity)
- Security configuration (admin CIDR, monitoring retention)
- Monitoring configuration (alarm thresholds, dashboard visibility)

### Resources Section:
1. Network Resources (VPC, Subnets, IGW, Route Tables)
2. Security Resources (Security Groups, IAM Roles)
3. Compute Resources (Launch Config, Auto Scaling Group, ECS Cluster)
4. Monitoring Resources (CloudWatch Alarms, Dashboard)

### Outputs Section:
- ECS Cluster ARN
- VPC ID
- Security Group IDs
- CloudWatch Dashboard URL

## Best Practices Implementation

### Resource Protection:
- **DeletionPolicy**: Retain for VPC, Security Groups, IAM Roles
- **UpdateReplacePolicy**: Retain for stateful resources

### Security Hardening:
- No SSH access by default (requires admin CIDR)
- Least privilege IAM policies
- Security group isolation between layers

### Cost Optimization:
- Tiered instance types based on workload
- Auto Scaling with cooldown periods
- CloudWatch retention configurable to reduce costs

### Scalability Considerations:
- Multi-AZ deployment for high availability
- Configurable instance counts
- Support for Spot instances (future enhancement)

This architecture provides a production-ready ECS foundation that can be extended with additional services like Application Load Balancers, RDS databases, or custom monitoring solutions.
