# Serverless AWS Organizations Data Collection Pipeline - Architecture Plan

## Key AWS Services Required

1. **S3** - For storing Organizations metadata
2. **Lambda** - For fetching Organizations data
3. **IAM** - For cross-account role assumption and permissions
4. **CloudWatch Events** - For scheduling weekly execution
5. **Glue** - For data cataloging
6. **Athena** - For SQL-based querying
7. **SSM Parameter Store** - For dynamic resource fetching

## Architecture Overview

The solution follows a serverless event-driven architecture with the following flow:
1. CloudWatch Events triggers Lambda weekly
2. Lambda assumes cross-account role to fetch Organizations data
3. Data is written to S3 in JSON format
4. Glue crawler catalogs data into Athena database
5. Users query data via Athena

## Detailed Component Analysis

### 1. S3 Bucket Configuration

**Properties:**
- Encryption: SSE-S3 (server-side encryption with S3-managed keys)
- Access Control: Bucket Policy with explicit deny for public access
- Versioning: Enabled for data retention
- Tags: Configurable via template parameters

**Dependencies:** None

**Defaults:** 
- Bucket name: `organizations-data-${AWS::AccountId}-${AWS::Region}`
- Encryption: SSE-S3

### 2. Lambda Function

**Properties:**
- Runtime: Python 3.12 (latest supported)
- Memory: 128 MB (conservative for metadata fetching)
- Timeout: 30 seconds (sufficient for Organizations API calls)
- Handler: `lambda_function.lambda_handler`
- Environment variables: Management account ID, target S3 bucket

**Dependencies:**
- IAM role with OrganizationsReadOnly and STS assume permissions
- S3 bucket for output

**Defaults:**
- Runtime: `python3.12`
- Memory: `128`
- Timeout: `30`

### 3. IAM Roles and Policies

**Lambda Execution Role:**
- Managed policies: `AWSLambdaBasicExecutionRole`
- Custom policy: OrganizationsReadOnly + STS assume role

**Cross-account Role:**
- Trust relationship: Lambda execution role
- Permissions: `organizations:Describe*` actions

**Glue Role:**
- Managed policies: `AWSGlueServiceRole`, `AmazonS3FullAccess` (scoped later)

**Dependencies:** S3 bucket ARNs for policy resources

### 4. CloudWatch Events Rule

**Properties:**
- Schedule: Weekly (cron expression `0 0 ? * 7#1 *` - first Sunday of month at midnight)
- Target: Lambda function

**Dependencies:** Lambda function ARN

### 5. Glue Crawler

**Properties:**
- Database: Configurable via parameter (default: `managementcur`)
- Targets: S3 path with JSON data
- Schedule: On-demand (triggered manually or via Lambda completion)

**Dependencies:** S3 bucket path, Athena database

### 6. Parameters Design

**Required Parameters:**
- `ManagementAccountId`: String, no default (must be provided)
- `TargetS3Bucket`: String, default generated from template
- `AthenaDatabaseName`: String, default `managementcur`
- `TagsToInclude`: Comma-separated list, default `Environment,Owner,Project`

**Optional Parameters:**
- `EncryptionType`: String, default `SSE-S3`
- `ScheduleExpression`: String, default weekly cron

## Security and Best Practices Implementation

### IAM Security
- Principle of least privilege for all roles
- Cross-account trust relationships explicitly defined
- No wildcard permissions in policies

### Data Protection
- S3 encryption at rest
- IAM roles for service-to-service access (no hardcoded credentials)
- Secure parameter handling

### Operational Excellence
- Structured logging in Lambda
- Error handling with retries
- Monitoring via CloudWatch Logs

## Dynamic Resource Fetching Strategy

### AMI Handling
Not applicable for this serverless architecture.

### Availability Zones
Not applicable for this serverless architecture.

### Other Dynamic Resources
- Use `AWS::AccountId` and `AWS::Region` pseudo parameters
- Construct ARNs using string concatenation
- Use SSM Parameter Store for any future AWS service parameters

## Error Handling and Resilience

### Lambda Error Handling
- Retry logic for Organizations API calls
- Dead-letter queue configuration for failed executions
- Structured error logging

### S3 Error Handling
- Eventual consistency handling for new bucket operations
- Retry logic for S3 write operations

### IAM Error Handling
- Proper error messages for missing cross-account role
- Validation of IAM role trust relationships

## Deployment Sequence

1. Create S3 bucket
2. Create IAM roles and policies
3. Deploy Lambda function
4. Create CloudWatch Events rule
5. Configure Glue crawler
6. Export resource ARNs

## Testing Strategy

### Unit Testing

- Test IAM role permissions
- Validate JSON output format

### Integration Testing
- End-to-end pipeline execution
- Cross-account role assumption
- S3 write permissions
- Glue crawler functionality

### Security Testing

- S3 bucket policies
- Cross-account access validation
