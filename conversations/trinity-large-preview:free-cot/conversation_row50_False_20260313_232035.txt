# Architectural Deployment Plan for URL-to-PNG Screenshot Service

## 1. Key AWS Services Required

Based on the business requirements, the following AWS services are essential:

- **VPC** with public and private subnets across two Availability Zones
- **EC2** instances for worker nodes and load test instance
- **Auto Scaling Group** with scaling policies
- **SQS** queue for message processing
- **S3** bucket for storing PNG screenshots
- **IAM** roles and policies for service permissions
- **CloudWatch** alarms and metrics for monitoring
- **Security Groups** for network security
- **Elastic Load Balancer** (optional, for distributing traffic)
- **SSM Parameter Store** for dynamic AMI fetching

## 2. Properties and Dependencies

### Network Dependencies
- VPC must be created before subnets
- Internet Gateway must be attached to VPC before route tables can reference it
- Route tables must be associated with subnets after creation

### IAM Dependencies
- IAM roles must be created before EC2 instances can reference them
- IAM policies must be attached to roles before instances launch

### SQS and S3 Dependencies
- SQS queue must exist before EC2 instances can consume from it
- S3 bucket must be created before EC2 instances can write to it

### Auto Scaling Dependencies
- Launch configuration or template must exist before Auto Scaling Group creation
- Scaling policies must be created after the Auto Scaling Group

## 3. Safe AWS Defaults

### VPC Configuration
- **CIDR Block**: `10.0.0.0/16` (standard private range)
- **Public Subnets**: `10.0.1.0/24` and `10.0.2.0/24`
- **Private Subnets**: `10.0.3.0/24` and `10.0.4.0/24`

### EC2 Configuration
- **Instance Type**: `t3.micro` (cost-effective for testing)
- **SSH Key Pair**: `default-key` (safe default parameter)
- **AMI**: Use SSM Parameter Store to fetch latest Amazon Linux 2 AMI
- **Availability Zones**: Use `Fn::GetAZs` and `Fn::Select` to dynamically select

### S3 Configuration
- **Bucket Name**: Use unique name with timestamp or random suffix
- **Encryption**: SSE-S3 by default for security

### SQS Configuration
- **Visibility Timeout**: `300` seconds (5 minutes for screenshot processing)
- **Message Retention**: `345600` seconds (4 days)

## 4. Dynamic Resource Fetching Strategy

### AMI Fetching
Use SSM Parameter Store to dynamically fetch the latest Amazon Linux 2 AMI:
```yaml
ImageId: !Sub 'resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2'
```

### Availability Zones
Use intrinsic functions to dynamically select AZs:
```yaml
!Select [0, !GetAZs !Ref 'AWS::Region']
!Select [1, !GetAZs !Ref 'AWS::Region']
```

## 5. Architecture Overview

### Network Architecture
- **VPC**: 10.0.0.0/16 with public and private subnets across two AZs
- **Internet Gateway**: For public internet access
- **NAT Gateways**: In public subnets for private instances to access internet
- **Route Tables**: Separate for public and private subnets

### Security Architecture
- **Security Groups**: 
  - Worker SG: Allow HTTP/HTTPS outbound, SSH from admin IP
  - Load Test SG: Allow HTTP/HTTPS outbound, SSH from admin IP
- **IAM Roles**: 
  - Worker Role: SQS read, S3 write permissions
  - Load Test Role: SQS write permissions

### Processing Architecture
- **SQS Queue**: Standard queue for URL requests
- **S3 Bucket**: For storing PNG screenshots with versioning enabled
- **Auto Scaling Group**: 1-2 instances based on queue depth
- **CloudWatch Alarms**: 
  - Scale up when queue depth > 10 messages
  - Scale down when queue depth < 2 messages

### Load Testing Architecture
- **Dedicated EC2 Instance**: For populating queue with test URLs
- **Cron Job**: Scheduled task to generate test traffic
- **Monitoring**: CloudWatch metrics for queue depth and processing time

## 6. Scaling Strategy

### CloudWatch Alarms
- **High Queue Depth Alarm**: `ApproximateNumberOfMessagesVisible > 10` for 5 minutes
- **Low Queue Depth Alarm**: `ApproximateNumberOfMessagesVisible < 2` for 5 minutes

### Scaling Policies
- **Scale Up**: Add 1 instance when high queue depth alarm triggers
- **Scale Down**: Remove 1 instance when low queue depth alarm triggers
- **Cooldown**: 300 seconds between scaling activities

## 7. Security Considerations

### IAM Policies
- **Least Privilege**: Workers only need SQS read and S3 write permissions
- **No Admin Access**: No instances have full admin privileges
- **Role Separation**: Different roles for workers and load test instance

### Network Security
- **Private Subnets**: Workers in private subnets for better security
- **Public Access**: Only load test instance in public subnet
- **SSH Restrictions**: SSH access limited to specific admin IP ranges

### Data Protection
- **S3 Encryption**: Server-side encryption enabled by default
- **VPC Endpoints**: Consider VPC endpoints for S3 access without internet
- **Logging**: Enable S3 and CloudWatch logging for audit trails

## 8. Monitoring and Logging

### CloudWatch Metrics
- **Queue Depth**: ApproximateNumberOfMessagesVisible
- **Processing Time**: Custom metrics from worker instances
- **Instance Health**: Auto Scaling Group metrics
- **S3 Operations**: Bucket metrics for write operations

### Logging Strategy
- **CloudWatch Logs**: For application logs from EC2 instances
- **S3 Access Logs**: For bucket access auditing
- **CloudTrail**: For API call logging and security analysis

This architectural plan provides a comprehensive, secure, and scalable foundation for the URL-to-PNG screenshot service while adhering to AWS best practices and CloudFormation linting rules.
