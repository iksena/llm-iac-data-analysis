# REST API Gateway to SQS Integration Architecture Plan

## Key AWS Services Required

1. **Amazon API Gateway** - REST API endpoint for HTTP-based message retrieval
2. **Amazon SQS** - Message queue for storing and retrieving messages
3. **AWS IAM** - Identity and Access Management for permissions
4. **AWS Systems Manager (SSM) Parameter Store** - For dynamic resource fetching (if needed)

## Architecture Overview

The solution will create a serverless architecture where:
- API Gateway exposes a GET endpoint that clients can call to retrieve messages
- API Gateway directly integrates with SQS using AWS service integration
- IAM permissions allow API Gateway to invoke SQS ReceiveMessage action
- SQS queue has a policy allowing API Gateway to access it

## Detailed Implementation Plan

### 1. API Gateway Configuration

**REST API Resource:**
- Create a new REST API with a single resource (e.g., `/messages`)
- Configure a GET method on this resource
- Use AWS service integration to connect directly to SQS

**Integration Setup:**
- Integration type: AWS Service
- AWS region: Use `AWS::Region` pseudo-parameter
- AWS Service: SQS
- HTTP method: POST (SQS requires POST for ReceiveMessage)
- Integration URI: `arn:aws:apigateway:${AWS::Region}:sqs:path/${AWS::AccountId}/${QueueName}`

**Integration Request Mapping:**
- Configure request parameters to map API Gateway query parameters to SQS ReceiveMessage parameters
- Map `QueueUrl` from the URI
- Map other SQS parameters (MaxNumberOfMessages, VisibilityTimeout, WaitTimeSeconds) from query string parameters

### 2. SQS Queue Configuration

**Queue Properties:**
- Standard queue (default)
- Configure appropriate visibility timeout (default: 30 seconds)
- Configure message retention period (default: 345,600 seconds/4 days)
- Configure maximum message size (default: 262,144 bytes)

**Queue Policy:**
- Create an IAM policy allowing API Gateway to send `ReceiveMessage` requests
- Principal: The API Gateway execution role
- Action: `sqs:ReceiveMessage`
- Resource: The SQS queue ARN

### 3. IAM Role Configuration

**API Gateway Execution Role:**
- Create an IAM role for API Gateway
- Trust policy allowing API Gateway to assume the role
- Permission policy allowing `sqs:ReceiveMessage` on the specific queue

**Permissions:**
- Use least privilege principle
- Only grant necessary permissions to the specific SQS queue
- Consider adding CloudWatch permissions for logging if needed

### 4. Dependencies and Properties

**Dependencies:**
1. IAM Role must be created before API Gateway (API Gateway needs the role ARN)
2. SQS Queue must be created before API Gateway integration (needs queue URL/ARN)
3. Queue Policy depends on both the queue and the IAM role

**Properties:**
- Use `AWS::Region` for region-specific configurations
- Use `AWS::AccountId` for account-specific ARNs
- Use `AWS::URLSuffix` for region-specific endpoints

### 5. Dynamic Resource Fetching

**SSM Parameter Store Usage:**
- Not required for this specific template as we're using service ARNs
- If needed for future extensions (like Lambda functions), use:
  - `resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2` for latest Amazon Linux AMI
  - Other SSM parameters for dynamic configurations

### 6. Safe Defaults (Based on Best Practices)

**API Gateway:**
- Deployment stage: 'prod' (default)
- Method timeout: 30000 ms (default)
- API endpoint type: REGIONAL (default)

**SQS Queue:**
- Visibility timeout: 30 seconds (default)
- Message retention period: 345600 seconds (default)
- Maximum message size: 262144 bytes (default)

**IAM Role:**
- Use AWS managed policies where applicable
- Create custom policies for specific permissions

### 7. Security Considerations

**Least Privilege:**
- Grant only necessary permissions to API Gateway
- Use resource-level permissions for SQS
- Consider VPC endpoint for SQS if using VPC

**Logging and Monitoring:**
- Enable API Gateway execution logging
- Consider CloudWatch metrics for monitoring
- Set up appropriate error handling

### 8. Error Handling and Validation

**API Gateway:**
- Configure appropriate error responses
- Set up integration timeouts
- Handle SQS-specific errors (queue not found, permission denied)

**SQS:**
- Configure dead-letter queue if needed
- Set up appropriate retry policies

### 9. Template Structure

The CloudFormation template will follow this logical order:
1. Parameters (with safe defaults)
2. IAM Role
3. SQS Queue
4. Queue Policy
5. API Gateway REST API
6. API Gateway Resource
7. API Gateway Method
8. API Gateway Integration
9. API Gateway Deployment
10. API Gateway Stage

This architecture provides a secure, scalable, and serverless solution for HTTP-based SQS message retrieval without requiring any compute resources or custom code.
