# Serverless Slot Machine Web Application - CloudFormation Architecture Plan

## Key AWS Services Required

1. **S3** - Static website hosting for frontend assets
2. **Lambda** - Serverless function to simulate slot machine spins
3. **Cognito Identity Pool** - Anonymous user authentication and temporary credentials
4. **IAM** - Role-based access control for Lambda and Cognito identities
5. **SSM Parameter Store** - Dynamic AMI fetching (if needed for future expansion)
6. **CloudFormation Custom Resource** - Automated deployment of frontend assets

## Architecture Overview

The solution follows a serverless architecture pattern where:
- Frontend assets are served from S3 as a static website
- Anonymous users authenticate via Cognito Identity Pool
- Frontend JavaScript uses temporary credentials to invoke Lambda
- Lambda function processes slot machine logic and returns results
- Custom Lambda resource automates deployment during stack creation

## Detailed Resource Breakdown

### 1. S3 Bucket Configuration

**Properties:**
- `AccessControl: PublicRead` (legacy, but required for static website)
- `OwnershipControls` (required when using AccessControl)
- `WebsiteConfiguration` with `IndexDocument: index.html`
- `AWS::S3::BucketPolicy` for public read access

**Dependencies:**
- Bucket must exist before applying policy
- Policy depends on bucket creation

### 2. Lambda Function

**Properties:**
- Runtime: `python3.12` (latest supported)
- Handler: `index.handler`
- Memory: 128 MB (minimum sufficient for this use case)
- Timeout: 10 seconds (adequate for slot simulation)

**Code Structure:**
```python
def handler(event, context):
    # Randomize slot results
    # Determine win/loss based on combinations
    # Return JSON response
```

**Dependencies:**
- IAM role must exist before function creation
- Role requires logging permissions

### 3. Cognito Identity Pool

**Properties:**
- `AllowUnauthenticatedIdentities: true`
- Linked to IAM roles for authenticated/unauthenticated users

**IAM Role for Unauthenticated Users:**
- Policy: `lambda:InvokeFunction` on the slot machine Lambda
- Least privilege principle applied

### 4. IAM Roles

**Lambda Execution Role:**
- Managed policy: `AWSLambdaBasicExecutionRole`
- Custom inline policy for CloudWatch Logs

**Cognito Unauthenticated Role:**
- Custom inline policy allowing only `lambda:InvokeFunction`
- Resource-specific ARN for the slot machine Lambda

### 5. Custom Lambda Resource for Deployment

**Purpose:** Automate frontend deployment during stack creation

**Properties:**
- Runtime: `python3.12`
- Handler: `index.handler`
- Timeout: 300 seconds (to allow for file operations)

**Functionality:**
1. Download pre-built ZIP from specified URL
2. Extract contents
3. Inject Cognito Identity Pool ID and region into frontend JavaScript
4. Upload modified files to S3 bucket

**Dependencies:**
- S3 bucket must exist before deployment
- Requires S3 read/write permissions

## Properties and Dependencies Analysis

### S3 Bucket Dependencies
- `AccessControl` requires `OwnershipControls` (linting rule)
- Bucket must be created before applying `AWS::S3::BucketPolicy`
- Website configuration requires bucket existence

### Lambda Dependencies
- Execution role must exist before function creation
- Function depends on IAM role
- Custom resource depends on S3 bucket

### IAM Dependencies
- Roles must be created before Lambda function
- Policies attached to roles before function creation

### Custom Resource Dependencies
- Depends on S3 bucket creation
- Depends on Lambda function creation (for policy injection)
- Requires S3 read/write permissions

## Safe Defaults and Dynamic Resource Fetching

### Missing Properties - Safe Defaults

**S3 Bucket:**
- `AccessControl: PublicRead` (required for static website)
- `OwnershipControls` with `Rules` allowing `ObjectWriter`

**Lambda Function:**
- `MemorySize: 128` (minimum viable)
- `Timeout: 10` (adequate for slot simulation)
- `Runtime: python3.12` (latest supported)

**Cognito Identity Pool:**
- No defaults needed - required properties provided

**IAM Roles:**
- Use AWS managed policies where applicable
- Custom inline policies for specific permissions

### Dynamic Resource Fetching

**AMIs:** Not required for this serverless architecture, but if needed:
```yaml
Fn::ImportValue: !Sub "resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2"
```

**AZs:** Not required for this architecture, but if needed:
```yaml
Fn::GetAZs: !Ref "AWS::Region"
```

## Security Considerations

1. **Least Privilege:** IAM roles grant only necessary permissions
2. **Anonymous Access:** Cognito provides temporary credentials without user accounts
3. **Public S3:** Only the static website bucket is public; no backend resources exposed
4. **No Hardcoded Secrets:** All configurations passed via CloudFormation parameters

## Deployment Sequence

1. Create S3 bucket with website configuration
2. Create IAM roles (Lambda execution and Cognito unauthenticated)
3. Create Lambda function with execution role
4. Create Cognito Identity Pool with linked roles
5. Deploy custom resource for frontend automation
6. Custom resource downloads, modifies, and uploads frontend assets

## Linting Compliance

The template will adhere to all specified linting rules:
- All Parameters have safe Default values
- No hardcoded AMIs or AZs
- S3 buckets use both AccessControl and OwnershipControls
- DeletionPolicy and UpdateReplacePolicy included where applicable
- No unnecessary Fn::Sub usage
- Latest Lambda runtime used
- Exact IAM policy ARNs and resource type names used

This architecture provides a secure, scalable, and automated serverless slot machine application that meets all business requirements while following AWS best practices and CloudFormation linting standards.
