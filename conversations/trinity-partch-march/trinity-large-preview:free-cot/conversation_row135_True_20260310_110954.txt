# Architectural Deployment Plan for CloudFront + API Gateway with Mock Backend

## 1. Key AWS Services Required

Based on the business need, the following AWS services are required:

- **Amazon CloudFront** - For global content delivery and HTTPS termination
- **Amazon API Gateway** - For REST API hosting with mock integration
- **AWS Lambda** - Not required (mock integration will be used)
- **AWS Systems Manager (SSM) Parameter Store** - For dynamic resource fetching (if needed)
- **AWS Certificate Manager (ACM)** - For SSL/TLS certificates (if needed)
- **AWS Identity and Access Management (IAM)** - For service permissions

## 2. Resource Properties and Dependencies

### CloudFront Distribution Properties:
- `DistributionConfig` with `Origins` pointing to API Gateway
- `DefaultCacheBehavior` with `TargetOriginId` and `ViewerProtocolPolicy`
- `Enabled: true` to activate the distribution
- `PriceClass` (default: 'PriceClass_100' for cost optimization)
- `ViewerCertificate` for HTTPS configuration

### API Gateway Properties:
- `RestApi` with `EndpointConfiguration` set to 'REGIONAL'
- `Resource` for `/hello` endpoint
- `Method` for GET with `Mock` integration type
- `Deployment` for production stage
- `Stage` named 'prod'

### Dependencies:
1. API Gateway must be created before CloudFront origin configuration
2. Deployment must exist before Stage can be created
3. Stage must exist before CloudFront can reference it

## 3. Safe AWS Defaults

Since specific properties weren't provided, I'll use these safe defaults:

- **API Gateway endpoint**: Regional (for lower latency within region)
- **CloudFront Price Class**: 'PriceClass_100' (North America and Europe only)
- **CloudFront Viewer Protocol Policy**: 'redirect-to-https'
- **API Gateway Mock Response**: Static JSON `{"message": "hello world"}`
- **CloudFront Origin Protocol Policy**: 'https-only'
- **Cache TTL**: 0 seconds (to disable caching as requested)

## 4. Dynamic Resource Fetching Strategy

For this specific use case, dynamic fetching isn't strictly necessary because:
- API Gateway endpoints are constructed using the API ID and stage name
- No AMIs or other dynamic resources are required

However, if we needed to fetch dynamic resources in the future, we would use:
- `Fn::ImportValue` for cross-stack references
- SSM Parameter Store for latest AMI IDs or other dynamic values
- `Fn::GetAZs` and `Fn::Select` for availability zone handling

## 5. Architecture Design

The solution will follow this structure:

```
CloudFront Distribution (HTTPS)
    ↓
API Gateway Regional Endpoint
    ↓
Mock Integration (static response)
```

### Key Design Decisions:

1. **Mock Integration**: Since the requirement is for a static "hello world" response, we'll use API Gateway's Mock integration type instead of Lambda, reducing complexity and cost.

2. **HTTPS Enforcement**: CloudFront will redirect HTTP to HTTPS, and communicate with API Gateway over HTTPS.

3. **No Caching**: Cache TTL will be set to 0 to ensure real-time responses.

4. **Regional API Gateway**: Using regional endpoints for lower latency and cost compared to edge-optimized endpoints.

5. **IAM Permissions**: Minimal permissions - only what's necessary for CloudFront to access the API Gateway.

## 6. Implementation Strategy

The CloudFormation template will be structured as follows:

1. **Parameters**: Define any configurable values with safe defaults
2. **Resources**: Create API Gateway resources first, then CloudFront
3. **Outputs**: Export the CloudFront distribution URL for easy access

## 7. Security Considerations

- CloudFront will use HTTPS for both viewer requests and origin requests
- API Gateway will be private to the CloudFront distribution (no public access)
- No IAM roles required for this simple mock setup
- All communication will be encrypted in transit

## 8. Cost Optimization

- Using Mock integration (no Lambda costs)
- Price Class 100 for CloudFront (limited geographic scope)
- Regional API Gateway (lower cost than edge-optimized)

This architecture provides a secure, cost-effective solution that meets all the specified requirements while following AWS best practices.
