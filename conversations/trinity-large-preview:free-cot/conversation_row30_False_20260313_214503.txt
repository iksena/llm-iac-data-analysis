# Detailed Architectural Deployment Plan for EFS with Cross-AZ EC2 Instances

## Key AWS Services Required

1. **VPC** - Virtual Private Cloud for network isolation
2. **EC2** - Elastic Compute Cloud instances for application hosting
3. **EFS** - Elastic File System for shared storage
4. **IAM** - Identity and Access Management for security roles
5. **CloudWatch** - Monitoring and alerting
6. **SSM** - Systems Manager for AMI resolution and instance management
7. **Auto Scaling** - Optional, for scaling capabilities
8. **S3** - Optional, for storing scripts or backups
9. **CloudFormation** - Infrastructure as Code deployment

## Architecture Overview

The solution will create a VPC with public subnets across multiple AZs, an EFS file system with mount targets in each AZ, two EC2 instances in separate AZs with IAM roles for EFS access and SSM permissions, automated EFS mounting via user data scripts with retry logic, and CloudWatch alarms for performance monitoring.

## Detailed Component Breakdown

### 1. VPC Configuration

**Properties:**
- CIDR block: 10.0.0.0/16 (default)
- Public subnets: 10.0.0.0/24, 10.0.1.0/24 (one per AZ)
- Internet Gateway attached to VPC
- Network ACLs allowing inbound/outbound traffic on ports 22, 80, 443, and NFS (2049)
- Route tables with default route pointing to Internet Gateway

**Dependencies:**
- Internet Gateway must be attached to VPC before creating route associations
- Subnets must exist before creating EFS mount targets

### 2. EFS File System

**Properties:**
- Performance mode: GeneralPurpose (default)
- Throughput mode: Bursting (default)
- Encrypted: True (default for security)
- Lifecycle policies: None (default)
- Cross-region replication: Not configured (default)

**Dependencies:**
- VPC must exist before creating EFS
- Subnets must exist before creating mount targets

### 3. EFS Mount Targets

**Properties:**
- One mount target per AZ in public subnets
- Security groups allowing NFS traffic (port 2049) from EC2 instances
- IP addresses assigned automatically

**Dependencies:**
- EFS file system must exist
- Subnets must exist
- Security groups must be created

### 4. IAM Roles and Instance Profiles

**IAM Role Properties:**
- Managed policies: AmazonSSMManagedInstanceCore, AmazonElasticFileSystemClientReadWrite
- Trust relationship allowing EC2 service to assume role
- Instance profile to attach role to EC2 instances

**Dependencies:**
- IAM role must be created before instance profile
- Instance profile must exist before launching EC2 instances

### 5. EC2 Instances

**Properties:**
- AMI: Latest Amazon Linux 2 using SSM parameter store
- Instance type: t3.micro (default, cost-effective)
- Key pair: default-key (safe default parameter)
- User data script for EFS mounting with retry logic
- CreationPolicy with cfn-signal for successful mount verification
- IAM instance profile for EFS and SSM access

**Dependencies:**
- VPC and subnets must exist
- Security groups must be created
- IAM instance profile must exist
- EFS mount targets must be available

### 6. CloudWatch Alarms

**Alarm Properties:**
- EFS I/O limits: Alarm when >95% of burst credits consumed
- Throughput utilization: Alarm when >80% of provisioned throughput used
- Metrics: FileSystemId, StorageBytes, BurstCreditBalance, PercentIOLimit, PercentThroughputLimit
- Actions: SNS topic for notifications

**Dependencies:**
- EFS file system must exist
- SNS topic must be created (if notifications required)

### 7. Security Groups

**Properties:**
- EFS security group: Allows NFS traffic (port 2049) from EC2 instances
- EC2 security group: Allows SSH (port 22) from anywhere, HTTP/HTTPS (ports 80/443) from anywhere
- Both security groups associated with appropriate resources

**Dependencies:**
- VPC must exist before creating security groups
- Security groups must exist before associating with resources

## Dynamic Resource Resolution

### AMI Resolution
Use SSM Parameter Store to fetch latest Amazon Linux 2 AMI:
```
resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2
```

### Availability Zones
Use Fn::GetAZs and Fn::Select to dynamically fetch AZs:
```
Fn::Select: [0, Fn::GetAZs: !Ref AWS::Region]
Fn::Select: [1, Fn::GetAZs: !Ref AWS::Region]
```

## User Data Script Logic

The EC2 user data script will:
1. Install EFS client utilities
2. Wait for EFS mount targets to become available (retry logic with timeout)
3. Create mount point directory
4. Mount EFS file system
5. Add entry to /etc/fstab for persistence
6. Signal CloudFormation with success/failure status

## CreationPolicy Implementation

Use CreationPolicy with cfn-signal to ensure:
- EC2 instances successfully mount EFS before stack creation completes
- Stack fails if mounting fails after timeout period
- Prevents partial deployments with instances that cannot access shared storage

## Security Considerations

- EFS encryption at rest enabled
- IAM roles with least privilege access
- Security groups controlling network access
- No hardcoded credentials or secrets
- SSM for secure instance management

## Cost Optimization

- Use t3.micro instances for cost-effectiveness
- EFS GeneralPurpose performance mode for balanced performance/cost
- Monitor CloudWatch metrics to optimize EFS provisioning
- Consider lifecycle policies for infrequent access data

## Failure Scenarios and Recovery

- Auto-healing via Auto Scaling group (if implemented)
- CloudWatch alarms for proactive monitoring
- IAM roles for secure access without hardcoded credentials
- User data retry logic for EFS mount availability
- CreationPolicy preventing incomplete deployments

This architecture provides a robust, production-ready EFS solution with cross-AZ accessibility, automated deployment, and comprehensive monitoring while adhering to AWS best practices and CloudFormation linting rules.
