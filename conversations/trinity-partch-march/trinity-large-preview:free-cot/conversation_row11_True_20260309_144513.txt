# DynamoDB Table for Mobile Notifications Reporting - Architecture Plan

## Key AWS Services Required
1. **AWS::DynamoDB::Table** - Core service for creating the DynamoDB table
2. **AWS::DynamoDB::Table/AttributeDefinition** - Define table attributes
3. **AWS::DynamoDB::Table/KeySchema** - Define primary key structure
4. **AWS::DynamoDB::Table/GlobalSecondaryIndex** - Create GSI for time-based queries
5. **AWS::DynamoDB::Table/ProvisionedThroughput** - Configure billing mode
6. **AWS::DynamoDB::Table/TimeToLiveSpecification** - Enable TTL for automatic cleanup

## Properties and Dependencies Analysis

### Table Configuration
- **TableName**: `MobileNotificationsReporting` (explicit naming for clarity)
- **BillingMode**: `PAY_PER_REQUEST` (cost-efficient for unpredictable workloads)
- **TimeToLiveSpecification**: Enabled with `AttributeName: "ttl"` and `Enabled: true`

### Primary Key Design
- **Partition Key**: `notificationId` (String) - Unique identifier for each notification
- **Sort Key**: `timestamp` (Number) - Event timestamp for ordering

### Global Secondary Index (GSI)
- **IndexName**: `TypeSentTimeIndex`
- **Partition Key**: `type` (String) - Notification type/category
- **Sort Key**: `sentTime` (Number) - Time when notification was sent
- **Projection**: `ALL` - Include all attributes for comprehensive queries

### TTL Configuration
- **AttributeName**: `ttl` (Number) - Unix timestamp for expiration
- **Enabled**: `true` - Activate TTL feature

## Best Practices Implementation

### 1. No Hardcoded Values
- No AMI IDs needed (DynamoDB is serverless)
- No Availability Zones needed (DynamoDB is region-scoped)

### 2. Security Considerations
- DynamoDB tables are private by default
- No IAM policies needed for basic table creation
- Consider adding resource-level permissions if integrating with applications

### 3. Cost Optimization
- **PAY_PER_REQUEST** billing mode automatically scales with traffic
- No provisioned capacity to manage
- TTL reduces storage costs by automatically deleting old data

### 4. Data Model Design
The schema supports:
- Point queries by notification ID
- Time-series queries by timestamp
- Type-based filtering with time range queries via GSI
- Automatic cleanup of expired notifications

### 5. Error Handling & Validation
- Attribute types are explicitly defined to prevent schema mismatches
- GSI keys match the attribute definitions
- TTL attribute is included in the table schema

## Deployment Considerations

- Verify GSI queries return expected results
- Confirm expired items are automatically deleted
- Test various query patterns (by ID, by type+time range)

This architecture provides a scalable, cost-effective solution for mobile notifications reporting with automatic data lifecycle management and efficient query patterns for common use cases.
