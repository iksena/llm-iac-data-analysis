## 4. Dynamic Resource Fetching

For this template, dynamic fetching is not required since:
- IAM roles don't need AMIs

- AWS managed policy ARNs are stable

## 5. Architecture Plan

I'll create a CloudFormation template with the following structure:

### Template Overview
- **AWSTemplateFormatVersion**: 2010-09-09
- **Description**: IAM roles with tiered permissions for FIS access control
- **Resources**: Four IAM roles with granular policies

### Role-by-Role Breakdown

#### Security Admins Role
- **Permissions**: Full FIS access, role-passing privileges, service-linked role creation
- **Policies**: 
  - AWS managed policy: AWSReadOnlyAccess
  - Custom inline policy for FIS full access
  - Custom inline policy for role-passing to workshop-specific service roles
  - Custom inline policy for service-linked role creation

#### Admins Role
- **Permissions**: Similar to Security Admins but with production FIS resource restrictions
- **Policies**:
  - AWS managed policy: AWSReadOnlyAccess
  - Custom inline policy for FIS access (excluding production-tagged resources)
  - Custom inline policy for role-passing to workshop-specific service roles
  - Custom inline policy for service-linked role creation

#### Users Role
- **Permissions**: Limited to experiment management and tagging, no production access
- **Policies**:
  - AWS managed policy: AWSReadOnlyAccess
  - Custom inline policy for experiment start/stop operations
  - Custom inline policy for FIS resource viewing
  - Custom inline policy for experiment tagging
  - Explicit deny for production FIS resources

#### Non-Users Role
- **Permissions**: Read-only AWS access, no FIS actions
- **Policies**:
  - AWS managed policy: AWSReadOnlyAccess
  - Explicit deny for all FIS actions

### Key Implementation Details

#### Policy Structure
- Use **IAM::Policy** resources for custom permissions
- Implement **Deny** statements using explicit `Effect: Deny` for production restrictions
- Use **Condition** blocks for tag-based access control
- Reference **AWS managed policies** using their exact ARNs

#### FIS Resource Access
- Use **Resource** blocks with `*` for broad access where appropriate
- Use **Condition** blocks with `aws:ResourceTag` for production environment detection
- Implement **tag-based filtering** for production resource exclusion

#### Role-Passing Implementation
- Create **custom inline policies** allowing specific service roles to be assumed
- Use **Principal** blocks with service-specific ARNs
- Implement **workshop-specific** service role references

#### Service-Linked Role Creation
- Use **iam:CreateServiceLinkedRole** permissions
- Specify **AWS service prefixes** for supported services
- Implement **conditional creation** based on service requirements

### Security Best Practices Applied

- **Principle of Least Privilege**: Each role has only necessary permissions
- **Explicit Denies**: Production resources are explicitly denied, not just omitted
- **Tag-Based Access Control**: Production detection uses resource tags
- **Managed Policies**: Foundation built on AWS managed policies
- **No Hardcoded Secrets**: No credentials or sensitive data in the template

### Template Validation Strategy

The template will be validated against:
- **AWS CloudFormation schema** for resource property compliance
- **cfn-lint** for best practices and linting rules
- **IAM policy simulator** for permission accuracy
- **Tag-based condition logic** for production environment detection

This architecture ensures a secure, scalable, and maintainable IAM role structure that meets the business requirements while adhering to AWS best practices and CloudFormation standards.
