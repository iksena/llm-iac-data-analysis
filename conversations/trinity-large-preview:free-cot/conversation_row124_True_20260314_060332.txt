- **Amazon DynamoDB** - For storing and retrieving music data
- **AWS IAM** - For creating roles and policies for service-to-service access
- **AWS Systems Manager (SSM) Parameter Store** - For dynamically fetching the latest Lambda runtime AMI (though Lambda is serverless, this is a best practice for any EC2 dependencies)
- **AWS CloudFormation** - For infrastructure as code deployment

### Resource Dependencies:
1. **DynamoDB Table** must be created before Lambda functions can be deployed
2. **IAM Roles and Policies** must exist before API Gateway and Lambda can assume them
3. **API Gateway** must be created before methods can be attached
4. **Usage Plan** must reference existing API keys and stages

### Critical Properties:
- **DynamoDB Table**: Partition key (`Artist`), Sort key (`Album`), Global Secondary Index (`Artist-Index`)
- **Lambda Functions**: Runtime (`python3.12`), Timeout (30 seconds), Memory (128 MB minimum)
- **API Gateway**: API key authentication, Usage plan with rate limits
- **IAM Roles**: Trust policies for API Gateway and Lambda, permissions to DynamoDB

## 3. Safe AWS Defaults for Missing Properties

Since specific properties like CIDR blocks or instance types are not mentioned, I will use the following safe defaults:

- **API Gateway**: Default stage name (`prod`), Default throttling (10,000 requests per second, 5,000 burst)
- **DynamoDB**: Default read/write capacity (5 RCUs/5 WCUs), Default index projection (ALL)
- **Lambda**: Runtime (`python3.12`), Timeout (30 seconds), Memory (128 MB)
- **IAM**: Default inline policies with least privilege
- **API Keys**: Default API key name (`music-api-key`), Default usage plan name (`music-usage-plan`)

## 4. Dynamic Resource Fetching Strategy

### Lambda Runtime:
- Use `resolve:ssm:/aws/service/lambda/python3.12` to fetch the latest Python 3.12 runtime
- This ensures the template always uses the most recent stable runtime without hardcoding

### No EC2 Dependencies:
- Since this is a serverless architecture (Lambda + API Gateway + DynamoDB), there are no EC2 instances or AMIs to fetch
- If EC2 were needed in the future, I would use SSM Parameter Store with patterns like:
  - `resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`

## 5. Security Architecture

### Authentication & Authorization:
- **API Key Authentication**: All API methods require a valid API key
- **IAM Roles**: Least-privilege policies for API Gateway and Lambda to access DynamoDB
- **Usage Plan**: Rate limiting (10,000 req/s) and throttling (5,000 burst) to prevent abuse

### Data Protection:
- **DynamoDB Encryption**: Server-side encryption enabled by default
- **IAM Policies**: Resource-level permissions to restrict access to the Music table only

## 6. Architecture Flow

1. **Create DynamoDB Table** with primary key (Artist, Album) and GSI (Artist-Index)
2. **Create IAM Roles**:
   - API Gateway role to invoke Lambda
   - Lambda role to access DynamoDB
3. **Deploy Lambda Functions**:
   - POST function to add music entries
   - GET function to query by artist
4. **Create API Gateway**:
   - Define resources (/music)
   - Add POST and GET methods with API key authentication
5. **Configure Usage Plan** with rate limits and associate with API key
6. **Deploy API** to production stage

## 7. Error Handling & Best Practices

### CloudFormation Best Practices:
- Use `DeletionPolicy: Retain` for DynamoDB to prevent accidental data loss
- Include `UpdateReplacePolicy` whenever `DeletionPolicy` is specified
- Use `Fn::Sub` only when variables are present in the string
- Validate all IAM policy ARNs and resource types against current AWS documentation

### Lambda Best Practices:
- Use environment variables for configuration (table name, region)
- Implement proper error handling and logging
- Set appropriate timeouts based on DynamoDB query patterns

This architecture provides a secure, scalable, and maintainable solution for managing music data through REST APIs, with proper authentication, authorization, and rate limiting to protect the backend resources.
