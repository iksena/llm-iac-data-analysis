# Detailed Architectural Deployment Plan

## 1. Key AWS Services Required

Based on the business need, the following AWS services are required:

- **Amazon VPC** - To create a virtual network for the EC2 instance
- **EC2** - To host the web application
- **S3** - To store static assets (images, jQuery libraries)
- **DynamoDB** - To store structured data in the `AWS-Services` table
- **IAM** - To create an instance profile with S3 and DynamoDB permissions
- **SSM Parameter Store** - To dynamically fetch the latest Amazon Linux 2 AMI
- **AWS SDK for PHP** - To be installed on the EC2 instance for programmatic access to AWS services

## 2. Properties and Dependencies

### VPC Dependencies
- VPC must be created before Subnets
- Internet Gateway must be attached to VPC before Route Table creation
- Route Table must have a route to the Internet Gateway before EC2 instance launch

### EC2 Dependencies
- Security Group must exist before EC2 instance creation
- IAM Instance Profile must be created before EC2 instance launch
- Subnet must exist before EC2 instance launch

### S3 Dependencies
- Bucket must be created before IAM policy references it
- IAM policy must exist before attaching to instance profile

### DynamoDB Dependencies
- Table must be created before IAM policy references it
- IAM policy must exist before attaching to instance profile

## 3. Safe AWS Defaults (Where Values Were Not Specified)

### VPC Configuration
- **VPC CIDR**: `10.0.0.0/16` (standard private range)
- **Public Subnet CIDR**: `10.0.1.0/24` (single public subnet)
- **Availability Zone**: Dynamically selected using `Fn::GetAZs` and `Fn::Select`

### EC2 Configuration
- **Instance Type**: `t3.micro` (free tier eligible, suitable for web applications)
- **AMI**: Dynamically fetched using SSM Parameter Store: `/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`
- **KeyName**: `default-key` (safe default, user can override)
- **SSH Location**: `0.0.0.0/0` (for simplicity, though this should be restricted in production)

### S3 Configuration
- **Bucket Name**: Dynamically generated using `!Sub` with stack name to ensure uniqueness
- **Region**: Dynamically fetched using `!Ref AWS::Region`

### DynamoDB Configuration
- **Table Name**: `AWS-Services` (as specified)
- **Partition Key**: `ServiceName` (String)
- **Sort Key**: `ServiceId` (String)

## 4. Dynamic Resource Fetching Strategy

### AMI Fetching
Use SSM Parameter Store to get the latest Amazon Linux 2 AMI:
```yaml
!Sub '/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2'
```

### Availability Zone Selection
Use `Fn::GetAZs` to get all AZs in the region, then `Fn::Select` to pick the first one:
```yaml
!Select [0, !GetAZs '']
```

### Region Detection
Use `!Ref AWS::Region` to dynamically get the deployment region.

## 5. Security and Best Practices Implementation

### IAM Role Configuration
- Create an IAM role with managed policies for S3 and DynamoDB access
- Attach the role to the EC2 instance via an Instance Profile
- Use least-privilege permissions (though the requirement states "full access")

### Security Group Configuration
- Allow SSH access from the specified source (or default 0.0.0.0/0)
- Allow HTTP access from anywhere (0.0.0.0/0) on port 80

### UserData Script Strategy
The UserData script will:
1. Install Apache, PHP, and AWS SDK for PHP
2. Download lab files from S3
3. Configure PHP files with dynamic region and bucket information
4. Populate DynamoDB table from JSON files
5. Start Apache service

## 6. Template Structure Plan

The CloudFormation template will be organized as follows:

1. **Parameters** - With safe defaults for all configurable values
2. **Resources** - In logical dependency order:
   - VPC and networking components
   - IAM role and instance profile
   - S3 bucket
   - DynamoDB table
   - Security Group
   - EC2 instance with UserData
3. **Outputs** - To display the public IP and S3 bucket URL

## 7. Error Prevention Strategy

To ensure the template passes cfn-lint and deploys successfully:

- All Parameters will have `Default` values matching their `AllowedPattern`
- No hardcoded AMI IDs or AZs will be used
- S3 buckets will use `BucketPolicy` instead of `AccessControl`
- All stateful resources will have both `DeletionPolicy` and `UpdateReplacePolicy`
- No unnecessary `Fn::Sub` usage
- All IAM ARNs will be verified for current validity

This architectural plan ensures a robust, automated deployment that meets all specified requirements while adhering to AWS best practices and CloudFormation linting rules.
