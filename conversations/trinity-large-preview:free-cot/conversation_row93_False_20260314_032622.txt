# Detailed Architectural Deployment Plan for Tag-Driven Nginx EC2 Instance with SSM and Monitoring

## 1. Key AWS Services Required

Based on the business need, the following AWS services are essential:

- **EC2**: To provision the Linux 2 instance
- **IAM**: To create an instance profile and role for SSM access and S3 permissions
- **SSM (AWS Systems Manager)**: For automated Nginx installation via associations and enhanced monitoring
- **S3**: For centralized logging storage
- **CloudWatch**: For performance monitoring and metrics
- **SSM Parameter Store**: To dynamically fetch the latest Amazon Linux 2 AMI

## 2. Properties and Dependencies

### Dependencies:
1. **IAM Role Creation**: Must be created before EC2 instance provisioning to attach to the instance
2. **S3 Bucket Creation**: Must exist before IAM policy creation that references it
3. **SSM Association**: Depends on both the EC2 instance and the SSM document being available

### Properties:
- **VPC and Networking**: While not explicitly mentioned, an EC2 instance requires a VPC, subnet, and security group
- **IAM Permissions**: Need SSM permissions and S3 read/write permissions for the dedicated bucket
- **Instance Tags**: Required to trigger the SSM association for Nginx installation

## 3. Safe AWS Defaults for Missing Properties

Since specific properties like CIDR blocks, instance types, or region are not provided, I'll use these safe defaults:

- **Instance Type**: `t3.micro` (free tier eligible, suitable for Nginx)
- **CIDR Blocks**: `10.0.0.0/16` for VPC, `10.0.0.0/24` for subnet
- **Key Pair**: Parameter with default value 'default-key' (user must have this key pair or create it)
- **Region**: Use `AWS::Region` pseudo parameter to make it dynamic
- **Availability Zone**: Use `Fn::GetAZs` and `Fn::Select` to dynamically select first AZ

## 4. Dynamic Resource Fetching Strategy

### AMI Fetching:
Use SSM Parameter Store to dynamically fetch the latest Amazon Linux 2 AMI:
```
resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2
```
This ensures the template always uses the latest AMI without hardcoding.

### Other Dynamic References:
- Use `AWS::Region` for region-specific resources
- Use `AWS::AccountId` for account-specific resources
- Use `AWS::URLSuffix` for region-specific endpoints

## 5. Architecture Overview

The deployment will follow this sequence:

1. **Create VPC and Networking Components**: VPC, Internet Gateway, Route Table, Subnet, and Security Group
2. **Create S3 Bucket**: For centralized logging with appropriate policy
3. **Create IAM Role and Instance Profile**: With SSM permissions and S3 access to the logging bucket
4. **Create EC2 Instance**: With the IAM role attached, proper tags, and user data
5. **Create SSM Document**: For Nginx installation script
6. **Create SSM Association**: That triggers on the instance tag to run the Nginx installation
7. **Configure CloudWatch Monitoring**: For performance insights

## 6. Security and Best Practices Implementation

- **Principle of Least Privilege**: IAM role will have only necessary permissions (SSM and specific S3 bucket access)
- **No Hardcoded Secrets**: Use dynamic references or parameters without defaults for sensitive data
- **Tag-Driven Automation**: SSM association will target instances with specific tags
- **Centralized Logging**: All logs will flow to the dedicated S3 bucket
- **Monitoring**: CloudWatch integration for performance metrics

## 7. Template Structure Plan

The CloudFormation template will be organized as follows:

1. **Parameters**: For customizable values (KeyName, BucketName, etc.) with safe defaults
2. **Resources**: All AWS resources in logical order of creation
3. **Outputs**: For useful information like instance ID, bucket name, and public IP

## 8. SSM Association Strategy

The SSM association will:
- Target instances with a specific tag (e.g., `InstallNginx: true`)
- Execute a shell script that installs, configures, and starts Nginx
- Ensure consistent configuration across all tagged instances
- Run automatically when the tag is applied or the instance is launched with the tag

## 9. Monitoring and Logging Strategy

- **CloudWatch Metrics**: EC2 instance metrics (CPU, network, disk usage)
- **CloudWatch Logs**: Integration with EC2 for system logs
- **S3 Logging**: All application and access logs stored in the dedicated S3 bucket
- **SSM Inventory**: Enhanced monitoring through SSM for software inventory and patch compliance

This architectural plan ensures a robust, automated, and observable deployment that meets all the specified requirements while adhering to AWS best practices and CloudFormation linting rules.
