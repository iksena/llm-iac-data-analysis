- Secrets Manager secret must be created before Aurora cluster

### Parameter Group Dependencies
- Custom parameter group must be created before attaching to Aurora cluster

- **Availability Zones**: Dynamically selected using `Fn::GetAZs` and `Fn::Select`
- **DB Instance Class**: `db.t3.large` (as specified)

- **Database Name**: `mydb` (default)
- **Master Username**: `admin` (default)
- **Key Pair**: `default-key` (safe default parameter)

For this Aurora deployment, I'll use:

- **SSM Parameter Store** for fetching the latest Aurora MySQL engine version if needed
- **AWS Secrets Manager** for storing database credentials securely
- **Dynamic AZ selection** using `Fn::GetAZs` and `Fn::Select` to ensure cross-region compatibility

## 5. Architecture Design

### Network Architecture
- Single VPC with two public subnets across two AZs
- Internet Gateway attached to VPC
- Route table with default route to IGW for public subnets
- Security group allowing MySQL traffic (port 3306) from anywhere (0.0.0.0/0)

### Database Architecture
- Custom DB parameter group with `time_zone` and `sql_mode` settings
- DB subnet group spanning both public subnets
- Aurora MySQL cluster with two db.t3.large instances
- Secrets Manager for master password storage

### Security Considerations
- Public accessibility as requested (though typically databases are private)
- Security group controlling access to port 3306
- Encrypted storage using AWS-managed KMS key
- IAM authentication enabled for Aurora

## 6. Parameter Group Configuration

The custom parameter group will enforce:
- `time_zone` = 'UTC'
- `sql_mode` = 'STRICT_TRANS_TABLES,NO_ZERO_IN_DATE,NO_ZERO_DATE,ERROR_FOR_DIVISION_BY_ZERO,NO_AUTO_CREATE_USER,NO_ENGINE_SUBSTITUTION'

## 7. Template Structure

The CloudFormation template will follow this logical order:
1. Network resources (VPC, subnets, IGW, route tables)
2. Database resources (parameter group, subnet group, secrets)
3. Aurora cluster and instances
4. Outputs for connection information

## 8. Compliance with Best Practices

The template will:
- Use safe default values for all parameters
- Dynamically select AZs for cross-region compatibility
- Include both `DeletionPolicy` and `UpdateReplacePolicy` for stateful resources
- Use dynamic references for secrets
- Follow proper resource dependency ordering
- Include appropriate IAM permissions
- Use latest Aurora MySQL engine version

This architecture provides a production-ready Aurora MySQL cluster with redundancy across two AZs, proper network isolation, and secure credential management while adhering to AWS best practices and CloudFormation linting rules.
