# EventBridge to MongoDB API Integration Architecture Plan

## Key AWS Services Required

1. **AWS::Events::EventBus** - Custom event bus for capturing application events
2. **AWS::Events::ApiDestination** - API destination for MongoDB endpoint
3. **AWS::Events::Connection** - Connection resource with API key authentication
4. **AWS::IAM::Role** - IAM role for EventBridge to invoke MongoDB API
5. **AWS::SQS::Queue** - Dead-letter queue for failed event deliveries
6. **AWS::SQS::QueuePolicy** - Queue policy for dead-letter queue
7. **AWS::SNS::Topic** - Optional notification topic for failed deliveries
8. **AWS::SNS::Subscription** - Optional subscription for notifications

## Properties and Dependencies

1. **EventBus** - No dependencies, creates custom event bus
2. **Connection** - Depends on IAM role being created first
3. **ApiDestination** - Depends on Connection and IAM role
4. **SQS Queue** - No dependencies, creates dead-letter queue
5. **Queue Policy** - Depends on SQS queue creation
6. **IAM Role** - No dependencies, but must be created before Connection and ApiDestination

## Safe AWS Defaults

- **MongoDB URL**: 'www.testanu.com' (as provided)
- **ApiKeyName**: 'test_name' (as provided)
- **ApiKeyValue**: 'test_value' (as provided)
- **EventBusName**: 'CustomEventBus' (default name)
- **ApiDestinationName**: 'MongoDBApiDestination' (default name)
- **ConnectionName**: 'MongoDBConnection' (default name)
- **QueueName**: 'EventBridgeDeadLetterQueue' (default name)
- **IAM RoleName**: 'EventBridgeMongoDBInvokerRole' (default name)

## Architecture Design

### 1. EventBridge Event Bus
- Create a custom event bus to capture application events
- No specific properties needed beyond the name

### 2. IAM Role Configuration
- Create an IAM role with permissions to invoke the MongoDB API endpoint
- Trust relationship allowing EventBridge to assume the role
- Policy allowing EventBridge to invoke the API destination

### 3. Connection Resource
- Create a connection with API key authentication
- Use the provided ApiKeyName and ApiKeyValue
- Reference the IAM role for permissions

### 4. API Destination
- Create an API destination pointing to the MongoDB URL
- Reference the connection resource
- Configure invocation rate limits (default: 1 request per second)

### 5. Dead-Letter Queue
- Create an SQS queue for failed event deliveries
- Configure appropriate retention period (default: 14 days)
- Create a queue policy allowing EventBridge to send messages

### 6. Event Rules (Optional Enhancement)
- Create event rules to route specific event patterns to the API destination
- Could include filtering criteria for different event types

## Security Considerations

1. **API Key Security**: The API key is hardcoded as 'test_value' per requirements, but in production this should use AWS Secrets Manager or Parameter Store with secure string type.

2. **IAM Permissions**: The IAM role should have the minimum necessary permissions to invoke the API destination only.

3. **Network Security**: Since this is an API call to an external endpoint, no VPC configuration is needed.

## Error Handling Strategy

1. **Dead-Letter Queue**: All failed event deliveries will be sent to the SQS queue for later analysis and reprocessing.

2. **Retry Policy**: EventBridge automatically retries failed invocations (default: 2 retries with exponential backoff).

3. **Monitoring**: CloudWatch metrics and alarms can be configured to monitor queue depth and delivery failures.

## Dynamic Resource Fetching

No dynamic resource fetching is needed for this template since:
- No AMIs are required (no EC2 instances)
- No AZ selection is needed (no VPC resources)
- The MongoDB URL is provided as a static value

## Template Structure Plan

The CloudFormation template will follow this logical order:

1. Parameters (with safe defaults)
2. Resources in dependency order:
   - IAM Role
   - SQS Queue and Policy
   - Connection
   - Event Bus
   - API Destination
3. Outputs (optional: queue URL, event bus ARN)

## Linting Compliance Strategy

1. All resources will have proper `UpdateReplacePolicy` matching `DeletionPolicy` where applicable
2. No hardcoded values except where explicitly provided in requirements
3. All IAM policies will use exact AWS managed policy ARNs
4. No use of deprecated properties or runtimes
5. All intrinsic functions will be used appropriately (no unnecessary `Fn::Sub`)

This architecture provides a robust, event-driven integration between EventBridge and MongoDB API with proper error handling and security considerations.
