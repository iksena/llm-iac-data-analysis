# AWS CloudFormation Architecture Plan

## 1. Key AWS Services Required

Based on the business need, I'll need to provision the following AWS services:

- **AWS::S3::Bucket** - To create the S3 bucket for object storage
- **AWS::SQS::Queue** - To create the SQS queue for receiving notifications
- **AWS::S3::BucketNotification** - To configure the S3 bucket to send notifications to the SQS queue
- **AWS::SQS::QueuePolicy** - To grant S3 permission to send messages to the SQS queue

## 2. Properties and Dependencies

The template will have the following dependencies and property relationships:

1. **SQS Queue Creation**: The SQS queue must be created first since the S3 bucket notification configuration needs the queue's ARN
2. **SQS Queue Policy**: The queue policy must be attached to the SQS queue to allow S3 to send messages
3. **S3 Bucket Notification**: The S3 bucket notification configuration must reference the SQS queue ARN
4. **Event Configuration**: The S3 bucket must be configured to send notifications for `s3:ObjectCreated:*` events

## 3. Safe AWS Defaults

Since no specific properties were provided, I'll use these safe defaults:

- **S3 Bucket Name**: No default (must be unique globally, so I'll make this a required parameter)
- **SQS Queue Name**: No default (will use CloudFormation-generated name)
- **S3 Event Type**: `s3:ObjectCreated:*` (as specified in the business need)
- **S3 Bucket Versioning**: Disabled (safe default)
- **SQS Queue Visibility Timeout**: 30 seconds (AWS default)
- **SQS Queue Message Retention Period**: 345,600 seconds (4 days, AWS default)

## 4. Dynamic Resource Fetching

For this template, I don't need to fetch any dynamic resources like AMIs since we're only creating S3 and SQS resources. However, I'll follow best practices by:

- Using `!Ref` to reference resource logical IDs
- Using `!GetAtt` to retrieve ARNs and other attributes
- Using `!Sub` only when necessary for string interpolation

## 5. Security and Permissions

I'll implement the following security measures:

- **SQS Queue Policy**: Grant S3 service principal permission to send messages to the queue
- **S3 Bucket Policy**: Not needed since we're only configuring notifications, not restricting access
- **Resource Protection**: No stateful resources requiring `DeletionPolicy` or `UpdateReplacePolicy`

## 6. Template Structure

The CloudFormation template will follow this logical structure:

1. **Parameters**: Define required parameters (S3 bucket name)
2. **Resources**:
   - SQS Queue
   - SQS Queue Policy
   - S3 Bucket
   - S3 Bucket Notification Configuration
3. **Outputs**: Export the S3 bucket name and SQS queue URL for reference

## 7. Compliance with AWS Best Practices

The template will adhere to all specified linting rules:

- ✅ All parameters will have safe default values
- ✅ No hardcoded AMI IDs or Availability Zones
- ✅ No legacy `AccessControl` properties for S3 buckets
- ✅ Proper use of `DeletionPolicy` and `UpdateReplacePolicy` where needed
- ✅ No unnecessary use of `Fn::Sub`
- ✅ No hallucinated properties
- ✅ Use of latest AWS managed policy ARNs
- ✅ No deprecated Lambda runtimes (not applicable here)

## 8. Event Configuration Details

The S3 bucket notification will be configured to send messages for all object creation events:

- Event types: `s3:ObjectCreated:*` (covers PUT, POST, COPY, and multipart upload completion)
- Filter: No prefix or suffix filters (all objects in the bucket)
- Queue destination: The SQS queue ARN

This configuration ensures that any object uploaded to the S3 bucket will trigger a notification to the SQS queue, enabling event-driven processing workflows.
