# SNS-to-SQS Fanout System with Filtered Message Routing - Architecture Plan

## 1. Key AWS Services Required

Based on the business requirements, I'll need the following AWS services:

- **SNS (Simple Notification Service)** - Central topic for message publishing
- **SQS (Simple Queue Service)** - Three queues: AllMetrics, TemperatureSqsQueue, HumiditySqsQueue
- **SQS Dead Letter Queues** - One per filtered queue for failure handling
- **IAM (Identity and Access Management)** - Policies for SNS-SQS permissions
- **SSM Parameter Store** - For dynamic resource fetching (though not explicitly needed here)

## 2. Properties and Dependencies

The template will have the following dependencies and property requirements:

1. **SNS Topic** must be created before SQS queues can subscribe to it
2. **SQS Queues** must exist before creating SNS subscriptions
3. **Dead Letter Queues** must be created before being referenced by main queues
4. **IAM Policies** must be created before being attached to resources

## 3. Safe AWS Defaults

Since specific properties weren't provided, I'll use these safe defaults:

- **Region**: `us-east-1` (default AWS region)
- **SNS Topic Name**: `MetricsFanoutTopic` (descriptive default)
- **Queue Names**: 
  - `AllMetricsQueue` (for all messages)
  - `TemperatureSqsQueue` (filtered for temperature)
  - `HumiditySqsQueue` (filtered for humidity)
- **Dead Letter Queue Names**: 
  - `TemperatureSqsQueueDLQ`
  - `HumiditySqsQueueDLQ`
- **Message Retention Period**: 345,600 seconds (4 days) - standard default
- **Visibility Timeout**: 30 seconds - standard default
- **Maximum Receives**: 5 (before sending to DLQ) - standard default

## 4. Dynamic Resource Fetching

For this specific template, dynamic fetching isn't required since we're using managed services (SNS/SQS) that don't need AMIs or specific resource IDs. However, if we were to extend this with Lambda functions or EC2 instances, I would use:

- **SSM Parameter Store** for latest Amazon Linux AMIs: `/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`
- **Fn::GetAZs** and **Fn::Select** for Availability Zones instead of hardcoding

## 5. Architecture Design

The template will follow this structure:

### SNS Topic Creation
- Create central SNS topic `MetricsFanoutTopic`
- No special properties needed beyond name

### SQS Queues Creation
1. **AllMetricsQueue** - Receives all messages without filtering
2. **TemperatureSqsQueue** - Filtered queue with redrive policy to DLQ
3. **HumiditySqsQueue** - Filtered queue with redrive policy to DLQ
4. **Dead Letter Queues** - One for each filtered queue

### SNS Subscriptions with Filtering
- Subscribe TemperatureSqsQueue to SNS topic with filter policy: `{ "MetricType": ["Temperature"] }`
- Subscribe HumiditySqsQueue to SNS topic with filter policy: `{ "MetricType": ["Humidity"] }`
- AllMetricsQueue will subscribe without filter policy to receive all messages

### IAM Policies
- Create IAM policies allowing SNS to send messages to SQS queues
- Attach policies to SNS topic

## 6. Critical Implementation Details

### Filter Policy Structure
The filter policies will use JSON structure:
```json
{
  "MetricType": ["Temperature"]
}
```

### Redrive Policy Configuration
Each filtered queue will have:
- `maxReceiveCount`: 5 (messages sent to DLQ after 5 failed receives)
- `deadLetterTargetArn`: ARN of corresponding DLQ

### Message Structure Assumption
Messages published to SNS should include:
```json
{
  "default": "message",
  "MetricType": "Temperature"  // or "Humidity"
}
```

## 7. Template Organization

The CloudFormation template will be organized as:

1. **Parameters** (with safe defaults)
2. **Resources** in logical order:
   - DLQ creation
   - Main SQS queues
   - SNS topic
   - IAM policies
   - SNS subscriptions
3. **Outputs** (optional) for ARNs and URLs

This architecture ensures scalable, selective message processing where publishers send to one SNS topic, and subscribers receive only relevant messages based on the `MetricType` attribute, with robust error handling through dead letter queues.
