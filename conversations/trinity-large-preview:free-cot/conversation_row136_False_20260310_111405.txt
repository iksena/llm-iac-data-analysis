# IAM Governance Control Architecture Plan

## Key AWS Services Required

1. **AWS Identity and Access Management (IAM)** - Core service for creating roles, policies, and permission boundaries
2. **AWS Systems Manager (SSM) Parameter Store** - For dynamic policy retrieval and configuration
3. **AWS CloudFormation** - Infrastructure as Code deployment
4. **AWS Lambda** (optional enhancement) - For automated governance enforcement
5. **AWS Config** (optional enhancement) - For continuous compliance monitoring

## Architecture Overview

The solution creates a governance framework with two operational modes:
- **Strict Mode**: Enforces permission boundaries and governance controls
- **Permissive Mode**: Provides broader IAM privileges without boundaries

## Detailed Component Design

### 1. Permission Boundary Policy

**Default Permission Boundary Policy** (restricts to S3 and read-only IAM):
- Allows all S3 actions (`s3:*`)
- Allows read-only IAM actions (`iam:Get*`, `iam:List*`, `iam:Generate*`)
- Denies IAM actions that could escalate permissions (`iam:CreatePolicy`, `iam:CreateRole` without boundary, etc.)
- Uses dynamic ARN resolution for maximum compatibility

### 2. IAM Role Creation Strategy

**Developer Role** (conditional based on enforcement toggle):
- **Strict Mode**: Creates role with permission boundary attached
- **Permissive Mode**: Creates role without boundary but with governance policies

**Governance Policies** (applied in both modes):
- **Naming Policy**: Validates resource names start with `MyTeam*`
- **Tagging Policy**: Requires `Owner` tag on all IAM resources
- **Boundary Policy**: Enforces permission boundary usage

### 3. Parameter Design with Safe Defaults

```yaml
Parameters:
  PermissionBoundaryEnforcement:
    Type: String
    Default: enabled
    AllowedValues:
      - enabled
      - disabled
    Description: Toggle permission boundary enforcement

  TeamOwnerTag:
    Type: String
    Default: MyTeam
    AllowedPattern: ^[a-zA-Z0-9-_]+$
    Description: Team identifier for Owner tag

  ResourcePrefix:
    Type: String
    Default: MyTeam
    AllowedPattern: ^[a-zA-Z0-9-_]+$
    Description: Required prefix for IAM resource names
```

### 4. Policy Construction Strategy

**Dynamic Policy Generation**:
- Use `Fn::Sub` to inject team-specific values
- Construct ARNs using `Fn::Join` and `Fn::Sub`
- Include conditional logic for enforcement mode

**Policy Structure**:
```yaml
- Version: '2012-10-17'
  Statement:
    - Effect: Allow
      Action:
        - iam:CreateRole
        - iam:DeleteRole
        - iam:UpdateRole
        - iam:AttachRolePolicy
        - iam:DetachRolePolicy
      Condition:
        StringEquals:
          iam:ResourceTag/Owner: !Ref TeamOwnerTag
          iam:PermissionBoundary: !If [IsStrictMode, !Ref PermissionBoundaryArn, !Ref "AWS::NoValue"]
```

### 5. Resource Dependencies and Ordering

1. **Permission Boundary Policy** (created first)
2. **Governance Policies** (created second)
3. **Developer Role** (created last, depends on policies)

### 6. Dynamic Resource Resolution

**AMI Resolution** (not directly applicable but demonstrates pattern):
```yaml
!Sub 'arn:${AWS::Partition}:iam::aws:policy/aws-service-role/AWSDenyAllExceptS3AndIAMRead'
```

**Region-Aware Configuration**:
- Use `AWS::Region` for region-specific ARNs
- Use `AWS::AccountId` for account-specific references

### 7. Security Best Practices Implementation

**No Hardcoded Values**:
- All ARNs constructed dynamically
- All policies use AWS-managed policy references where possible
- No hardcoded account IDs or region-specific values

**Tag Enforcement**:
- Policies include conditions requiring `Owner` tag
- Tag key/value pairs validated through IAM policy conditions

### 8. Error Handling and Validation

**Policy Validation**:
- Use IAM policy simulator for testing
- Include conditions that fail closed (deny by default)

**Resource Validation**:
- Naming prefix validation through IAM policy conditions
- Tag validation through IAM policy conditions

### 9. Optional Enhancements

**Lambda-Based Enforcement** (for production environments):
- Custom Lambda function to validate IAM resource creation
- Integration with AWS Config for compliance monitoring
- Automated remediation of non-compliant resources

**Config Rules** (for continuous compliance):
- Managed rules for IAM best practices
- Custom rules for team-specific governance
- Automated reporting and alerting

### 10. Deployment Strategy

**Phase 1**: Deploy governance framework in permissive mode
**Phase 2**: Test with developer teams
**Phase 3**: Enable strict mode with permission boundaries
**Phase 4**: Monitor and adjust based on feedback

This architecture provides a robust governance framework that can be toggled between permissive and strict modes, ensuring developers have the necessary IAM privileges while maintaining organizational control over resource creation and permissions.
