For this template, dynamic fetching isn't required since we're using AWS-managed policies that are always available by ARN. However, if we needed to fetch the latest policy ARNs dynamically, we could use SSM Parameter Store, though AWS-managed policies have stable ARNs.

## Architecture Plan

The template will create the following resources in order:

1. **IAM Role**: A role that EC2 instances can assume, with trust policy allowing EC2 service to assume it
2. **IAM Managed Policies**: Two AWS-managed policies attached to the role:
   - `AmazonSSMManagedInstanceCore` for full SSM functionality
   - `CloudWatchAgentServerPolicy` for CloudWatch agent operations
3. **Instance Profile**: A container for the IAM role that can be assigned to EC2 instances

## Security Considerations

- The IAM role will have the minimum necessary permissions for CloudWatch and SSM integration
- No hardcoded credentials or secrets are needed
- The trust policy is restricted to EC2 service only
- All resources will use AWS best practices for naming and configuration

1. **Parameters**: Minimal parameters with safe defaults
2. **Resources**: 
   - IAM Role with trust policy
   - Two managed policy attachments
   - Instance profile
3. **Outputs**: Export the role ARN and instance profile name for use in other templates

## Linting Compliance

The template will adhere to all specified linting rules:
- No hardcoded AMIs or AZs (not applicable here)
- No legacy S3 properties (not applicable here)
- Proper use of intrinsic functions where needed
- No hallucinated properties
