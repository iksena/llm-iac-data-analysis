# SNS-to-SQS Fanout System with Filtered Message Routing - Architecture Plan

## 1. Key AWS Services Required

Based on the business requirement, the following AWS services are needed:

- **SNS (Simple Notification Service)**: Central topic for message publishing
- **SQS (Simple Queue Service)**: Three queues (all metrics, temperature-filtered, humidity-filtered)
- **SQS Dead Letter Queues**: One per SQS queue for handling processing failures
- **IAM (Identity and Access Management)**: Service roles and policies for SNS-SQS integration

## 2. Properties and Dependencies

The template will follow this dependency chain:

1. **IAM Roles/Policies** → SNS Topic → SQS Queues → SNS Topic Subscriptions
2. **Dead Letter Queues** must be created before their corresponding SQS queues
3. **SQS Queues** must exist before SNS subscriptions can be created

## 3. Safe AWS Defaults (where values are unspecified)

- **CIDR Blocks**: Not applicable (no VPC required for this service-only architecture)
- **Instance Types**: Not applicable (no EC2 instances)
- **Region**: Use `AWS::Region` pseudo-parameter for region-specific resources
- **Account**: Use `AWS::AccountId` pseudo-parameter for account-specific resources

## 4. Dynamic Resource Fetching Strategy

- **AMIs**: Not required (no EC2 instances)
- **Availability Zones**: Not required (no VPC/Subnet resources)
- **SNS Topic ARNs**: Use `Ref` function to reference topic names
- **SQS Queue URLs**: Use `Fn::GetAtt` to retrieve queue URLs after creation

## 5. Architecture Design

### Core Components

1. **SNS Topic**: Central message hub for all metric data
2. **SQS Queues**:
   - `AllMetricsSqsQueue`: Receives all messages from SNS
   - `TemperatureSqsQueue`: Receives only temperature metrics
   - `HumiditySqsQueue`: Receives only humidity metrics
3. **Dead Letter Queues**: One per SQS queue (e.g., `AllMetricsDlq`, `TemperatureDlq`, `HumidityDlq`)
4. **IAM Policies**: Allow SNS to send messages to SQS queues

### Message Filtering Strategy

Each filtered queue will use SQS's message filtering capability with the following criteria:

- **TemperatureSqsQueue**: `MetricType = "Temperature"`
- **HumiditySqsQueue**: `MetricType = "Humidity"`

### SNS Topic Subscriptions

Three subscriptions will be created:
1. **AllMetricsSubscription**: Raw subscription (no filtering) to `AllMetricsSqsQueue`
2. **TemperatureSubscription**: Filtered subscription to `TemperatureSqsQueue`
3. **HumiditySubscription**: Filtered subscription to `HumiditySqsQueue`

## 6. Implementation Plan

### Step 1: IAM Setup
- Create IAM policies allowing SNS to send messages to SQS queues
- Create IAM roles if needed (though SNS-SQS integration typically uses resource-based policies)

### Step 2: Dead Letter Queues
- Create three SQS DLQs with appropriate names and default settings
- Set reasonable `MaxReceiveCount` (e.g., 5) for DLQ triggering

### Step 3: SQS Queues
- Create `AllMetricsSqsQueue` with `AllMetricsDlq` as its dead letter queue
- Create `TemperatureSqsQueue` with `TemperatureDlq` as its dead letter queue
- Create `HumiditySqsQueue` with `HumidityDlq` as its dead letter queue

### Step 4: SNS Topic
- Create central SNS topic with a descriptive name (e.g., `MetricsFanoutTopic`)

### Step 5: SNS Subscriptions
- Create raw subscription from SNS topic to `AllMetricsSqsQueue`
- Create filtered subscription from SNS topic to `TemperatureSqsQueue` with `MetricType = "Temperature"` filter
- Create filtered subscription from SNS topic to `HumiditySqsQueue` with `MetricType = "Humidity"` filter

## 7. Best Practices Compliance

This architecture adheres to all specified best practices:

- **No hardcoded AMIs or AZs** (not applicable for this service-only template)
- **No Parameters without defaults** (template will use hardcoded names with safe defaults)
- **No legacy S3 properties** (not applicable)
- **Proper resource protection** (will include appropriate policies)
- **No deprecated runtimes** (not applicable)
- **Secure credential handling** (not applicable for this service-only template)

## 8. Template Structure

The CloudFormation template will be organized as follows:

1. **Parameters** (if any, with safe defaults)
2. **Resources**:
   - IAM Policies/Roles
   - SQS Dead Letter Queues
   - SQS Queues
   - SNS Topic
   - SNS Subscriptions
3. **Outputs** (optional: queue URLs, topic ARN for reference)

This architecture provides a scalable, fault-tolerant message routing system that allows different processing pipelines to consume only the metric types they need, while maintaining a complete message archive in the all-metrics queue.
