# AWS IAM Password Policy CloudFormation Architecture Plan

## 1. Key AWS Services Required

Based on the business requirements, the following AWS services are essential:

- **AWS Identity and Access Management (IAM)** - For creating and managing the account-wide password policy
- **AWS Lambda** - For dynamic policy application/removal via AWS SDK
- **AWS Systems Manager (SSM) Parameter Store** - For storing Lambda function code and configuration parameters
- **AWS Identity and Access Management (IAM)** - For creating Lambda execution role and optional permission boundaries
- **AWS CloudFormation** - For infrastructure as code deployment

## 2. Properties and Dependencies

The deployment follows a linear dependency chain:

1. **IAM Role Creation** → Lambda execution role must exist before Lambda function
2. **Lambda Function Creation** → Requires IAM role and SSM-stored code
3. **Password Policy Application** → Lambda function must be deployed before it can be invoked

No VPC or networking components are required since Lambda runs in a serverless environment.

## 3. Safe AWS Defaults

Since specific properties weren't provided, I'll use these safe defaults:

- **Minimum Password Length**: Default = 14 (meets compliance requirements)
- **Password Expiration**: Default = 90 days (standard enterprise policy)
- **Password Reuse Prevention**: Default = 24 (prevents cycling through old passwords)
- **IAM Permission Boundary**: Optional (not set by default)
- **Lambda Runtime**: python3.12 (latest supported runtime)
- **Lambda Timeout**: 30 seconds (sufficient for IAM operations)
- **Lambda Memory**: 128 MB (adequate for simple IAM SDK calls)

## 4. Dynamic Resource Fetching Strategy

**AMI Fetching**: Not applicable since Lambda runs in a managed environment without EC2 instances.

**Lambda Function Code**: Store the Lambda function code in SSM Parameter Store as a secure string, allowing updates without template modifications.

## Architecture Design

The template will implement a modular approach with these key components:

### Parameters (with safe defaults)
```yaml
Parameters:
  MinimumPasswordLength:
    Type: Number
    Default: 14
    MinValue: 6
    MaxValue: 128
  RequireUppercaseCharacters:
    Type: String
    Default: true
  RequireLowercaseCharacters:
    Type: String
    Default: true
  RequireNumbers:
    Type: String
    Default: true
  RequireSymbols:
    Type: String
    Default: true
  MaxPasswordAge:
    Type: Number
    Default: 90
  PasswordReusePrevention:
    Type: Number
    Default: 24
  AllowUsersToChangePassword:
    Type: String
    Default: true
  PermissionBoundaryArn:
    Type: String
    Default: ""
```

### Resources

1. **IAM Role for Lambda** - Execution role with IAM permissions
2. **Lambda Function** - Python function to apply/remove password policy
3. **SSM Parameter** - Stores Lambda code as secure string
4. **Custom Resource** - CloudFormation custom resource to invoke Lambda

### Lambda Function Logic

The Lambda function will:
- Use boto3 IAM client to set password policy
- Handle both creation and deletion requests
- Validate parameter values before applying
- Log operations for audit trail

### Security Considerations

- Lambda execution role follows principle of least privilege
- Parameters use appropriate types and constraints
- No hardcoded secrets or credentials
- Optional permission boundary support for additional security

### Error Handling

The template will include:
- Parameter validation for logical constraints
- Lambda retry logic for eventual consistency
- CloudFormation rollback on failure
- Comprehensive logging for troubleshooting

This architecture ensures a secure, maintainable, and compliant IAM password policy deployment that can be easily updated through CloudFormation without manual intervention.
