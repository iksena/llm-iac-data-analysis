- **Security Group** - To control SSH access to the instance
- **IAM Role** - For instance permissions (if needed for SSM or other services)
- **SSM Parameter Store** - To dynamically fetch the latest Amazon Linux AMI

## 2. Properties and Dependencies

### Dependencies:
1. **VPC and Subnet** - The default VPC and subnet must exist before creating the EC2 instance
2. **Internet Gateway** - Must be attached to the VPC before creating route tables
3. **Route Table** - Must be associated with the subnet to enable internet connectivity
4. **Security Group** - Must be created before associating it with the EC2 instance
5. **EIP** - Can only be associated with an EC2 instance after the instance is created

### Properties:
- **EC2 Instance**: Instance type, AMI, key pair, security group, subnet
- **EIP**: Association with EC2 instance
- **Security Group**: SSH ingress rules with source IP restrictions
- **IAM Role**: Instance profile for SSM access (optional but recommended)

## 3. Safe AWS Defaults

Since specific properties are not provided, I will use the following safe defaults:

- **AMI**: Use SSM Parameter Store to fetch the latest Amazon Linux 2 AMI dynamically
- **Instance Type**: `t3.micro` (free tier eligible, suitable for basic workloads)
- **VPC**: Default VPC (no need to create a new one)
- **Subnet**: Default subnet in the region
- **Key Pair**: Parameter with default value 'default-key' (user must create this key pair or change the default)
- **CIDR Blocks**: 
  - VPC: Default VPC CIDR (172.31.0.0/16)
  - Subnet: Default subnet CIDR (172.31.x.0/20)
  - SSH Access: `0.0.0.0/0` (can be restricted later)
- **Region**: Use `AWS::Region` pseudo parameter to ensure template works in any region

## 4. Dynamic Resource Fetching Strategy

### AMI Fetching using SSM Parameter Store:
Instead of hardcoding an AMI ID, I will use the SSM Parameter Store to fetch the latest Amazon Linux 2 AMI:

```yaml

```

This ensures the template always uses the latest patched AMI without manual updates.

### Availability Zones:
Use `Fn::GetAZs` and `Fn::Select` to dynamically fetch availability zones instead of hardcoding:

```yaml
!Select [0, !GetAZs !Ref 'AWS::Region']
```

## 5. Security Considerations

- **SSH Access**: The security group will allow SSH access. By default, I'll use `0.0.0.0/0` but this should be restricted to specific IP ranges in production.
- **IAM Role**: I'll create an IAM role with the AmazonSSMManagedInstanceCore policy to allow SSM access for remote management without SSH keys.
- **EIP**: While EIPs are useful for static IPs, they come with hourly charges when not in use. Consider if this is necessary for your use case.

## 6. Template Structure Plan

The CloudFormation template will be structured as follows:

1. **Parameters**: For key pair name and SSH source CIDR
2. **Resources**:
   - EC2 Instance with latest Amazon Linux 2 AMI
   - Elastic IP Address
   - Security Group with SSH access
   - IAM Role and Instance Profile for SSM
   - EIP Association
3. **Outputs**: Public IP address and instance ID for easy access

## 7. Potential Issues and Mitigations

- **Default Key Pair**: If the user doesn't have a key pair named 'default-key', the deployment will fail. I'll include clear documentation about this requirement.
- **SSH Access Restrictions**: Using `0.0.0.0/0` is insecure for production. I'll make this a parameter with a safe default but encourage users to restrict it.
- **EIP Costs**: EIPs incur hourly charges when not associated with a running instance. I'll note this in the documentation.

This architectural plan ensures a production-ready, secure, and maintainable CloudFormation template that follows AWS best practices and will pass cfn-lint validation.
