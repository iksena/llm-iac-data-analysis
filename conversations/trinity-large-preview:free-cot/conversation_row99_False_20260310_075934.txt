# AWS CloudFormation Architecture Plan: Secure Multi-AZ VPC with Public/Private Subnets

## 1. Key AWS Services Required

Based on the business requirements, the following AWS services and resources are needed:

- **VPC** - Core networking construct
- **Internet Gateway (IGW)** - For public internet connectivity
- **NAT Gateway** - For private subnet outbound internet access
- **Elastic IP (EIP)** - Required for NAT Gateway
- **Subnets** - Public and private subnets across two AZs
- **Route Tables** - Separate route tables for public and private traffic
- **Route Table Associations** - Linking subnets to appropriate route tables
- **VPC Endpoints** - For private AWS service access (optional enhancement)
- **Tags** - For cost tracking and governance

## 2. Properties and Dependencies

### Critical Dependencies:
1. **VPC must be created before** Internet Gateway, NAT Gateway, and Subnets
2. **Internet Gateway must be attached to VPC** before creating route table entries
3. **Elastic IP must be created before NAT Gateway**
4. **Route Tables must be created before associations**
5. **Subnets must exist before route table associations**

### Property Requirements:
- CIDR blocks for VPC and subnets (need defaults)
- Availability Zones (must use Fn::GetAZs and Fn::Select)
- Tags for all resources
- Route table entries with proper destination CIDR blocks

- **Private Subnet CIDRs**: `10.0.2.0/24` and `10.0.3.0/24` (one per AZ)
- **Availability Zones**: Use `Fn::GetAZs` to dynamically fetch available AZs in region, then `Fn::Select` to pick first two

- **Network ACLs**: Implement NACL rules for additional security layer
- **Security Groups**: Plan for default deny-all inbound, allow-all outbound
- **Private DNS**: Enable DNS hostnames and DNS support in VPC
- **IAM Permissions**: Use least privilege for any IAM roles

## 6. Tagging Strategy

Implement consistent tagging across all resources:
- **Environment**: Production/Development
- **Project**: VPC-Infrastructure
- **Owner**: Cloud-Operations
- **CostCenter**: 1234
- **ManagedBy**: CloudFormation

## 7. Template Structure Plan

The CloudFormation template will follow this logical order:

1. **Parameters** (with safe defaults)
2. **Mappings** (if needed for region-specific configs)
3. **Resources** in dependency order:
   - VPC
   - Internet Gateway + Attachment
   - Elastic IP
   - NAT Gateway

   - Route Tables (public + private)
   - Route Table Associations
4. **Outputs** (VPC ID, Subnet IDs, etc.)

## 8. Linting Compliance Strategy

To ensure cfn-lint passes:
- All Parameters have Default values matching AllowedPattern
- No hardcoded AZs or AMI IDs
- No legacy S3 AccessControl properties
- All stateful resources have both DeletionPolicy and UpdateReplacePolicy if either is specified
- No unnecessary Fn::Sub usage
- Only valid CloudFormation properties used

This architecture ensures high availability across two AZs, proper traffic segregation between public and private subnets, and follows AWS Well-Architected security best practices while being fully automatable through CloudFormation.
