# Architectural Deployment Plan for Secure Web Application Environment

## Key AWS Services Required

1. **VPC** - Virtual Private Cloud for network isolation
2. **Internet Gateway** - Enables internet connectivity for the VPC
3. **Subnet** - Public subnet within the VPC for hosting resources
4. **Route Table** - Routes traffic to the internet gateway
5. **EC2 Instance** - Web server running Amazon Linux with Apache and PHP
6. **Security Group** - Controls inbound/outbound traffic to the EC2 instance
7. **IAM Role** - Grants permissions for CloudWatch monitoring and SSM access
8. **S3 Bucket** - Centralized log storage with encryption
9. **S3 Bucket Policy** - Controls access to the log bucket
10. **CloudWatch Logs** - Monitoring and log aggregation

## Properties and Dependencies

### Network Infrastructure Dependencies
- VPC must be created before Internet Gateway, Subnet, and Route Table
- Internet Gateway must be attached to VPC before Route Table can reference it
- Route Table must be associated with Subnet after both are created

### EC2 Dependencies
- Security Group must be created before EC2 instance can reference it
- IAM Role must be created before EC2 instance can reference it
- Subnet must exist before EC2 instance can be launched in it

### S3 Dependencies
- S3 Bucket must be created before Bucket Policy can reference it

## Dynamic Resource Resolution Strategy

### AMI Resolution
I will use SSM Parameter Store to dynamically fetch the latest Amazon Linux 2 AMI:
- Parameter: `/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`
- This ensures the template always uses the latest AMI without hardcoding

### Availability Zones
I will use `Fn::GetAZs` and `Fn::Select` to dynamically determine available AZs:
- `Fn::GetAZs: ''` returns all AZs in the region
- `Fn::Select: 0` selects the first AZ (safe default)

## Safe Default Values

### Network Configuration
- **VPC CIDR**: `10.0.0.0/16` (standard private range)
- **Public Subnet CIDR**: `10.0.1.0/24` (within VPC range)
- **SSH Source**: `0.0.0.0/0` (can be restricted later)

### EC2 Configuration
- **Instance Type**: `t3.micro` (free tier eligible, suitable for web server)
- **KeyName**: `default-key` (safe placeholder)
- **SSH Source**: `0.0.0.0/0` (can be restricted later)

### S3 Configuration
- **Bucket Name**: Dynamically generated using stack name and random suffix
- **Encryption**: SSE-S3 (server-side encryption with S3-managed keys)

## Security Implementation

### Network Security
- Security Group allows HTTP (port 80) from anywhere
- Security Group allows SSH (port 22) from specified source
- All outbound traffic allowed (can be restricted if needed)

### Instance Security
- IAM Role grants CloudWatch Agent and SSM Agent permissions
- No direct IAM user credentials stored in template
- User data script installs only necessary packages

### Data Security
- S3 bucket uses server-side encryption by default
- Public access blocked at bucket level
- Bucket policy restricts log ingestion to authorized sources

## User Data Script Strategy

The EC2 instance will use a user data script to:
1. Update system packages
2. Install Apache web server
3. Install PHP
4. Create a simple PHP info page
5. Start Apache service
6. Enable Apache to start on boot

## Monitoring and Logging Strategy

### CloudWatch Integration
- IAM role allows CloudWatch agent installation and configuration
- EC2 instance will automatically send logs to CloudWatch
- S3 bucket configured to receive CloudWatch logs

### Log Management
- S3 bucket with versioning enabled for log retention
- Server-side encryption protects log data at rest
- Bucket policy controls who can write logs

## Template Structure Plan

The CloudFormation template will follow this logical order:

1. **Parameters** - All configurable values with safe defaults
2. **Mappings** - Any lookup tables needed
3. **Resources** - All AWS resources in dependency order:
   - VPC and networking components
   - Security groups
   - IAM role and instance profile
   - EC2 instance
   - S3 bucket and policy
4. **Outputs** - Useful information like instance public IP and S3 bucket name

## Compliance Considerations

The architecture ensures:
- Network isolation through VPC
- Least privilege access through security groups
- Centralized logging for audit trails
- Encryption for data at rest
- Automated patching through SSM
- Monitoring through CloudWatch

This plan provides a secure, monitored, and compliant foundation that can be extended for production workloads while maintaining automation and best practices.
