# AWS CloudFormation Deployment Plan for Production VPC Infrastructure

## 1. Key AWS Services Required

Based on the business requirements, the following AWS services and resources are needed:

- **VPC** - Core networking infrastructure
- **Internet Gateway** - Public internet access for public subnets
- **NAT Gateway** - Outbound internet access for private subnets
- **Elastic IP** - Static IP for NAT Gateway
- **Subnets** - 1 public and 2 private subnets across 3 AZs
- **Route Tables** - Separate routing for public and private subnets
- **Security Groups** - Network access control for different traffic types
- **Application Load Balancer** - Internal ALB for containerized services
- **ECR Repository** - Container image storage with scanning
- **SSM Parameter Store** - Cross-service resource referencing

## 2. Properties and Dependencies

### Critical Dependencies:
1. **VPC must be created before** Internet Gateway, NAT Gateway, Subnets, and Security Groups
2. **Internet Gateway must be attached to VPC** before creating public route table
3. **Elastic IP must be allocated** before creating NAT Gateway
4. **NAT Gateway must be created** before private route table can reference it
5. **Subnets must exist** before ALB can be deployed across them
6. **Security Groups must reference existing VPC**

### Property Specifications:

**VPC:**
- CIDR Block: `10.0.0.0/16` (safe default)
- Enable DNS hostnames: `true`
- Enable DNS support: `true`

**Subnets:**
- Public Subnet CIDR: `10.0.1.0/24`, `10.0.2.0/24`, `10.0.3.0/24`
- Private Subnet CIDR: `10.0.4.0/24`, `10.0.5.0/24`, `10.0.6.0/24`
- Map Public IP on Launch: `true` for public subnets, `false` for private

**Security Groups:**
- HTTP/HTTPS: Port 80 and 443, source `0.0.0.0/0`
- SSH: Port 22, source `0.0.0.0/0` (for simplicity)
- PostgreSQL: Port 5432, source `10.0.0.0/16` (internal VPC)
- All Traffic: Port range 0-65535, source `10.0.0.0/16` (internal VPC)

**ALB:**
- Scheme: `internal`
- Type: `application`
- Default Action: Fixed response (HTTP 404) on port 80

**ECR Repository:**
- Repository name: `my-container-app`
- Image scanning: `true` (enabled on push)

## 3. Safe AWS Defaults for Missing Properties

**CIDR Blocks:**
- VPC: `10.0.0.0/16`
- Public Subnets: `10.0.1.0/24`, `10.0.2.0/24`, `10.0.3.0/24`
- Private Subnets: `10.0.4.0/24`, `10.0.5.0/24`, `10.0.6.0/24`

**Availability Zones:**
- Use `Fn::GetAZs` and `Fn::Select` to dynamically fetch AZs instead of hardcoding

**AMI IDs:**
- Use SSM Parameter Store to fetch latest Amazon Linux 2 AMI dynamically

**Instance Types:**
- Not required for this networking-focused template

**AMI Retrieval:**
```yaml

**Availability Zones:**

!Select [1, !GetAZs '']
!Select [2, !GetAZs '']
```

## 5. SSM Parameter Store Strategy

Store the following resource IDs for cross-service referencing:

- `/MyApp/VPCId` - VPC ID
- `/MyApp/PublicSubnet1` - Public subnet 1 ID
- `/MyApp/PublicSubnet2` - Public subnet 2 ID
- `/MyApp/PublicSubnet3` - Public subnet 3 ID
- `/MyApp/PrivateSubnet1` - Private subnet 1 ID
- `/MyApp/PrivateSubnet2` - Private subnet 2 ID
- `/MyApp/PrivateSubnet3` - Private subnet 3 ID
- `/MyApp/InternalALBListener` - ALB listener ARN
- `/MyApp/SecurityGroupHTTP` - HTTP security group ID
- `/MyApp/SecurityGroupSSH` - SSH security group ID
- `/MyApp/SecurityGroupPostgreSQL` - PostgreSQL security group ID
- `/MyApp/SecurityGroupInternal` - Internal VPC security group ID

## 6. Security Group Configuration Plan

Create four separate security groups for clarity and maintainability:

1. **HTTP Security Group:** Allows HTTP/HTTPS from anywhere
2. **SSH Security Group:** Allows SSH from anywhere (for simplicity)
3. **PostgreSQL Security Group:** Allows PostgreSQL from VPC CIDR
4. **Internal Security Group:** Allows all traffic within VPC

## 7. ALB Configuration Plan

- **Scheme:** Internal (private)
- **Subnets:** All three private subnets for high availability
- **Listener:** Port 80 with fixed response action returning HTTP 404
- **Security Groups:** Attach HTTP and Internal security groups

## 8. ECR Repository Configuration

- **Repository Name:** `my-container-app`
- **Image Scanning:** Enabled on push for security
- **KMS Key:** Use default AWS managed CMK

## 9. Template Structure Plan

The CloudFormation template will follow this logical order:

1. Parameters (with safe defaults)
2. Resources in dependency order:
   - VPC
   - Internet Gateway + attachment
   - Elastic IP
   - NAT Gateway
   - Public Subnets + Route Table + routes
   - Private Subnets + Route Table + routes
   - Security Groups
   - ALB + Listener
   - ECR Repository
   - SSM Parameters
3. Outputs (optional, for convenience)

This architecture ensures high availability across three AZs, proper network segmentation, and provides all necessary infrastructure components for deploying containerized applications in a production environment.
