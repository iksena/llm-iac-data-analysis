# Architectural Deployment Plan for URL-to-PNG Screenshot Service

## 1. Key AWS Services Required

Based on the business requirements, I'll need the following AWS services:

- **VPC** (Virtual Private Cloud) - Network isolation
- **EC2** (Elastic Compute Cloud) - Worker instances for screenshot processing
- **Auto Scaling** - Dynamic scaling based on queue depth
- **SQS** (Simple Queue Service) - Message queue for URL requests
- **S3** (Simple Storage Service) - Storage for PNG screenshots
- **IAM** (Identity and Access Management) - Security roles and policies
- **CloudWatch** - Monitoring, alarms, and scaling policies
- **SSM Parameter Store** - Dynamic AMI fetching
- **Security Groups** - Network access control
- **Load Balancer** (optional) - For load testing instance communication

## 2. Properties and Dependencies

### Critical Dependencies:
1. **VPC must be created before** EC2 instances, Auto Scaling groups, and Load Balancers
2. **IAM Roles must be created before** EC2 instances that reference them
3. **S3 Bucket must exist before** IAM policies that grant access to it
4. **Security Groups must be created before** EC2 instances that reference them
5. **SQS Queue must be created before** IAM policies that grant access to it

### Property Dependencies:
- Auto Scaling Group depends on Launch Configuration or Launch Template
- CloudWatch Alarms depend on Metrics from SQS and EC2
- Scaling Policies depend on CloudWatch Alarms

## 3. Safe AWS Defaults (where specific values are missing)

### Network Configuration:
- **VPC CIDR**: `10.0.0.0/16` (standard private range)
- **Public Subnet CIDRs**: `10.0.0.0/24` and `10.0.1.0/24` (two AZs)
- **Private Subnet CIDRs**: `10.0.10.0/24` and `10.0.11.0/24` (two AZs)
- **Availability Zones**: Dynamically selected using `Fn::GetAZs` and `Fn::Select`

### EC2 Configuration:
- **Instance Type**: `t3.medium` (balanced performance for Chromium processing)
- **AMI**: Dynamically fetched using SSM Parameter Store for Amazon Linux 2
- **SSH Key**: Parameter with default value 'default-key' (user must create this key pair)

### S3 Configuration:
- **Bucket Name**: Dynamically generated using `AWS::AccountId` and `AWS::Region`
- **Encryption**: SSE-S3 (server-side encryption with S3-managed keys)

### SQS Configuration:
- **Visibility Timeout**: `300` seconds (5 minutes for screenshot processing)
- **Message Retention**: `1209600` seconds (14 days)

## 4. Dynamic Resource Fetching Strategy

### AMI Fetching:
```yaml
AMazonLinux2HVM:
  Type: AWS::SSM::Parameter::Value<AWS::EC2::Image::Id>
  Default: /aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2
```

### Availability Zones:
```yaml
Fn::Select:
  - 0
  Fn::GetAZs: !Ref AWS::Region
```

## 5. Architecture Overview

The solution will consist of:

1. **Network Layer**: VPC with public and private subnets across two AZs
2. **Storage Layer**: S3 bucket for PNG storage with encryption
3. **Messaging Layer**: SQS queue for URL requests with appropriate visibility timeout
4. **Compute Layer**: 
   - Auto Scaling Group of worker instances (1-2 instances)
   - Dedicated load test instance
5. **Security Layer**: IAM roles, security groups, and policies
6. **Monitoring Layer**: CloudWatch alarms and scaling policies

## 6. Scaling Strategy

- **Scale-out**: When SQS queue depth exceeds threshold (e.g., 10 messages)
- **Scale-in**: When SQS queue depth is below threshold (e.g., 2 messages)
- **Min Instances**: 1 (always available for processing)
- **Max Instances**: 2 (cost control while maintaining performance)

## 7. Security Considerations

- **IAM Roles**: Least-privilege access for EC2 instances to S3 and SQS
- **Security Groups**: Restrict traffic to necessary ports only
- **S3 Bucket Policy**: Restrict access to authorized IAM roles and principals
- **No SSH Access**: Worker instances won't have direct SSH access for security

## 8. Implementation Sequence

1. Create VPC, subnets, and internet gateway
2. Create IAM roles and policies
3. Create S3 bucket and SQS queue
4. Create security groups
5. Create launch configuration/template
6. Create Auto Scaling group
7. Create CloudWatch alarms and scaling policies
8. Create load test instance

This architecture ensures high availability, scalability, and security while meeting the business requirements for a URL-to-PNG screenshot service.
