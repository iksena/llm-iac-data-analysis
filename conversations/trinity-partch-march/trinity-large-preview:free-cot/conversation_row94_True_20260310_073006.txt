# Detailed Architectural Deployment Plan for EC2 with SSM Automation

## Key AWS Services Required

1. **EC2** - For the Linux 2 instance
2. **IAM** - For instance roles and policies
3. **SSM (Systems Manager)** - For automation documents and command execution
4. **S3** - For centralized logging
5. **VPC** - For network isolation (with subnets, IGW, route tables)
6. **CloudWatch** - For monitoring and logging
7. **SSM Parameter Store** - For dynamic AMI fetching

## Architecture Overview

The deployment will create a self-contained environment with:
- A VPC with public subnet
- EC2 instance with SSM permissions
- S3 bucket for logs
- Custom SSM Automation document
- IAM roles and policies with least privilege

## Detailed Component Breakdown

### 1. Network Infrastructure

**VPC:**
- CIDR: 10.0.0.0/16 (default safe value)
- 1 public subnet: 10.0.1.0/24
- Internet Gateway attached to VPC
- Route table with default route (0.0.0.0/0) pointing to IGW

### 2. EC2 Instance Configuration

**Instance Details:**
- AMI: Dynamic fetch using SSM Parameter Store
- Instance Type: t3.micro (safe default for free tier)
- Key Pair: Parameter with default 'default-key' (safe fallback)
- IAM Role: EC2SSMRole with necessary permissions

**IAM Role Permissions:**
- SSM: ssm:GetParameter, ssm:SendCommand, ssm:ListAssociations
- S3: s3:PutObject, s3:GetBucketAcl, s3:ListBucket
- CloudWatch: logs:CreateLogGroup, logs:CreateLogStream, logs:PutLogEvents

### 3. S3 Bucket for Logging

**Configuration:**
- Bucket name: dynamic with random suffix to ensure uniqueness
- Policy: AWS::S3::BucketPolicy for controlled access
- OwnershipControls: Required if using legacy AccessControl
- DeletionPolicy/UpdateReplacePolicy: Retain (to prevent accidental data loss)

### 4. SSM Automation Document

**Document Structure:**
- Schema: AWS::SSM::Document type: Automation
- Steps:
  1. OS Updates: aws:runPatchBaseline with reboot option
  2. Nginx Installation: aws:downloadContent + aws:executeScript
  3. Validation: aws:executeScript with curl command

**Permissions:**
- Document requires IAM role with SSM execution permissions
- Targets EC2 instance by tag or instance ID

### 5. Dependencies and Resource Ordering

**Critical Dependencies:**
1. VPC created before IGW
2. IGW attached to VPC before route creation
3. Subnet created before EC2 instance
4. IAM Role and Instance Profile created before EC2
5. S3 Bucket created before SSM document that references it

## Dynamic Resource Fetching Strategy

### AMI Resolution
```yaml
Fn::ImportValue:
  Fn::Sub: "arn:aws:ssm:${AWS::Region}:${AWS::AccountId}:parameter/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2"
```

### Availability Zones
```yaml
Fn::GetAZs: ""
```

## Security and Compliance

### IAM Policies
- Least privilege principle applied
- No wildcard permissions except where necessary (e.g., s3:PutObject on specific bucket)
- SSM permissions scoped to specific documents and instances

### Network Security
- Security Group allowing HTTP (port 80) from anywhere
- SSH access controlled via parameter (default restricted)

## Validation and Testing

### SSM Automation Validation
- Step 1: Verify OS updates applied successfully
- Step 2: Confirm Nginx installation and service status
- Step 3: Validate web server responds on port 80

### Logging Verification
- Confirm logs written to S3 bucket
- Verify CloudWatch metrics for instance health

## Error Handling and Rollback

### CloudFormation Properties
- DeletionPolicy: Retain for S3 bucket
- UpdateReplacePolicy: Retain for stateful resources
- DependsOn clauses to ensure proper resource creation order

### SSM Automation Error Handling
- Built-in retry logic for failed steps
- Conditional execution based on previous step success

This architecture ensures a robust, automated deployment that meets all specified requirements while adhering to AWS best practices and CloudFormation linting rules.
