# Architectural Deployment Plan for Custom Resource with Lambda Arithmetic Operations

## 1. Key AWS Services Required

Based on the business need, the following AWS services are required:

- **AWS Lambda**: To host the custom resource handler function that performs arithmetic operations
- **AWS CloudFormation**: To deploy the infrastructure and invoke the custom resource
- **AWS Identity and Access Management (IAM)**: To create execution roles for Lambda with necessary permissions
- **AWS Systems Manager (SSM) Parameter Store**: To dynamically fetch Lambda runtime information (though not strictly needed for Lambda itself, it's a best practice for other resources)

## 2. Properties and Dependencies

The template will have the following key properties and dependencies:

- **Lambda Function Properties**:
  - Runtime: `python3.12` (latest supported runtime)
  - Handler: `index.handler`
  - Timeout: 30 seconds (default, sufficient for arithmetic operations)
  - MemorySize: 128 MB (default, sufficient for this use case)

- **IAM Role Properties**:
  - Trust policy allowing Lambda service to assume the role
  - Permissions policy allowing:
    - `lambda:InvokeFunction` (for custom resource invocation)
    - `logs:CreateLogGroup`, `logs:CreateLogStream`, `logs:PutLogEvents` (for CloudWatch logging)

- **Custom Resource Properties**:
  - ServiceToken: ARN of the Lambda function
  - Parameters: `FirstNumber`, `SecondNumber`, `Operator`, `SimulateFailure`

## 3. Safe AWS Defaults

Since specific properties are not provided, I will use the following safe defaults:

- **Lambda Runtime**: `python3.12` (latest supported)
- **Lambda Timeout**: 30 seconds
- **Lambda Memory**: 128 MB
- **IAM Role Name**: Dynamically generated using `!Sub` with stack name and resource name
- **Log Group**: Automatically created by Lambda

## 4. Dynamic Resource Fetching

For this specific use case, dynamic fetching using SSM Parameter Store is not strictly necessary since we're using the latest Lambda runtime. However, I will follow the best practice of using dynamic references for any secrets or sensitive data if needed in the future.

## 5. Architecture Plan

The CloudFormation template will consist of the following resources:

1. **IAM Role**: Execution role for the Lambda function
2. **Lambda Function**: Custom resource handler for arithmetic operations
3. **Custom Resource**: `Custom::helper` that invokes the Lambda function
4. **Outputs**: To display the calculation result

### Lambda Function Logic

The Lambda function will:
1. Parse the `FirstNumber`, `SecondNumber`, and `Operator` from the `ResourceProperties`
2. Perform the arithmetic operation based on the `Operator` (addition, subtraction, multiplication, division)
3. Check the `SimulateFailure` parameter to conditionally simulate a failure
4. Return the result or error to CloudFormation

### Custom Resource Invocation

The `Custom::helper` resource will:
1. Invoke the Lambda function with the provided parameters
2. Pass the `RequestType` (Create, Update, Delete) to the Lambda function
3. Handle the response from the Lambda function and propagate it to CloudFormation

### Error Handling and Simulation

The `SimulateFailure` parameter will allow testing of custom resource failure scenarios:
- If `SimulateFailure` is `true`, the Lambda function will throw an error
- If `SimulateFailure` is `false` or not provided, the Lambda function will perform the calculation

### Outputs

The template will output the result of the arithmetic operation, which will be available in the CloudFormation stack outputs after successful creation or update.

## 6. Security Considerations

- The Lambda function will have minimal permissions (only what's necessary for logging and custom resource invocation)
- No sensitive data will be hardcoded in the template
- The IAM role will follow the principle of least privilege

## 7. Testing Strategy

The template will be tested with various scenarios:
1. Successful arithmetic operations (addition, subtraction, multiplication, division)
2. Division by zero error handling
3. Simulated failure scenarios
4. Stack deletion to ensure proper cleanup

This architecture ensures a robust, secure, and testable custom resource implementation that meets the business requirements while adhering to AWS best practices.
