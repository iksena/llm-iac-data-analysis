# AWS Web Server Environment Deployment Plan

## Key AWS Services Required

1. **VPC** - Virtual Private Cloud for network isolation
2. **Internet Gateway** - Enables internet connectivity for the VPC
3. **Route Table** - Routes traffic to the internet gateway
4. **Subnet** - Network segment within the VPC
5. **Security Group** - Controls inbound/outbound traffic to the EC2 instance
6. **EC2 Instance** - Web server running Apache and PHP
7. **IAM Role** - Grants EC2 instance permission to access S3
8. **S3 Bucket** - Stores the sample web content ZIP file
9. **S3 Bucket Policy** - Controls access to the S3 bucket

## Properties and Dependencies

### Critical Dependencies
1. **VPC must be created before** the Internet Gateway, Subnet, and Security Group
2. **Internet Gateway must be attached to VPC** before creating Route Table entries
3. **Route Table must be associated with Subnet** after both are created
4. **Security Group must be created within VPC context**
5. **IAM Role must exist before EC2 instance creation**
6. **S3 Bucket must exist before EC2 instance creation** (for user data script)

### Property Dependencies
- Subnet requires VPC ID
- Security Group requires VPC ID
- Route Table requires VPC ID and Internet Gateway ID
- EC2 Instance requires Subnet ID, Security Group IDs, and IAM Role

## Safe AWS Defaults (for missing properties)

### Network Configuration
- **VPC CIDR**: `10.0.0.0/16` (standard private range)
- **Subnet CIDR**: `10.0.1.0/24` (single subnet within VPC)
- **Availability Zone**: First AZ in region using `Fn::GetAZs` and `Fn::Select`

### Instance Configuration
- **Instance Type**: `t2.micro` (as specified)
- **AMI**: Latest Amazon Linux 2 using SSM Parameter Store
- **KeyName**: `default-key` (safe default, user can override)
- **IAM Instance Profile**: Custom role with S3 read permissions

### Security Configuration
- **SSH Port**: `22` (for management access)
- **HTTP Port**: `80` (for web traffic)
- **CIDR for access**: `0.0.0.0/0` (open to world, can be restricted)

## Dynamic Resource Fetching Strategy

### AMI Retrieval
Use SSM Parameter Store to fetch the latest Amazon Linux 2 AMI:
```yaml
ImageId: !Sub 'arn:aws:ssm:${AWS::Region}:${AWS::AccountId}:parameter/aws/service/ami-amazon-linux-latest/amzn2-ami-hvn-x86_64-gp2'
```

### Availability Zone Selection
```yaml
!Select [0, !GetAZs '']
```

## Architecture Flow

1. **Create VPC** with 10.0.0.0/16 CIDR
2. **Create Internet Gateway** and attach to VPC
3. **Create Route Table** with default route to Internet Gateway
4. **Create Subnet** in first AZ with 10.0.1.0/24 CIDR
5. **Associate Route Table with Subnet**
6. **Create Security Group** allowing HTTP (80) and SSH (22)
7. **Create IAM Role** with S3 read-only policy
8. **Create S3 Bucket** for web content
9. **Create S3 Bucket Policy** to allow public read if needed
10. **Create EC2 Instance** with:
    - Latest Amazon Linux 2 AMI
    - t2.micro instance type
    - IAM role for S3 access
    - User data script for Apache/PHP installation and content deployment

## User Data Script Strategy

The user data script will:
1. Update system packages
2. Install Apache and PHP
3. Start Apache service
4. Install AWS CLI
5. Download ZIP file from S3
6. Extract content to `/var/www/html/`
7. Set proper permissions

## Security Considerations

- Use IAM roles instead of access keys for S3 access
- Security group restricts traffic to necessary ports
- User data script runs as root (standard for EC2 initialization)
- S3 bucket policy can be configured for public read if content should be publicly accessible

## Error Prevention Measures

- All resources have proper dependencies defined
- No hardcoded values for AMIs or AZs
- Parameters have safe defaults
- S3 bucket policy uses modern approach instead of legacy AccessControl
- EC2 instance uses IAM role instead of hardcoded credentials

This architecture ensures a fully automated, self-contained web server deployment that can be repeated across environments with minimal configuration changes.
