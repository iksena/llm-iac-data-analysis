# Event-Driven DynamoDB to EventBridge Integration Architecture Plan

## Key AWS Services Required

1. **DynamoDB** - To create the table with streaming enabled
2. **EventBridge** - For the event bus and pipes
3. **IAM** - For service roles and permissions
4. **Lambda** (optional) - For custom event transformation if needed

## Architecture Overview

The solution will create a DynamoDB table with streams enabled, an EventBridge event bus, and two EventBridge Pipes that filter DynamoDB stream records and forward them to the bus. The pipes will use filter patterns to route INSERT and MODIFY events separately.

## Detailed Resource Breakdown

### 1. DynamoDB Table with Streams

**Properties:**
- Table name: `OrdersTable` (default)
- Primary key: `OrderId` (string)
- Stream specification: `NEW_AND_OLD_IMAGES` to capture full item state
- Billing mode: `PAY_PER_REQUEST` (default for serverless)

**Dependencies:** None

### 2. EventBridge Event Bus

**Properties:**
- Name: `OrdersEventBus` (default)
- EventSourceName: `dynamodb` (default)

**Dependencies:** None

### 3. EventBridge Pipes

**Pipe 1 - INSERT Events:**
- Name: `DynamoDBInsertPipe`
- Source: DynamoDB stream from OrdersTable
- Destination: OrdersEventBus
- Filter pattern: `detail-type="DynamoDB Stream"` with `eventName="INSERT"`

**Pipe 2 - MODIFY Events:**
- Name: `DynamoDBModifyPipe`
- Source: DynamoDB stream from OrdersTable
- Destination: OrdersEventBus
- Filter pattern: `detail-type="DynamoDB Stream"` with `eventName="MODIFY"`

**Dependencies:** Both pipes depend on the DynamoDB table and event bus being created first.

### 4. IAM Roles

**Pipe Execution Role:**
- Trust policy: EventBridge Pipes service
- Permissions: DynamoDB stream read access, EventBridge PutEvents

**Dependencies:** Created before pipes are provisioned.

## Parameter Strategy

Since no specific values are provided, I'll use safe defaults:

```yaml
Parameters:
  TableName:
    Type: String
    Default: OrdersTable
    Description: DynamoDB table name
  EventBusName:
    Type: String
    Default: OrdersEventBus
    Description: EventBridge event bus name
```

## Dynamic Resource Fetching

No AMIs or AZs are needed for this serverless architecture, so no SSM Parameter Store calls are required.

## Security & Permissions

The pipes will use IAM roles with least-privilege access:
- DynamoDB stream read-only access to the specific table
- EventBridge PutEvents permission to the specific bus only

## Best Practices Implementation

1. **No hardcoded values** - All resource names use parameters with defaults
2. **No deprecated runtimes** - Not applicable (serverless)
3. **No legacy S3 properties** - Not applicable
4. **Proper IAM policies** - Using least-privilege principle
5. **No Fn::Sub without variables** - Will use Fn::Join or direct string concatenation where needed

## Resource Dependencies

1. DynamoDB table must be created before pipes
2. Event bus must be created before pipes
3. IAM role must be created before pipes

## Event Schema

The pipes will transform DynamoDB stream records into EventBridge events with:
- Source: `dynamodb`
- Detail-type: `DynamoDB Stream`
- Detail: Original DynamoDB stream record
- Custom attributes: `eventName` (INSERT/MODIFY) and `tableName`

This architecture provides a scalable, event-driven integration that captures real-time DynamoDB changes and routes them to EventBridge for downstream processing without requiring custom Lambda functions for basic routing.
