# Detailed Architectural Deployment Plan for VPC with Security Group

## Key AWS Services Required
1. **VPC** - Virtual Private Cloud for network isolation
2. **Internet Gateway** - For internet connectivity
3. **Security Group** - Network firewall for instance-level traffic control
4. **Route Table** - Network traffic routing
5. **Subnet** - Network segment within the VPC

## Properties and Dependencies Analysis

### VPC Dependencies
- VPC must be created before any dependent resources
- CIDR block is required for VPC creation
- Default safe CIDR: `10.0.0.0/16` (private, non-routable)

### Internet Gateway Dependencies
- Must be attached to VPC before use
- Creates bi-directional dependency with VPC
- No additional properties required beyond VPC attachment

### Security Group Dependencies
- Must be created within a VPC
- Requires VPC ID as dependency
- Default inbound rule: allow SSH (port 22) from anywhere for management access
- Default outbound rule: allow all traffic for internet access

### Route Table Dependencies
- Must be associated with a VPC
- Requires route to Internet Gateway for internet access
- Default route: `0.0.0.0/0` via Internet Gateway

### Subnet Dependencies
- Must be created within a VPC
- Requires Availability Zone
- Default CIDR: `10.0.1.0/24` (subset of VPC range)

## Dynamic Resource Fetching Strategy

### AMI Selection
- Use SSM Parameter Store to fetch latest Amazon Linux 2 AMI
- Parameter: `/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`
- This ensures always using the latest patched AMI without manual updates

### Availability Zones
- Use `Fn::GetAZs` to dynamically fetch available AZs for the region
- Use `Fn::Select` to pick the first available AZ
- This ensures template works across all regions without hardcoding

## Security Best Practices

### Security Group Configuration
- Default inbound: Allow SSH (port 22) from anywhere (0.0.0.0/0)
- Default outbound: Allow all traffic (0.0.0.0/0)
- These defaults can be overridden via parameters

### VPC Configuration
- Default CIDR: `10.0.0.0/16` (private, non-routable)
- This provides 65,536 IP addresses, sufficient for most use cases

## Parameter Strategy

### Required Parameters
- **KeyName**: EC2 Key Pair name for SSH access
  - Default: `default-key` (safe fallback)
  - This allows template deployment without manual input

### Optional Parameters
- **VPCCIDR**: VPC CIDR block
  - Default: `10.0.0.0/16`
  - Allows customization while providing safe default

- **SubnetCIDR**: Subnet CIDR block
  - Default: `10.0.1.0/24`
  - Must be subset of VPC CIDR

- **SSHLocation**: Source IP range for SSH access
  - Default: `0.0.0.0/0` (allow from anywhere)
  - Can be restricted for production environments

## Template Structure Plan

### Resources Section
1. VPC (with default CIDR)
2. Internet Gateway
3. VPC Gateway Attachment
4. Security Group (with default rules)
5. Route Table
6. Route (default route to Internet Gateway)
7. Subnet (with default CIDR)
8. Subnet Route Table Association

### Parameters Section
- KeyName (with default)
- VPCCIDR (with default)
- SubnetCIDR (with default)
- SSHLocation (with default)

### Metadata Section
- AWS::CloudFormation::Interface for parameter grouping and ordering

## Linting Compliance Strategy

### Parameter Validation
- All parameters have safe defaults
- Allowed patterns match default values
- No required manual input for deployment

### Resource Configuration
- No hardcoded AMIs or AZs
- All dependencies properly ordered
- No legacy properties used
- All required properties included

### Security Considerations
- Default security group rules are permissive but configurable
- VPC uses private CIDR range
- No hardcoded credentials or secrets

This architecture provides a solid foundation for a VPC with security group that can be deployed automatically without manual intervention, while maintaining flexibility for customization through parameters.
