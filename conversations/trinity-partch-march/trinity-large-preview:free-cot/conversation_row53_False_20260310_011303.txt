# AWS EMR Serverless Deployment Architecture Plan

## Key AWS Services Required

1. **AWS::IAM::Role** - For EMR Serverless execution permissions
2. **AWS::S3::Bucket** - Private bucket for scripts and data storage
3. **AWS::EMR::ServerlessApplication** - EMR Serverless application with Spark runtime
4. **AWS::EC2::VPC** - Virtual network for EMR Serverless (required for private S3 access)
5. **AWS::EC2::Subnet** - Subnets within VPC
6. **AWS::EC2::InternetGateway** - For VPC internet access
7. **AWS::EC2::VPCGatewayAttachment** - Attach IGW to VPC
8. **AWS::EC2::RouteTable** - Route configuration
9. **AWS::EC2::Route** - Default route to IGW
10. **AWS::EC2::SubnetRouteTableAssociation** - Associate subnets with route table

## Properties and Dependencies

### Critical Dependencies:
1. VPC must be created before Subnets
2. Internet Gateway must be created before VPCGatewayAttachment
3. VPCGatewayAttachment must be created before Route
4. RouteTable must be created before Route
5. Route must be created before SubnetRouteTableAssociation

### Property Dependencies:
- EMR Serverless requires IAM Role with proper permissions before application creation
- S3 Bucket must exist before EMR Serverless application configuration
- VPC and Subnets must exist before EMR Serverless application creation

## Safe AWS Defaults

### Network Configuration:
- **VPC CIDR**: `10.0.0.0/16` (standard private range)
- **Subnet CIDR**: `10.0.1.0/24` (single subnet for simplicity)
- **Availability Zone**: First AZ in region using `Fn::GetAZs` and `Fn::Select`

### EMR Configuration:
- **EMR Runtime**: `emr-6.15.0` (as specified)
- **Auto-start/Auto-stop**: Enabled via `AutoStartConfig` and `AutoStopConfig`
- **Initial Capacity**: 1 worker for cost efficiency

### S3 Configuration:
- **Bucket Name**: Dynamically generated using stack name and timestamp
- **Encryption**: SSE-S3 (server-side encryption with S3-managed keys)
- **Access**: Private with bucket policy for EMR Serverless role

## Dynamic Resource Fetching Strategy

### AMI Retrieval:
- Not required for EMR Serverless (serverless runtime)
- If needed for any EC2 resources, use: `resolve:ssm:/aws/service/ami-amazon-linux-latest/amzn2-ami-hvm-x86_64-gp2`

### AZ Selection:
```yaml
!Select [0, !GetAZs !Ref 'AWS::Region']
```

## Security Architecture

### IAM Role Configuration:
- **EMR Serverless Managed Policy**: `AmazonEMRServerlessRole`
- **S3 Permissions**: Custom policy allowing EMR Serverless role to access the private bucket
- **CloudWatch Logs**: Permissions for log streaming

### S3 Bucket Security:
- **Ownership Controls**: Enabled to prevent unauthorized access
- **Bucket Policy**: Restricts access to EMR Serverless role only
- **Encryption**: SSE-S3 for data at rest

## Cost Optimization Strategy

### Auto-start/Auto-stop Configuration:
- **AutoStart**: Enabled with 5-minute delay after job submission
- **AutoStop**: Enabled with 5-minute idle timeout
- **Initial Capacity**: Minimum 1 worker to handle immediate requests

### Resource Configuration:
- **Worker Type**: Use memory-optimized instances for Spark workloads
- **Network Configuration**: Private VPC to prevent data exfiltration

## Template Structure Plan

1. **Parameters**: Safe defaults for all configurable values
2. **Resources**:
   - IAM Role with EMR Serverless permissions
   - S3 Bucket with encryption and access controls
   - VPC with single public subnet
   - EMR Serverless Application with Spark runtime
3. **Outputs**: Application ARN, S3 bucket name, and IAM role ARN

## Error Prevention Measures

1. **Linting Compliance**: All properties will match CloudFormation schema
2. **Dynamic References**: No hardcoded values except safe defaults
3. **Resource Protection**: Proper `DeletionPolicy` and `UpdateReplacePolicy` for stateful resources
4. **Security Best Practices**: No raw secrets, proper IAM least privilege

This architecture ensures a production-ready EMR Serverless deployment with proper security, cost optimization, and compliance with AWS best practices.
